TCP — широко используемый сетевой протокол. Это «надёжный» протокол, работающий поверх ненадёжного протокола: IP (сокращение от Internet Protocol).

Давайте копнем глубже и разберемся, как работает TCP «под капотом».

Начнем с рассмотрения IP (сокращение от Internet Protocol), протокола, на основе которого создан TCP.

IP не зависит от TCP, поскольку IP — это протокол более низкого уровня, чем TCP.
TCP использует IP для маршрутизации пакетов между системами.

## IP — почтовая служба Интернета

Когда программа отправляет данные по сети с использованием IP, данные разбиваются и отправляются в виде нескольких «пакетов».

В каждом пакете содержится:

- раздел заголовка
- раздел данных

Заголовок содержит адрес отправителя и получателя, как на конверте, который вы отправляете через местную почтовую службу.

![](../../../assets/obsidian-images/Pasted%20image%2020251103113505.png)

Важное сходство между IP и почтовой службой заключается в том, что пакеты **не гарантированно** дойдут до адресата. Несмотря на все усилия, предпринимаемые для этого, пакеты иногда теряются в пути.

Более того, если вы отправляете 5 пакетов одновременно, нет гарантии, что они прибудут по назначению в одно и то же время или в одном и том же порядке.

Пакеты могут быть потеряны и доставлены в неправильном порядке, поскольку IP — это протокол с максимальными усилиями, то есть он не гарантирует доставку пакетов или соблюдение порядка

Протокол TCP был создан для устранения ограничений IP.

## Гарантии TCP

В первую очередь TCP предлагает две гарантии: **(а)** надежную доставку пакетов и **(б)** упорядоченную доставку пакетов.

**Гарантия №1 — Надежная доставка**

TCP гарантирует отсутствие потерь пакетов при передаче. Это достигается за счёт запроса у получателя подтверждения всех отправленных пакетов и повторной передачи пакетов, если подтверждение не получено.

**Гарантия №2 — Доставка по заказу**

Помимо гарантии того, что пакеты достигнут получателя, TCP также гарантирует их правильную доставку. Это достигается путём присвоения каждому пакету порядкового номера. Получатель отслеживает эти номера и переупорядочивает пакеты, идущие не по порядку. Если пакет отсутствует, получатель ожидает его повторной передачи.

TCP использует порядковые номера для поддержания правильного порядка передачи данных и повторно собирает пакеты в пункте назначения.

## TCP-соединения

TCP — это протокол, ориентированный на соединение. Это означает, что для взаимодействия по TCP программа должна сначала «установить соединение». Для этого одна программа берёт на себя роль «сервера», а другая — роль «клиента».

Сервер ожидает подключения, а клиент инициирует соединение. После установки соединения клиент и сервер могут как получать, так и отправлять данные (это двусторонний канал).

TCP — это двунаправленный протокол связи, означающий, что как клиент, так и сервер могут отправлять и получать данные.

TCP-соединение идентифицируется с помощью уникальной комбинации четырех значений:

- IP-адрес назначения
- номер порта назначения
- исходный IP-адрес
- номер исходного порта

Например, вы подключаетесь к google.com. Значения для вашего TCP-подключения будут выглядеть следующим образом:

- **IP-адрес назначения** : xxxx (веб-сервер Google)
- **Номер порта назначения** : 443 (порт по умолчанию, используемый HTTPS)
- **исходный IP-адрес** : yyyy (ваш компьютер)
- **номер исходного порта** : 26789 (случайное число, не используемое никакими другими соединениями на вашем компьютере)
![](../../../assets/obsidian-images/Pasted%20image%2020251103113723.png)

Если ваш браузер открывает несколько подключений к серверу Google, изменится только «номер исходного порта», остальное останется прежним.

MAC-адреса не используются для идентификации TCP-соединения, поскольку они относятся к канальному уровню, а не к транспортному уровню, на котором работает TCP.


## TCP-рукопожатие

TCP-рукопожатие — это способ установления соединения между клиентами и серверами. Этот процесс состоит из трёх этапов.

**Шаг 1: СИН**

Сначала клиент инициирует соединение, отправляя серверу пакет SYN (synchronize), указывающий на запрос на установление соединения. Этот пакет также содержит порядковый номер, позволяющий отслеживать порядок отправки пакетов.
![](../../../assets/obsidian-images/Pasted%20image%2020251103113826.png)

**Шаг 2: SYN-ACK**

Сервер, получив этот пакет SYN, отправляет обратно пакет SYN-ACK (синхронизация-подтверждение).
![](../../../assets/obsidian-images/Pasted%20image%2020251103113851.png)

**Шаг 3: ПОДТВЕРЖДЕНИЕ**

На последнем этапе этого трёхстороннего рукопожатия клиент подтверждает пакет SYN-ACK сервера, отправляя пакет ACK (подтверждение). Соединение считается установленным после получения сервером этого последнего пакета.
![](../../../assets/obsidian-images/Pasted%20image%2020251103113913.png)

Клиент отправляет пакет SYN, сервер отвечает пакетом SYN-ACK, а клиент отвечает пакетом ACK.

Трехстороннее рукопожатие обеспечивает создание надежного соединения, проверяет готовность обоих устройств к передаче данных и устанавливает начальные порядковые номера для правильного порядка пакетов данных.
