# Troubleshooting, subnet hisoblash va wildcard mask

## Amaliy troubleshooting commandlar

Linux:

```bash
# IP addresslar
ip addr

# Routing table
ip route

# Default gateway
ip route | grep default

# ARP/neighbor table
ip neigh

# Ping
ping 8.8.8.8

# DNS tekshirish
dig example.com A
nslookup example.com

# Path tekshirish
traceroute 8.8.8.8

# Packet capture
sudo tcpdump -i any -n host 8.8.8.8
```

Windows:

```cmd
ipconfig /all
route print
arp -a
ping 8.8.8.8
tracert 8.8.8.8
nslookup example.com
```

Cisco IOS:

```text
show ip interface brief
show ip route
show arp
ping 8.8.8.8
traceroute 8.8.8.8
show running-config interface gigabitEthernet 0/0
```

## IPv4 muammoni topish algoritmi

Agar host Internetga chiqmayotgan bo'lsa:

1. **IP bor-yo'qligini tekshir**
   - `ip addr`
   - IP `169.254.x.x` bo'lsa, DHCP ishlamagan bo'lishi mumkin.

2. **Mask to'g'riligini tekshir**
   - noto'g'ri mask hostni noto'g'ri subnetda deb o'ylatadi.

3. **Default gateway borligini tekshir**
   - `ip route`
   - gateway host bilan bir subnetda bo'lishi kerak.

4. **Gateway ping bo'ladimi?**
   - bo'lmasa L2, VLAN, Wi-Fi, kabel, ARP yoki gateway muammosi.

5. **Public IP ping bo'ladimi?**
   - `ping 8.8.8.8`
   - bo'lmasa routing/NAT/firewall muammosi.

6. **DNS ishlaydimi?**
   - `dig google.com`
   - IP ping bo'lsa-yu, hostname ishlamasa, DNS muammosi.

7. **Traceroute bilan qayerda uzilishini ko'r**
   - local gatewaydami, ISPdami, remote tomondami.

## Ko'p uchraydigan xatolar

### Gateway boshqa subnetda bo'lishi

Noto'g'ri:

```text
Host IP: 192.168.1.10/24
Gateway: 192.168.2.1
```

Host gateway'ga ARP qila olmaydi, chunki gateway boshqa subnetda.

To'g'ri:

```text
Host IP: 192.168.1.10/24
Gateway: 192.168.1.1
```

### Network yoki broadcast addressni hostga berish

Noto'g'ri:

```text
192.168.1.0/24
192.168.1.255/24
```

Bu addresslar odatda hostga berilmaydi.

### Mask mismatch

Host A:

```text
192.168.1.10/24
```

Host B:

```text
192.168.1.20/25
```

Ba'zan ular bir-birini bir subnetda deb o'ylaydi, ba'zan yo'q. Bu juda noaniq troubleshooting holatlarini keltirib chiqaradi.

### Classful fikrlash

Noto'g'ri fikr:

```text
10.x.x.x doim /8 bo'ladi
192.168.x.x doim /24 bo'ladi
```

To'g'ri:

```text
Mask yoki CIDR ko'rsatilmasa, aniq xulosa qilib bo'lmaydi.
```

### Public va private IPni adashtirish

`192.168.1.10` Internetda global unique emas. Minglab uy routerlarda shu IP bo'lishi mumkin. Public Internetga chiqish uchun NAT yoki public IP kerak.

## Subnet hisoblash uchun tezkor usul

Misol:

```text
192.168.10.77/27
```

1. `/27` mask:

```text
255.255.255.224
```

2. Qiziq octet oxirgisi, chunki mask oxirgi octetda `224`.

3. Block size:

```text
256 - 224 = 32
```

4. Subnetlar:

```text
192.168.10.0
192.168.10.32
192.168.10.64
192.168.10.96
192.168.10.128
192.168.10.160
192.168.10.192
192.168.10.224
```

5. `77` qaysi oraliqda?

```text
64 - 95
```

Natija:

```text
Network:   192.168.10.64
Broadcast: 192.168.10.95
Hosts:     192.168.10.65 - 192.168.10.94
```

Subnetting bo'yicha yanada chuqur misollar: [subnetting-cidr.md](../../subnetting-cidr.md).

## Cisco wildcard mask

Cisco ACL va OSPF configlarida ko'pincha subnet mask emas, **wildcard mask** ishlatiladi.

Wildcard mask - subnet maskning teskarisi.

```text
Subnet mask:   255.255.255.0
Wildcard mask: 0.0.0.255
```

Misollar:

| CIDR | Subnet mask | Wildcard mask |
|---|---|---|
| /8 | 255.0.0.0 | 0.255.255.255 |
| /16 | 255.255.0.0 | 0.0.255.255 |
| /24 | 255.255.255.0 | 0.0.0.255 |
| /30 | 255.255.255.252 | 0.0.0.3 |
| /32 | 255.255.255.255 | 0.0.0.0 |

Cisco ACL misol:

```text
access-list 10 permit 192.168.1.0 0.0.0.255
```

Bu:

```text
192.168.1.0/24 ga ruxsat ber
```

degani.

Wildcard maskni octet bo'yicha hisoblash:

```text
255 - subnet mask octet = wildcard octet
```

Masalan:

```text
255.255.255.192
0.0.0.63
```

## Tezkor checklist

```text
1. IP address bormi?
2. Mask to'g'rimi?
3. Gateway shu subnetdami?
4. Gateway ARP/ping bo'ladimi?
5. Routing table'da default route bormi?
6. NAT/firewall ishlayaptimi?
7. DNS ishlayaptimi?
8. Traceroute qayerda to'xtayapti?
```

## Qisqa xulosa

IPv4 troubleshootingda tartib muhim: avval host IP/mask/gateway, keyin local gateway, keyin routing/NAT/firewall, oxirida DNS tekshiriladi. Subnet hisoblashda esa block size va host bitlar soni asosiy tezkor usul bo'lib xizmat qiladi.
