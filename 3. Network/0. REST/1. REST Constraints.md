
REST’ni **to‘liq RESTful** deb atash uchun u quyidagi **6 ta arxitektura cheklovi (constraint)** ga amal qilishi kerak:

| #   | Nomi                                                              | Maqsadi                                                |
| --- | ----------------------------------------------------------------- | ------------------------------------------------------ |
| 1️⃣ | **Uniform Interface**                          (Yagona interfeys) | Barcha resurslarga bir xil usulda murojaat qilish      |
| 2️⃣ | **Client–Server**                                                 | Mijoz va serverning rolini ajratish                    |
| 3️⃣ | **Stateless**                                                     | Server har bir so‘rovni mustaqil tarzda qayta ishlashi |
| 4️⃣ | **Cacheable**                                                     | Javoblarni keshlash (saqlash) imkonini berish          |
| 5️⃣ | **Layered System**                                                | Tizimni qatlamlarga ajratish imkonini berish           |
| 6️⃣ | **Code on Demand** _(ixtiyoriy)_                                  | Server mijozga bajariladigan kod yuborishi mumkin      |

---
## 1. Uniform Interface (Yagona Interfeys)

**Client va server o‘rtasidagi o‘zaro aloqani yagona va izchil shaklda tashkil etish**.

Uniform interfeysni ta’minlaydigan 4 qism bor:

1️⃣ **Resource identification**

Har bir resurs (ma’lumot) noyob **URI** orqali aniqlanadi.  
Masalan:  

```txt
✅ To'g'ri:
GET /users/123
GET /users/123/orders
GET /orders/456

❌ Noto'g'ri:
GET /getUserById?id=123
GET /user-orders?userId=123
```


2️⃣ **Manipulation through representations**  

Client serverdagi resurs holatini **representation)** orqali o‘zgartiradi.  
Masalan, JSON ma’lumotni yuborish orqali.

3️⃣ **Self-descriptive messages**  

Har bir so‘rov (request) va javob (response) o‘zida yetarli ma’lumotni tutadi.  
Client xabarni qanday ishlatishni bilishi kerak. Yani bir xil yondashuv barcha resurslar uchun:

- Naming convention (nomlash konvensiyasi)
- Link format
- Ma'lumot formati (XML va/yoki JSON)
- HTTP metodlari (GET, POST, PUT, DELETE)

**Maqsad:** Dasturchi bitta API bilan tanishgandan keyin, boshqa API'lar uchun ham xuddi shunday yondashuvdan foydalana olishi kerak.

4️⃣ **Hypermedia as the engine of application state (HATEOAS)**  

Client dastur faqat **bitta boshlang‘ich URI** ni biladi.  
Qolgan barcha amallarni server yuborgan **linklar (hypermedia)** orqali aniqlaydi.  
Masalan, javobda `next_page` yoki `comments_url` kabi havolalar bo‘ladi.

👉 **Oddiy qilib aytganda:**  
REST interfeysi — **bir xil metodlar va URL-lar orqali ishlaydigan yagona tizim**.

---
## 2. Client–Server

REST’da **client** (frontend yoki mobil ilova) va **server** (backend) qat’iyan **ajratilgan** bo‘ladi.

- Client — faqat interfeys bilan shug‘ullanadi (ko‘rsatish, yuborish).
- Server — ma’lumotni saqlash va qayta ishlash bilan shug‘ullanadi.
- Client faqat resurs URI'larini bilishi kerak - boshqa hech narsa
- Ular orasidagi interfeys o'zgarmagan holatda qolishi shart

---

## 3. Stateless (Holatsiz)

Server oldingi so‘rovlarni **eslab qolmaydi**.

- Client har safar barcha kerakli ma’lumotni (masalan, token) yuboradi.
- Server hech qanday “session” yoki “history” saqlamaydi.
- Har bir so'rov yangi so'rov sifatida ko'rib chiqiladi

**Har bir so'rov o'zida kerakli barcha ma'lumotni o'z ichiga olishi kerak:**
- Authentication ma'lumotlari
- Authorization ma'lumotlari
- Boshqa zarur context

**Misol:**
```
GET /api/orders
Headers:
  Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
  Content-Type: application/json
```

**Muhim:** Client ilovaning holatini boshqarish uchun mas'ul. Serverda so'rovlar orasida hech qanday client context saqlanmaydi.

---

## 4. Cacheable (Keshlanuvchi)

Serverdan qaytgan javoblar **keshlanadigan** yoki **keshlanmaydigan** bo‘lishi kerakligini bildiradi.
Response ma'lumotlari cacheable yoki non-cacheable deb belgilanadi. 
Client cache keyinchalik bir xil requestlar uchun javobni qayta ishlatishi mumkin.

Agar keshlanadigan bo‘lsa:

- Client keyingi safar xuddi shu so‘rovni yubormasdan, **oldingi javobni qayta ishlatishi** mumkin.
- Bu tarmoq yukini kamaytiradi va ishlash tezligini oshiradi.
- **Client uchun:** Performance yaxshilanadi
- **Server uchun:** Yukni kamaytiradi, kengaytiriluvchanlikni oshiradi
- **Tarmoq uchun:** Traffic kamayadi

### REST'da Keshlash:

Resurslar mos kelganda keshlanishi kerak va o'zlarini **cacheable** deb e'lon qilishlari shart.

**Keshlash ikki joyda amalga oshirilishi mumkin:**
1. Server tomonida
2. Client tomonida

**HTTP header misoli:**
```
Cache-Control: max-age=3600
Cache-Control: no-cache
Cache-Control: no-store
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
```

---

## 5. Layered System (Qatlamli Tizim)

REST arxitekturasi **bir necha qatlamlardan** tashkil topadi:  
masalan:

- Client → Proxy → Load Balancer → API Gateway → Server



```
Client → [Proxy] → [Load Balancer] → [API Gateway] → [API Server] → [Database Server]
````

**Foydalari:**

- Xavfsizlikni oshirish (firewall qatlamlarini qo'shish)
- Load balancing
- Tizimni kengaytirish osonlashadi
- Har bir qatlam o'z vazifasiga e'tibor qaratadi

---

## 6. Code on Demand (Ixtiyoriy)

Server ba’zida klientga **bajariladigan kod (script yoki applet)** yuborishi mumkin.  
Masalan, JavaScript funksiyalarni yuklab ishlatish.
