## 1-Bo'lim: Resource Tushunchasi

### 1.1 Resource nima?

REST'da **resource** - nomlanishi mumkin bo'lgan har qanday ma'lumot yoki kontseptsiya:

- Document yoki image
- Vaqtinchalik service (masalan: "Toshkentdagi bugungi ob-havo")
- Boshqa resourcelar to'plami (collection)
- Real obyekt (masalan: foydalanuvchi, mahsulot)

**Muhim prinsip**: Resource - bu entity o'ziga emas, balki entitylar to'plamiga **conceptual mapping**.

**Real hayotdan misol**:

- "Eng so'nggi versiya" - bu har doim yangilanib turuvchi resource
- "2024-yil 15-mayda chop etilgan versiya" - bu statik resource

Ikkalasi ham turli resourcelar, garchi ma'lum vaqtda bir xil ma'lumotni ko'rsatsa ham.

### 1.2 Resource Turlari

**a) Singleton Resource** - yagona obyekt:
```txt
/customers/12345        // 12345 ID'li bitta mijoz
/users/admin            // Admin foydalanuvchi
```


**b) Collection Resource** - obyektlar to'plami:
```
/customers              // Barcha mijozlar
/products               // Barcha mahsulotlar
```


**c) Sub-collection Resource** - ichma-ich to'plam:
```
/customers/12345/orders              // 12345 mijozning barcha buyurtmalari
/customers/12345/orders/98765        // Konkret buyurtma
```

## 2-Bo'lim: URI Dizayni - Best Practices

### 2.1 Otlardan (Nouns) foydalaning, fe'llardan emas!

✅ **To'g'ri:**
```
GET    /users                  // Foydalanuvchilar ro'yxati
POST   /users                  // Yangi foydalanuvchi yaratish
GET    /users/123              // 123-foydalanuvchi ma'lumotlari
PUT    /users/123              // 123-foydalanuvchini yangilash
DELETE /users/123              // 123-foydalanuvchini o'chirish
```

❌ **Noto'g'ri:**
```
/getUsers
/createUser
/deleteUser/123
/updateUser/123
```

**Sabab**: HTTP metodlari (GET, POST, PUT, DELETE) allaqachon actionlarni ifodalaydi - URI faqat resourceni identifikatsiya qilishi kerak.

---

### 2.2 Resource Archetypes (3 ta asosiy turi)

#### **2.2.1 Document** - yagona obyekt (Singular)
```
/users/admin
/devices/D-12345
/articles/rest-best-practices
```

Document - bu collection ichidagi bitta element. O'zining fieldlari va boshqa resourcelarga linklari bor.

#### **2.2.2 Collection** - server tomonidan boshqariladigan to'plam (Plural)
```
/users
/devices
/articles
/users/123/orders          // Sub-collection
```

Collection yangi resourcelarni qabul qilishi mumkin, lekin ularni yaratish yoki yaratmaslik - collection o'zi hal qiladi.

#### **2.2.3 Store** - client tomonidan boshqariladigan repository (Plural)
```
/users/123/playlists
/users/123/favorites
/users/123/bookmarks
```

Store'da client o'zi resource qo'yadi, olib chiqadi va o'chiradi. Store yangi URI yaratmaydi - har bir resourcening URI'si client tomonidan beriladi.

---

### 2.3 Consistency - Izchillik Qoidalari

#### **2.3.1 Forward slash (/) - ierarxiyani ko'rsatadi**
```
/device-management
/device-management/devices
/device-management/devices/D-123
/device-management/devices/D-123/sensors
/device-management/devices/D-123/sensors/S-456
```

#### **2.3.2 Oxirgi forward slash ishlatmang**
```
❌ /device-management/devices/
✅ /device-management/devices
```

#### **2.3.3 Hyphen (-) ishlatib o'qilishi oson qiling**
```
❌ /devicemanagement/manageddevices
✅ /device-management/managed-devices
```

#### **2.3.4 Underscore (_) ishlatmang**
```
❌ /inventory_management/managed_entities
✅ /inventory-management/managed-entities
```

Ba'zi fontlarda underscore ko'rinmasligi yoki qisman yashirinishi mumkin.

#### **2.3.5 Lowercase (kichik harflar) ishlatng**
```
✅ /my-folder/my-documents
❌ /My-Folder/My-Documents
❌ /MY-FOLDER/MY-DOCUMENTS
```

**Eslatma**: URI path case-sensitive! `http://api.example.com/Users` va `http://api.example.com/users` - turli endpointlar.

---

### 2.4 File Extension ishlatmang
```
❌ /devices.json
❌ /devices.xml
✅ /devices
```

Media type'ni URI'da emas, `Content-Type` header'ida bildiring:
```
GET /devices
Accept: application/json

Response:
Content-Type: application/json
```

---

### 2.5 CRUD operatsiyalarini URI'ga qo'shmang

HTTP metodlari actionni belgilaydi, URI faqat resourceni!
```
❌ GET  /users/create
❌ POST /users/delete/123
❌ GET  /users/update/123

✅ POST   /users              // Yaratish
✅ DELETE /users/123          // O'chirish
✅ PUT    /users/123          // To'liq yangilash
✅ PATCH  /users/123          // Qisman yangilash
✅ GET    /users/123          // O'qish
```

---

### 2.6 Query Parameters - Filtrlash, Sorting, Pagination

Yangi API yaratmang - query parameterlari ishlatng!
```
✅ GET /devices                                    // Barcha qurilmalar
✅ GET /devices?region=uzbekistan                  // Filtrlash
✅ GET /devices?region=uzbekistan&status=active    // Bir nechta filtr
✅ GET /devices?sort=installation-date&order=desc  // Saralash
✅ GET /devices?page=2&limit=20                    // Pagination
✅ GET /devices?region=uz&status=active&page=1&limit=10&sort=name
```

## 3-Bo'lim: Fe'llardan qochish (Anti-pattern)

### 3.1 Nima uchun fe'llar noto'g'ri?
```
❌ /devices/123/scripts/456/execute
❌ /users/123/activate
❌ /orders/789/cancel
```

Bu RPC-style, REST emas!

### 3.2 To'g'ri yechim: Noun sifatida action yaratish

**Noto'g'ri usul:**
```
❌ POST /scripts/456/execute
```

**To'g'ri usul 1: Status resource yaratish**
```
✅ POST /scripts/456/status
Body: { "action": "execute" }
```

**To'g'ri usul 2: Execution resource yaratish**
```
✅ POST /script-executions
Body: { 
  "script_id": 456,
  "device_id": 123
}
```

Bu yangi resource yaratadi - execution obyekti. Keyin uning statusini tekshirish mumkin:
```
GET /script-executions/789
Response: {
  "id": 789,
  "script_id": 456,
  "status": "running",
  "started_at": "2025-10-24T10:30:00Z"
}
```

## 4-Bo'lim: Real Loyiha Misoli

E-commerce tizimi uchun API dizayni:
```
# Foydalanuvchilar
GET    /users                          # Barcha foydalanuvchilar
POST   /users                          # Ro'yxatdan o'tish
GET    /users/123                      # Profil ma'lumotlari
PUT    /users/123                      # Profilni yangilash
DELETE /users/123                      # Hisobni o'chirish

# Mahsulotlar
GET    /products                       # Mahsulotlar katalogi
GET    /products?category=electronics  # Kategoriya bo'yicha
GET    /products/456                   # Konkret mahsulot
POST   /products                       # Yangi mahsulot (admin)
PUT    /products/456                   # Mahsulotni yangilash

# Buyurtmalar (user'ning sub-collection'i)
GET    /users/123/orders               # Foydalanuvchining buyurtmalari
POST   /users/123/orders               # Yangi buyurtma
GET    /users/123/orders/789           # Konkret buyurtma
PATCH  /users/123/orders/789           # Buyurtma statusini yangilash

# Savatcha (store archetype - client-managed)
GET    /users/123/cart                 # Savatcha
POST   /users/123/cart/items           # Mahsulot qo'shish
DELETE /users/123/cart/items/456       # Mahsulotni olib tashlash

# Izohlar (mahsulotning sub-collection'i)
GET    /products/456/reviews           # Mahsulot izohlari
POST   /products/456/reviews           # Izoh qoldirish
GET    /products/456/reviews/321       # Konkret izoh
```

## 5-Bo'lim: Advanced Patterns

### 5.1 Murakkabroq actions uchun yechimlar

**Google Cloud'ning Custom Methods pattern'i** (munozarali):
```
POST /devices/123:restart
POST /files/456:duplicate
POST /emails/789:send
```

Bu ba'zilar uchun REST prinsiplariga zid, lekin Google keng qo'llaydi.

**Alternativ - Noun sifatida action:**
```
POST /device-restarts        Body: { "device_id": 123 }
POST /file-duplications      Body: { "file_id": 456 }
POST /email-deliveries       Body: { "email_id": 789 }
```

### 5.2 Bulk Operations

#### Bir nechta resourceni bir vaqtda o'chirish

```
DELETE /users?ids=1,2,3
```

#### Yoki bulk endpoint

```
POST /users/bulk-delete
Body: { "ids": [1, 2, 3] }
```

### ❌ Noto'g'ri

```http
GET  /getUsers
POST /createUser
PUT  /updateUser/123
DELETE /deleteUser/123
```


### ✅ To'g'ri

```http
GET    /users          # Barcha userlarni olish
```

### 🤔 Nega?
- **HTTP metodlar allaqachon actionni bildiradi**: 
	- GET = olish
	- POST = yaratish
	- PUT = yangilash
	- DELETE = o'chirish
- **URI resourceni identifikatsiya qilishi kerak**, action emas
- **RESTful vs RPC**: Fe'l ishlatganingizda bu RPC-style bo'ladi, REST emas

### 💡 Real misol
```
# E-commerce
❌ /getProducts
✅ /products

# Blog
❌ /createPost
✅ /posts

# Autentifikatsiya
❌ /loginUser          # Bu muammo!

   POST /sessions     # Login qilish
   DELETE /sessions   # Logout qilish
```

## 2️⃣ Collection uchun ko'plik (plural) shakl ishlatng

### ❌ Noto'g'ri
```http
/product        # Bir mahsulotmi yoki hammasi?
/product/123    # Chalkash!
```

### ✅ To'g'ri
``` HTTP
/products       # Collection - aniq!
/products/123   # Collectiondagi bitta element
```

### 🤔 Nega?
- **Semantic aniqlik**: `/products` - bu to'plam deb tushunarli
- **Konsistentlik**: Barcha collectionlar uchun bir xil pattern
- **Natural til**: Ingliz tilida ham "users", "products" tabiiy eshitiladi

### 💡 Istisnolar?
``` http
# Singleton resourcelar uchun singular
/profile        # Joriy foydalanuvchining profili (yagona)
/cart           # Joriy foydalanuvchining savatchasi (yagona)
/dashboard      # Dashboard (yagona)

# Lekin collectionlar uchun - plural!
/users/123/orders    # 123-userning buyurtmalari (ko'plik)
```

## 3️⃣ Hyphen (-) bilan so'zlarni ajratng

### ❌ Noto'g'ri
``` http
/userprofiles        # O'qish qiyin
/userProfiles        # camelCase - URI'da noto'g'ri
/user_profiles       # Underscore ko'rinmasligi mumkin
```

### ✅ To'g'ri
```http
/user-profiles       # Aniq, o'qilishi oson
/device-management

```

### 🤔 Nega?
- **Readability**: Hyphen ko'zga yaqqol ko'rinadi
- **SEO**: Search enginelar hyphen'ni so'z ajratuvchi deb tushunadi
- **Convention**: Web'da qabul qilingan standart

### 💡 Real ssenariy
``` http
# Dashboard sahifasi
❌ /admindashboard
❌ /adminDashboard
✅ /admin-dashboard

# User profili
❌ /useraccountsettings
✅ /user-account-settings
```

## 4️⃣ Lowercase (kichik harflar) ishlatng

### ❌ Noto'g'ri
``` http
/Users              # Case-sensitive!
/Products/123
/Order-History      # Mixed case
```

### ✅ To'g'ri
``` http
/users
/products/123
/order-history
```

### 🤔 Nega?
- **URI path case-sensitive**: `/Users` ≠ `/users` - bu ikki xil endpoint!
- **Xatoliklardan saqlaydi**: Katta-kichik aralashmasligi confusion kamaytiradi
- **Convention**: Industry standard

### ⚠️ Muhim!
``` http
# Bu ikkala URI turlicha!
http://api.example.com/Users      # 404 error berishi mumkin
http://api.example.com/users      # To'g'ri ishlaydi

# Domain case-insensitive, lekin path emas!
HTTP://API.EXAMPLE.COM/users  ✅   # Domain ixtiyoriy
http://api.example.com/Users  ❌   # Path kichik bo'lishi kerak
```

## 5️⃣ Forward slash (/) - ierarxiyani bildiradi

### ✅ To'g'ri struktura
``` http
/users                           # 1-daraja: Userlar
/users/123                       # 2-daraja: Konkret user
/users/123/orders                # 3-daraja: User orderlari
/users/123/orders/456            # 4-daraja: Konkret order
/users/123/orders/456/items      # 5-daraja: Order itemlari
```

### 🤔 Mantiq?
Har bir slash - "ning ichidagi" degan ma'noni bildiradi:
``` http
/users/123/orders/456
```

### 💡 Real e-commerce misol
``` http
# Kategoriya → Mahsulot → Reviewlar
/categories/electronics
/categories/electronics/products
/categories/electronics/products/789
/categories/electronics/products/789/reviews

Yoki qisqaroq:
/products/789
/products/789/reviews
/products/789/reviews/321        # Konkret review
```

---

## 6️⃣ Oxirgi forward slash (/) ishlatmang

### ❌ Noto'g'ri
``` http
/users/             # Qo'shimcha slash kerak emas
/products/123/      # Semantic qiymat bermaydi
```

### ✅ To'g'ri
```http
/users
/products/123
```

### 🤔 Nega?
- **Semantic qiymat yo'q**: Oxirgi slash hech narsa bildirmaydi
- **Duplicate content muammosi**: Ba'zi serverlar buni turli URL deb hisoblashi mumkin
- **Chalkashlik**: `/users` va `/users/` bir xilmi? Client bilmaydi

### ⚠️ Server konfiguratsiyasi
``` http
# Nginx redirect misoli
/users/   →  301 Redirect  →  /users

# Shuning uchun oxiridan slash qo'ymang!
```

## 7️⃣ File extension (.json, .xml) ishlatmang

### ❌ Noto'g'ri
``` http
/users.json
/products/123.xml
/orders.pdf
```

### ✅ To'g'ri
``` http
/users              # Format HTTP headerda
/products/123
/orders
````

### 🤔 Qanday formatni bildirish kerak?

**Client so'raydi (Request header):**

```http
GET /users
Accept: application/json
```

**Server javob beradi (Response header):**

```http
HTTP/1.1 200 OK
Content-Type: application/json

{ "users": [...] }
```

### 💡 Content Negotiation misoli

#### JSON so'rash
```http
GET /products/123
Accept: application/json
→ Response: { "id": 123, "name": "..." }
```

#### XML so'rash
```http
GET /products/123
Accept: application/xml
→ Response: <product><id>123</id>...</product>
```

#### PDF so'rash
```http
GET /orders/456
Accept: application/pdf
→ Response: [PDF binary data]
```

**Afzalliklari:**
- Bir xil URI - turli formatlar
- Cleaner URLs
- Format o'zgarganda URI o'zgarmaydi

## 8️⃣ CRUD operatsiyalarini URI'ga qo'shmang

### ❌ Noto'g'ri (RPC-style)
``` http
POST /users/create
GET  /users/get/123
PUT  /users/update/123
DELETE /users/delete/123
POST /products/search        # Bu ham muammoli!
```

### ✅ To'g'ri (RESTful)
``` http
POST   /users               # Create
GET    /users/123           # Read
PUT    /users/123           # Update (full)
PATCH  /users/123           # Update (partial)
DELETE /users/123           # Delete

GET /products?search=laptop # Search
```

### 🤔 HTTP Metodlar = Fe'llar

| HTTP Method | CRUD Operation   | Misol               |
| ----------- | ---------------- | ------------------- |
| POST        | Create           | `POST /users`       |
| GET         | Read             | `GET /users/123`    |
| PUT         | Update (full)    | `PUT /users/123`    |
| PATCH       | Update (partial) | `PATCH /users/123`  |
| DELETE      | Delete           | `DELETE /users/123` |

### 💡 Murakkabroq actionlar?

**Noto'g'ri:**
``` http
POST /orders/456/cancel
POST /users/123/activate
POST /devices/789/restart
```

**To'g'ri variant 1: Status o'zgartirish**
```http
PATCH /orders/456
Body: { "status": "cancelled" }

PATCH /users/123
Body: { "status": "active" }
```

**To'g'ri variant 2: Yangi resource yaratish**
```http
POST /order-cancellations
Body: { "order_id": 456, "reason": "..." }

POST /device-restarts
Body: { "device_id": 789 }
```

## 🔟 Izchillik (Consistency) - eng muhimi!

### 🎯 Nima uchun izchillik muhim?

API'da **bir xil patternni** qo'llasangiz:
- ✅ O'rganish oson
- ✅ Xatolar kamayadi
- ✅ Maintainability yaxshilanadi
- ✅ Developer experience (DX) yaxshi

### ❌ Izchil emas (har bir endpoint o'z uslubida)
```http
GET  /getUsers                    # Fe'l ishlatilgan
POST /user/create                 # Singular + fe'l
GET  /Users/123                   # Bosh harf
PUT  /user-profile/update/123     # Aralash
GET  /products.json               # Extension
DELETE /DeleteProduct/456         # camelCase + fe'l
```

### ✅ Izchil (bir xil pattern)
```http
GET    /users
POST   /users
GET    /users/123
PUT    /users/123
DELETE /users/123

GET    /products
POST   /products
GET    /products/456
PUT    /products/456
DELETE /products/456

GET    /orders
GET    /orders/789
POST   /users/123/orders
```

### 📋 Izchillik Checklist

Har bir yangi endpoint uchun tekshiring:

- [ ] Plural shakl ishlatilganmi? (`/users` ✅ `/user` ❌)
- [ ] Lowercase? (`/users` ✅ `/Users` ❌)
- [ ] Hyphen bilan ajratilganmi? (`/user-profiles` ✅ `/userProfiles` ❌)
- [ ] Fe'l yo'qmi? (`/users` ✅ `/getUsers` ❌)
- [ ] Oxirgi slash yo'qmi? (`/users` ✅ `/users/` ❌)
- [ ] Extension yo'qmi? (`/users` ✅ `/users.json` ❌)
- [ ] HTTP metod to'g'rimi? (`GET /users` ✅ `GET /users/get` ❌)
- [ ] Ierarxiya mantiqiymi? (`/users/123/orders` ✅)
