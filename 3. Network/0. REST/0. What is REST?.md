
**REST**(Representational State Transfer) — bu **tarmoq orqali ishlaydigan ilovalar (masalan, web-xizmatlar)** uchun mo‘ljallangan **arxitektura uslubi (architecture style)**.  
U **Roy Fielding** tomonidan 2000-yilda taqdim etilgan.

**REST API (yoki RESTful API)** — bu **REST prinsiplari asosida qurilgan API**

Bugungi kunda REST — **web API (Application Programming Interface)** yaratishning eng mashhur yondashuvlaridan biridir.

> 🔹 REST bu **protokol** yoki **standart** emas.  
> U — faqat **arxitektura uslubi**, ya’ni **qanday ishlash kerakligi haqida qoida va tamoyillar to‘plami**(ya’ni, dizayn prinsiplari)


REST’ni **to‘liq RESTful** deb atash uchun u quyidagi **6 ta arxitektura cheklovi (constraint)** ga amal qilishi kerak:

| #   | Nomi                                                              | Maqsadi                                                |
| --- | ----------------------------------------------------------------- | ------------------------------------------------------ |
| 1️⃣ | **Uniform Interface**                          (Yagona interfeys) | Barcha resurslarga bir xil usulda murojaat qilish      |
| 2️⃣ | **Client–Server**                                                 | Mijoz va serverning rolini ajratish                    |
| 3️⃣ | **Stateless**                                                     | Server har bir so‘rovni mustaqil tarzda qayta ishlashi |
| 4️⃣ | **Cacheable**                                                     | Javoblarni keshlash (saqlash) imkonini berish          |
| 5️⃣ | **Layered System**                                                | Tizimni qatlamlarga ajratish imkonini berish           |
| 6️⃣ | **Code on Demand** _(ixtiyoriy)_                                  | Server mijozga bajariladigan kod yuborishi mumkin      |


**Resurs** — bu API orqali boshqariladigan **obyekt yoki ma’lumot turi**.  
Masalan:

| Resurs  | Tavsif        | URL namunasi |     |
| ------- | ------------- | ------------ | --- |
| User    | Foydalanuvchi | `/users`     |     |
| Post    | Post (maqola) | `/posts`     |     |
| Product | Mahsulot      | `/products`  |     |
|         |               |              |     |
Har bir resursga **HTTP metodlari** orqali murojaat qilinadi:

## 1. RESTning Olti Asosiy Prinsipi

REST arxitekturasi oddiylik, kengaytiriluvchanlik va stateless (holatni saqlamaslik)ni ta'minlaydigan bir necha prinsiplar va cheklovlarga asoslangan.

### 1.1. Uniform Interface (Yagona Interfeys)

Bu printsip komponentlar orasidagi interfeysni soddalashtiradi va tizim arxitekturasini tushunarli qiladi.

**To'rtta asosiy talab:**

**a) Resurslarni identifikatsiya qilish**

- Har bir resurs (ma'lumot) client va server o'rtasida noyob identifikatorga ega bo'lishi kerak
- Masalan: `/users/123`, `/posts/456` kabi URI'lar

**b) Representation orqali resurslarni boshqarish**

- Resurslar serverda yagona ko'rinishda (representation) bo'lishi kerak
- Client bu ko'rinishlardan foydalanib resurs holatini o'zgartiradi

**c) Self-descriptive xabarlar**

- Har bir resurs tasvirida uni qanday qayta ishlash kerakligi haqida yetarli ma'lumot bo'lishi kerak
- Qo'shimcha amallarni bajarish imkoniyatlari ham ko'rsatilishi lozim

**d) Hypermedia ilovaning harakatlantiruvchi kuchi**

- Client faqat boshlang'ich URI'ni bilishi kerak
- Boshqa barcha resurslar va o'tishlar hyperlink'lar orqali dinamik ravishda boshqariladi

**Oddiy qilib aytganda:** REST client va server o'rtasida izchil va yagona interfeys yaratadi. Masalan, HTTP-asosli REST API'larda standart HTTP metodlari (GET, POST, PUT, DELETE) va URI'lar ishlatiladi.

### 1.2. Client-Server (Client-Server Modeli)

Bu printsip client va server o'rtasida vazifalarni ajratishni talab qiladi.

**Foydalari:**

- Client (foydalanuvchi interfeysi) va Server (ma'lumot saqlash) mustaqil ravishda rivojlanadi
- Interfeys turli platformalarda ishlatilishi mumkin
- Server komponentlari soddalashadi va kengaytiriluvchanlik oshadi

**Real hayotdan misol:** Restaurant modelini tasavvur qiling. Mijoz (client) menyudan taom buyurtma qiladi, oshpaz (server) esa taomni tayyorlaydi. Ular bir-biridan mustaqil ishlaydi, lekin standart menyu (interface) orqali bog'lanadi.

**Muhim shart:** Client va server rivojlanayotganda ularning o'rtasidagi shartnoma (interface) buzilmasligi kerak.

### 1.3. Stateless (Holatsiz)

Serverda hech qanday client holati saqlanmaydi. Har bir so'rov to'liq va mustaqil bo'lishi kerak.

**Qanday ishlaydi:**

- Client yuborgan har bir so'rov o'zida kerakli barcha ma'lumotni o'z ichiga olishi kerak
- Server avvalgi so'rovlar haqida hech narsa eslamaydi
- Session holati to'liq client tomonida saqlanadi

**Real hayotdan misol:** Har safar do'konga borganingizda, sotuvchi sizni eslamaydi. Siz har gal o'zingiz haqingizda (nima sotib olmoqchi ekanligingiz) barcha ma'lumotni aytishingiz kerak.

**Foydasi:** Server soddalashadi, load balancing osonlashadi, tizim kengaytiriluvchanlik oshadi.

### 1.4. Cacheable (Keshlanuvchi)

Response (javob) o'zini keshlash mumkin yoki mumkin emasligini aniq ko'rsatishi kerak.

**Qanday ishlaydi:**

- Agar javob cacheable bo'lsa, client uni ma'lum vaqt davomida qayta ishlatishi mumkin
- Bu xuddi shu so'rovlar uchun server yukini kamaytiradi
- Tezlikni oshiradi

**Real hayotdan misol:** Siz kitobxonada kitobning nusxasini olganingizda, uni bir hafta ushlab turishingiz mumkin. Har safar bir xil ma'lumot uchun kitobxonaga borish shart emas.

### 1.5. Layered System (Qatlamli Tizim)

Arxitektura ierarxik qatlamlarda tashkil etilishi mumkin. Har bir komponent faqat o'zi bilan bevosita bog'langan qatlam bilan ishlaydi.

**Real hayotdan misol:** MVC pattern (Model-View-Controller) - bu qatlamli tizimning yaqqol namunasi. Har bir qatlam o'z vazifasini bajaradi va boshqa qatlamlarning ichki ishlashini bilmaydi.

**Foydasi:**

- Tizimni rivojlantirish osonlashadi
- Xavfsizlikni oshirish mumkin (masalan, load balancer, firewall qatlamlarini qo'shish)
- Kengaytiriluvchanlik yaxshilanadi

### 1.6. Code on Demand (Ixtiyoriy)

Bu yagona ixtiyoriy printsip. Client funksionalligini kod (applet yoki script) yuklab olish orqali kengaytirish mumkin.

**Qanday ishlaydi:**

- Server clientga kod yuborishi mumkin
- Client bu kodni bajaradi
- Bu clientning oldindan tayyorlanishi kerak bo'lgan funksiyalarni kamaytiradi

**Misol:** Veb-sahifada JavaScript kodi yuklanib, u brauzerda bajariladi.

## 2. Resource (Resurs) Nima?

**Resource** - RESTda ma'lumotning asosiy abstraktsiyasi. Nomlash mumkin bo'lgan har qanday ma'lumot resurs bo'lishi mumkin.

**Resurs misollari:**

- Hujjat yoki rasm
- Vaqtinchalik servis
- Boshqa resurslar to'plami
- Real obyekt (masalan, foydalanuvchi)

### Resource Representation (Resurs Tasviri)

Resursning ma'lum bir vaqtdagi holati - bu uning representation'i.

**Representation tarkibi:**

1. **Ma'lumot** - asosiy kontent
2. **Metadata** - ma'lumot haqida ma'lumot
3. **Hypermedia links** - clientni keyingi holatga o'tkazadigan havolalar

### 2.1. Resource Identifiers

REST har bir resursni identifikatsiya qilish uchun identifikatorlardan foydalanadi. Odatda bu URI (Uniform Resource Identifier) bo'ladi.

**Misollar:**

```
https://api.example.com/users/123
https://api.example.com/posts/456
https://api.example.com/posts/456/comments
```

### 2.2. Hypermedia

**Media type** - representation'ning ma'lumot formati. Bu format qanday qayta ishlash kerakligini belgilaydi.

**Muhim tushuncha:** Hypertext (yoki hypermedia) - bu ma'lumot va boshqaruvning bir vaqtda taqdim etilishi. Ma'lumotning o'zi foydalanuvchiga (yoki mashinalarga) tanlov va harakatlarni taqdim etadi.

**Roy Fieldingning so'zlari:** Hypertext faqat brauzerda HTML (yoki XML, JSON) bo'lishi shart emas. Mashinalar ham ma'lumot formatini va bog'lanish turlarini tushunsa, link'larni kuzatib borishi mumkin.

### 2.3. Self-Descriptive (O'z-o'zini Tasvirlovchi)

Resurs tavsiflari o'z-o'zini tushuntiruvchi bo'lishi kerak. Client resursning xodim yoki qurilma ekanligini bilishi shart emas - u faqat media type'ga qarab harakat qiladi.

**Amalda:** Ko'pincha har bir resurs uchun maxsus media type yaratiladi.

### 2.4. To'liq Misol

Blog posti haqida REST resurs misoli:

json

```json
{
  "id": 123,
  "title": "What is REST",
  "content": "REST is an architectural style...",
  "published_at": "2023-11-04T14:30:00Z",
  "author": {
    "id": 456,
    "name": "John Doe",
    "profile_url": "https://example.com/authors/456"
  },
  "comments": {
    "count": 5,
    "comments_url": "https://example.com/posts/123/comments"
  },
  "self": {
    "link": "https://example.com/posts/123"
  }
}
```

**Bu misolda:**

- Post haqida to'liq ma'lumot bor
- Muallif (author) va izohlar (comments) uchun hypermedia link'lar berilgan
- Client bu link'larni kuzatib, qo'shimcha ma'lumot olishi yoki amallarni bajarishi mumkin

## 3. Resource Methods (Resurs Metodlari)

Resource metodlar resursning bir holatdan ikkinchi holatga o'tishini ta'minlaydi.

**Keng tarqalgan noto'g'ri tushuncha:** Ko'p odamlar resource metodlarni HTTP metodlari (GET/PUT/POST/DELETE) bilan aralashtirib yuborishadi. Roy Fielding qaysi metoddan qayerda foydalanish kerakligi haqida aniq tavsiya bermagan. U faqat **uniform interface** bo'lishini ta'kidlaydi.

**Misol:** Agar siz resursni yangilash uchun HTTP POST ishlatishga qaror qilsangiz (HTTP PUT o'rniga), bu ham to'g'ri. Interfeys baribir RESTful bo'ladi.

**Ideal holat:** Resurs holatini o'zgartirish uchun kerak bo'lgan hamma narsa resurs representation'ida bo'lishi kerak - barcha qo'llab-quvvatlanadigan metodlar va ularning qanday ishlashi ham.

**Asosiy qoida:** REST API'ga faqat boshlang'ich URI va standart media type'lar bilan kirish kerak. Undan keyin barcha o'tishlar client tomonidan serverdagi representation'larda mavjud tanlovlar asosida amalga oshiriladi.

## 4. REST va HTTP Bir Xil Emasmi?

Ko'p odamlar REST va HTTP ni bir xil deb o'ylashadi. Bu **noto'g'ri**.

### REST ≠ HTTP

REST printsiplari HTTP bilan cheklanmaydi. Roy Fielding o'z dissertatsiyasida hech qanday protokol, hatto HTTP haqida ham aniq yo'nalish bermagan.

**Asosiy qoida:** Agar siz oltita asosiy printsipga rioya qilsangiz, interfeysni RESTful deb atash mumkin - qaysi protokol ishlatilganidan qat'iy nazar.

**Amalda:** HTTP eng ko'p ishlatiladigan protokol, lekin REST uni majburiy qilmaydi.

---

## 5. Xulosa

**REST arxitektura uslubida:**

1. **Ma'lumot va funksionallik** resurs sifatida ko'rib chiqiladi
2. **URI** (Uniform Resource Identifier) orqali resurslar identifikatsiya qilinadi
3. **Oddiy va aniq amallar** to'plami orqali resurslar bilan ishlash amalga oshiriladi
4. **Resurslar representation'dan ajratilgan** - clientlar turli formatlarda (HTML, XML, JSON, PDF, JPEG va boshqalar) kontentga kirishi mumkin
5. **Standartlashtirilgan interfeys va protokol** - odatda HTTP, lekin majburiy emas
6. **Metadata** keshlanishni boshqarish, xatolarni aniqlash va autentifikatsiya uchun ishlatiladi
7. **Har bir o'zaro ta'sir stateless** bo'lishi shart

**REST'ning afzalliklari:**

- ✅ Sodda
- ✅ Yengil (lightweight)
- ✅ Tez
- ✅ Kengaytiriluvchan
- ✅ Platformadan mustaqil
