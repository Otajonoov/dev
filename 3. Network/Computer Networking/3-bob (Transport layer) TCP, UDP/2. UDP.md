### UDP (User Datagram Protocol)

- Logical connection o'rnatmasdan ishlaydigan protokol
- Ishonchsiz ma'lumot uzatish xizmatini taqdim etadi
- Application'lar o'rtasida connectionless aloqa

UDP da traffic boshqarilmaydi - application istalgan tezlikda jo'natishi mumkin.

Internet kontekstida transport layer paketlarini **segment** deb ataymiz. Lekin esda tuting:

- RFC standartlarida TCP paketlari "segment"
- UDP paketlari ko'pincha "datagram" deb ataladi
- Network layer paketlari ham "datagram" deb ataladi!

UDP ning kamchiliklari :

1. No acknowledgment
2. No guaranteed delivery
3. Connectionless
4. No congestion Control
5. No order packets
6. Security

UDP ning afzalliklari :

1. Smaller packets
2. Less bandwidth
3. faster than tcp
4. Stateless

### UDP ning minimal xizmatlari:

- Process'lar o'rtasida ma'lumot uzatish
- Xatolarni aniqlash
- **Bu hammasi!** Qo'shimcha xizmatlar yo'q
- IP kabi unreliable - kafolat bermaydi

 **UDP** serverda ozi uchun connection saqlamaydi bu va **==_stateless connection_==** deyiladi.
 Bu degani agar server ochib yonsa ham UDP client bu haqida sezmaydi . 
 
 **UDP** sizdan malumotlarni qabul qilganligi haqida malumot bermaydi (==_no acknowledgment_==).
 
 U faqat siz uzatgan packet to'g'ri yoki noto'g'riligi haqida malumot berishi mumkin.
 Hamda sizni malumotlaringiz yetib borishi kafolatlanmaydi .U siz qanchalik ko'p malumot uzatsangiz ham serverga trafik ko'p yoki kamligini hisobga olmaydi. Malumotni olib uzataveradi. Packetlar bunda tartiblanmaydi ketma-ket uzataveriladi. 
 
 **UDP xavfsizlik** katta muammo bo'ladi chunki server bilan aloqa ushlab turilmaganligi uchun server so'rov qayerdan kelayotganligini aniqlab ololmaydi. 
 
 Agarda qaysidir port ochiq bo'lsa istalgan odam malumot uzatishi mumkin. 
 Shuning uchun juda ko'p **==_firewalllar_==** boshlang'ich holatda UDP ni bloklaydi. 
 UDP malumot yetib borgan bormaganligini tekshirmaganligi hamda tartiblamaganligi uchun packetlarga qo'shimcha malumot qo'shmaydi va buning yordami kichkina packetlarga ega bo'ladi. Va buning natijasida kamroq internet trafigi ishlatilinadi. TCP dan tezroq chunki ular malumotni qayta uztishga yoki trafik ko'pligida sekinlashishlar yo'ligi uchun tezroq ishlaydi.

#### 3.3. UDP — Ulanishsiz Transport Darajasi Protokoli

UDP (User Datagram Protocol) — bu eng sodda transport darajasi protokoli bo'lib, u minimal funksiyalarni bajaradi. UDP protokolining asosiy xususiyatlari:

##### UDP'ning asosiy xarakteristikalari:

1. **Ulanishsiz protokol** - UDP ma'lumot uzatishdan oldin jo'natuvchi va qabul qiluvchi o'rtasida ulanish o'rnatmaydi (handshake yo'q)
2. **Minimal funksionallik** - UDP faqat:
    - Multiplekslash/demultiplekslash
    - Oddiy xatolarni tekshirish (checksum)
    - Port raqamlarini qo'shish funksiyalarini bajaradi
3. **IP protokolga qo'shimcha** - UDP IP protokolga faqat port raqamlari va checksum qo'shadi, boshqa hech narsa qo'shmaydi

##### UDP'dan foydalanadigan ilovalar:

- **DNS** (Domain Name System)
- **RIP** (Routing Information Protocol)
- **SNMP** (Simple Network Management Protocol)
- **Real-time ilovalar** (IP-telefoniya, video konferensiya)
- **Streaming media** (audio/video oqimi)

##### UDP'ning TCP'dan afzalliklari:

1. **Ilovaning to'liq nazorati** - UDP ma'lumotlarni darhol yuboradi, TCP esa congestion control tufayli sekinlashtirishi mumkin
2. **Ulanish o'rnatish yo'q** - TCP uchlik handshake talab qiladi, UDP esa darhol ma'lumot yuboradi (kechikish yo'q)
3. **Holat saqlanmaydi** - TCP connection state saqlaydi (buferlar, sequence numbers), UDP esa saqlamaydi.
4. **Kichik header** - UDP header atigi 8 bayt, TCP header esa 20 bayt

#### 3.3.1. UDP Segmentining Tuzilishi

UDP segmenti quyidagi qismlardan iborat:

```
┌─────────────────────────────────┐
│     32 bit                      │
├─────────────────┬───────────────┤
│ Source Port     │ Dest Port     │ (har biri 16 bit)
│ (Jo'natuvchi)   │ (Qabul qiluvchi)│
├─────────────────┼───────────────┤
│ Length          │ Checksum      │ (har biri 16 bit)
│ (Uzunlik)       │ (Nazorat yig'indisi)│
├─────────────────┴───────────────┤
│                                 │
│     Application Data            │
│     (Ilova ma'lumotlari)       │
│                                 │
└─────────────────────────────────┘
```

**Header maydonlari:**

- **Source Port** (16 bit) - jo'natuvchi port raqami
- **Destination Port** (16 bit) - qabul qiluvchi port raqami
- **Length** (16 bit) - butun UDP segmentining uzunligi (header + data) baytlarda
- **Checksum** (16 bit) - xatolarni tekshirish uchun nazorat yig'indisi

## 3.3.2. UDP Checksum (Nazorat yig'indisi)

Checksum xatolarni aniqlash uchun ishlatiladi. U segmentdagi bitlar uzatish jarayonida buzilgan-buzilmaganligini tekshiradi.

### Checksum hisoblash jarayoni:

**Jo'natuvchida:**

1. Segment barcha 16-bitlik so'zlariga bo'linadi
2. Barcha 16-bitlik so'zlar qo'shiladi
3. Natijaning 1-to'ldiruvchisi (one's complement) hisoblanadi
4. Bu qiymat checksum maydoniga yoziladi

**Misol:**

```
So'z 1: 0110011001100000
So'z 2: 0101010101010101  
So'z 3: 1000111100001100

1 va 2 yig'indisi:
  0110011001100000
+ 0101010101010101
─────────────────
  1011101110110101

3-so'z bilan yig'indisi:
  1011101110110101
+ 1000111100001100
─────────────────
  0100101011000010

1-to'ldiruvchi (0↔1 almashtirish):
0100101011000010 → 1011010100111101 (checksum)
```

**Qabul qiluvchida:**

1. Barcha 16-bitlik so'zlar (data + checksum) qo'shiladi
2. Agar natija 1111111111111111 bo'lsa - xato yo'q
3. Agar biror bit 0 bo'lsa - xato bor

### Nima uchun UDP checksum kerak?

- Barcha link-layer protokollar xatolarni tekshirmaydi
- Router xotirasida saqlashda xato bo'lishi mumkin
- End-to-end tamoyili: oxirgi funksionallik end-system'larda bo'lishi kerak

**Muhim:** UDP xatolarni aniqlaydi, lekin tuzatmaydi! Ba'zi implementatsiyalar buzilgan segmentni tashlab yuboradi, ba'zilari ilovaga ogohlantirish bilan uzatadi.

## Xulosa

UDP - bu sodda, tez, lekin ishonchsiz protokol. U real-time ilovalar, DNS so'rovlari va boshqa kechikishga sezgir ilovalar uchun juda mos keladi. UDP'ning soddaligi uning kuchi - u minimal overhead qo'shadi va ilovaga to'liq nazorat beradi. Lekin ishonchli uzatish kerak bo'lsa, bu funksiyani ilova darajasida implementatsiya qilish kerak bo'ladi.

---

### Ishonchli ma'lumot uzatish modeli

Ishonchli ma'lumot uzatish xizmati yuqori darajadagi ob'ektlarga **ishonchli kanal** taqdim etadi. Ishonchli kanal orqali:

- Ma'lumotlar buzilmaydi (bitlar 0 dan 1 ga yoki aksincha o'zgarmaydi)
- Ma'lumotlar yo'qolmaydi
- Ma'lumotlar jo'natilgan tartibda yetkaziladi

Bu aynan TCP protokoli Internet ilovalari uchun taqdim etadigan xizmat modelidir.

### rdt (reliable data transfer) protokoli

Ishonchli ma'lumot uzatish protokoli (rdt) ishonchsiz kanal ustida ishonchli xizmat yaratadi. Masalan, TCP ishonchsiz IP protokoli ustida ishonchli uzatishni ta'minlaydi.

## 3.4.1. Ishonchli ma'lumot uzatish protokolini yaratish

### rdt1.0 - Mukammal ishonchli kanal uchun protokol

Eng oddiy holat - pastki kanal mutlaqo ishonchli bo'lgan vaziyat. Bu holda:

- Jo'natuvchi ma'lumotlarni qabul qiladi va paketga o'rab yuboradi
- Qabul qiluvchi paketni qabul qilib, ma'lumotlarni chiqarib oladi va yuqori darajaga uzatadi
- Hech qanday tasdiqlash yoki qayta yuborish kerak emas

### rdt2.0 - Bitlar buzilishi mumkin bo'lgan kanal

Kanal bitlarni buzishi mumkin, lekin paketlarni yo'qotmaydi. Bu holda **ARQ (Automatic Repeat reQuest)** protokollari qo'llaniladi.

ARQ protokollari uchta asosiy mexanizmdan foydalanadi:

1. **Xatolikni aniqlash** - Checksum yordamida buzilgan bitlarni aniqlash
2. **Teskari aloqa** - ACK (ijobiy) va NAK (salbiy) kvitansiyalar
3. **Qayta uzatish** - Xatolik aniqlangan paketlarni qayta yuborish

Bu **stop-and-wait** protokol - yangi paket yuborishdan oldin joriy paket uchun tasdiqlashni kutadi.

### rdt2.1 - Kvitansiyalar ham buzilishi mumkin

rdt2.0 ning kamchiligi - ACK/NAK kvitansiyalari ham buzilishi mumkin. Yechim:

- Paketlarga **sequence number** (tartib raqami) qo'shish
- Oddiy holat uchun 1 bit yetarli (0 yoki 1)
- Takroriy paketni yangi paketdan ajratish mumkin bo'ladi

### rdt2.2 - NAK siz protokol

NAK o'rniga oldingi paket uchun ACK yuborish mumkin. Duplicate ACK lar xatolikni bildiradi.

### rdt3.0 - Paketlar yo'qolishi mumkin bo'lgan kanal

Kanal nafaqat bitlarni buzishi, balki paketlarni ham yo'qotishi mumkin. Yechim:

**Timer mexanizmi:**

1. Paket yuborilganda timer ishga tushiriladi
2. Belgilangan vaqt ichida ACK kelmasa - timeout
3. Timeout bo'lganda paket qayta yuboriladi
4. ACK kelganda timer to'xtatiladi

Bu **alternating bit protocol** deb ham ataladi (sequence number 0 va 1 orasida almashadi).

### Asosiy tushunchalar xulosasi:

Protocol elementlari va ularning vazifalari:

- **Checksum** - xatoliklarni aniqlash
- **Sequence numbers** - paketlarni tartibga solish va dublikatlarni aniqlash
- **Timer** - yo'qolgan paketlarni aniqlash
- **ACK/NAK** - qabul qilish haqida xabar berish

Bu elementlarning har biri protokolda muhim va zaruriy rol o'ynaydi. Endi bizda ishonchli ma'lumot uzatish protokoli mavjud!

### 3.4.2. Konveyerlashgan ishonchli ma'lumot uzatish protokoli

rdt3.0 protokoli funksional jihatdan to'g'ri ishlaydi, ammo bu protokolning xizmat ko'rsatish sifati, ayniqsa zamonaviy yuqori tezlikli kompyuter tarmoqlarida, ko'p foydalanuvchilarni qoniqtirmaydi. Asosiy muammo shundaki, u tasdiqlashni kutish protokollariga tegishli.

Kutishning oqibatlarini yaxshiroq tushunish uchun quyidagi vaziyatni tasavvur qilaylik: bitta host AQShning g'arbiy qirg'og'ida, ikkinchisi esa sharqiy qirg'og'ida joylashgan. Bu ikki host o'rtasida signal yorug'lik tezligida tarqalganda aylanma vaqt (RTT) taxminan 30 ms ni tashkil qiladi. Hostlar 1 Gbit/s (10⁹ bit/s) uzatish tezligiga ega kanal bilan bog'langan deb faraz qilaylik. Sarlavha va ma'lumotlarni o'z ichiga olgan paket hajmi L = 1000 bayt yoki 8000 bit bo'lsa, gigabit kanalda paketni yetkazib berish uchun zarur vaqt:

d_uzatish = L/R = 8000 bit/paket / 10⁹ bit/s = 8 mks

Jo'natuvchi t=0 vaqtda paket jo'natishni boshlasa, oxirgi bit jo'natuvchi kanaliga t = L/R = 8 mks vaqtda tushadi. Keyin paketga mamlakatni kesib o'tish uchun 15 ms kerak bo'ladi va paketning oxirgi biti qabul qiluvchiga t = RTT/2 + L/R = 15,008 ms vaqtda yetib keladi. Soddalik uchun tasdiqlash paketlari juda kichik (ularning uzatish vaqtini e'tiborsiz qoldirishimiz mumkin) va qabul qiluvchi ma'lumot paketining oxirgi biti qabul qilingan zahoti tasdiqlash jo'nata oladi deb faraz qilaylik. U holda tasdiqlash jo'natuvchiga t = RTT + L/R = 30,008 ms vaqtda yetib keladi. Shu vaqtda jo'natuvchi keyingi xabarni uzatishni boshlashi mumkin. Shunday qilib, 30,008 ms dan jo'natuvchi faqat 0,008 ms davomida to'g'ridan-to'g'ri jo'natish bilan band bo'ladi.

Agar biz jo'natuvchining "foydali ish koeffitsiyenti"ni (yoki kanalning) jo'natuvchi kanalda haqiqatda ma'lumot jo'natish bilan band bo'lgan vaqt ulushi sifatida aniqlaydigan bo'lsak, tasdiqlashni kutish protokoli jo'natuvchining juda achinarlisi "FIK"ga ega:

d_jo'natuvchi = (L/R)/(RTT + L/R) = 0,008/30,008 = 0,00027

Ya'ni jo'natuvchi vaqtning faqat 0,027 foizida band bo'lgan! Bundan tashqari, bizning hisob-kitoblarimiz shuni ko'rsatadiki, ma'lumotlar uzatish tezligi 1000 bayt/30,008 ms ni tashkil qiladi, bu 267 Kbit/s - aloqa kanalining ta'minlagan tezligidan minglab marta past ko'rsatkich.

Aynan shu tavsiflan muammoning yechimi juda oddiy: tasdiqlashni kutish protokoli uslubida uzatish o'rniga, jo'natuvchi tasdiqlashlarni kutmasdan bir nechta paketlarni jo'natishi mumkin , uch paketni ketma-ket uzatish bizning protokolimizga nisbatan foydalilikni uch baravar oshiradi. Guruh bo'lib jo'natiladigan paketlarni konveyer ko'rinishida tasvirlash qulay bo'lgani uchun, bu uzatish usuli konveyerlashgan uzatish deb atalgan.

Ishonchli ma'lumot uzatish protokollarida konveyerlashni qo'llash quyidagi oqibatlarga olib keladi:

- Ketma-ket raqamlar diapazoni kengayadi, chunki barcha jo'natilgan paketlar (takroriy uzatishdan tashqari) bir qiymatli aniqlanishi kerak.
- Uzatuvchi va qabul qiluvchi tomonda buferlar hajmini oshirish zarur bo'ladi. Jo'natuvchi tomonining bufer hajmi tasdiqlash kutilayotgan barcha jo'natilgan paketlarni saqlash uchun yetarli bo'lishi kerak. Qabul qiluvchi tomonda muvaffaqiyatli qabul qilingan paketlarni buferga olish zarurati yuzaga kelishi mumkin.
- Ketma-ket raqamlar diapazoni va bufer hajmi talablari protokol paketlarning buzilishi, yo'qolishi va kechikishiga javob sifatida qabul qilgan harakatlarga bog'liq. Konveyerlash holatida xatolarni tuzatishning ikki usuli mavjud: N paketga qaytish va tanlab takrorlash.


**Muammo**: Oddiy "jo'nat va kutish" protokoli (har bir paketdan keyin tasdiqlashni kutish) juda sekin ishlaydi. Misol:

- AQShning bir qirg'og'idan ikkinchisiga ma'lumot jo'natish
- 1 Gigabit/s tezlikdagi kanal
- Ammo haqiqiy tezlik atiga 267 Kilobit/s (ming marta sekinroq!)

**Sabab**: Jo'natuvchi har bir paketdan keyin tasdiqlashni kutishga majbur, bu vaqtda kanal bo'sh turadi.

**Yechim**: Konveyerlash - bir vaqtning o'zida bir nechta paketni jo'natish, tasdiqlashni kutmasdan. Bu usul samaradorlikni bir necha barobar oshiradi.

**Konveyerlashning talablari**:

1. Ko'proq raqam berish tizimi kerak (paketlarni farqlash uchun)
2. Jo'natuvchi va qabul qiluvchida katta xotira (bufer) kerak
3. Xatolarni tuzatish uchun murakkab algoritm kerak

###  N paketga qaytish (GBN protokoli)

Go-Back-N (GBN) protokolida jo'natuvchi bir vaqtning o'zida bir nechta mavjud ma'lumot paketlarini tasdiqlashni kutmasdan uzatishi mumkin, tasdiqlashni kutayotgan paketlarning maksimal soni N dan oshmasligi kerak. 

Agar biz eng katta tasdiqlashni kutayotgan paket raqamini base, va eng kichik ishlatilmagan ketma-ket raqamni nextseqnum deb ataydigan bo'lsak, ya'ni keyingi jo'natiladigan paketning ketma-ket raqami, ketma-ket raqamlar diapazonida to'rtta oraliqni ajratish mumkin. [0,base-1] oraliqdagi ketma-ket raqamlar allaqachon uzatilgan va ularga tasdiqlash olingan paketlarga to'g'ri keladi. [base,nextseqnum-1] oralig'i allaqachon jo'natilgan, ammo ularga hali tasdiqlash olinmagan paketlarga to'g'ri keladi. [nextseqnum, base+N-1] oraliqdagi ketma-ket raqamlar yuqori darajadan ma'lumotlar kelishi bilan darhol jo'natilishi mumkin bo'lgan paketlarga tayinlanadi. Nihoyat, base+N ga teng yoki undan katta ketma-ket raqamlar konveyerdagi joriy tasdiqlanmagan paket (aniqrog'i, base ketma-ket raqamli paket) tasdiqlanmaguncha ishlatilishi mumkin emas.

Oyna hajmi - N

[Diagramma tavsifi: base va nextseqnum ko'rsatkichlari, ketma-ket raqamlar oynasi]

Tavsiflar:

- Allaqachon ACK olingan
- Jo'natilgan, ammo hali ACK olinmagan
- [Jo'natishga] tayyor, ammo hali jo'natilmagan
- Tayyor emas

jo'natilgan, ammo hali tasdiqlanmagan paketlar uchun mavjud ketma-ket raqamlar diapazonini ketma-ket raqamlar diapazonidagi N o'lchamdagi "oyna" sifatida tasavvur qilish mumkin. Protokol ishlashi davomida u ketma-ket raqamlar fazosi bo'ylab oldinga siljiydi. Shu sababli N soni oyna o'lchamini bildiradi va GBN protokoli sirpanuvchi oyna protokoli deb ataladi. Ehtimol siz nima uchun avvalo N soni bilan jo'natilmagan, tasdiqlanmagan paketlarning maksimal sonini cheklaganimizdan hayron bo'lsangiz. Nega ularning sonini cheklanmagan qoldirmadik? 3.5-bo'limda ko'ramizki, sabablardan biri oqimni boshqarishdir. 3.7-bo'limda to'lib ketishni boshqarishni o'rganayotganda boshqa sababni ko'rib chiqamiz.

Amalda paketning ketma-ket raqami paket sarlavhasidagi sobit o'lchamdagi maydonida joylashtiriladi. Agar k paket ketma-ket raqami maydonidagi bitlar soni bo'lsa, ketma-ket raqamlar diapazoni [0,2^k-1] bo'ladi. Cheklangan ketma-ket raqamlar diapazoni ishlatilgani uchun, ular bilan barcha arifmetik amallar 2^k moduli bo'yicha bajarilishi kerak, ya'ni ketma-ket raqamlar fazosini 2^k o'lchamdagi halqa ko'rinishida tasavvur qilish mumkin, bu erda 2^k-1 ketma-ket raqamidan keyin bevosita 0 ketma-ket raqami keladi. Eslatib o'tamiz, rdt3.0 protokolida bir bitli ketma-ket raqam va [0,1] ketma-ket raqamlar diapazoni ishlatiladi. Ushbu bobning oxiridagi bir nechta masalalarda cheklangan ketma-ket raqamlar ketma-ketligi ko'rib chiqiladi. 3.5-bo'limda ko'ramizki, TCP protokoli 32-bitli ketma-ket raqamlarni ishlatadi, ular ma'lumot paketlari emas, balki oqimdagi baytlar sonini hisoblash uchun qo'llaniladi.

[Algoritm diagrammasi - Jo'natuvchi uchun]

```
rdt_send(data)
if(nextseqnum<base+N){
    sndpkt[nextseqnum]=make_pkt(nextseqnum,data,checksum)
    udt_send(sndpkt[nextseqnum])
    if(base==nextseqnum)
        start_timer
    nextseqnum++
}
else
    refuse_data(data)
```

base=1, nextseqnum=1

Kutish holati

timeout:

```
start_timer
udt_send(sndpkt[base])
udt_send(sndpkt[base+1])
...
udt_send(sndpkt[nextseqnum-1])
```

rdt_rcv va tasdiqlash olish:


GBN protokoli jo'natuvchisi uch turdagi hodisalarni qo'llab-quvvatlashi kerak:

- **Yuqori daraja protokoli tomonidan chaqirish.** "Yuqoridan" rdt_send() funksiyasi chaqirilganda, uzatuvchi tomon avval oyna to'lganlik darajasini tekshiradi (ya'ni tasdiqlash kutilayotgan N ta jo'natilgan xabarning mavjudligi). Agar oyna to'lmagan bo'lsa, yangi paket hosil qilinadi va uzatiladi, o'zgaruvchilar qiymatlari yangilanadi. Aks holda uzatuvchi tomon ma'lumotlarni yuqori darajaga qaytaradi, bu oyna to'lganligi haqidagi yashirin ko'rsatmadir. Odatda yuqori daraja ma'lum vaqtdan keyin ma'lumotlarni uzatish uchun qayta urinish qiladi. Haqiqiy ilovada jo'natuvchi ma'lumotlarni buferga oladi (darhol jo'natish o'rniga) yoki sinxronlash mexanizmi (masalan, semafor yoki bayroq)ga ega bo'ladi, bu yuqori darajaga faqat oyna to'lmagan holatda rdt_send()ni chaqirish imkonini beradi.
- **Tasdiqlash olish.** Bizning GBN protokolimizda n ketma-ket raqamli paket uchun umumiy tasdiqlash beriladi, bu n dan oldingi barcha ketma-ket raqamli paketlar muvaffaqiyatli qabul qilinganligini ko'rsatadi. GBN protokolining qabul qiluvchi tomonini ko'rib chiqishda qo'l berib ko'rishuv mexanizmiga qaytamiz.
- **Kutish vaqti tugashi.** GBN protokoli o'z nomi bilan ma'lumotlar yo'qolishi va kechikishiga javob berish mexanizmiga majburdir. Tasdiqlashni kutish protokoli kabi, paket va tasdiqlashlar yo'qolishi va kechikishi faktlarini aniqlash uchun GBN protokoli taymerni ishlatadi. Agar kutish vaqti tugasa, uzatuvchi tomon barcha jo'natilgan tasdiqlanmagan paketlarni qayta jo'natadi.

Bizning misolimizda (3.19-rasmga qarang) uzatuvchi tomon tasdiqlanmagan eng "eski" paket jo'natilgan vaqtdan boshlab vaqtni hisoblaydigan yagona taymerni ishlatadi. Agar tasdiqlash olinsa, ammo uzatilgan tasdiqlanmagan paketlar mavjud bo'lsa, taymer qayta ishga tushiriladi. Tasdiqlanmagan paketlar yo'qligi taymerni to'xtatishga olib keladi.

GBN protokolida qabul qiluvchining harakatlari ham oddiy. Agar n ketma-ket raqamli paket to'g'ri va ketma-ket tartibda olinsa, ya'ni yuqori darajaga jo'natilgan oxirgi ma'lumotlar n-1 ketma-ket raqamli paketdan kelgan bo'lsa, qabul qiluvchi n-chi paket uchun ACK jo'natadi va paket ma'lumotlarining bir qismini yuqori darajaga uzatadi. Boshqa barcha hollarda qabul qiluvchi paketni qabul qilmaydi va to'g'ri tartibda olingan oxirgi paket uchun ACKni qayta jo'natadi. E'tibor bering, paketlar yuqori darajaga bittadan va tartibli ravishda uzatiladi, va agar k paketi olingan va yetkazilgan bo'lsa, k dan kichik ketma-ket raqamli barcha paketlar ham yetkazilgan. Ya'ni yig'ma tasdiqlashdan foydalanish GBN protokoli uchun eng mos variant.

Bizning GBN protokolimizda qabul qiluvchi noto'g'ri tartibda kelgan paketlarni rad etadi. Garchi to'g'ri yetkazilgan, ammo noto'g'ri tartibda kelgan paketlarni rad etish ahmoqona va isrof bo'lib tuyulishi mumkin bo'lsa ham, buning ma'lum asoslari bor. Eslatib o'tamiz, qabul qiluvchi ma'lumotlarni yuqori darajaga to'g'ri tartibda yetkazishi kerak. Endi n paketi kutilayotgan, ammo n+1 paketi kelgan deb faraz qilaylik. Ma'lumotlar tartibli uzatilishi kerakligi sababli, qabul qiluvchi n+1 paketini buferga jo'natishi va keyin n paketi olinishi va yetkazilgandan keyin yuqori darajaga yetkazishi mumkin. Biroq agar n paketi yo'qolgan bo'lsa, n va n+1 paketlari ham GBN protokoli jo'natuvchisining qayta uzatish qoidasi natijasida oxir-oqibat qayta uzatiladi. Ya'ni qabul qiluvchi shunchaki n+1 paketini rad etishi mumkin. Bunday yondashuvning afzalligi qabul qiluvchi tomonida oddiy buferlashda: noto'g'ri tartibda olingan har bir keyingi paket uchun buferdan foydalanish shart emas. Ya'ni jo'natuvchi oyna chegaralari va bu oyna ichidagi nextseqnum pozitsiyasini qo'llab-quvvatlashi kerak bo'lgan vaqtda, qabul qiluvchi qo'llab-quvvatlashi kerak bo'lgan yagona ma'lumot qismi - keyingi tartibli paketning ketma-ket raqami. Bu qiymat expectedseqnum o'zgaruvchisida saqlanadi, 3.21-rasmdagi qabul qiluvchi FSM sxemasida ko'rsatilganidek. Albatta, to'g'ri olingan paketlarni tashlab yuborishning kamchiligi shundaki, bu paketning keyingi uzatilishi yo'qolishi yoki buzilishi va yanada ko'proq qayta uzatishlar talab qilinishi mumkin.

3.22-rasmda oyna hajmi to'rtta paketga teng bo'lgan holatda GBN protokolining harakatlari tasvirlangan. Bunday oyna cheklovi tufayli jo'natuvchi 0 dan 3 gacha raqamli paketlarni jo'natadi, so'ngra ushbu paketlardan biri yoki bir nechtasi tan olinguncha kutadi. Birorta ijobiy ACK olinganida (masalan, ACK0 yoki ACK1), oyna oldinga siljiydi va jo'natuvchi yana bitta yangi paket jo'natishi mumkin (mos ravishda pkt4 va pkt5). Qabul qiluvchi tomonda 2-paket yo'qoladi va shuning uchun 3, 4, 5 paketlar noto'g'ri tartibda keladi va tashlab yuboriladi.

GBN protokolini o'rganishni yakunlab, uning stekda amalga oshirilishi 3.20-rasmdagi kengaytirilgan FSM sxemasiga strukturaviy jihatdan o'xshashligini ta'kidlaymiz. Amalga oshirish turli hodisalarga javob sifatida harakatlari sodir bo'lishi kerak bo'lgan turli protseduralr ko'rinishida ham bajariladi. Bunday hodisa boshqariladigan dasturlashda turli protseduralar protokol stekining boshqa protseduralari tomonidan yoki uzilish natijasida chaqiriladi. Jo'natuvchi uchun bunday hodisalar quyidagilar bo'lishi mumkin: (1) rdt_send()ni bajarish uchun yuqori daraja obyekti tomonidan chaqirish, (2) taymer uzilishi va (3) paketlar kelganda quyi darajadan rdt_rcv() chaqiruvi. Ushbu bobning oxiridagi dasturlash mashqlari sizga model, ammo real tarmoq sozlamalari uchun ushbu protseduralarni amalda amalga oshirish imkonini beradi.

Bu yerda ta'kidlaymizki, GBN protokoli 3.5-bo'limda ishonchli ma'lumot yetkazish tarkibiy qismlarini o'rganishda sanab o'tgan deyarli barcha usullarni o'z ichiga oladi. Ularga ketma-ketlik raqamlaridan foydalanish, yig'ma tasdiqlashlar, nazorat yig'indilari va vaqt tugashi hamda qayta uzatish operatsiyalari kiradi.

### Asosiy g'oyalar:

**1. Oyna tushunchasi (Sliding Window):**

- Jo'natuvchi bir vaqtning o'zida N ta paketgacha jo'natishi mumkin
- Bu "oyna" deb ataladigan diapazon bo'ylab harakat qiladi
- Masalan, N=4 bo'lsa, 0,1,2,3 paketlarni jo'natib, tasdiqlashni kutadi

**2. Ketma-ket raqamlash:**

- Har bir paketga raqam beriladi (0, 1, 2, 3, ...)
- Raqamlar aylanma tarzda takrorlanadi (0,1,2,3,0,1,2,3...)
- Bu paketlarni farqlash va tartibini saqlash uchun

**3. Tasdiqlash mexanizmi:**

- Qabul qiluvchi faqat ketma-ket kelgan paketlarni qabul qiladi
- Agar paket yo'qolsa yoki tartibsiz kelsa - rad etadi
- "Yig'ma tasdiqlash" ishlatiladi: ACK2 degani 0,1,2 paketlar olingan

**4. Xato bilan kurash:**

- Agar paket yo'qolsa yoki vaqt tugasa - barcha tasdiqlanmagan paketlar qayta jo'natiladi
- Shuning uchun "Go-Back-N" (N ga qaytish) deb ataladi

### Misol:

- Jo'natuvchi: pkt0, pkt1, pkt2, pkt3 jo'natadi
- pkt2 yo'qoladi
- pkt3 keladi, lekin rad etiladi (pkt2 kelmagan)
- Vaqt tugagach, pkt2, pkt3 qayta jo'natiladi

### Afzallik va kamchiliklar:

**Afzalliklar:**

- Oddiy amalga oshirish
- Qabul qiluvchida kam xotira kerak

**Kamchiliklar:**

- To'g'ri kelgan paketlar ham qayta jo'natiladi
- Tarmoq resurslarini ko'p ishlatadi

Bu protokol internetning poydevori bo'lib, TCP protokolining asosini tashkil qiladi.

**Selective Repeat (SR) - Tanlab takrorlash** protokoli, u GBN protokolining kamchiliklarini bartaraf etish uchun yaratilgan.

### Asosiy muammo va yechim

**GBN ning kamchiligi:**

- Agar bitta paket yo'qolsa, barcha keyingi paketlar ham qayta jo'natiladi
- Masalan: 1000 ta paket jo'natilgan, faqat bitta xato bo'lsa - barchasi qayta jo'natiladi
- Bu juda ko'p resurs isrof qiladi

**SR ning yechimi:**

- Faqat yo'qolgan yoki buzilgan paketlarni qayta jo'natish
- Har bir paket uchun alohida tasdiqlash
- To'g'ri kelgan paketlarni saqlash (hatto tartibsiz bo'lsa ham)

### SR protokolining ishlash prinsipi

**1. Jo'natuvchi tomoni:**

- Har bir paket uchun alohida taymer
- Faqat vaqti tugagan paketni qayta jo'natish
- Oyna (window) ichida tasdiqlangan va tasdiqlanmagan paketlar aralash bo'lishi mumkin

**2. Qabul qiluvchi tomoni:**

- Har bir to'g'ri paket uchun alohida ACK jo'natish
- Tartibsiz kelgan paketlarni bufferda saqlash
- Ketma-ket paketlar to'planganda yuqori darajaga berish

### Amaliy misol

```
Jo'natish ketma-ketligi: pkt0, pkt1, pkt2, pkt3, pkt4, pkt5
Holat: pkt2 yo'qoladi

GBN da: pkt2, pkt3, pkt4, pkt5 - barchasi qayta jo'natiladi
SR da: faqat pkt2 qayta jo'natiladi

Qabul qiluvchi:
- pkt0, pkt1 - darhol yuqoriga beradi
- pkt3, pkt4, pkt5 - bufferda saqlaydi
- pkt2 kelganda - barchasini birga yuqoriga beradi
```

### Oyna o'lchami cheklovi

**Muhim qoida:** SR protokolida oyna o'lchami ketma-ket raqamlar sonining yarmidan oshmasligi kerak.

**Sabab:** Agar oyna juda katta bo'lsa, qabul qiluvchi yangi paketni takroriy paketdan farqlay olmaydi.

Misol: Agar raqamlar 0,1,2,3,0,1,2,3... takrorlansa va oyna=3 bo'lsa:

- Birinchi 0-paket keldi va tasdiqlandi
- Keyingi 0-paket (yangi ma'lumot) kelganda - bu birinchisining takrori yoki yangimi?

### Afzallik va kamchiliklar

**SR ning afzalliklari:**

- Samarali tarmoq foydalanish
- Kam qayta uzatish
- Yuqori unumdorlik

**SR ning kamchiliklari:**

- Murakkab amalga oshirish
- Har ikkala tomonda ko'p xotira kerak
- Har bir paket uchun alohida taymer

### Texnik tafsilotlar

**Jo'natuvchi harakatlari:**

1. Ma'lumot kelganda - oyna ichida bo'lsa jo'natish
2. Vaqt tugaganda - faqat shu paketni qayta jo'natish
3. ACK kelganda - paketni belgilash va oynani siljitish

**Qabul qiluvchi harakatlari:**

1. To'g'ri paket kelsa - ACK jo'natish va bufferga saqlash
2. Ketma-ket bo'lsa - yuqori darajaga berish
3. Avvalgi paket kelsa - ACK ni takrorlash

### Xulosa

SR protokoli zamonaviy tarmoq protokollarining asosi bo'lib, TCP protokolida qo'llaniladi. U samaradorlik va ishonchlilik o'rtasida muvozanat o'rnatadi, ammo GBN ga nisbatan murakkabroq amalga oshiriladi. Bu protokol internetning yuqori tezlik va ishonchliligini ta'minlashda muhim rol o'ynaydi.
