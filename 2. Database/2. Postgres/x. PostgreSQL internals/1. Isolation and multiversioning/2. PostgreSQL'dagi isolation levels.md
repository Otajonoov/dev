
Vaqt o'tishi bilan tranzaksiyalarni boshqarishning blоkirovka protokollari o'rniga snapshot'lar asosidagi izolyatsiya protokoli (Snapshot Isolation, SI) keldi. 
Uning g'oyasi shundan iboratki, har bir tranzaksiya ma'lum vaqt momentidagi ma'lumotlarning consistency snapshot'i bilan ishlaydi. 
Snapshot'ga snapshot yaratilish momentigacha tasdiqlangan barcha dolzarb o'zgarishlar tushadi.

Snapshot'lar asosidagi izolyatsiya minimal blоkirovkalar bilan chiqib ketishga imkon beradi. Aslida faqat bir xil qatorning takroriy o'zgartirilishi blоkirovka qilinadi. 
Boshqa barcha operatsiyalar bir vaqtning o'zida bajarilishi mumkin: 
- yozuvchi tranzaksiyalar hech qachon o'quvchi tranzaksiyalarni blоkirovka qilmaydi,
- o'quvchilar esa umuman hech kimni hech qachon blоkirovka qilmaydi.

PostgreSQL'da SI protokolining ko'p versiyali varianti amalga oshirilgan. 
Ko'p versiyalilik shuni anglatadiki, DBMS'da bir vaqt momentida bir xil qatorning bir nechta versiyalari birga mavjud bo'lishi mumkin. 
Bu snapshot'ga mos versiyani kiritishga imkon beradi, eskirgan ma'lumotlarni o'qishga urinayotgan tranzaksiyalarni to'xtatish o'rniga.

Ma'lumotlar snapshot'laridan foydalanish hisobiga PostgreSQL'dagi izolyatsiya standart talab qilganidan farq qiladi va umuman olganda u qattiqroq. 
Dirty read ta'rif bo'yicha ruxsat etilmaydi. 
Rasman PostgreSQL'da Read Uncommitted darajasini ko'rsatish mumkin, lekin u xuddi Read Committed kabi ishlaydi, shuning uchun men bundan keyin bu daraja haqida umuman gapirmayman. 
Repeatable Read darajasi nafaqat non-repeatable read, balki phantom read ham ruxsat etilmaydi (garchi to'liq izolyatsiyani ta'minlamasa ham). 
To'g'ri, Read Committed darajasida bir qator hollarda o'zgarishlarni yo'qotish mumkin.

### Read Committed

**Dirty read yo'qligi.** Iflos ma'lumotlarni o'qib bo'lmasligiga ishonch hosil qilish oson. 
Tranzaksiya boshlaymiz. Standart bo'yicha u Read Committed izolyatsiya darajasidan foydalanadi:

```sql
=> BEGIN;

=> SHOW transaction_isolation;

 transaction_isolation
-----------------------
 read committed
(1 row)
```

Aniqroq aytganda, standart daraja parametr bilan beriladi, uni kerak bo'lganda o'zgartirish mumkin:

```sql
=> SHOW default_transaction_isolation;

 default_transaction_isolation
-------------------------------
 read committed
(1 row)
```

Xo'sh, ochilgan tranzaksiyada hisobdan mablag' yechib olamiz, lekin o'zgarishlarni tasdiqlamaymiz. Tranzaksiya o'z o'zgarishlarini doimo ko'radi:

````sql
=> UPDATE accounts SET amount = amount - 200 WHERE id = 1;

=> SELECT * FROM accounts WHERE client = 'alice';
```
```
 id | client | amount
----+--------+--------
  1 | alice  | 800.00
(1 row)
````

Ikkinchi seansda xuddi shu Read Committed darajasi bilan yana bir tranzaksiya boshlaymiz:

sql

````sql
=> BEGIN;

=> SELECT * FROM accounts WHERE client = 'alice';
```
```
 id | client | amount
----+--------+---------
  1 | alice  | 1000.00
(1 row)
````

Kutilganidek, boshqa tranzaksiya tasdiqlanmagan o'zgarishlarni ko'rmaydi — грязное чтение ruxsat etilmaydi.

**Неповторяющееся чтение.** Endi birinchi tranzaksiya o'zgarishlarni tasdiqlasa va ikkinchisi xuddi shu so'rovni takroriy bajarsa:

sql

```sql
=> COMMIT;
```

sql

````sql
=> SELECT * FROM accounts WHERE client = 'alice';
```
```
 id | client | amount
----+--------+--------
  1 | alice  | 800.00
(1 row)
````

sql

```sql

```

So'rov allaqachon yangi ma'lumotlarni oladi — bu Read Committed darajasida ruxsat etilgan неповторяющееся чтение anomaliyasidir.

Amaliy xulosa: tranzaksiyada oldingi operator o'qigan ma'lumotlar asosida qaror qabul qilib bo'lmaydi, chunki operatorlar bajarilishi orasidagi vaqtda hamma narsa o'zgarishi mumkin. Mana dastur kodida shunchalik tez-tez uchraydigan misolki, u klassik antipattern hisoblanadi:

sql

```sql
IF (SELECT amount FROM accounts WHERE id = 1) >= 1000 THEN
    UPDATE accounts SET amount = amount - 1000 WHERE id = 1;
END IF;
```

Tekshirish va yangilash o'rtasida o'tadigan vaqt davomida boshqa tranzaksiyalar hisobning holatini istalgancha o'zgartirishi mumkin, shuning uchun bunday "tekshirish" hech narsadan saqlamaydi. Bir tranzaksiyaning operatorlari orasida boshqa tranzaksiyalarning ixtiyoriy operatorlari "kirib kelishi" mumkinligini tasavvur qilish qulay, masalan shunday:

sql

```sql
IF (SELECT amount FROM accounts WHERE id = 1) >= 1000 THEN
    UPDATE accounts SET amount = amount - 200 WHERE id = 1;
    COMMIT;
    UPDATE accounts SET amount = amount - 1000 WHERE id = 1;
END IF;
```

Agar operatorlarni almashtirib, hamma narsani buzish mumkin bo'lsa, demak kod noto'g'ri yozilgan. O'zimizni bunday vaziyatlar to'qnashmasi bilan duch kelmasligimiz bilan aldamaslik kerak: agar baxtsizlik sodir bo'lishi mumkin bo'lsa, u albatta sodir bo'ladi. Bunday xatolarni takrorlash va shuning uchun tuzatish juda qiyin.

Kodni to'g'ri qanday yozish mumkin? Bir nechta imkoniyatlar bor:

- **Protsedura kodini deklarativ kod bilan almashtirish.**

Masalan, bu holda tekshiruvni yaxlitlik chekloviga aylantirish oson:

sql

```sql
ALTER TABLE accounts ADD CHECK amount >= 0;
```

Endi kodda hech qanday tekshiruvlar kerak emas: oddiy amaliyotni bajarish va kerak bo'lganda yaxlitlikni buzishga urinish holatida yuzaga keladi gan istisno qayta ishlash kifoya.

- **Bitta SQL operatoridan foydalanish.**

Согласованность bilan muammolar operatorlar orasida boshqa tranzaksiya tugashi va ko'rinadigan ma'lumotlar o'zgarishi tufayli yuzaga keladi. Agar operator bitta bo'lsa, unda hech qanday oraliqlar yo'q.

PostgreSQL'da bitta SQL operator bilan murakkab vazifalarni hal qilishga yetarli vositalar bor. Umumiy jadval ifodalari (CTE)ni qayd etaman, ularda shu jumladan INSERT, UPDATE, DELETE, MERGE v.15 operatorlarini, shuningdek atomik ravishda "kiritish, agar qator allaqachon mavjud bo'lsa, yangilash" mantiqini amalga oshiradigan INSERT ON CONFLICT operatorini ishlatish mumkin.

- **Foydalanuvchi blоkirovkalarini ishlatish.**

Oxirgi vosita — barcha kerakli qatorlarga qo'lda eksklyuziv blоkirovka o'rnatish (SELECT FOR UPDATE, sah. 258) yoki umuman butun jadvalga (LOCK TABLE, sah. 251). Bu har doim ishlaydi, lekin ko'p versiyalilikning afzalliklarini yo'qotadi: bir vaqtda bajarilish o'rniga operatsiyalarning bir qismi ketma-ket bajariladi.

**Несогласованное чтение.** Biroq hammasi unchalik oddiy emas. PostgreSQL'ning amalga oshirilishi standartda tartibga solinmagan boshqa, kamroq ma'lum anomaliyalarga yo'l qo'yadi.

Bu anomaliyadan Read Committed darajasida qolgan holda qanday qochish mumkin? Albatta, bitta operator ishlatish. Masalan shunday:

Shu paytgacha men ma'lumotlarning ko'rinishi faqat operatorlar orasida o'zgarishi mumkinligini ta'kidladim, lekin bu shunchalik ravshmi? Va agar so'rov uzoq vaqt bajarilsa, u ma'lumotlarning bir qismini bir holatda, bir qismini esa boshqa holatda ko'rishi mumkinmi?

Tekshiramiz. Buning uchun qulay usul — operatorga pg_sleep funktsiyasini chaqirish orqali sun'iy kechikishni kiritish. Birinchi qator darhol o'qiladi, ikkinchisi esa — ikki soniyadan keyin:

sql

```sql
=> SELECT amount, pg_sleep(2) -- 2 soniya
   FROM accounts WHERE client = 'bob';
```

Bu konstruksiya bajarilayotganda, boshqa tranzaksiyada mablag'ni qaytarib o'tkazamiz:

sql

````sql
=> BEGIN;
=> UPDATE accounts SET amount = amount + 100 WHERE id = 2;
=> UPDATE accounts SET amount = amount - 100 WHERE id = 3;
=> COMMIT;
```

Natija operator ma'lumotlarni uning bajarilishini boshlash momentida bo'lgan holatda ko'rishini ko'rsatadi, bu, shubhasiz, to'g'ri:
```

---------+----------

(2 rows)
````

Lekin bu yerda ham hammasi unchalik oddiy emas. Agar so'rovda o'zgaruvchan funktsiya (VOLATILE o'zgaruvchanlik toifasi bilan) chaqirilsa va bu funksiyada boshqa so'rov bajarilsa, bu ichki so'rov asosiy so'rov ma'lumotlari bilan согласованный bo'lmagan ma'lumotlarni ko'radi.

Funktsiyadan foydalanib Bob hisoblarining holatini tekshiramiz:

sql

```sql

AS $$
    SELECT amount FROM accounts a WHERE a.id = get_amount.id;
$$ VOLATILE LANGUAGE sql;

=> SELECT get_amount(id), pg_sleep(2)
   FROM accounts WHERE client = 'bob';
```

Va yana hisoblar orasida pulni o'tkazamiz, kechikishli so'rov bajarilayotgan paytda:

sql

````sql
=> BEGIN;
=> UPDATE accounts SET amount = amount + 100 WHERE id = 2;
=> UPDATE accounts SET amount = amount - 100 WHERE id = 3;
=> COMMIT;
```

Bu holda несогласованный ma'lumotlarni olamiz — 100 ₽ yo'qoldi:
```

------------+----------

(2 rows)
````

Ta'kidlaymanki, bunday effekt faqat Read Committed izolyatsiya darajasida va faqat VOLATILE o'zgaruvchanlik toifasi bilan mumkin. Muammo shundaki, standart bo'yicha aynan shu izolyatsiya darajasi va aynan shu o'zgaruvchanlik toifasi ishlatiladi, shuning uchun etiraf etish kerak — grabl lar juda qulay joylashgan.

Shunday qilib, Bob hisoblari shartga mos keladi va yangilash bajarilgandan keyin uning balansi 10 ₽ ga oshishi kerak.

Blоkirovka olib tashlangandan keyin UPDATE operatori yangilashga urinayotgan qatorni qayta o'qiydi (sah. 269) (lekin faqat uni bitta!). Natijada Bob'ga 900 ₽ dan kelib chiqib 9 ₽ hisoblanadi. Lekin agar Bob'da 900 ₽ bo'lganida, uning hisoblari umuman tanlovga tushmasligi kerak edi.

Shunday qilib, tranzaksiya noto'g'ri ma'lumotlarni o'qidi: qatorlarning bir qismi — bir vaqt momentida, bir qismi — boshqasida. Потерянное обновление o'rniga biz yana несогласованное чтение anomaliyasini olamiz.

### Repeatable Read

**Неповторяющееся va фантомное чтение yo'qligi.** Izolyatsiya darajasining nomi Repeatable Read¹ o'qishning takrorlanuvchanligi haqida gapiradi. Buni tekshiramiz va bir vaqtning o'zida fantomlar yo'qligiga ishonch hosil qilamiz. Buning uchun birinchi tranzaksiyada Bob hisoblarini oldingi holatga qaytaramiz va Charlie uchun yangi hisob yaratamiz:

¹ postgrespro.ru/docs/postgresql/17/transaction-iso#XACT-REPEATABLE-READ.

sql

````sql
=> BEGIN;

=> UPDATE accounts SET amount = 200.00 WHERE id = 2;
=> UPDATE accounts SET amount = 800.00 WHERE id = 3;

=> INSERT INTO accounts VALUES
    (4, 'charlie', 100.00);

=> SELECT * FROM accounts ORDER BY id;

----+----------+--------
````

````sql
(3 rows)
````

**Несогласованная запись.** Xo'sh, PostgreSQL'da Repeatable Read izolyatsiya darajasida standartda tasvirlangan barcha anomaliyalarning oldini olinadi. Lekin umuman barcha emas — fanga hali ularning nechta mavjudligi noma'lum. Biroq muhim fakt isbotlangan: snapshot'lar asosidagi izolyatsiya ularning jami qancha bo'lmasin, aynan ikkita anomaliyani mumkin qoldiradi.

Bu anomaliyalarning birinchisi — несогласованная запись (write skew).

Bob'ning balansini minusga olib ketishga muvaffaq bo'ldik, garchi alohida-alohida har bir tranzaksiya to'g'ri ishlagan bo'lsa ham.

**Фақat o'qувchi tranzaksiya anomaliyasi.** Faqat o'quvchi tranzaksiya anomaliyasi — Repeatable Read darajasida mumkin bo'lgan anomaliyalardan ikkinchisi va oxirgisi. Uni namoyish qilish uchun uchta tranzaksiya kerak bo'ladi, ularning ikkitasi ma'lumotlarni o'zgartiradi, uchinchisi esa — faqat o'qiydi.

Lekin avval Bob hisoblarining holatini tiklaymiz:

````sql

(2 rows)
````

sql

```sql
=> BEGIN ISOLATION LEVEL REPEATABLE READ; -- 1

=> UPDATE accounts SET amount = amount + (
       SELECT sum(amount) FROM accounts WHERE client = 'bob'
   ) * 0.01
   WHERE id = 2;
```

Keyin boshqa tranzaksiya Bob'ning boshqa hisobidan pulni yechib oladi va o'z o'zgarishlarini tasdiqlaydi:

sql

```sql
=> BEGIN ISOLATION LEVEL REPEATABLE READ; -- 2

=> UPDATE accounts SET amount = amount - 100.00 WHERE id = 3;

=> COMMIT;
```

Agar shu momentda birinchi tranzaksiya tasdiqlangan bo'lsa, hech qanday anomaliya yuzaga kelmaydi: biz avval birinchi tranzaksiya, keyin ikkinchisi bajarilgan deb hisoblashimiz mumkin edi (lekin aksincha emas, chunki birinchi tranzaksiya id = 3 hisobning holatini ikkinchi tranzaksiya tomonidan bu hisob o'zgartirilishidan oldin ko'rgan).

Lekin tasavvur qiling, shu momentda uchinchi (faqat o'quvchi) tranzaksiya Bob hisoblaridagi summalarni so'raydi. U ikkinchi tranzaksiyaning o'zgarishlarini (allaqachon tasdiqlangan) ko'rishi mumkin, lekin birinchisiniki (hali tasdiqlanmagan)ni emas:

````sql
=> SELECT * FROM accounts WHERE client = 'bob'; -- 3
```
```

----+--------+--------
  2 | bob    | 900.00
  3 | bob    |   0.00
(2 rows)
````

### Serializable

Serializable¹ darajasida barcha mumkin bo'lgan anomaliyalarning oldini olinadi. Aslida Serializable ma'lumotlar snapshot'lari asosidagi izolyatsiya ustiga qo'shimcha sifatida amalga oshirilgan. Repeatable Read'da yuzaga kelmaydigan anomaliyalar (masalan грязное, неповторяющееся, фантомное чтение) Serializable darajasida ham yuzaga kelmaydi. Yuzaga keladigan ikkita anomaliya (несогласованная запись va faqat o'quvchi tranzaksiya anomaliyasi) esa maxsus ravishda aniqlanadi va bu holda tranzaksiya to'xtatiladi: bizga tanish serializatsiya xatosi yuzaga keladi.

¹ postgrespro.ru/docs/postgresql/17/transaction-iso#XACT-SERIALIZABLE.

Shunday qilib, Serializable izolyatsiya darajasidan foydalanuvchi dastur serializatsiya xatosi bilan tugagan tranzaksiyalarni takrorlashi kerak. (Xuddi shunday Repeatable Read darajasida ham qilish kerak, agar faqat o'quvchi tranzaksiyalar bilan cheklanmasa.)

Serializable darajasi dasturlashning soddaligini beradi, lekin uning narxi — mumkin bo'lgan anomaliyalarni aniqlash va tranzaksiyalarning ma'lum bir qismini to'xtatishga sarflanadigan qo'shimcha xarajatlar. Qo'shimcha xarajatlarni kamaytirish faqat o'quvchi tranzaksiyalarni aniq READ ONLY sifatida belgilash orqali mumkin. Lekin asosiy savol, albatta, to'xtatilgan tranzaksiyalarning ulushi qanchalik katta, chunki ularni takroriy bajarish kerak bo'ladi. Agar faqat ma'lumotlarga ko'ra haqiqatan ham mos kelmaydigan kesishuvchi tranzaksiyalar to'xtatilganida edi, hamma narsa yaxshi bo'lar edi. Lekin bunday amalga oshirish muqarrar ravishda juda resurs sig'imli bo'lib chiqadi, chunki har bir qator bilan operatsiyalarni kuzatish kerak bo'ladi.

Haqiqatda amalga oshirish yolg'on ijobiy ishga tushishga yo'l qo'yadi (sah. 288): ba'zi mutlaqo normal tranzaksiyalar ham to'xtatiladi, ularga "shunchaki omad yetmaydi". Omad ko'plab sabablarga bog'liq, masalan mos indekslarning mavjudligi yoki mavjud operativ xotira hajmiga va xatti-harakatni oldindan bashorat qilish qiyin.

Agar Serializable izolyatsiya darajasi ishlatilsa, u dasturning barcha tranzaksiyalari uchun qo'llanilishi kerak. Turli izolyatsiya darajalarining tranzaksiyalarini aralashtirish Serializable darajasi (hech qanday ogohlantirmasdan) Repeatable Read kabi harakat qilishiga olib keladi. Shuning uchun Serializable ishlatilganda default_transaction_isolation parametri qiymatini o'zgartirish ma'noli, garchi, albatta, bu noto'g'ri darajani aniq ko'rsatishga to'sqinlik qilmasa ham.

Amalga oshirishning boshqa cheklovlari ham bor, v.12 masalan Serializable darajasidagi so'rovlar replikalarda ishlamaydi. Va garchi funksionallikni yaxshilash ustida ish to'xtamaydigan bo'lsa ham, mavjud cheklovlar va qo'shimcha xarajatlar bunday izolyatsiya darajasining jozibadorligini kamaytiradi.

## 2.4. Qaysi izolyatsiya darajasidan foydalanish kerak?

Read Committed izolyatsiya darajasi PostgreSQL'da standart ishlatiladi va ehtimol aynan shu daraja dasturlarning mutlaq ko'pchiligida qo'llaniladi. U tranzaksiyani to'xtatish faqat nosozlik holatida mumkin, lekin несогласованность'ning oldini olish uchun to'xtatish qo'llanilmasligida qulay. Boshqacha qilib aytganda, serializatsiya xatosi yuzaga kelishi mumkin emas va tranzaksiyalarni takrorlash haqida g'amxo'rlik qilishga hojat yo'q.

Tanganing teskari tomoni — yuqorida batafsil ko'rib chiqilgan mumkin bo'lgan anomaliyalarning katta soni. Dasturchi doimiy ravishda ularni yodda tutishga majbur va ularning paydo bo'lishiga yo'l qo'ymaslik uchun kodni shunday yozishga majbur. Agar kerakli harakatlarni bitta SQL operatorida shakllantirish mumkin bo'lmasa, aniq blоkirovka o'rnatishga murojaat qilishga to'g'ri keladi. Eng yoqimsiz narsa — kodni несогласованный ma'lumotlar olish bilan bog'liq xatolar mavjudligi uchun test qilish qiyin va xatolarning o'zi bashorat qilib bo'lmaydigan va takrorlanmaydigan tarzda yuzaga kelishi mumkin va shuning uchun tuzatishda murakkab.

Repeatable Read izolyatsiya darajasi несогласованность'ning bir qismi muammolarini bartaraf etadi, lekin, afsuski, hammasini emas. Shuning uchun qolgan anomaliyalar haqida esda tutish ham kerak, dasturni serializatsiya xatolarini to'g'ri qayta ishlashi uchun o'zgartirish ham kerak. Bu, albatta, noqulay. Lekin faqat o'quvchi tranzaksiyalar uchun bu daraja Read Committed'ni ajoyib tarzda to'ldiradi va masalan, bir nechta SQL so'rovlardan foydalanadigan hisobotlarni qurish uchun foydali.

Nihoyat, Serializable darajasi несогласованность haqida umuman g'amxo'rlik qilmaslikka imkon beradi, bu kodni yozishni sezilarli darajada soddalashtiradi. Dasturdan talab qilinadigan yagona narsa — serializatsiya xatosi olganda har qanday tranzaksiyani takrorlash qobiliyati. Lekin to'xtatilgan tranzaksiyalar ulushi va qo'shimcha qo'shimcha xarajatlar throughput'ni sezilarli darajada kamaytirishi mumkin. Shuningdek, Serializable darajasi replikalarda qo'llanilmasligini va uni boshqa izolyatsiya darajalari bilan aralashtirib bo'lmasligini hisobga olish kerak.

# Xulosa

Bu PostgreSQL'dagi izolyatsiya mexanizmlarining qanday ishlashini va turli darajalarning qanday kafolatlar berishini chuqur tushuntiruvchi fundamental bobdir. Asosiy xulosalar:

**1. Согласованность — murakkab tushuncha:** DBMS yaxlitlikni (integrity) ta'minlashi mumkin, lekin to'liq согласованность (consistency) uchun dastur mas'ul. Bu juda muhim farq.

**2. Anomaliyalar — haqiqiy muammo:** Standart faqat bir nechta anomaliyalarni tavsiflaydi, lekin haqiqatda ko'proq mavjud. PostgreSQL'ning o'z amalga oshirilishi ba'zi hollarda standartdan farq qiladi.

**3. Read Committed — eng keng qo'llaniladigan, lekin xavfli:**

- Грязное чтение yo'q, lekin неповторяющееся чтение va несогласованное чтение mumkin
- Bitta SQL operator strategiyasi yoki explicit blоkirovkalar kerak
- Kod yozish va test qilish qiyin

**4. Repeatable Read — oraliq yechim:**

- Snapshot asosida ishlaydi
- Ko'proq kafolatlar beradi, lekin serializatsiya xatolari bilan ishlash kerak
- Read-only tranzaksiyalar uchun ideal

**5. Serializable — eng qat'iy, lekin qimmat:**

- Barcha anomaliyalarni oldini oladi
- Dasturlashni soddalashtiradi, lekin xarajatli
- Yolg'on ijobiy natijalar muammo bo'lishi mumkin
- Replikalarda ishlamaydi

**6. Amaliy maslahatlar:**

- Dekl

arativ constraint'lardan foydalaning

- Bitta SQL operator bilan ishlashga harakat qiling
- Volatile funktsiyalardan ehtiyot bo'ling Read Committed'da
- Transaction retry mexanizmini Repeatable Read va Serializable uchun amalga oshiring

Bu bob backend dasturchi sifatida database bilan to'g'ri ishlash uchun juda muhim. Izolyatsiya darajalarini tushunish production'da ma'lumotlar yo'qotish va data corruption'dan saqlanish uchun kalit hisoblanadi.
