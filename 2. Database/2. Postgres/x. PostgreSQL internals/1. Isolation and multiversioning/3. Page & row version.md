
Har bir page ichki markup'ga ega va, qoida tariqasida, quyidagi bo'limlarni o'z ichiga oladi:

- header
- pointer massivi - row version'larga
- bo'sh joy (free space);
- row versions
- special area

---
### Page header'i

Page header'i kichik address'larda joylashgan va fixed size'ga ega. U page haqida turli ma'lumotlarni saqlaydi, masalan checksum, shuningdek barcha boshqa area'larning size'lari.

Size'larni `pageinspect` extension yordamida osongina olish mumkin. 
Table'ning eng birinchi page'iga qarab chiqamiz (numeratsiya noldan boshlanadi):

```sql
=> CREATE EXTENSION pageinspect;

=> SELECT lower, upper, special, pagesize
   FROM page_header(get_raw_page('accounts',0));

 lower | upper | special | pagesize
-------+-------+---------+----------
   152 |  6904 |    8192 |     8192
(1 row)
```

### Special area

Special area page'ning qarama-qarshi tomonida, katta address'larda joylashgan. U ba'zi index type'lari tomonidan yordamchi ma'lumotlarni saqlash uchun ishlatiladi. Qolgan hollarda, jumladan table page'larida, bu area nol size'ga ega.

Umuman olganda, index page'lari ancha xilma-xil tuzilgan va ularning tarkibi aniq index type'iga bog'liq. Hatto bitta index type'ida ham turli xil page'lar bo'ladi: masalan, B-tree'da maxsus strukturali nol metadata page'i va table page'laridek tashkil etilgan "oddiy" page'lar mavjud.

### Row version'lar

Special area oldida row'lar joylashgan — database'da saqlanadigan ma'lumotlarning o'zi, ba'zi xizmat (service) ma'lumotlari qo'shilgan holda.

Table'lar holatida biz shunchaki row'lar haqida emas, balki row version'lar (row versions, tuples) haqida gapiramiz, chunki multiversion'lilik bir xil row'ning bir nechta version'lari mavjud bo'lishini nazarda tutadi. Index'larga multiversion'lilik tarqalmaydi; buning o'rniga index'lar barcha mumkin bo'lgan table row version'lariga havola qiladi, ular orasidan visibility qoidalari bo'yicha mos keladiganlari tanlanadi.

> **Izoh:** _Tuple_ atamasi relyatsion nazariyadan olingan. Bu PostgreSQL'ning akademik o'tmishidan qolgan yana bir meros. Nazariyani DBMS tuzilishi bilan aralashtirib yubormaslik uchun, men _row version_ (ba'zan, agar noaniqlik tug'dirmasa, qisqaroq _row_ so'zi bilan almashtirib) tarjimasini ishlataman.

### Row version'larga pointer'lar

Row version'larga pointer'lar massivi page'ning mundarijasi vazifasini bajaradi. U header'dan keyin joylashgan.

Index row'lari qandaydir tarzda table'dagi row version'larga havola qilishi kerak. Buning uchun olti byte'lik row version identifier'lari (tuple id, tid) ishlatiladi. Identifier page fayl'ining asosiy layer'idagi page raqamidan iborat va bu page'dagi row version'ga qandaydir ko'rsatma ham o'z ichiga olishi kerak.

Bunday ko'rsatma sifatida page boshiga nisbatan offset ishlatilishi mumkin edi. Lekin shunda row version'ni page ichida ko'chirish mumkin bo'lmas edi, chunki index'lardan kelayotgan havolalar buzilardi. Bu esa page ichida bo'sh joyning fragment'lanishiga va boshqa yoqimsiz oqibatlarga olib kelardi.

Shuning uchun bilvosita addressing (indirect addressing) ishlatiladi: version identifier'i pointer raqamiga havola qiladi, pointer esa row version'ning page'dagi joriy pozitsiyasiga. Row version ko'chirilganda uning identifier'i o'zgarmaydi; faqat xuddi shu page'da joylashgan pointer'ni o'zgartirish kifoya.

Har bir pointer aynan to'rt byte egallaydi va quyidagilarni o'z ichiga oladi:

- row version'ning page boshiga nisbatan offset'i;
- row version'ning uzunligi (length);
- row version'ning status'ini belgilaydigan bir nechta bit'lar.

### Bo'sh joy (Free space)

Pointer'lar va row version'lar orasida bo'sh joy qolishi mumkin (bu free space map'da belgilangan). Page ichida hech qanday fragmentation bo'lmaydi, barcha bo'sh joy har doim bitta qism sifatida ifodalanadi.

## 3.2. Row version'lar strukturasi

Row version header'dan iborat, undan keyin asl data keladi.

Version header'i ko'plab field'larni o'z ichiga oladi, ular orasida:

**xmin, xmax** — transaction raqamlari, ular berilgan version'ni xuddi shu row'ning boshqa version'laridan farqlaydi;

**infomask** — version xususiyatlarini belgilaydigan bir qator informatsion bit'lar;

**ctid** — xuddi shu row'ning keyingi, yangi version'iga havola;

**null value'larning bitmap'i** — NULL qiymatlarni qabul qiluvchi column'larni belgilaydigan bit'lar massivi.

Natijada header ancha katta bo'ladi — har bir row version uchun minimum 23 byte, odatda esa null value bitmap'i va data boshlanishining majburiy alignment'i tufayli bundan ko'proq. "Tor" (narrow) table uchun xizmat ma'lumotlari hajmi foydali ma'lumotlar hajmidan ortib ketishi mumkin.

Disk'dagi data formati operativ xotiradagi data tasviri (representation) bilan to'liq mos keladi. Page row version'lar bilan birga buffer cache'ga "boricha", hech qanday o'zgartirishlarsiz o'qiladi. Shuning uchun bir platformadan olingan data file'lar boshqa platformalar bilan mos kelmaydi.

Mos kelmaslikning sabablaridan biri — byte tartib'i (byte order). Masalan, x86 arxitekturasida kichik razryadlardan kattaga qarab tartib (little-endian) qabul qilingan, z/Architecture teskari tartibni (big-endian) ishlatadi, ARM'da esa tartib almashtirilishi mumkin.

Mos kelmaslikka shuningdek ko'plab arxitekturalar talab qiladigan mashina so'zlari chegaralari bo'yicha data alignment ham sabab bo'ladi. Masalan, 32-bit x86 arxitektura tizimida butun sonlar (integer type, to'rt byte egallaydi) to'rt byte'li so'zlar chegarasiga tekislangan bo'ladi, xuddi ikkilangan aniqlikdagi suzuvchi nuqtali sonlar (double precision type, sakkiz byte) kabi. 64-bit tizimda esa double qiymatlar sakkiz byte'li so'zlar chegarasiga tekislanadi.

Alignment tufayli table row'ning size'i field'larning joylashish tartibiga bog'liq. Odatda bu ta'sir unchalik sezilmaydi, lekin ba'zi hollarda size'ning sezilarli oshishiga olib kelishi mumkin. Mana misol:

sql

```sql
=> CREATE TABLE padding(
     b1 boolean,
     i1 integer,
     b2 boolean,
     i2 integer
   );

=> INSERT INTO padding VALUES (true,1,false,2);

=> SELECT lp_len FROM heap_page_items(get_raw_page('padding', 0));

 lp_len
--------
     40
(1 row)
```

Men `pageinspect` extension'ining `heap_page_items` funksiyasini ishlatdim, u pointer'lar va row version'lar haqida ma'lumot olish imkonini beradi.

> **Izoh:** PostgreSQL'da **heap** (uyum) so'zi bilan table'lar belgilanadi. Bu atamaning yana bir unchalik aniq bo'lmagan qo'llanishi bo'lib, row version'lar uchun joy ajratish mexanizmining operativ xotirani dinamik taqsimlash bilan o'xshashligiga ishora qiladi. Albatta, ba'zi o'xshashlikni topish mumkin, lekin table'ni boshqarish uchun butunlay boshqa algoritmlar ishlatiladi. Bu so'z "hammasi uyumga tashlab qo'yilgan" ma'nosida ishlatiladi deb hisoblash mumkin, bu tartiblangan index'lardan farqni ta'kidlaydi.

Row 40 byte egallaydi. Ulardan 24 byte header'ga ketadi, integer type'dagi column'lar har biri 4 byte, boolean — har biri 1 byte egallaydi. Jami 34 ga teng, 6 byte esa integer'ni to'rt byte'li so'zlar chegarasiga alignment qilish tufayli isrof bo'ladi.

Table'ni qayta qurib, joyni yanada samarali ishlatish mumkin:

sql

```sql
=> DROP TABLE padding;

=> CREATE TABLE padding(
     i1 integer,
     i2 integer,
     b1 boolean,
     b2 boolean
   );

=> INSERT INTO padding VALUES (1,2,true,false);

=> SELECT lp_len FROM heap_page_items(get_raw_page('padding', 0));

 lp_len
--------
     34
(1 row)
```

Yana bir mumkin bo'lgan mikrooptimizatsiya — table boshiga barcha fixed size'dagi, null qiymatlarni qabul qilmaydigan column'larni ko'chirish. Bunday field'larga murojaat row version boshidan field offset'ini cache'lash imkoniyati tufayli yanada samarali bo'ladi.

## 3.3. Row version'lar ustida amallar bajarish

Bir xil row'ning turli version'larini farqlash uchun har bir version ikkita belgi (mark) ga ega bo'lib, ular uning "amal qilish muddatini" belgilaydi — xmin va xmax. Lekin vaqtning o'zi emas, balki doimiy oshib boradigan transaction raqamlari hisoblagichi ishlatiladi.

Row yaratilganda, xmin qiymati INSERT buyrug'ini bajargan transaction raqamiga tenglashtiriladi.

Row o'chirilganda, joriy version'ning xmax qiymatiga DELETE buyrug'ini bajargan transaction raqami yoziladi.

UPDATE buyrug'ini ma'lum darajada ikkita amal sifatida ko'rish mumkin: DELETE va INSERT. Joriy row version'da xmax qiymati UPDATE ni bajargan transaction raqamiga tenglashtiriladi. So'ng xuddi shu row'ning yangi version'i yaratiladi; undagi xmin qiymati oldingi version'ning xmax qiymati bilan mos keladi. Xuddi shunday MERGE buyrug'i ham elementar insert va delete'larga "parchalanadi".

Keyingi qismda men row version'lar bilan turli amallar qanday past darajada (low level) bajarilishini batafsil ko'rsataman.

Tajribalar uchun ikkita column'li table va ulardan biri bo'yicha index kerak bo'ladi:

sql

```sql
=> CREATE TABLE t(
     id integer GENERATED ALWAYS AS IDENTITY,
     s text
   );

=> CREATE INDEX ON t(s);
```

### Insert

Bitta row insert qilamiz, oldin transaction boshlagan holda:

sql

```sql
=> BEGIN;

=> INSERT INTO t(s) VALUES ('FOO');
```

Mana bizning joriy transaction raqamimiz:

sql

```sql
=> -- txid_current() v.13 gacha
   SELECT pg_current_xact_id();

 pg_current_xact_id
--------------------
                781
(1 row)
```

> **Izoh:** PostgreSQL'da transaction tushunchasi uchun ko'pincha **xact** qisqartmasi ishlatiladi, uni funksiya nomlarida ham, kodda ham topish mumkin. Mos ravishda, transaction raqami xact id, txid yoki hatto shunchaki xid deb belgilanadi. Bu nomlar bizga yana uchraydi.

Page tarkibiga qarab chiqamiz. `heap_page_items` funksiyasi bizga barcha kerakli ma'lumotni beradi, lekin data'ni "boricha", idrok qilish uchun murakkab formatda ko'rsatadi:

sql

```sql
=> SELECT *
   FROM heap_page_items(get_raw_page('t',0))
   \gx

-[ RECORD 1 ]-------------------
lp          | 1
lp_off      | 8160
lp_flags    | 1
lp_len      | 32
t_xmin      | 781
t_xmax      | 0
t_field3    | 0
t_ctid      | (0,1)
t_infomask2 | 2
t_infomask  | 2050
t_hoff      | 24
t_bits      |
t_oid       |
t_data      | \x0100000009464f4f
```

Tushunish uchun ma'lumotning faqat bir qismini qoldiramiz va ba'zi column'larni shifrlaymiz:

sql

```sql
=> SELECT '(0,'||lp||')' AS ctid,
     CASE lp_flags
       WHEN 0 THEN 'unused'
       WHEN 1 THEN 'normal'
       WHEN 2 THEN 'redirect to '||lp_off
       WHEN 3 THEN 'dead'
     END AS state,
     t_xmin as xmin,
     t_xmax as xmax,
     (t_infomask & 256) > 0  AS xmin_committed,
     (t_infomask & 512) > 0  AS xmin_aborted,
     (t_infomask & 1024) > 0 AS xmax_committed,
     (t_infomask & 2048) > 0 AS xmax_aborted
   FROM heap_page_items(get_raw_page('t',0))
   \gx

-[ RECORD 1 ]--+-------
ctid           | (0,1)
state          | normal
xmin           | 781
xmax           | 0
xmin_committed | f
xmin_aborted   | f
xmax_committed | f
xmax_aborted   | t
```

Mana bu yerda nima qilindi:

- `lp` pointer'i row version identifier'ining oddiy ko'rinishiga keltirildi: (page raqami, pointer raqami).
- `lp_flags` pointer holati shifrlab chiqildi. Bu yerda u `normal` qiymatga ega — bu pointer haqiqatan ham row version'ga havola qilayotganini bildiradi.
- Barcha informatsion bit'lardan hozircha faqat ikkita juftlik ajratildi. `xmin_committed` va `xmin_aborted` bit'lari xmin raqamli transaction commit qilinganmi yoki abort qilinganmi ko'rsatadi. xmax transaction haqida shunga o'xshash ma'lumotni `xmax_committed` va `xmax_aborted` bit'lari beradi.

> **Izoh:** `pageinspect` extension'ida barcha informatsion bit'larni shifrlash uchun `heap_tuple_infomask_flags` funksiyasi bor, lekin men faqat shu paytda kerak bo'lganlarini va yanada ixcham ko'rinishda chiqaraman.

Shunday qilib, insert paytida table page'ida 1-raqamli pointer paydo bo'ldi, u birinchi va hozircha yagona row version'ga havola qiladi.

Row version'dagi xmin field'i joriy transaction raqami bilan to'ldirildi. Transaction hali faol (active), shuning uchun `xmin_committed` va `xmin_aborted` bit'lari o'rnatilmagan.

xmax field'i soxta 0 raqami bilan to'ldirildi, chunki berilgan row version o'chirilmagan va aktual hisoblanadi. Transaction'lar bu raqamga e'tibor bermaydi, chunki `xmax_aborted` bit'i o'rnatilgan.

> **Izoh:** Abort qilingan transaction bit'i umuman mavjud bo'lmagan transaction uchun o'rnatilganligi g'alati tuyulishi mumkin. Lekin isolation nuqtai nazaridan bu bir xil narsa: bekor qilingan transaction iz qoldirmaydi va, demak, mavjud bo'lmagan.

Bunday query bizga yana ko'p marta kerak bo'ladi, shuning uchun men uni funksiyaga o'rayman. Shu bilan birga chiqish ixchamligi uchun informatsion bit'lar column'larini olib tashlab, transaction status'ini uning raqamiga qo'shaman:

sql

```sql
=> CREATE FUNCTION heap_page(relname text, pageno integer)
   RETURNS TABLE(ctid tid, state text, xmin text, xmax text)
   AS $$
   SELECT (pageno,lp)::text::tid AS ctid,
     CASE lp_flags
       WHEN 0 THEN 'unused'
       WHEN 1 THEN 'normal'
       WHEN 2 THEN 'redirect to '||lp_off
       WHEN 3 THEN 'dead'
     END AS state,
     t_xmin || CASE
       WHEN (t_infomask & 256) > 0 THEN ' c'
       WHEN (t_infomask & 512) > 0 THEN ' a'
       ELSE ''
     END AS xmin,
     t_xmax || CASE
       WHEN (t_infomask & 1024) > 0 THEN ' c'
       WHEN (t_infomask & 2048) > 0 THEN ' a'
       ELSE ''
     END AS xmax
   FROM heap_page_items(get_raw_page(relname,pageno))
   ORDER BY lp;
   $$ LANGUAGE sql;
```

Bu ko'rinishda version header'ida nima bo'layotgani ancha tushunarli:

sql

```sql
=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin | xmax
-------+--------+------+------
 (0,1) | normal | 781  | 0 a
(1 row)
```

Shunga o'xshash, lekin ancha kam detalli ma'lumotni table'ning o'zidan, xmin va xmax pseudo-column'lar yordamida olish mumkin:

sql

```sql
=> SELECT xmin, xmax, * FROM t;

 xmin | xmax | id |  s
------+------+----+-----
  781 |    0 |  1 | FOO
(1 row)
```

### Commit (Fiksatsiya)

Transaction muvaffaqiyatli yakunlanganda uning status'ini eslab qolish kerak — commit qilinganini belgilash lozim. Buning uchun **clog** (commit log) deb ataladigan struktura ishlatiladi. Bu system catalog table'i emas, balki PGDATA/pg_xact katalogidagi maxsus file'lar.

clog bir nechta file'larga faqat qulaylik uchun bo'lingan. Bu file'lar bilan ishlash page'lar bo'yicha, server'ning umumiy (shared) memory'sidagi buffer'lar orqali olib boriladi. clog cache'ining size'i `transaction_buffers` parametri bilan sozlanadi, default qiymati esa buffer cache size'ining 1/512 qismini tashkil qiladi (v. 17).

clog'da, row version header'ida bo'lgani kabi, har bir transaction uchun ikkita bit ajratilgan: committed va aborted.

Shunday qilib, transaction commit qilinganda clog'da berilgan transaction uchun committed bit'i o'rnatiladi. Boshqa biror transaction bizning table page'imizga murojaat qilganda, u savolga javob berishi kerak bo'ladi: xmin raqamli transaction yakunlanganmi?

- **Agar yo'q** bo'lsa, yaratilgan row version ko'rinmasligi kerak. Transaction bajarilayotganligini tekshirish uchun yana bitta struktura ko'rib chiqiladi — u instance'ning shared memory'sida joylashgan va **ProcArray** deb ataladi. Unda barcha faol (active) process'larning ro'yxati bor va har biri uchun uning joriy (active) transaction raqami ko'rsatilgan.
- **Agar yakunlangan bo'lsa**, commit bilan yoki abort bilan? Abort bo'lsa, row version ham ko'rinmasligi kerak. Aynan shu tekshirish uchun clog strukturasi kerak. Garchi clog'ning oxirgi page'lari operativ xotiradagi buffer'larda saqlanayotgan bo'lsa ham, bunday tekshirishni har safar bajarish qimmatga tushadi. Shuning uchun bir marta aniqlangan transaction status'i row version header'iga informatsion bit'lar — `xmin_committed` va `xmin_aborted` — ga yoziladi; ularni **hint bit'lar** deb ham atashadi. Agar bu bit'lardan biri o'rnatilgan bo'lsa, xmin transaction'ining holati ma'lum hisoblanadi va keyingi transaction'ga clog'ga ham, ProcArray'ga ham murojaat qilish kerak bo'lmaydi.

Nima uchun bu bit'lar row'ni insert qilayotgan transaction tomonidan o'rnatilmaydi? Gap shundaki, bu vaqtda transaction hali muvaffaqiyatli yakunlanishi yoki yakunlanmasligini bilmaydi. Commit paytida esa qaysi page'lardagi qaysi row'lar o'zgartirilganligini tushunib bo'lmaydi. Bunday page'lar ko'p bo'lishi mumkin va ularni eslab qolish foydasiz. Bundan tashqari, page'larning bir qismi buffer cache'dan disk'ga siqib chiqarilgan bo'lishi mumkin; bit'larni o'zgartirish uchun ularni qayta o'qish commit'ni sezilarli sekinlashtirgan bo'lardi.

Tejashning teskari tomoni shundan iboratki, har qanday transaction (hatto oddiy o'qish — SELECT bajarayotgan ham) data page'larida informatsion bit'larni o'rnatishni boshlashi mumkin. Albatta, buffer cache'dagi page'lar bu holda dirty bo'lib qoladi.

Nihoyat, biz transaction boshlagan row insert'ni commit qilamiz:

sql

```sql
=> COMMIT;
```

Page'da hech narsa o'zgarmadi (lekin biz bilamizki, transaction status'i allaqachon clog'ga yozilgan):

sql

```sql
=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin | xmax
-------+--------+------+------
 (0,1) | normal | 781  | 0 a
(1 row)
```

Endi page'ga birinchi bo'lib murojaat qilgan transaction ("oddiy" tarzda, `pageinspect` yordamida emas) xmin transaction'ining status'ini aniqlashi va uni informatsion bit'larga yozishi kerak bo'ladi:

sql

```sql
=> SELECT * FROM t;

 id |  s
----+-----
  1 | FOO
(1 row)

=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+------
 (0,1) | normal | 781 c | 0 a
(1 row)
```

### Delete (O'chirish)

Row o'chirilganda, aktual version'ning xmax field'iga o'chiruvchi transaction raqami yoziladi va `xmax_aborted` bit'i qayta o'rnatiladi (reset).

> **Izoh:** Faol (active) transaction'ga mos keladigan xmax qiymati row lock vazifasini ham bajaradi. Agar boshqa transaction bu row'ni update yoki delete qilmoqchi bo'lsa, u xmax transaction'ining yakunlanishini kutishga majbur bo'ladi.

Row'ni o'chiramiz:

sql

```sql
=> BEGIN;

=> DELETE FROM t;

=> SELECT pg_current_xact_id();

 pg_current_xact_id
--------------------
                782
(1 row)
```

Transaction raqami xmax field'iga yozildi, lekin informatsion bit'lar hali o'rnatilmagan:

sql

```sql
=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+------
 (0,1) | normal | 781 c | 782
(1 row)
```

### Abort (Bekor qilish)

O'zgarishlarni abort qilish commit'ga o'xshash ishlaydi va xuddi shunday tez bajariladi, faqat clog'da committed bit'i o'rniga aborted bit'i o'rnatiladi. Buyruq ROLLBACK deb atalsa ham, o'zgarishlarning rollback'i bo'lmaydi: transaction data page'larida nimani o'zgartirishga ulgurgan bo'lsa, hammasi joyida qoladi.

sql

```sql
=> ROLLBACK;

=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+------
 (0,1) | normal | 781 c | 782
(1 row)
```

Page'ga murojaat qilganda status tekshiriladi va row version'da `xmax_aborted` hint bit'i o'rnatiladi. xmax raqamining o'zi page'da qoladi, lekin unga endi hech kim qaramaydi:

sql

```sql
=> SELECT * FROM t;

 id |  s
----+-----
  1 | FOO
(1 row)

=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+-------
 (0,1) | normal | 781 c | 782 a
(1 row)
```

### Update (Yangilash)

Update shunday ishlaydi, go'yo avval joriy row version'ning delete'i, keyin yangi version'ning insert'i bajarilar edi:

sql

```sql
=> BEGIN;

=> UPDATE t SET s = 'BAR';

=> SELECT pg_current_xact_id();

 pg_current_xact_id
--------------------
                783
(1 row)
```

Query bitta row'ni (yangi version'ni) qaytaradi:

sql

```sql
=> SELECT * FROM t;

 id |  s
----+-----
  1 | BAR
(1 row)
```

Lekin page'da ikkala version ham saqlanadi:

sql

```sql
=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+------
 (0,1) | normal | 781 c | 783
 (0,2) | normal | 783   | 0 a
(2 rows)
```

Oldin o'chirilgan version xmax field'ida joriy transaction raqami bilan belgilandi. Bu qiymat eski qiymatning ustiga yozildi, chunki avvalgi transaction abort qilingan edi. `xmax_aborted` bit'i esa qayta o'rnatildi (reset), chunki joriy transaction'ning status'i hali noma'lum.

Va nihoyat, transaction'ni yakunlaymiz.

sql

```sql
=> COMMIT;
```

## Xulosa

Bu bobda PostgreSQL'ning ichki page strukturasi va row version'lar mexanizmi batafsil ko'rib chiqildi. Quyidagi asosiy fikrlarni ta'kidlash mumkin:

**Page strukturasi** — har bir page header, pointer'lar massivi, free space, row version'lar va special area'dan iborat. Bu tuzilish page ichida samarali joy boshqaruvini ta'minlaydi. Pointer'lar orqali indirect addressing ishlatilishi — juda aqlli yechim. Bu row version'larni page ichida ko'chirish imkonini beradi, bunda index'lardagi havola (tid) o'zgarmaydi.

**Row version header'i** — har bir row version kamida 23 byte header egallaydi. Bu katta overhead, ayniqsa narrow table'lar uchun. Shuning uchun table loyihalashda column'lar tartibini to'g'ri tanlash (alignment optimization) muhim — integer va boshqa fixed-size type'larni boshiga, boolean kabi kichik type'larni oxiriga qo'yish orqali bir necha byte'larni tejash mumkin.

**MVCC mexanizmi** — xmin va xmax orqali har bir row version'ning "hayot davri" belgilanadi. INSERT xmin'ni o'rnatadi, DELETE xmax'ni o'rnatadi, UPDATE esa aslida DELETE + INSERT. Bu mexanizm reader'lar writer'larni block qilmasligini ta'minlaydi — PostgreSQL'ning MVCC ustunliklaridan biri.

**Hint bit'lar va clog** — transaction status'ini aniqlash clog orqali amalga oshiriladi, lekin samaradorlik uchun bir marta aniqlangan status hint bit'lar sifatida row version header'iga yoziladi. Qiziq tomoni shundaki, hatto SELECT query ham dirty page'lar yaratishi mumkin, chunki u hint bit'larni o'rnatishi mumkin.

**ROLLBACK = tez** — PostgreSQL'da abort qilish commit qilish kabi tez, chunki haqiqiy rollback bo'lmaydi. Data page'lardagi o'zgarishlar joyida qoladi, faqat clog'da aborted bit o'rnatiladi. Keyingi transaction'lar hint bit'lar orqali bu version'larni "ko'rmay" o'tadi.

Bu bilimlar vacuum, snapshot isolation va boshqa PostgreSQL internal mexanizmlarini tushunish uchun poydevor bo'lib xizmat qiladi.

# Index'lar, TOAST, Virtual transaction'lar, Nested transaction'lar

## 3.4. Index'lar

Har qanday turdagi index'larda hech qachon row version'lar bo'lmaydi, har bir row aynan bitta nusxada ifodalanadi. Boshqacha aytganda, index row header'ida xmin va xmax field'lari bo'lmaydi. Index'dan havola barcha table row version'lariga yo'naltiriladi. Transaction qaysi version unga ko'rinishini aniqlash uchun table'ga qarashi kerak (agar page visibility map'da belgilanmagan bo'lsa).

Qulaylik uchun `pageinspect` extension yordamida page'dagi barcha index yozuvlarini ko'rsatadigan oddiy funksiya yaratamiz (B-tree page'i doirasida yozuvlar tekis ro'yxat ko'rinishida joylashadi):

sql

```sql
=> CREATE FUNCTION index_page(relname text, pageno integer)
   RETURNS TABLE(itemoffset smallint, htid tid)
   AS $$
   SELECT itemoffset,
          htid   -- ctid v.13 gacha
   FROM bt_page_items(relname,pageno);
   $$ LANGUAGE sql;
```

Page'da table row'ning ikkala version'iga — aktual va eski — pointer'larni topamiz:

sql

```sql
=> SELECT * FROM index_page('t_s_idx',1);

 itemoffset | htid
------------+-------
          1 | (0,2)
          2 | (0,1)
(2 rows)
```

BAR < FOO bo'lgani uchun, ikkinchi row version'ga pointer index'da birinchi o'rinda turadi.

## 3.5. TOAST

TOAST-table mohiyatan oddiy table hisoblanadi va u uchun o'zining version'ligi qo'llab-quvvatlanadi: "toast"larning version'lari asosiy table row version'lari bilan bog'lanmagan. Lekin TOAST-table bilan ichki ishlash shunday qurilganki, row'lar hech qachon update qilinmaydi, faqat qo'shiladi va o'chiriladi, shuning uchun version'lilik bu holda biroz degenerativ (soddalashgan) bo'ladi.

Data o'zgarganda, asosiy table'da har doim yangi row version yaratiladi. Lekin agar update TOAST'da saqlanayotgan "uzun" qiymatga tegmasa, yangi row version TOAST-table'dagi avvalgi qiymatga havola qiladi. Faqat update "uzun" qiymatni o'zgartirganda, asosiy table'da yangi row version ham, yangi "toast"lar ham yaratiladi.

## 3.6. Virtual transaction'lar

PostgreSQL transaction raqamlarini "tejash" imkonini beruvchi optimizatsiyadan foydalanadi.

Agar transaction faqat data o'qisa, u row version'larning ko'rinishiga (visibility) hech qanday ta'sir qilmaydi. Shuning uchun avvaliga xizmat ko'rsatuvchi process transaction'ga virtual raqam (virtual xid) beradi. Raqam xizmat ko'rsatuvchi process identifier'i va ketma-ket sondan iborat. Bu raqamni berish barcha process'lar o'rtasida synchronization'ni talab qilmaydi va shuning uchun juda tez bajariladi. Transaction'ning haqiqiy raqami hali yo'q:

sql

```sql
=> BEGIN;

=> -- txid_current_if_assigned() v.13 gacha
   SELECT pg_current_xact_id_if_assigned();

 pg_current_xact_id_if_assigned
--------------------------------

(1 row)
```

Turli vaqt lahzalarida tizimda allaqachon ishlatilgan raqamlarga ega virtual transaction'lar bo'lishi mumkin. Va bu normal, chunki virtual raqamlar faqat operativ xotirada, transaction faol (active) bo'lgan paytda mavjud; ular hech qachon data page'lariga yozilmaydi va disk'ka tushmaydi.

Agar transaction data o'zgartira boshlasa, unga haqiqiy, unique transaction raqami beriladi:

sql

```sql
=> UPDATE accounts
   SET amount = amount - 1.00;

=> SELECT pg_current_xact_id_if_assigned();

 pg_current_xact_id_if_assigned
--------------------------------
                            785
(1 row)

=> COMMIT;
```

## 3.7. Nested transaction'lar (Ichma-ich transaction'lar)

### Savepoint'lar

SQL'da savepoint'lar (saqlash nuqtalari) belgilangan bo'lib, ular transaction'ning bir qism amallarini bekor qilish imkonini beradi, transaction'ni to'liq to'xtatmasdan. Lekin bu yuqorida keltirilgan sxemaga sig'maydi, chunki transaction status'i barcha o'zgarishlar uchun bitta, fizik jihatdan esa hech qanday data rollback qilinmaydi.

Bunday funksionallikni amalga oshirish uchun savepoint'li transaction bir nechta nested transaction'larga (subtransaction) bo'linadi, ularning status'ini alohida boshqarish mumkin.

Nested transaction'lar o'zlarining raqamiga ega (asosiy transaction raqamidan kattaroq). Nested transaction'larning status'i clog'ga oddiy tarzda yoziladi, lekin commit qilingan nested transaction'lar bir vaqtning o'zida ikkita bit — committed va aborted — bilan belgilanadi. Yakuniy status asosiy transaction status'iga bog'liq: agar u bekor qilingan bo'lsa, barcha nested transaction'lar ham bekor qilingan hisoblanadi.

Nested'lik haqidagi ma'lumot PGDATA/pg_subtrans katalogi ichidagi file'larda saqlanadi, bu file'larning page'lariga murojaat SLRU-cache orqali amalga oshiriladi. Uning size'i `subtransaction_buffers` parametri bilan sozlanadi, default qiymati esa buffer cache size'ining 1/512 qismini tashkil qiladi.

> **Izoh:** Nested transaction'larni autonomous transaction'lar bilan adashtirmang. Autonomous transaction'lar bir-biridan hech qanday tarzda bog'liq emas, nested'lardan farqli ravishda. Oddiy PostgreSQL'da autonomous transaction'lar mavjud emas, va bu yaxshi tomonga: ular amalda juda kam kerak, boshqa DBMS'larda esa ularning mavjudligi suiiste'mollikka olib keladi, bu hamma uchun muammoga aylanadi.

Table'ni tozalaymiz, transaction boshlaymiz va row insert qilamiz:

sql

```sql
=> TRUNCATE TABLE t;

=> BEGIN;

=> INSERT INTO t(s) VALUES ('FOO');

=> SELECT pg_current_xact_id();

 pg_current_xact_id
--------------------
                787
(1 row)
```

Endi savepoint o'rnatamiz va yana bitta row insert qilamiz:

sql

```sql
=> SAVEPOINT sp;

=> INSERT INTO t(s) VALUES ('XYZ');

=> SELECT pg_current_xact_id();

 pg_current_xact_id
--------------------
                787
(1 row)
```

E'tibor bering, `pg_current_xact_id` funksiyasi asosiy transaction raqamini qaytaradi, nested transaction raqamini emas.

sql

```sql
=> SELECT *
   FROM heap_page('t',0) p
   LEFT JOIN t ON p.ctid = t.ctid;

 ctid  | state  | xmin | xmax | id |  s
-------+--------+------+------+----+-----
 (0,1) | normal | 787  | 0 a  |  2 | FOO
 (0,2) | normal | 788  | 0 a  |  3 | XYZ
(2 rows)
```

Savepoint'ga rollback qilamiz va uchinchi row insert qilamiz:

sql

```sql
=> ROLLBACK TO sp;

=> INSERT INTO t(s) VALUES ('BAR');

=> SELECT *
   FROM heap_page('t',0) p
   LEFT JOIN t ON p.ctid = t.ctid;

 ctid  | state  | xmin | xmax | id |  s
-------+--------+------+------+----+-----
 (0,1) | normal | 787  | 0 a  |  2 | FOO
 (0,2) | normal | 788  | 0 a  |    |
 (0,3) | normal | 789  | 0 a  |  4 | BAR
(3 rows)
```

Page'da biz abort qilingan nested transaction tomonidan qo'shilgan row'ni ko'rishda davom etayapmiz.

O'zgarishlarni commit qilamiz:

sql

```sql
=> COMMIT;

=> SELECT * FROM t;

 id |  s
----+-----
  2 | FOO
  4 | BAR
(2 rows)

=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+------
 (0,1) | normal | 787 c | 0 a
 (0,2) | normal | 788 a | 0 a
 (0,3) | normal | 789 c | 0 a
(3 rows)
```

Endi yaxshi ko'rinib turibdi — har bir nested transaction o'zining status'iga ega.

Nested transaction'larni SQL'da bevosita ishlatib bo'lmaydi, ya'ni joriy transaction'ni tugatmasdan yangi transaction boshlash mumkin emas:

sql

```sql
=> BEGIN;
BEGIN

=> BEGIN;
WARNING:  there is already a transaction in progress
BEGIN

=> COMMIT;
COMMIT

=> COMMIT;
WARNING:  there is no transaction in progress
COMMIT
```

Bu mexanizm bilvosita (implicit) ishga tushiriladi: savepoint'lar ishlatilganda, PL/pgSQL exception handling'da va ba'zi boshqa, yanada ekzotik holatlarda.

### Error'lar va amallarning atomicity'si

Agar amal bajarilish jarayonida error yuz bersa nima bo'ladi?

Masalan:

sql

```sql
=> BEGIN;

=> SELECT * FROM t;

 id |  s
----+-----
  2 | FOO
  4 | BAR
(2 rows)

=> UPDATE t SET s = repeat('X', 1/(id-4));
ERROR:  division by zero
```

Xatolikdan so'ng transaction uzilgan (aborted) hisoblanadi va unda boshqa birorta ham amal ruxsat etilmaydi:

sql

```sql
=> SELECT * FROM t;
ERROR:  current transaction is aborted, commands ignored until end
of transaction block
```

Hatto o'zgarishlarni commit qilishga urinib ko'rsak ham, PostgreSQL bekor qilish haqida xabar beradi:

sql

```sql
=> COMMIT;
ROLLBACK
```

Nima uchun xatolikdan so'ng transaction'ni davom ettirish mumkin emas? Allaqachon bajarilgan amallar hech qachon rollback qilinmasligi sababli, biz error'dan oldin bajarilgan o'zgarishlarning bir qismiga kirish huquqiga ega bo'lar edik — bu nafaqat transaction'ning, balki hatto operator'ning atomicity'sini buzgan bo'lardi.

Bizning misolda bo'lgani kabi, operator error'gacha ikkita row'dan birini update qilishga ulgurgan:

sql

```sql
=> SELECT * FROM heap_page('t',0);

 ctid  | state  | xmin  | xmax
-------+--------+-------+------
 (0,1) | normal | 787 c | 790
 (0,2) | normal | 788 a | 0 a
 (0,3) | normal | 789 c | 0 a
 (0,4) | normal | 790   | 0 a
(4 rows)
```

Aytgancha, psql'da xatolikdan so'ng ham transaction ishini davom ettirish imkonini beruvchi rejim mavjud, go'yo xato operator'ning amallari rollback qilinganday:

sql

```sql
=> \set ON_ERROR_ROLLBACK on

=> BEGIN;

=> UPDATE t SET s = repeat('X', 1/(id-4));
ERROR:  division by zero

=> SELECT * FROM t;

 id |  s
----+-----
  2 | FOO
  4 | BAR
(2 rows)

=> COMMIT;
COMMIT
```

Taxmin qilish qiyin emas — bunday rejimda psql aslida har bir buyruq oldiga implicit savepoint o'rnatadi va xatolik yuz berganda unga rollback qiladi. Bu rejim default bo'yicha ishlatilmaydi, chunki savepoint'larni o'rnatish (hatto ularga rollback qilmasdan ham) sezilarli overhead bilan bog'liq.

## Xulosa

Bu bo'limlarda PostgreSQL'ning bir nechta muhim mexanizmlari ko'rib chiqildi:

**Index'lar va MVCC** — Index'larda row version'lar tushunchasi umuman yo'q. Har bir index entry faqat bitta nusxada mavjud, xmin/xmax header field'lari bo'lmaydi. Index barcha table row version'lariga havola qiladi, keyin transaction o'zi table'ga qarab visibility'ni aniqlaydi. Bu dizayn oddiy va samarali, lekin bitta kamchilik bor: dead tuple'lar index'da ham joy egallaydi, shuning uchun VACUUM index'larni ham tozalashi kerak.

**TOAST** — Uzun qiymatlar alohida TOAST-table'da saqlanadi, u o'zining version'ligiga ega, lekin soddalashtirilgan: row'lar faqat insert va delete qilinadi, update bo'lmaydi. Agar UPDATE "uzun" qiymatga tegmasa, yangi row version eski TOAST entry'ga havola qiladi — bu katta BLOB'lar bilan ishlashda juda samarali.

**Virtual transaction'lar** — Read-only transaction'larga haqiqiy transaction raqam berilmaydi, faqat virtual xid (process ID + sequential number). Bu raqamlar operativ xotirada yashaydi, disk'ka hech qachon yozilmaydi. Haqiqiy raqam faqat data o'zgartirganda beriladi. Bu optimizatsiya transaction counter'ning keraksiz o'sishini oldini oladi — bu 32-bit xid wraparound muammosini kamaytirish uchun muhim.

**Nested transaction'lar va savepoint'lar** — PostgreSQL'da nested transaction'lar SQL'da bevosita emas, faqat savepoint'lar, PL/pgSQL exception handling va boshqa mexanizmlar orqali implicit ishlatiladi. Har bir nested transaction o'z raqami va clog'dagi status'iga ega, lekin yakuniy natija asosiy transaction'ga bog'liq. Muhim nuqta: savepoint'lar overhead keltirib chiqaradi, shuning uchun `ON_ERROR_ROLLBACK` rejimi default emas.

**Error va atomicity** — PostgreSQL'da error bo'lganda transaction to'liq abort qilinadi, davom ettirish mumkin emas. Sabab: bajarilgan amallar fizik rollback qilinmagani uchun, davom ettirsak operator atomicity'si buziladi. Bu qat'iy yondashuv data consistency'ni kafolatlaydi.
