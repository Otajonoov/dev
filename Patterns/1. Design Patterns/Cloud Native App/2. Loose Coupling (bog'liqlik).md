
"**==Coupling==**" (bog'liqlik) termini - komponentlar o'rtasidagi o'zaro bog'liqlik darajasini ifodalovchi atama.

##### **High Cohesion**
> Modul yoki komponent faqat **bitta vazifani aniq va to‘liq bajaradi.**  
> ➤ Bu uni tushunishni, test qilishni va qo‘llab-quvvatlashni osonlashtiradi.

---
#####  **Loose Coupling**
> Modul yoki komponent boshqa komponentlarga minimal darajada bog‘langan
> ➤ Bu uni mustaqil o‘zgartirish yoki almashtirishni osonlashtiradi.


### Ideal kombinatsiya:

**High Cohesion + Low Coupling** — bu professional arxitekturaning maqsadi.

> 👉 **Modullar ichida**: Har bir modul faqat **bir vazifani** qiladi (**high cohesion**)  
> 👉 **Modullar o‘rtasida**: Ular faqat **interface yoki API orqali** aloqa qiladi (**low coupling**)


# **Coupling va Cohesion – To‘liq Tushuncha**
### Coupling turlari:

|Coupling turi|Tavsif|Misol|
|---|---|---|
|**Tight Coupling**|Modullar bir-birining ichki tuzilmasiga qattiq bog‘langan|A modul B modulning struct’ini to‘g‘ridan-to‘g‘ri ishlatadi|
|**Loose Coupling**|Modullar faqat interface yoki signal orqali aloqa qiladi|A modul B modulga `Storage` interfeysi orqali murojaat qiladi|

---

### Cohesion turlari:

|Cohesion turi|Tavsif|Misol|
|---|---|---|
|**Low Cohesion**|Modul turli vazifalarni bajaradi|`UserService` ichida ham `DB`, ham `email`, ham `auth`|
|**High Cohesion**|Modul **faqat bitta ishni** qiladi|`UserService` faqat `user` bilan ishlaydi, `AuthService` faqat `auth`|

---

## 🏗️ **Ideal kombinatsiya: High Cohesion + Loose Coupling**

> 💡 Bu **SOLID** va **Clean Architecture** tamoyillariga mos keladi.

- **High Cohesion**: Har bir modul aniq bitta maqsadga xizmat qiladi
    
- **Loose Coupling**: Modullar bir-biridan mustaqil ishlaydi, interfeyslar orqali bog‘lanadi

# 🔧 Go Tilida Bunga Qanday Erishiladi?

Go — bu **soddalik va interfeysga asoslangan dizayn** bilan low coupling va high cohesion uchun qulay til.

---

## 🔹 Go’da Coupling’ni kamaytirish patternlari

### 1. **Interface Segregation Principle**

```go
type Notifier interface {
	Notify(message string) error 
}
```

- `EmailNotifier`, `SlackNotifier`, `SMSNotifier` hammasi shu interfeysni implement qiladi.
- **A modul** `Notifier` interfeysi bilan ishlaydi, qaysi implementatsiya berilishi esa unga bog‘liq emas.

---

### 2. **Factory Function (Constructor pattern)**
```go
func NewService(repo Repository) *Service {
	return &Service{
		repo: repo
	}
}
```

- Har xil `Repository` implementatsiyasini kiritish mumkin
- Dependency injection orqali **loose coupling**

---

### 3. **Strategy Pattern (behavior parametr sifatida)**

```go
type AuthStrategy interface {  
	Authenticate(user string, pass string) bool 
}
```

- Har xil autentifikatsiya strategiyalarini modullar o‘zgartirmasdan almashtirish mumkin

---

### 4. **Adapter Pattern**
```go
type PaymentAdapter interface {     
	Pay(amount float64) error 
}  
// StripeAdapter, PayPalAdapter implement qiladi
```

- Tashqi servislar o‘zgaradi → faqat adapter o‘zgaradi, qolgan kod emas

---

### 5. **Middleware Pattern (HTTP/gRPC)**

Go'da middlewarelar orqali **responsibility’larni ajratish** — coupling’ni pasaytiradi.
```go
func LoggingMiddleware(next http.Handler) http.Handler {     
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {               log.Println("Request received")         
		next.ServeHTTP(w, r)     
	}) 
}
```

## 🧱 Arxitekturaviy yondashuvlar

### 🔹 Clean Architecture (Onion Layered)


`Entities → UseCases → InterfaceAdapters → Infrastructure`

- Ichki qatlamlar hech qachon tashqi qatlamlarni chaqirmaydi
    
- Har bir qatlam **faqat o‘z ishi bilan shug‘ullanadi** → High Cohesion
    
- Aloqa faqat **interfeyslar orqali** → Loose Coupling
    

### 🔹 Hexagonal Architecture (Ports & Adapters)

- `Port` — interfeyslar (input/output)
    
- `Adapter` — tashqi tizimga bog‘lovchi implementatsiya
    
- Yadro logikasi adapterlarga bog‘liq emas

## 🛠️ Go’da Bunday Dizaynni Qo‘llagan Misol
```go
// Port
type Storage interface {
	Save(data string) error 
}  

// Adapter 
type FileStorage struct{} 
func (f *FileStorage) Save(data string) error {     
	// Faylga yozish     
	return nil 
}  

// Service 
type ReportService struct { 
	storage Storage 
}  

func NewReportService(s Storage) *ReportService {     
	return &ReportService{
		storage: s
	} 
}
```
- `ReportService` hech qachon `FileStorage` yoki `S3Storage`ni bilmaydi
    
- Bu **Loose Coupling + High Cohesion**

## Kuchsiz bog'liqlikning afzalliklari

1. **Mustaqil rivojlanish** - bir komponentdagi o'zgarishlar boshqa komponentlarga ta'sir qilmaydi
2. **Mustaqil deploy qilish** - har bir komponentni alohida yangilash mumkin
3. **Tizim moslashuvchanligi** - komponentlarni almashtirish osonroq
4. **Xatoliklarga chidamlilik** - bir komponentdagi xatolik butun tizimga ta'sir qilmaydi
5. **Masshtablilik** - har bir komponentni alohida masshtablash imkoniyati

## Go’da foydali patternlar ro‘yxati:
