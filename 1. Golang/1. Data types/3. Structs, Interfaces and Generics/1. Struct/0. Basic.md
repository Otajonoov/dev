
```go
type Book struct {
    ID     int
    Title  string
    Author string `tagKey:"tagValue"`
}
```


Обратите внимание на использование struct tag в этом примере (`tagKey:"tagValue"`). Struct tag — это строковые литералы, прикреплённые к полям, как тот, что рядом с `Author`. Они действуют как metadata и могут использоваться вашим кодом Go или внешними инструментами. Это позволяет реализовать дополнительное поведение, такое как encoding, validation или database mapping.

Распространённый случай использования tag — **сериализация**, при которой данные конвертируются между различными форматами. 
Tag управляют тем, как поля структуры кодируются или декодируются при использовании библиотек вроде `encoding/json`:

```go
type User struct {
    Username string `json:"custom_username"`
    Age      int    `json:"age"`
}
// {"custom_username":"Bob","age":18}
```

Tag `json` не является специальным ключевым словом; это просто конвенция, распознаваемая пакетом `encoding/json`. Вы можете определять пользовательские tag для поддержки других сценариев использования.

Пакет `reflect` в Go позволяет исследовать и манипулировать типами во время выполнения, включая чтение struct tag:

```go
type Person struct {
    Name string `json:"name" xml:"nameElement"`
    Age  int    `json:"age,omitempty"`
}

func main() {
    p := Person{Name: "John Doe", Age: 30}
    t := reflect.TypeOf(p)
    for i := 0; i < t.NumField(); i++ {
        field := t.Field(i)
        jsonTag := field.Tag.Get("json")
        xmlTag := field.Tag.Get("xml")
        fmt.Printf("%q field: json tag: %q, xml tag: %q\n", 
        field.Name, jsonTag, xmlTag)
    }
}
// Output:
// "Name" field: json tag: "name", xml tag: "nameElement"
// "Age" field: json tag: "age,omitempty", xml tag: ""
```

Этот пример показывает, как пакет `encoding/json` использует tag для сериализации, но тот же принцип может быть применён к любой функциональности, которая извлекает пользу из runtime metadata.

### Инициализация структур с помощью Composite Literal

Этот сокращённый синтаксис позволяет присваивать значения структуре непосредственно в момент создания.

Есть два распространённых подхода:

- Предоставить значения в точном порядке расположения полей в структуре.
- Явно указать имя каждого поля при присваивании значений.

Именование полей особенно полезно при работе с большими структурами или когда вы хотите задать только несколько полей, оставив остальные на их zero value:

```go
a := Book{ID: 100, Title: "Great Expectations", Author: "Charles Dickens"}
b := Book{Title: "The Anatomy of Go"}
```


 Это называется **позиционной инициализацией**:
```go
b := Book{100, "Great Expectations", "Charles Dickens"}
```

Тем не менее, позиционная инициализация может быть практичной в конкретных случаях, таких как:
- Маленькие структуры с очевидным порядком полей, например `Point{X, Y}` или `Pair{Key, Value}`.
- Table-driven тесты, где быстрое создание множества экземпляров структур без повторения имён полей помогает сохранить код кратким.

Вот пример table-driven теста для простой функции `Add`:
```go
func TestAdd(t *testing.T) {
    // Определяем slice тестовых случаев
    var tests = []struct {
        input1   int
        input2   int
        expected int
    }{
        {1, 1, 2},
        {2, 3, 5},
        {10, -5, 5},
        {0, 0, 0},
    }

    // Итерируем по тестовым случаям
    for _, tt := range tests {
        result := Add(tt.input1, tt.input2)
        if result != tt.expected {
            t.Errorf("Add(%d, %d) = %d; want %d", 
            tt.input1, tt.input2, result, tt.expected)
        }
    }
}
```

Итак, как Go подготавливает память и обеспечивает правильную инициализацию при использовании composite literal?
Kомпилятор генерирует инструкции для присваивания значений каждому полю или для установки их в zero value, если значение не предоставлено.

### Доступ к полям структуры

Доступ к полям структуры осуществляется через точечную (`.`) нотацию, как при чтении, так и при обновлении значений:

```go
type User struct {
    ID   int
    Name string
}

func main() {
    b1 := User{ID: 100, Name: "John"}
    b2 := &User{ID: 101, Name: "Doe"}

    b1.Name = "Jane"
    fmt.Println(b1.Name)  // Output: Jane
    fmt.Println(b2.Name)  // Output: Doe
}
```

Компилятор знает точное расположение каждого поля внутри структуры. Он вычисляет **offset** (смещение) каждого поля от начала структуры в памяти:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212104255.png)

При доступе к полю через точечную нотацию Go вычисляет offset поля от базового адреса структуры. Рассмотрим `b2`, который является pointer на struct. Можно было бы ожидать, что нужно разыменовать его вручную, написав `(*b2).ID`.

Хотя такой синтаксис допустим, Go выполняет разыменование **автоматически**. 
Компилятор обнаруживает, что `b2` является pointer, и прозрачно разрешает доступ к полю.
Это означает, что вы можете просто написать `b2.ID`, и Go выполнит необходимое разыменование pointer. 
Именно поэтому, если `b2` равен `nil`, попытка доступа к полю приведёт к **nil pointer dereference** во время выполнения.

Вот как компилятор обрабатывает доступ к полям на этапе проверки типов:

```go
// DotField возвращает выражение selector, которое выбирает
// index-ое поле данного выражения, которое должно быть типа struct
// или pointer-to-struct.
func DotField(pos src.XPos, x ir.Node, index int) *ir.SelectorExpr {
    op, typ := ir.ODOT, x.Type()
    if typ.IsPtr() {
        op, typ = ir.ODOTPTR, typ.Elem()
    }
    if !typ.IsStruct() {
        base.FatalfAt(pos, "DotField of non-struct: %L", x)
    }
    types.CalcSize(typ)
    field := typ.Field(index)
    return dot(pos, field.Type, op, x, field)
}
// _Figure 56. DotField (src/cmd/compile/internal/typecheck.go)_
```

Когда компилятор распознаёт pointer на struct, он меняет операцию с обычного dot node (`ODOT`) на dot pointer node (`ODOTPTR`). 
Это различие важно на этапе генерации кода, где абстрактная операция транслируется в низкоуровневые инструкции:

```go
func (s *state) exprCheckPtr(n ir.Node, checkPtrOK bool) *ssa.Value {
    ...
    switch n.Op() {
    ...
    case ir.ODOTPTR:
        n := n.(*ir.SelectorExpr)
        // Получаем значение pointer
        p := s.exprPtr(n.X, n.Bounded(), n.Pos())
        // Добавляем offset поля к pointer
        p = s.newValue1I(ssa.OpOffPtr, types.NewPtr(n.Type()), n.Offset(), p)
        // Загружаем значение из памяти (разыменование)
        return s.load(n.Type(), p)
    }
    ...
}
// _Figure 57. exprCheckPtr (src/cmd/compile/internal/ssagen/ssa.go)_
```


Здесь компилятор получает значение pointer, добавляет offset поля, а затем загружает данные из памяти, разыменовывая вычисленный адрес.

Вы можете заметить на практике, что нельзя **индексировать pointer** на slice или map. Поддерживаются только pointer на массивы:

```go
func main() {
    var a *[5]int
    var b *[]int
    var c *map[string]int

    // valid
    fmt.Println(a[2])
    
    // invalid operation: cannot index b (variable of type *[]int)
    fmt.Println(b[2]) 
    
    // invalid operation: cannot index c (variable of type *map[string]int)   
    fmt.Println(c["a"])
}
```

Такое поведение существует потому, что slice и map фундаментально отличаются от простых непрерывных типов памяти, таких как массивы и структуры.
Они уже включают indirection (косвенность) как часть своего дизайна. 
Хотя slice сами по себе не являются pointer, они опираются на descriptor, содержащий pointer на underlying array вместе с информацией о length и capacity.

Если бы Go позволял индексацию pointer на slice, компилятору пришлось бы выполнять **два уровня разыменования**: сначала для доступа к slice header (`data`, `len`, `cap`), затем для извлечения элемента из data pointer. 
Та же логика применима к map, которые внутренне ссылаются на hash table (`hmap`).

Разрешение такого синтаксиса добавило бы лишнюю сложность в язык. 
Это ввело бы ещё один уровень indirection без предоставления явной пользы, делая поведение slice и map менее предсказуемым и более трудным для анализа.

В отличие от массивов, которые хранят множество значений одного типа, структуры позволяют объединять **различные типы данных** в единый блок. Хотя и массивы, и структуры хранят данные в непрерывном блоке памяти, структуры специально предназначены для работы со смешанными типами данных.

Если сравнить расположение массивов и структур в памяти, их структура удивительно похожа. В обоих случаях данные расположены линейно — один блок за другим:

```go
type Struct struct {
    First  int
    Second bool
    Third  string
}
```

![](../../../../assets/obsidian-images/Pasted%20image%2020260212100929.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Если посмотреть на layout структуры `Struct`, происходит нечто неожиданное. Поле `Second bool` начинается с адреса памяти `0x14000094068`. Можно было бы ожидать, что следующее поле начнётся сразу после — по адресу `0x14000094069`. 
Вместо этого оно перепрыгивает на `0x14000094070`. 
Это оставляет **7-байтный промежуток** после 1-байтного поля `bool`:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212101059.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Это происходит из-за **выравнивания памяти и padding**. 
Go выравнивает данные в памяти для повышения производительности. 
Иногда он добавляет неиспользуемое пространство, называемое padding, чтобы сделать доступ более эффективным.
