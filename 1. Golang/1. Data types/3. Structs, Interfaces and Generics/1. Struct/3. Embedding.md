
Embedding позволяет одной структуре включать другую, давая им возможность работать вместе чистым и эффективным способом. 
В Go поля и методы embedded struct **автоматически продвигаются**, делая их доступными так, как если бы они были частью внешней структуры.

Эти embedded struct часто называют «анонимными полями», потому что они включаются без присвоения отдельного имени. 
Технически они не безымянные — **их тип служит именем поля**. 
При необходимости к ним по-прежнему можно обращаться напрямую по имени типа:

```go
type Engine struct {
    Horsepower int
}

type Car struct {
    Engine
    Brand string
}

func main() {
    myCar := Car{
        Engine: Engine{
            Horsepower: 200,
        },
        Brand: "Toyota",
    }

    // Доступ к полям Engine напрямую через Car
    fmt.Println(myCar.Horsepower)          // Outputs: 200
    fmt.Println(myCar.Brand)               // Outputs: "Toyota"

    // Или доступ к полям через embedded struct
    fmt.Println(myCar.Engine.Horsepower)   // Outputs: 200
}
```


Если вы знакомы с наследованием из объектно-ориентированных языков, это поведение может показаться похожим. 
Однако Go **не поддерживает наследование** в традиционном смысле. 
Вместо этого он полагается на **composition**. 

Когда и внешняя, и embedded структуры определяют поля или методы с одинаковым именем, Go следует определённому набору правил для разрешения конфликта.
Это поведение известно как **shadowing** (затенение):

1. **Внешняя структура имеет приоритет:** Go сначала проверяет поля внешней структуры. Если встречается embedded struct, Go не погружается в неё сразу. Вместо этого он отмечает embedded struct для последующей оценки. Если нужное поле найдено во внешней структуре, этот результат используется немедленно.
2. **Проверка embedded структур:** Если поле не найдено во внешней структуре, Go переходит к проверке embedded структур. Он делает это в порядке их появления.
3. **Разрешение неоднозначности:** Если потенциальное совпадение найдено, Go проверяет, что не существует других совпадений на том же уровне глубины. Если на этом уровне обнаружено несколько кандидатов, возникает **ошибка компиляции** из-за неоднозначности.
4. **Возврат разрешённого совпадения:** Как только уникальное совпадение определено, Go выбирает его и использует как результат.

Следующий фрагмент показывает, как работает shadowing при совпадении имён полей:

```go
type Base struct {
    A
    B
    Name string
}

type A struct {
    Shared string
    Name   string
}

type B struct {
    Shared string
}

base.Name      // valid
base.Shared    // invalid: Error: Ambiguous (Shared exists in both A and B)
base.A.Shared  // valid
base.B.Shared  // valid
```

Вот как Go разрешает каждый случай:

- Доступ к `base.Name` работает без проблем. Go находит поле `Name` непосредственно в структуре `Base`, поэтому сразу возвращает это значение.
- Попытка доступа к `base.Shared` завершается ошибкой компиляции. И `A`, и `B` определяют поле `Shared`, и поскольку они embedded на одном уровне, Go не может определить, какое использовать. Это приводит к **неоднозначности**.

### Shadowing методов в Embedded структурах

Разработчики с опытом объектно-ориентированного программирования (ООП) часто ожидают, что Go будет обрабатывать методы аналогично классическому наследованию. 
На первый взгляд, embedding может напоминать наследование, но Go делает акцент на **композиции**, а не на наследовании — это ключевое концептуальное различие.

Методы в Go следуют тому же поведению shadowing, что и поля. 
Рассмотрим следующий пример:

```go
type Vehicle struct {
    Name string
}

func (v Vehicle) Operate() string {
    return fmt.Sprintf("%s operates on %s", v.Name, v.Surface())
}

func (v Vehicle) Surface() string {
    return "land"
}

Vehicle{Name: "Car"}.Operate()
// Output: "Car operates on land"
```

В этом случае метод `Operate` вызывает `Surface`, который возвращает `"land"`. 
Результат предсказуем: `"Car operates on land"`.

Теперь предположим, что мы создаём тип `Boat`, который содержит embedded `Vehicle` и определяет свой собственный метод `Surface`:

```go
type Boat struct {
    Vehicle
}

func (b Boat) Surface() string {
    return "water"
}
```

Это может выглядеть как переопределение метода.
Вызов `boat.Surface()` вернёт `"water"`, как и ожидается. 
Однако Go **не поддерживает** традиционное переопределение методов. 
Вместо этого метод `Surface` в `Boat` **затеняет** метод в `Vehicle`.

Важное различие проявляется при вызове `boat.Operate()`. 
Несмотря на то, что `Boat` определяет свой собственный метод `Surface`, метод `Operate` принадлежит `Vehicle`. 
Внутри этого метода вызов `Surface()` ссылается на `Vehicle.Surface`, **а не** на `Boat.Surface`.

Это потому, что методы, определённые на embedded типе, видят **только свой собственный method set**:

```go
func main() {
    boat := Boat{Vehicle{Name: "Speedboat"}}
    fmt.Println(boat.Surface())  // Output: water
    fmt.Println(boat.Operate())  // Output: Speedboat operates on land
}
```

В результате вывод может быть удивительным: `"Speedboat operates on land"`. 
Несмотря на то, что `Boat` имеет свой собственный метод `Surface`, `Operate` не использует его, потому что принадлежит embedded типу `Vehicle` и разрешает вызовы методов **в рамках своей области видимости**.
