
Когда вы определяете структуру, Go выделяет память для её полей на основе их типов и располагает их в определённой последовательности. 
В отличие от массивов, которые плотно упакованы в памяти, структуры могут содержать **неиспользуемые промежутки**. 
Эти промежутки вводятся из-за требований выравнивания, обеспечивающих эффективный доступ к данным.

Для выполнения ограничений выравнивания компилятор может вставлять **padding байты** между полями. 
Эти padding байты не видны в коде, но необходимы для выравнивания данных способом, оптимизирующим производительность для CPU. 
Компромисс за лучшую скорость доступа — **увеличение использования памяти**:

```go
type A struct {
    A byte
    B int32
    C byte
    D int64
    E byte
}

type OptimizedA struct {
    D int64
    B int32
    A byte
    C byte
    E byte
}

unsafe.Sizeof(A{})          // 32
unsafe.Sizeof(OptimizedA{}) // 16
```

Хотя и `A`, и `OptimizedA` содержат одни и те же поля, их использование памяти сильно различается. `A` занимает **32 байта**, тогда как `OptimizedA` — только **16**. 
Разница сводится к **порядку расположения полей**.

Этот layout иллюстрирует четыре ключевые концепции, связанные с памятью:

1. **Word size** (размер слова): объём данных, который CPU может прочитать или записать за одну операцию, на основе архитектуры системы.
2. **Cache line** (строка кеша): блок памяти, который CPU загружает в свой кеш за один раз, обычно 64 или 128 байт на современных системах.
3. **Alignment** (выравнивание): каждое поле должно начинаться с адреса памяти, соответствующего требованию выравнивания его типа (например, `int32` должен начинаться на 4-байтной границе).
4. **Padding** (заполнение): дополнительное пространство, автоматически добавляемое компилятором для удовлетворения правил выравнивания и обеспечения быстрого, выровненного доступа к памяти.
---
---

**Word size** определяет количество бит, которые CPU может обработать за одну операцию. Он влияет на то, как данные обрабатываются, хранятся и адресуются в системе. Архитектура процессора определяет word size. 
Например, на 32-битной Linux-системе word size составляет 32 бита. 
На 64-битной системе — 64 бита.

Это различие означает, что 32-битный процессор использует 32-битные регистры общего назначения, тогда как 64-битный процессор оперирует 64-битными регистрами.

Эти регистры выполняют арифметические, логические операции и адресацию памяти. 
В результате 64-битная машина может выполнять операции с 64-битными integer за один шаг. Напротив, работа со 128-битными числами может потребовать нескольких инструкций.

Производительность, как правило, лучше, когда данные выровнены по native word size CPU. Рассмотрим пример:

```go
var a uint64 = 100
var b uint64 = 200
var result uint64 = a + b
```

На 64-битной системе это сложение выполняется за **одну CPU-инструкцию**, поскольку `uint64` соответствует word size. 
Если использовать более крупный тип, такой как `complex128` (128 бит), CPU потребуется дополнительная обработка для операций, выходящих за пределы native word size.

Word size также напрямую влияет на **адресацию памяти**. 
32-битная система может адресовать до 4 ГБ памяти:

```
2^32 = 4,294,967,296 байт = 4 ГБ
```

В то время как 64-битная система теоретически может адресовать:

```
2^64 = 18,446,744,073,709,551,616 байт
```

Хотя большинство 64-битных систем не используют полный диапазон 2^64 из-за аппаратных ограничений, адресуемое пространство всё равно значительно больше, чем у 32-битной системы.

### Выравнивание Cache Line

Cache line — критически важная для производительности концепция на аппаратном уровне. 
**L1 cache** — самый маленький и быстрый кеш, ближайший к ядру CPU — хранит данные в блоках фиксированного размера, называемых cache line. 
Когда CPU читает из памяти или записывает в неё, он оперирует **целыми cache line**, а не отдельными байтами.

Размер cache line зависит от архитектуры процессора. 
Распространённые размеры: 32, 64, 128 или даже 256 байт:

```go
src/internal/cpu/cpu_loong64.go:  const CacheLinePadSize = 64
src/internal/cpu/cpu_ppc64x.go:   const CacheLinePadSize = 128
src/internal/cpu/cpu_riscv64.go:  const CacheLinePadSize = 64
src/internal/cpu/cpu_mips.go:     const CacheLinePadSize = 32
src/internal/cpu/cpu_wasm.go:     const CacheLinePadSize = 64
src/internal/cpu/cpu_arm.go:      const CacheLinePadSize = 32
src/internal/cpu/cpu_mips64x.go:  const CacheLinePadSize = 32
src/internal/cpu/cpu_mipsle.go:   const CacheLinePadSize = 32
src/internal/cpu/cpu_arm64.go:    const CacheLinePadSize = 128
src/internal/cpu/cpu_s390x.go:    const CacheLinePadSize = 256
src/internal/cpu/cpu_x86.go:      const CacheLinePadSize = 64
// Размер Cache Line на различных архитектурах (src/internal/cpu/cpu__.go)*
```

Мы фокусируемся на ARM64, поэтому будем использовать 128 байт как reference размер cache line. 
Рассмотрим следующий фрагмент:

```go
var arr [200]byte
x := arr[10]
```

Хотя осуществляется доступ только к одному байту, CPU загружает **весь 128-байтный cache line**, содержащий `arr[10]`.
Эта загрузка включает диапазон соседних значений:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212121515.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Этот подход извлекает выгоду из принципа, называемого **spatial locality** (пространственная локальность), который предполагает, что соседние адреса памяти, скорее всего, будут использованы в ближайшее время. Если значения вроде `arr[8]`, `arr[9]`, `arr[11]` или `arr[12]` будут запрошены вскоре после этого, они уже загружены в cache. Это уменьшает количество обращений к памяти и повышает производительность.

Но как определить, начинается ли cache line с `arr[8]` или с какой-то другой позиции?

Cache line **никогда** не начинается с произвольного байтового смещения. 
Он всегда начинается с адреса памяти, который является **кратным** размеру cache line. 
Это свойство называется **выравниванием**. 
Например, при 64-байтном cache line допустимые начальные адреса — 0, 64, 128 и так далее. 
Это выравнивание — фундаментальная часть устройства современных систем памяти.

Когда структура данных **пересекает границу cache line** — то есть начинается близко к концу одного cache line и продолжается в следующем — CPU должен загрузить оба cache line для доступа к ней. Это называется **split load** (или **split access**).

Например, предположим, что ваша система использует 128-байтные cache line, и вы обращаетесь к 32-байтной структуре, такой как наш struct `A`. 
Если эта структура начинается со смещения 112:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212121900.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Поскольку cache line выровнены по адресам, где `address % 128 == 0`, структура начинается за 16 байт до конца одного cache line и продолжается на 16 байт в следующий. В результате CPU должен загрузить **два cache line** для получения полной структуры.

Правильное выравнивание помогает предотвратить эту проблему. 
Если структура начинается по cache-aligned адресу и достаточно мала, чтобы целиком поместиться в один cache line, CPU может получить к ней доступ за **одну загрузку из памяти**
Это снижает latency и улучшает общую производительность.

Одна из стратегий избежания split access — **уменьшение размера структуры**. 
Вот оптимизированный layout структуры `A`:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212122306.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Когда структура меньше cache line и не пересекает его границу, она может быть загружена в L1 cache за одно обращение к памяти. Хотя несколько дополнительных операций с памятью могут показаться незначительными, эти маленькие неэффективности могут быстро накапливаться — особенно при обработке больших объёмов данных или при повторном обращении к памяти в performance-critical коде.

### False Sharing

> **Примечание.** Cache line могут приводить к проблемам производительности, таким как **false sharing**. Это происходит, когда несколько ядер CPU обращаются к **разным переменным**, которые случайно находятся в **одном cache line**, и хотя бы одно ядро выполняет запись.

Причина в том, что любая операция записи приводит к **инвалидации всего cache line** на всех ядрах, вызывая ненужные перезагрузки кеша на ядрах, которые обращаются только к несвязанным переменным в том же cache line. Это создаёт contention, значительно снижающее производительность в многопоточных приложениях

Рассмотрим два thread, выполняющихся на разных ядрах:

```go
type SharedData struct {
    x int32
    y int32
}
```

Если каждый thread часто обновляет либо `x`, либо `y`, и оба поля расположены в одном cache line, false sharing становится реальной проблемой. 
Обновление `x` инвалидирует весь cache line, затрагивая thread, обращающийся к `y`, хотя `y` не был изменён.

Для mitigation false sharing в Go можно ввести **padding между полями**:

```go
type SharedData struct {
    x int32
    _ [124]byte  // Padding для разделения x и y (при 128-байтном cache line)
    y int32
}
```

Такой дизайн гарантирует, что `x` и `y` хранятся в **разных cache line**, предотвращая cache invalidation между thread, обращающимися к ним конкурентно. 
Более переиспользуемый и надёжный подход — использование выделенного padded типа:

```go
type PaddedInt32 struct {
    value int32
    _     [124]byte  // Гарантируем, что общий размер 128 байт
}

type SharedData struct {
    x PaddedInt32
    y PaddedInt32
}
```

Этот приём padding также применяется в **стандартной библиотеке Go**. 
Например, в реализации `sync.Pool`:

```go
type poolLocal struct {
    poolLocalInternal
    // Предотвращает false sharing на распространённых платформах
    // 128 mod (cache line size) = 0.
    pad [128 - unsafe.Sizeof(poolLocalInternal{})%128]byte
}
// Padding poolLocal в sync.Pool (src/sync/pool.go)_
```

Padding гарантирует, что каждый экземпляр `poolLocal` выровнен по cache line, избегая interference между различными процессорами, обращающимися к своим данным.

Аналогичный паттерн присутствует в пакете runtime в реализации semaphore:

```go
type semTable [semTabSize]struct {
    root semaRoot
    pad  [cpu.CacheLinePadSize - unsafe.Sizeof(semaRoot{})]byte
}
```

### Выравнивание и Padding в структурах

**Выравнивание** (alignment) гарантирует, что адреса памяти соответствуют требованиям размера каждого поля. 
Когда размер типа равен или меньше word size CPU, его выравнивание обычно основано на его собственном размере. 
Например, 4-байтный `int32` выравнивается по 4-байтным границам. Это означает, что его адрес должен быть кратен 4 — например, 0, 4, 8 и так далее.

Тот же принцип применяется к меньшим типам. 
`byte` (1 байт) выравнивается по 1-байтным границам, а `int16` (2 байта) — по 2-байтным.

Для типов, **больших** чем word size системы, выравнивание обычно следует word size, а не фактическому размеру типа. На 64-битной системе типы вроде `complex128` (16 байт) или более крупные структуры обычно выравниваются по **8-байтным** границам. Это потому, что CPU оптимизирован для доступа к памяти порциями, соответствующими word size.

Для поддержания выравнивания компилятор вставляет **padding** между полями. 
Цель padding — обеспечить, чтобы каждое поле начиналось с правильно выровненного адреса памяти.

Структура `A` даёт хороший пример:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212123733.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Выравнивание происходит не случайно. Padding заполняет неиспользуемое пространство между полями для соответствия правилам выравнивания. Вот как это работает в структуре `A`:

- **A:** `byte` начинается со смещения 0, что полностью выровнено.
- **B:** `int32` должен начинаться на 4-байтной границе. Поскольку `A` использовал только 1 байт, компилятор вставляет **3 байта padding** после него.
- **C:** Ещё один `byte`, размещённый сразу после `B`, без дополнительного padding.
- **D:** `int64` требует 8-байтного выравнивания. После `C` добавляются **7 байт padding**, чтобы `D` начался на допустимой 8-байтной границе.
- **E:** Финальный `byte`, размещённый сразу после `D`.

В общей сложности структура `A` потребляет **32 байта памяти**. 
Значительная часть этого — padding, а не реальные данные. 
Это дополнительное пространство необходимо для удовлетворения требований выравнивания и помощи CPU в эффективном доступе к полям.



```go
// size = 1, align = 1
type A struct {
    A byte
}

// size = 12, align = 4
type B struct {
    A byte
    B int32
    C int32
}
```

Здесь `A` требует только 1-байтного выравнивания, поскольку содержит один `byte`. 
С другой стороны, `B` выравнивается по 4 байтам, потому что содержит два поля `int32`, каждое из которых требует 4-байтного выравнивания. Это различие напрямую влияет на то, как выделяется память для структуры в целом:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212124354.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

> ### Инструкции и загрузка из памяти
> 
> При доступе CPU к памяти следует учитывать **два уровня**: передача cache line, выполняемая аппаратно, и доступ к памяти на уровне инструкций, выполняемый CPU.
> 
> На **аппаратном уровне** подсистема памяти загружает полные cache line (часто 64 или 128 байт) из RAM в кеш CPU. Это выполняется cache controller.
> 
> На **уровне инструкций** операции вроде `MOV`, `LOAD` или `ADD` работают с порциями размером в word. На 64-битной системе чтение значения типа `int64` запускает 8-байтную load-инструкцию. Хотя был загружен 64-байтный cache line, CPU-инструкция обращается только к необходимым 8 байтам.

Рассмотрим структуру с двумя полями: `A` — 1 байт, и `B` — 8 байт. 
Если бы padding не добавлялся, layout памяти выглядел бы так:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212124822.png)

На определённых архитектурах такое невыравнивание не позволяет CPU прочитать все 8 байт `B` за одну операцию. Причина в том, что CPU спроектированы для загрузки данных порциями размером в word, что означает, что они ожидают загрузки 8 байт (на 64-битных системах) начиная с выровненных адресов (0, 8, 16 и т.д.).

Даже несмотря на то, что все 8 байт присутствуют в памяти, аппаратура не может обработать доступ как единую эффективную загрузку. Она должна извлечь перекрывающиеся области памяти за **несколько шагов** и собрать значение с помощью внутренних регистров. Этот процесс увеличивает latency и снижает производительность.

### Работа с пустыми структурами

Начиная с Go 1.22, итерация фиксированное количество раз стала более прямолинейной. Теперь можно использовать `for range` непосредственно с integer:

```go
for range 1000 {
    // ...
}
```


До добавления этой возможности существовал распространённый трюк с использованием массива пустых структур для достижения того же результата. Он позволял написать `for range` loop кратко:

```go
empty := [1000]struct{}{}
for range empty {
    // ...
}

unsafe.Sizeof(empty) // 0
```

Несмотря на то, что массив содержит 1000 элементов, он занимает **ноль байт** в памяти. Вызов `unsafe.Sizeof` подтверждает это. Это работает, потому что `struct{}` не имеет полей и, следовательно, не занимает места.

Пустые структуры остаются полезными помимо итерации. Распространённый паттерн использования — в map, таких как `map[T]struct{}`, для представления **set** (множеств), где важны только ключи. Этот подход экономит память и ясно выражает намерение.
Напротив, использование `map[T]bool` хранит дополнительные данные, которые часто не нужны.

Однако ситуация становится более тонкой, когда пустые структуры включены в более крупную структуру. Если пустая структура **стоит в конце**, размер содержащей структуры может быть больше ожидаемого:

```go
type First struct {
    B struct{}
    A int64
}

type Last struct {
    A int64
    B struct{}
}

// unsafe.Sizeof(First{}) // 8
// unsafe.Sizeof(Last{})  // 16
```

Хотя и `First`, и `Last` содержат одни и те же поля, и пустая структура `B` технически не занимает места, их **размеры различаются**. `First` — 8 байт, тогда как `Last` занимает 16 байт.

Это происходит потому, что Go вычисляет адреса полей, начиная с базового адреса структуры и добавляя offset каждого поля. В `First` пустая структура размещена в начале, поэтому её адрес совпадает с началом структуры. В `Last` пустая структура стоит в конце, что приводит к тому, что её адрес оказывается **прямо на границе** выделенного блока памяти.

Это создаёт потенциальную проблему. Если взять pointer на `Last.B`, он будет указывать **за пределы** памяти, зарезервированной для `Last`. В Go это может привести к unsafe или undefined поведению из-за потенциального перекрытия с несвязанной памятью.

Чтобы избежать этого, компилятор добавляет padding к структуре, гарантируя, что она заканчивается на безопасной границе. Эта корректировка увеличивает размер `Last` до 16 байт:

![](../../../../assets/obsidian-images/Pasted%20image%2020260212145128.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Имейте в виду: zero-size значения не ограничиваются пустыми структурами. 
Массивы с нулём элементов (например, `[0]T`) также трактуются как имеющие нулевой размер — всё, что не резервирует пространства в памяти, попадает в эту категорию.

### Zero-Size значения на Stack

Ещё одно интересное поведение zero-size значений — несколько переменных могут **разделять один и тот же адрес памяти**. Это оптимизация компилятора, предназначенная для уменьшения использования памяти при работе со значениями, не требующими хранения.

Хотя это полезно для понимания, такое поведение является деталью реализации и на него **не следует полагаться** во всех ситуациях:

```go
func main() {
    var a, b = struct{}{}, struct{}{}
    var c int = 100

    println("a:", &a) // 0x1400004e747
    println("b:", &b) // 0x1400004e747
    println("c:", &c) // 0x1400004e747
}
```

Здесь и `a`, и `b` — пустые структуры. 
Поскольку они не занимают памяти, компилятор переиспользует **один и тот же адрес** для обоих на stack. Дополнительные zero-size переменные также могут получить этот же адрес.

![](../../../../assets/obsidian-images/Pasted%20image%2020260212150159.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Теперь посмотрите на переменную `c`, которая является `int`. 
Несмотря на то, что она служит другой цели и хранит реальные данные, она, похоже, разделяет тот же адрес, что и `a` и `b`. На самом деле `c` занимает следующие 8 байт памяти, начиная с этого адреса, тогда как `a` и `b` не занимают места вообще.

### Zero-Size значения на Heap

Выделение zero-size значений на heap включает немного другую стратегию. 
Используя директиву `//go:noinline` и возвращая pointer, вы гарантируете, что эти значения переживут scope функции, где они были созданы:

```go
type Empty struct{}

//go:noinline
func GetZeroSizeStruct() *Empty {
    return &Empty{}
}

//go:noinline
func GetZeroSizeArray() *[0]int {
    return &[0]int{}
}

func main() {
    // heap
    println("zero-size struct:", GetZeroSizeStruct()) // 0x100859a40
    println("zero-size array:", GetZeroSizeArray())    // 0x100859a40
}
```

В этом примере оба heap-allocated zero-size значения указывают на **один и тот же адрес**. Это не совпадение — Go runtime использует **предопределённый адрес памяти** для всех zero-size heap-allocations. Это специальное местоположение называется **«zero base address»** (`zerobase`).

Если вам интересны внутренние детали, этот адрес управляется переменной `zerobase` в пакете runtime. Это часть оптимизаций runtime Go для избежания ненужного использования памяти при выделении значений, не требующих пространства.

## Xulosa

**1. Struct field tartibi — bu faqat stil emas, bu performance.** Ko'pchilik dasturchilar struct field'larini mantiqiy tartibda yozadi (masalan, `Name`, `Age`, `Email`). Lekin bu bob ko'rsatdiki, field tartibini **alignment bo'yicha optimallashtirish** memory usage'ni **ikki baravar** kamaytirishi mumkin (32 → 16 bayt). Real-world loyihalarda, ayniqsa katta hajmdagi slice'larda (`[]MyStruct`) bu farq sezilarli bo'ladi.

**2. Padding — ko'rinmas, lekin haqiqiy narx.** Kompilator har bir field'ni to'g'ri alignment'ga joylashtirish uchun **avtomatik padding** qo'shadi. Bu padding kodda ko'rinmaydi, lekin memory'da bor. `unsafe.Sizeof()` yordamida struct o'lchamini tekshirish — yaxshi odat. Katta struct'larda `int64` va `float64` kabi katta field'larni boshiga qo'yib, `byte` va `bool` kabi kichik field'larni oxiriga qo'yish — oddiy, lekin ta'sirli optimizatsiya.

**3. Empty struct — nol memory, lekin nozik holatlar bor.** `struct{}` nol bayt egallaydi — bu `map[T]struct{}` set pattern uchun ideal. Lekin **struct oxirida** turgan empty struct'ga Go qo'shimcha padding qo'shadi (`Last` struct misoli — 8 o'rniga 16 bayt). Sababi: agar pointer empty struct'ga olinsa va u struct'ning eng oxirida tursa, bu pointer struct'dan **tashqariga** ko'rsatishi mumkin. Bu Go'ning memory safety garantiyasini buzadi.

**4. Embedding — bu inheritance emas, bu composition.** Go'da `Boat` struct'ga `Vehicle` embed qilsangiz, `Boat.Operate()` ichida `Surface()` chaqirilganda **Vehicle.Surface()** ishlatiladi, Boat.Surface() emas. Bu OOP'dagi virtual method yoki method overriding'dan tubdan farq qiladi. Embedding — bu **method forwarding**, receiver o'zgarmaydi. Shu sababli Go'da «parent class method'ini override qilish» tushunchasi yo'q.

**5. Cache line va false sharing — concurrent dasturlarda muhim.** Agar ikkita goroutine bitta struct'ning turli field'lariga yozsa va bu field'lar **bitta cache line**'da joylashgan bo'lsa, CPU har safar butun cache line'ni invalidate qiladi. Bu false sharing — hech qanday real data dependency bo'lmasa ham, performance dramatik tushadi. Yechim: field'lar orasiga `[128]byte` padding qo'yish. Bu pattern Go standart kutubxonasida ham ishlatiladi (`sync.Pool`, `runtime/sema.go`).

**6. Pointer auto-dereference — qulay, lekin tushunib ishlating.** `b2.ID` yozganingizda, Go avtomatik `(*b2).ID` ga aylantiradi. Bu qulay, lekin `b2` nil bo'lsa, runtime'da **nil pointer dereference** panic bo'ladi. Kompilator `ODOT` ni `ODOTPTR` ga aylantiradi va SSA'da pointer + offset + load operatsiyalarini generatsiya qiladi.

**Amaliy maslahatlar:**

- Struct field'larini **katta type'lardan kichikka** tartiblab joylashtiring — padding kamayadi
- `unsafe.Sizeof()` va `unsafe.Alignof()` bilan struct o'lchamini tekshiring
- Concurrent kod uchun shared struct'larda **cache line padding** ni o'ylab ko'ring
- Empty struct'ni har doim struct **boshiga** qo'ying, oxiriga emas
- Tag'lar comparison'da hisobga olinadi, lekin assignment/conversion'da emas — bu real-world integration'da foydali
- Embedding'dan foydalanganda, method resolution **receiver type scope**'da ishlashini unutmang
