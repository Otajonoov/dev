
По своей сути, interface — это **контракт**: коллекция сигнатур методов, которые тип может реализовать. Если тип предоставляет реализации для всех методов, определённых в interface, говорят, что он **удовлетворяет** этому interface. 
Эти сигнатуры методов должны точно совпадать по имени, типам параметров и типам возвращаемых значений — хотя сами имена параметров не являются частью контракта и могут отличаться.

---

Interface определяет набор ожиданий к поведению. 
Он указывает, какие методы тип должен предоставить, но **не как** эти методы реализованы:

```go
type Worker interface {
    Work()
}
```

Любой тип — будь то struct или пользовательский тип — который реализует метод `Work()`, считается `Worker`.

Отличительная черта системы interface в Go, по сравнению с языками вроде Java или C#, — использование **неявной реализации** (implicit implementation). Нет необходимости явно заявлять, что тип удовлетворяет interface. Пока присутствуют необходимые методы, тип **автоматически** соответствует interface:

```go
type Developer struct {
    Name  string
    Skill string
}

func (d Developer) Work() {
    fmt.Printf("%s is coding in %s\n", d.Name, d.Skill)
}

type Designer struct {
    Name string
    Tool string
}

func (d Designer) Work() {
    fmt.Printf("%s is designing with %s\n", d.Name, d.Tool)
}
```

И `Developer`, и `Designer` удовлетворяют interface `Worker` просто благодаря реализации метода `Work()` с совпадающей сигнатурой. 
Это позволяет значениям любого из типов быть присвоенными переменной типа `Worker`:

```go
var work Worker
work = Developer{"Alice", "Go"}
work.Work()
work = Designer{"Bob", "Figma"}
work.Work()
```

Что отсутствует? Нет явного объявления вроде «Developer implements Worker». 

---

Например, рассмотрим пакет `bufio`. 
При выполнении буферизованного чтения он ожидает input, удовлетворяющий interface `io.Reader`:

```go
package bufio

// NewReader returns a new [Reader] whose buffer has the default size.
func NewReader(rd io.Reader) *Reader {
    return NewReaderSize(rd, defaultBufSize)
}
```

Вашему типу не нужно напрямую импортировать пакет `io` для совместимости. 
Ему достаточно реализовать метод `Read([]byte) (int, error)`. 
Go обработает остальное автоматически:

```go
type MyReader struct {}

func (r MyReader) Read(p []byte) (n int, err error) {
    // Пользовательская логика чтения
}

func main() {
    reader := bufio.NewReader(MyReader{})
}
```

В этом примере `bufio` принимает `MyReader` без проблем. 
Конкретный тип не важен — важно то, что он предоставляет метод, соответствующий interface `io.Reader`.

Тем не менее, бывают ситуации, когда вы хотите **явно утвердить**, что тип удовлетворяет определённому interface. Это обычно возникает, когда компилятор не может вывести связь, например, при работе с reflection или кодом, генерируемым во время выполнения.

В таких сценариях вы можете принудительно проверить контракт простой проверкой:

```go
import "io"

// Синтаксис: var _ <interface> = (*type)(nil)
var _ io.Reader = (*MyReader)(nil)
```

Эта строка говорит компилятору: «Я ожидаю, что `*MyReader` реализует `io.Reader`. 
Проверь это сейчас.» Если реализация неполная или некорректная, Go выдаст ошибку компиляции:

```
cannot use (*MyReader)(nil) (value of type *MyReader) as io.Reader value in variable
declaration: *MyReader does not implement io.Reader (missing method Read)
```

Теперь, когда основы ясны, вернёмся к interface `Worker` и рассмотрим его в реальном сценарии:

```go
func main() {
    var workers = []Worker{
        Developer{Name: "Alice", Skill: "Go"},
        Designer{Name: "Bob", Tool: "Photoshop"},
    }

    for _, w := range workers {
        w.Work()
    }
}
```

Выполнение этого кода производит следующий вывод:

```
Alice is coding in Go
Bob is designing with Photoshop
```

Здесь slice `workers` содержит значения разных типов: `Developer` и `Designer`. 
Оба реализуют interface `Worker`, демонстрируя **полиморфизм** в действии.
Interface позволяют писать код, который единообразно работает с различными типами, при этом каждый тип определяет своё уникальное поведение.

Go **не ведёт** мастер-список всех типов, реализующих данный interface. 
Если тип никогда не используется с interface, между ними нет связи. 
Вместо этого, когда тип фактически присваивается interface, компилятор проверяет, соответствует ли он требованиям interface. Если да, Go генерирует то, что называется **interface table** (`itab`). Эта структура связывает interface с конкретным типом и ассоциирует необходимые реализации методов.

Чтобы понять, как это работает за кулисами, давайте кратко взглянем на assembly, генерируемый из предыдущего примера:

| Go Code                                    | Go Assembly                                        |
| ------------------------------------------ | -------------------------------------------------- |
| `Developer{Name: "Alice", Skill: "Go"}`    | `MOVD $type:main.Developer(SB), R0`                |
|                                            | `MOVD $go:itab.main.Developer,main.Worker(SB), R2` |
| `Designer{Name: "Bob", Tool: "Photoshop"}` | `MOVD $type:main.Designer(SB), R0`                 |
|                                            | `MOVD $go:itab.main.Designer,main.Worker(SB), R2`  |

Символы `$type` и `$go:itab` встроены непосредственно в скомпилированный бинарник. Они **не вычисляются** во время выполнения программы; они разрешаются на этапе компиляции. К моменту запуска вашего кода Go уже установил связи между конкретными типами и interface, которые они реализуют. Во время выполнения эти связи просто **используются по ссылке**.

С этим механизмом в уме рассмотрим практическое использование interface:

```go
func announce(s Worker) {
    s.Work()
}

func main() {
    announce(Developer{Name: "Alice", Skill: "Go"})
    announce(Designer{Name: "Bob", Tool: "Photoshop"})
}
```

Функция `announce` принимает любое значение, удовлетворяющее interface `Worker`. 
Это способствует гибкости при сохранении ясности контракта — любой тип, реализующий `Work()`, допустим здесь.
