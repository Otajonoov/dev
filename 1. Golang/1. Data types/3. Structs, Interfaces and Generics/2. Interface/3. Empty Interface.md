
Ещё одна важная концепция — **empty Interface**, записываемый как `interface{}` или `any`. 
	Этот interface не требует никаких методов, поэтому может хранить значения **любого типа**. Даже начинающие часто используют его, не осознавая.

Хорошо известный пример — функция `fmt.Println`:

```go
package fmt

func Println(a ...any) (n int, err error) {
    return Fprintln(os.Stdout, a...)
}
```

Здесь `any` — просто более лаконичный alias для `interface{}`, определённый так:

```go
type any = interface{}
```

Хотя пустой interface предлагает максимальную гибкость, он также вводит ограничения. 
Как только значение сохранено в `interface{}`, его конкретный тип **больше не доступен напрямую**.

Чтобы осмысленно взаимодействовать с ним, нужно извлечь его оригинальный тип с помощью **type assertion** или **type switch**:

```go
func doSomething(i interface{}) {
    switch v := i.(type) {
    case string:
        fmt.Println("String:", v)
    case int:
        fmt.Println("Int:", v)
    default:
        fmt.Println("Unknown type")
    }
}
```

Функция `doSomething` проверяет underlying тип во время выполнения и действует соответственно.

## Внутренняя структура пустых интерфейсов

Одна концепция, часто вызывающая путаницу в Go — так называемый **«nil, но не nil» interface**.

Рассмотрим пример:

```go
func main() {
    var x interface{}
    var y *int = nil

    x = y

    if x != nil {
        fmt.Println("x != nil")
    } else {
        fmt.Println("x == nil")
    }
    fmt.Println(x)
}
```

На первый взгляд можно ожидать, что `x` будет nil после присвоения ему nil pointer `y`. 
Однако вывод:

```
"x != nil"
```

**Почему?** Даже несмотря на то, что `y` равен nil, `x` хранит **больше**, чем просто nil-значение — он хранит **тип** `*int` вместе с nil pointer. Эта информация о типе делает всё различие. Поскольку `x` содержит конкретный тип, Go трактует его как **не-nil**.

Пустой interface (`interface{}`) считается **по-настоящему nil** только когда и тип, и значение **не установлены**. Чтобы увидеть это в действии, можно исследовать сырое содержимое interface с помощью `println`:

```go
func main() {
    var x interface{}
    println("prev:", x)

    var y *int = nil
    x = y
    println("next:", x)
}
```

```
prev: (0x0,0x0)
next: (0x1024174a0,0x0)
```

Изначально `x` пуст — нет типа, нет значения — представлен как `(0x0,0x0)`. 
После присвоения `y`, `x` теперь хранит **две вещи**: pointer на type descriptor для `*int` и второй pointer на (nil) значение. Даже несмотря на то, что значение nil, **наличие типа** означает, что interface не пуст. Go трактует его как не-nil, потому что он содержит информацию о типе.

Теперь рассмотрим более прикладной пример с пользовательскими типами:

```go
type Developer struct {
    Name  string
    Skill string
}

type Designer struct {
    Name string
    Tool string
}

func main() {
    var workers = []interface{}{
        Developer{Name: "Alice", Skill: "Go"},
        Designer{Name: "Bob", Tool: "Photoshop"},
        Developer{Name: "Charlie", Skill: "Python"},
    }

    for i := range workers {
        println(workers[i])
    }
}
```

Этот код даёт представление о том, как значения interface представлены внутренне:

```
(0x1047ca100,0x140000606a0)  # Alice
(0x1047ca060,0x14000060718)  # Bob
(0x1047ca100,0x140000606f8)  # Charlie
```

Обратите внимание, что первый pointer **одинаков** для Alice и Charlie (`0x1047ca100`). 
Это подтверждает, что type descriptor для `Developer` **переиспользуется** для всех экземпляров.

Каждое значение interface состоит из **двух pointer**. 
Чтобы понять, что эти pointer представляют, обратимся к исходному коду Go:

```go
type eface struct {
    _type *_type
    data  unsafe.Pointer
}
// eface (src/runtime/runtime2.go)_
```

Под капотом пустой interface реализован как **двухсловная структура**. 
- `_type`, указывает на type descriptor
- фактические данные

Go поддерживает **единый общий descriptor** для каждого конкретного типа:

![](../../../../assets/obsidian-images/Pasted%20image%2020260213124555.png)


---
---

```go
a := 10
var b interface{} = a
a = 20
fmt.Println(a) // 20
fmt.Println(b) // всё ещё 10
```

Здесь изменение `a` на 20 не влияет на `b`, потому что `b` хранит **отдельную копию** оригинального значения `a`. Это прямолинейно, но не универсально последовательно. Содержит ли interface копию значения или ссылку, зависит от **типа**, который присваивается.

Чтобы увидеть это яснее, рассмотрим, что происходит при присвоении integer interface:

```go
func main() {
    var a int = 5
    var var1 interface{} = a
    var var2 interface{} = a
    println("var1:", var1)
    println("var2:", var2)

    var const1 interface{} = 5
    var const2 interface{} = 5
    println("const1:", const1)
    println("const2:", const2)
}
```

```
var1:   (0x100265320,0x1400004e710)
var2:   (0x100265320,0x1400004e708)
const1: (0x100265320,0x10025a9d8)
const2: (0x100265320,0x10025a9d8)
```

Этот вывод раскрывает, что type descriptor для `int` (`0x100265320`) **переиспользуется** во всех случаях. Это ожидаемо — Go глобально переиспользует один и тот же descriptor для каждого типа.

Наиболее интересная деталь кроется в **data pointer**. 
Для `var1` и `var2` data-адреса **различаются**. 
Но для `const1` и `const2` они **идентичны**:

![](../../../../assets/obsidian-images/Pasted%20image%2020260213153737.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Это различие подчёркивает, как Go обрабатывает память под капотом:

- `var1` и `var2` имеют **разные** data pointer (`0x1400004e710` и `0x1400004e708`), указывая, что каждое присваивание приводит к **отдельной копии** значения.
- `const1` и `const2`, напротив, оба указывают на **один и тот же** адрес (`0x10025a9d8`), хотя были присвоены независимо с одной и той же константой.

Pointer `eface.data` для констант также **заметно меньше**, чем для переменных, и даже меньше, чем pointer type descriptor (`0x100265320`). Это потому, что константы ссылаются на **предварительно выделенную память** в read-only секции бинарника.

---
---

Go следует определённой стратегии при присвоении значений полю `data` interface:

1. Если конкретное значение является **pointer**, Go использует этот pointer напрямую без копирования данных. Это оптимизация, которую стоит запомнить.
2. Если значение имеет **нулевой размер**, `eface.data` указывает на специальное расположение `zerobase`, которое также используется для пустых struct.
3. Для **однобайтовых типов** вроде `uint8`, `int8` или `bool`, `eface.data` может указывать на общее расположение в памяти `staticuint64s`
4. Если значение — **read-only global** (такое как литерал или константа), pointer ссылается на его фиксированное расположение в read-only памяти бинарника.
5. Если значение **маленькое** (≤ 1024 байт) и не escapes на heap, компилятор создаёт временную копию на **stack**, и `eface.data` указывает на неё.
6. Для **больших значений** или значений, которые escape, Go выделяет новую копию на **heap**.


Довольно много теории, так что давайте посмотрим, как это работает на практике:

```go
func main() {
    // eface.data использует оригинальный pointer напрямую
    pointer := new(int)
    pointerEface := interface{}(pointer)
    println("pointer:", pointer)
    println("pointerEface:", pointerEface)

    // eface.data указывает на zerobase для zero-size типов
    println("zeroSizeArrayEface:", interface{}([0]int{}))
    println("emptyStructEface:", interface{}(struct{}{}))
}
```

```
pointer:            0x1400005e710
pointerEface:       (0x100e0fce0,0x1400005e710)
zeroSizeArrayEface: (0x100e12580,0x100ea1a20)
emptyStructEface:   (0x100e13720,0x100ea1a20)
```

> ![](../../../../assets/obsidian-images/Pasted%20image%2020260213155810.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

В первой части `pointerEface` **напрямую ссылается** на адрес памяти pointer, как и ожидалось.

Во втором и третьем выводах data pointer (`eface.data`) указывает на `runtime.zerobase`. 
Это происходит потому, что `[0]int{}` и `struct{}{}` занимают ноль байт. 
Компилятор знает, что нет причин выделять память для значений, не содержащих данных.
