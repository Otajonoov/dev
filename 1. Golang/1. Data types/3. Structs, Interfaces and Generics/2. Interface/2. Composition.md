
В Go embedding interface проявляется в двух основных формах: 

- embedding одного interface в другой
- embedding interface в struct

### 1. Embedding interface в interface

Embedding interface в другие interface позволяет строить более гибкие и модульные дизайны путём **композиции** меньших, целенаправленных interface. 

```go
type Speaker interface {
    Speak() string
}

type Mover interface {
    Move() string
}

// Embedding interface
type Animal interface {
    Speaker
    Mover
}
```

Здесь interface `Animal` содержит embedded `Speaker` и `Mover`. 
Чтобы удовлетворить `Animal`, тип должен реализовать **оба** метода: `Speak()` и `Move()`. 

Этот паттерн широко применяется в стандартной библиотеке Go, особенно в пакете `io`:

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type Closer interface {
    Close() error
}

// ReadWriter группирует базовые методы Read и Write.
type ReadWriter interface {
    Reader
    Writer
}

// ReadCloser группирует базовые методы Read и Close.
type ReadCloser interface {
    Reader
    Closer
}

// WriteCloser группирует базовые методы Write и Close.
type WriteCloser interface {
    Writer
    Closer
}

// ReadWriteCloser группирует базовые методы Read, Write и Close.
type ReadWriteCloser interface {
    Reader
    Writer
    Closer
}
```

Ситуация становится ещё интереснее, когда embedded interface включают методы **с одинаковым именем**. Это ведёт себя аналогично embedding struct, но вводит дополнительные соображения:

```go
type Speaker interface {
    Speak() string
}

type Singer interface {
    Speak() string
}

type DoubleSpeak interface {
    Speaker
    Singer
}
```

### 2. Embedding interface в struct

```go
type Speaker interface {
    Speak() string
}

type Dog struct {
    Speaker
    Name string
}
```

Здесь struct `Dog` содержит embedded interface `Speaker`. 
На первый взгляд это выглядит разумно — если `Dog` может говорить, embedding `Speaker` должен это подразумевать.

Однако ситуация меняется при попытке использования:

```go
func main() {
    dog := Dog{Name: "Molly"}
    
    // panic: runtime error: invalid memory address or nil pointer dereference
    fmt.Println(dog.Speak())
}
```

Вместо дружелюбного лая программа **аварийно завершается** с runtime panic. 
Проблема в том, как Go обрабатывает embedded interface внутри struct.

Embedded interface **не приносит с собой реализацию**. 
Если конкретное значение не присвоено interface, оно по умолчанию равно `nil`. 
Эта тонкая деталь критична — без фактического типа, присвоенного interface, вызов его метода приводит к **nil pointer dereference**.

Исправим проблему. Одно решение - реализовать метод `Speak` непосредственно на типе `Dog`:

```go
func (d Dog) Speak() string {
    return "Woof!"
}

func main() {
    dog := Dog{}
    fmt.Println(dog.Speak()) // Output: Woof!
}
```

Эта версия работает и даёт ожидаемый вывод. 
Но это может привести к недопониманию. 
Может показаться, что embedding interface **требует** от struct реализации методов interface. 
На самом деле это не так.

Исходная проблема не в самом `Dog` — она в embedded поле `Speaker`, которое **остаётся неинициализированным**:

Добавляя `Speak` непосредственно к `Dog`, вы **не удовлетворяете** embedded interface.
Вы просто определяете метод с тем же именем, который **shadow** ожидаемый от `Speaker`. 
Это исправляет panic в данном контексте, но не решает underlying проблему.

Если попытаться вызвать `Speak` через поле `Speaker`, оригинальный panic возвращается:

```go
func main() {
    dog := Dog{}
    
    // Output: Woof!
    fmt.Println(dog.Speak())
    
    // panic: runtime error: invalid memory address or nil pointer dereference
    fmt.Println(dog.Speaker.Speak())
}
```

Для правильного решения нужно присвоить **конкретную реализацию** embedded interface. Иначе поле остаётся nil interface и завершится аварийно при выполнении:

```go
type DogVoice struct {}

func (w DogVoice) Speak() string {
    return "Grrr!"
}

func main() {
    dog := Dog{Speaker: DogVoice{}}
    fmt.Println(dog.Speak())         // Output: Grrr!
    fmt.Println(dog.Speaker.Speak()) // Output: Grrr!
}
```

Теперь код выполняется безопасно. 
Поле `Speaker` инициализировано фактической реализацией (`DogVoice{}`), устраняя риск nil dereference.

Стоит отметить, что предыдущее поведение — когда `Dog`, казалось, отвечал на `Speak()` — было вызвано не enforcement'ом interface. 
Это было просто **method promotion**. 
Метод, определённый на `Dog`, имел приоритет и маскировал отсутствие фактической реализации interface.

Истинная ценность embedding interface в struct заключается:
- предоставлении struct доступа к методам interface
- обеспечении возможности его использования там, где ожидается этот interface

Рассмотрим пакет `context` Go как практический пример. 
Interface `context.Context` embedded в несколько внутренних struct, таких как `valueCtx`, `cancelCtx` и `stopCtx`:

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key any) any
}

type valueCtx struct {
    Context
    key, val any
}

func WithValue(parent Context, key, val any) Context {
    // Детали реализации...
    return &valueCtx{parent, key, val}
}

type cancelCtx struct {
    Context
    mu   sync.Mutex
    done atomic.Value
    // Дополнительные поля...
}
```

Этот подход работает, потому что interface `Context` устанавливает контракт, тогда как типы вроде `valueCtx` и `cancelCtx` управляют внутренней логикой и состоянием.

Это сильный пример **разделения ответственностей** (separation of concerns): interface определяет, какое поведение требуется, а struct определяет, **как** это поведение осуществляется.
