
Теперь, рассмотрев пустые interface, перейдём к **непустым interface**, где всё становится немного сложнее. Часто описывают interface как пару «тип» и «значение» — и это справедливо для пустых interface. Но непустые interface вводят **дополнительный слой**:

```go
type Worker interface {
    Work()
}

func main() {
    println(interface{}(Developer{}))  // Пустой interface
    println(Worker(Developer{}))       // Непустой interface
}
```

```
(0x100eab960,0x14000060718)
(0x100eb27b8,0x140000606f8)
```

Оба interface-значения оборачивают один и тот же конкретный тип: `Developer`. 
Однако **первые pointer различаются**.

Это различие не означает, что type descriptor для `Developer` (`$type:Developer`) изменился. Вместо этого непустые interface используют **другой механизм** хранения информации о типе.

В непустом interface первый pointer не ссылается напрямую на type descriptor. 
Вместо этого он указывает на **interface table**, или `itab`.

```go
type iface struct {
    tab  *itab
    data unsafe.Pointer
}
```

Кто-то может спросить, почему Go просто не добавит больше полей в struct `iface` — возможно, что-то вроде:

```go
type iface struct {
    tab   *itab
    _type *_type
    data  unsafe.Pointer
}
```

Ответ кроется в **последовательном представлении** interface в Go. 
Хотя непустые interface содержат больше информации (такой как pointer на методы), язык **не делает различия** между пустыми и непустыми interface по размеру структуры. 
Оба должны занимать одинаковый размер — **16 байт**.
Добавление дополнительных полей нарушило бы эту последовательность.

Когда значение вроде `Dog` присваивается переменной типа `Speaker`, компилятор строит структуру `iface`, представляющую этот interface. 
Поле `tab` затем указывает на `itab`, который включает подробную информацию о типе `Dog` — конкретно, как он удовлетворяет метод `Speak`, определённый в interface `Speaker`.

Думайте об `itab` как о **runtime-справочнике**, который сообщает Go, как именно конкретный тип удовлетворяет требования interface:

```go
type itab struct {
    inter *interfacetype  // на 0 байте
    _type *_type          // на 8 байте
    hash  uint32          // на 16 байте
    _     [4]byte         // на 20 байте
    fun   [1]uintptr      // на 24 байте
}
```

`itab` служит **посредником**. 
Как и пустой interface, он включает pointer на type descriptor, но также несёт дополнительные metadata, помогающие runtime вызывать правильные реализации методов.

>![](../../../../assets/obsidian-images/Pasted%20image%2020260213162309.png)

Вот что представляет каждое поле:

- **inter:** Указывает на `interfacetype`, который хранит metadata об interface, включая его обязательные методы.
- **_type:** Descriptor для конкретного типа, присваиваемого interface.
- **hash:** Кешированное значение `_type.hash`, используемое для быстрых поисков, особенно в type switch.
- **fun:** Массив function pointer, которые непосредственно связаны с методами, необходимыми для удовлетворения interface.

Среди них массив `fun` **особенно важен**. 
Начиная с byte offset 24, он содержит фактические function pointer для методов, требуемых interface. Эти pointer используются при **method dispatch** во время выполнения

Рассмотрим конкретный пример:

```go
type Worker interface {
    Work()
}

var worker = Worker(
    Developer{
        Name:  "Aiden",
        Skill: "Go",
    },
)

func main() {
    worker.Work()
}
```

>![](../../../../assets/obsidian-images/Pasted%20image%2020260213165753.png)

Разбор ключевых assembly-инструкций:

- `MOVD main.worker(SB), R1`: Загружает itab pointer из структуры `iface` в регистр R1.
- `MOVD main.worker+8(SB), R0`: Загружает data pointer (конкретное значение `Developer`) в R0.
- `MOVD 24(R1), R1`: Извлекает method pointer из слота `fun[0]` в itab, соответствующий методу `(*Developer).Work`.
- `CALL (R1)`: Вызывает метод, используя function pointer.

Вот как Go выполняет **dynamic dispatch**. 
Запись `fun[0]` в itab предоставляет прямую связь с правильной реализацией метода для конкретного типа. Так runtime **перебрасывает мост** между контрактом interface и фактической логикой метода.

Чтобы продемонстрировать ту же идею более доступным способом, можно полностью полагаться на код Go, используя техники из пакета `unsafe`:

```go
type iface struct {
    tab  *itab
    data unsafe.Pointer
}

type itab struct {
    inter unsafe.Pointer
    _type unsafe.Pointer
    _     uint32
    _     [4]byte
    fun   [1]unsafe.Pointer
}

func main() {
    dev := Developer{Name: "Alice", Skill: "Go"}
    worker := Worker(dev)

    // Доступ к внутренностям iface
    ifaceData := *(*iface)(unsafe.Pointer(&worker))
    println("- iface.tab.inter:", ifaceData.tab.inter)
    println("- iface.tab._type:", ifaceData.tab._type)
    println("- iface.tab.fun[0]:", unsafe.Pointer(ifaceData.tab.fun[0]))

    // Проверка фактического адреса метода
    fmt.Printf("- (*Developer).Work address: %p\n", (*Developer).Work)
}
```

Вывод даёт нам ещё более близкий взгляд:

```
- iface.tab.inter:  0x1007a6f20
- iface.tab._type:  0x1007ae620
- iface.tab.fun[0]: 0x10076e050
- (*Developer).Work address: 0x10076e050
```

Как и ожидалось, запись `fun[0]` внутри struct `itab` указывает на **тот же адрес памяти**, что и метод `(*Developer).Work`: `0x10076e050`.

### Поле fun: массив переменной длины

Есть нечто тонкое, но важное в поле `fun`.

Хотя объявлено как `[1]uintptr`, оно **не ограничено** одним методом. Этот паттерн сигнализирует, что массив имеет **переменную длину**. 
Количество записей зависит от того, сколько методов определяет interface.

Эта техника, известная как **variable-length array** или **runtime-sized array**, пришла из C и аналогичных низкоуровневых языков, которые сформировали internal'ы Go. 
Вы часто увидите её, когда количество элементов неизвестно на этапе компиляции. Объявление с одним элементом отмечает **начальную точку**, но в памяти массив **расширяется дальше**.

Например, если interface определяет три метода, фактический layout в памяти будет содержать три function pointer. `[1]` просто указывает на базу массива.

Go runtime использует **pointer arithmetic** для безопасного доступа к полному массиву:

```go
ni := len(inter.Methods)
...
methods := (*[1 << 16]unsafe.Pointer)(unsafe.Pointer(&m.Fun[0]))[:ni:ni]
```

Это позволяет runtime трактовать slice `fun` так, как если бы он был объявлен с правильным количеством записей изначально.

### Автоматическая обработка Receiver-метода

Интересная деталь, которая может быть неочевидной — как поле `fun[0]` в itab указывает на метод `(*Developer).Work`, хотя `Developer.Work` определён с **value receiver**:

```go
//go:noinline
func (d Developer) Work() {
    println("%s is working", d.Name)
}
```

Что здесь происходит?

Компилятор Go **автоматически создаёт** wrapper-метод с pointer receiver, который внутренне вызывает оригинальный value receiver. Это поведение не ограничено interface — оно часть общей обработки методов Go. Вот почему вы можете вызывать метод, определённый на value receiver, используя pointer на тип без ручного разыменования:

```go
d := &Developer{Name: "John"}
d.Work() // Работает, хотя Work() имеет value receiver
```

Компилятор обрабатывает конвертацию под капотом, гарантируя, что вызов метода остаётся валидным независимо от того, является ли receiver value или pointer.

### Как Go строит Itab

Большинство записей itab **генерируются на этапе компиляции**. Например:

```go
var d Worker = Developer{}
```

Когда значение `Developer` присваивается interface `Worker`, компилятор подготавливает itab, связывающий два типа (`itab: {Developer, Worker}`) во время компиляции.

Однако бывают ситуации, когда компилятор **не может** определить связь тип-interface заранее. В таких случаях itab должен создаваться **во время выполнения**. 
Это происходит при использовании reflection, динамической генерации типов или конвертации interface с несколькими уровнями indirection.

Compile-time записи itab хранятся в выделенной секции бинарника — **`.itablink`**. 
Для подтверждения рассмотрим layout бинарника:

```bash
$ objdump -h main

Sections:

  0 __text          000665e4 0000000100001000 TEXT
  1 __symbol_stub1  000001f8 0000000100067600 TEXT
  2 __rodata        0001ec44 0000000100067800 DATA
  3 __rodata        00012e40 0000000100088000 DATA
  4 __typelink      000003f4 000000010009ae40 DATA
  5 __itablink      00000018 000000010009b240 DATA
  6 __gosymtab      00000000 000000010009b258 DATA
  7 __gopclntab     000528d0 000000010009b260 DATA
  ...
```

Секция `.itablink` имеет размер `0x18` (24 байта) и расположена по адресу `0x000000010009b240`. Поскольку каждый pointer занимает 8 байт на macOS (Darwin), эта секция может хранить ровно **три** записи itablink (24 / 8 = 3).

Мы определили только один interface (`Worker`) и использовали его с типом `Developer`, так почему три записи?

```
$ go tool nm theanatomyofgo | grep go:itab

10009a7d8 R go:itab.runtime.errorString,error
10009a7f8 R go:itab.runtime.plainError,error
```

Помимо ожидаемой записи Developer → Worker, есть две дополнительные: `errorString` и `plainError`, обе реализующие interface `error`.

Эти две **всегда включаются** в бинарники Go, даже когда программа не использует interface `error` напрямую. Они являются частью runtime и поддерживают базовую обработку ошибок.

При подключении внешних пакетов количество записей itab **увеличивается**. 
Например, замена `println` на `fmt.Println` требует импорта пакета `fmt`, что добавляет множество записей:

```

1000ecb08 R go:itab.*fmt.pp,fmt.State
1000ecac0 R go:itab.*internal/bisect.parseError,error
1000ecaa0 R go:itab.*internal/godebug.runtimeStderr,internal/bisect.Writer
1000eca60 R go:itab.*internal/poll.DeadlineExceededError,error
1000ec9a0 R go:itab.*io/fs.PathError,error
1000ec8c8 R go:itab.*os.File,io.Writer
1000ec8e8 R go:itab.main.Developer,main.Worker
1000eca00 R go:itab.runtime.errorString,error
1000eca20 R go:itab.runtime.plainError,error
...
```

Эти interface table **создаются во время компиляции**, но **инициализируются при запуске** runtime.

Как часть настройки runtime — **до выполнения** вашей функции `main()` — Go строит глобальную hash table, называемую `itabTable`. Эта таблица начинается с **512 слотов**, как определено `runtime.itabInitSize`. Она использует комбинацию hash-значений типа interface и конкретного типа для определения правильного расположения хранения каждого itab.

Runtime обходит все активные модули и регистрирует каждый itab в глобальной таблице:

```go
func itabsinit() {
    ...
    for _, md := range activeModules() {
        for _, i := range md.itablinks {
            itabAdd(i)
        }
    }
    ...
}

// itabLock must be held.
func itabAdd(m *itab) {
    ...
    t := itabTable
    if t.count >= 3*(t.size/4) { // 75% load factor
        // Grow hash table.
        t2 := (*itabTableType)(mallocgc((2+2*t.size)*goarch.PtrSize, nil, true))
        t2.size = t.size * 2
        iterate_itabs(t2.add)
        ...
        atomicstorep(unsafe.Pointer(&itabTable), unsafe.Pointer(t2))
        t = itabTable
    }
    t.add(m)
}
// itabsinit (src/runtime/iface.go)_
```

Для большинства программ, не использующих plugin или shared library, обычно есть только **один активный модуль**: `firstmoduledata`. Он включает всё от вашего main package до стандартных библиотек, таких как `fmt` и `runtime`.

Этот процесс инициализации — то, как Go runtime **узнаёт**, какой itab использовать для конкретной пары тип-interface.

### Lazy initialization для неизвестных пар

Однако не все комбинации тип-interface известны при компиляции. 
Когда компилятор не может предвидеть пару, runtime обрабатывает это через **lazy initialization**. Itab создаётся только когда комбинация **фактически используется**.

Эта логика обрабатывается в `runtime.getitab`. 
Если запрошенный itab не найден в глобальной таблице, runtime следует чёткой последовательности:

```go
func getitab(inter *interfacetype, typ *_type, canfail bool) *itab {
    var m *itab

    // 1. Сначала ищем в hash table
    t := (*itabTableType)(atomic.Loadp(unsafe.Pointer(&itabTable)))
    if m = t.find(inter, typ); m != nil {
        goto finish
    }
    ...

    // 2. Если не найдено, создаём новый itab
    m = (*itab)(persistentalloc(
        unsafe.Sizeof(itab{})+uintptr(len(inter.Methods)-1)*goarch.PtrSize,
        0, &memstats.other_sys))
    m.inter = inter
    m._type = typ
    m.hash = 0
    m.init()
    itabAdd(m)
    ...
}
// getitab (src/runtime/iface.go)_
```

Runtime сначала проверяет глобальную `itabTable` на существующее совпадение. Если найдено, немедленно возвращает. Если нет — **выделяет** новый itab, заполняет необходимые поля и **регистрирует** его в глобальной таблице для будущих поисков.

Одна деталь, достойная внимания — строка:

```go
// Выделяет память для struct itab плюс пространство для всех method pointer
persistentalloc(unsafe.Sizeof(itab{})+uintptr(len(inter.Methods)-1)*goarch.PtrSize, 0,
    &memstats.other_sys)
```

Это вычисление подтверждает то, что мы видели ранее: поле `fun [1]uintptr` в struct `itab` — лишь **начальная точка**. Фактический размер массива method pointer зависит от количества методов, определяемых interface.

Ещё один важный аспект — как **резервируется** память. Функция `persistentalloc` выделяет память, живущую **весь срок программы**. Всё, что выделено таким образом, **не управляется garbage collector**.

В совокупности это даёт чёткое представление о том, как Go runtime строит и управляет связями тип-interface, как при запуске, так и по требованию.
