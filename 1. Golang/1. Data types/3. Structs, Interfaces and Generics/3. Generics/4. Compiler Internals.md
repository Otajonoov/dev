
Go использует стратегию, называемую **monomorphization** — термин может звучать причудливо, но концепция проста.

Когда вы пишете generic-функцию, компилятор анализирует конкретные типы, которые вы используете, и **генерирует специфическую версию** этой функции для каждого типа.

Возьмём функцию `Max` как пример:

- Если вы вызываете её с integer, компилятор генерирует версию для `int`.
- Если позже вы используете её с `float64`, создаётся отдельная версия для `float64`.

В переводе на обычный Go это выглядело бы примерно так:

```go
func Max(a, b int) int {
    if a > b {
        return a
    }
    return b
}

func Max(a, b float64) float64 {
    if a > b {
        return a
    }
    return b
}
```

Это как написать каждую версию вручную, но компилятор генерирует их за вас, подобно **function overloading**.

**Почему этот подход?**

Генерируя конкретные реализации на этапе компиляции, Go **избегает runtime overhead**. 
В других языках generic-функции могут полагаться на virtual dispatch для определения нужной версии во время выполнения. Это добавляет стоимость в runtime. 
Go обходит это, **разрешая всё во время компиляции**.

Однако этот метод имеет **компромисс: code bloat**.

Каждый новый тип, используемый с generic-функцией, приводит к отдельной скомпилированной версии. Это может увеличить **размер бинарника**, добавить время компиляции и использовать больше памяти.

Для mitigation Go применяет технику, известную как **shape stenciling**, часто в комбинации с **dictionaries**. Этот подход представляет золотую середину — избегая ненужного дублирования при сохранении преимуществ type safety и производительности.

### Генерация кода: Stenciling и Dictionaries

Развивая monomorphization, Go идёт более эффективным путём, комбинируя **shape stenciling** с **dictionaries** для управления generics.

- **Shape Stenciling:** Вместо генерации полностью специализированной версии функции для каждого типа, компилятор создаёт **generic shape** во время компиляции. Эта общая shape поддерживает любой тип, соответствующий constraint. Вместо дублирования машинного кода для каждой вариации Go **переиспользует единую реализацию** где возможно.
- **Dictionaries:** Наряду со stenciled-кодом Go передаёт **type-specific metadata** во время выполнения через структуры, известные как dictionaries. Эти dictionaries включают method set и детали поведения для каждого конкретного типа, позволяя generic-коду работать корректно с множеством типов.

Вместе эти механизмы помогают уменьшить дублирование кода и сохранить **размеры бинарников меньшими** по сравнению с полной monomorphization. Хотя передача dictionaries вносит небольшую runtime-стоимость, компромисс приемлем для сохранения производительности и компактных бинарников.

### Пример с функцией Max

| Go                                      | Pseudo Go Code                                          |
| --------------------------------------- | ------------------------------------------------------- |
| `func Max[T Ordered](a, b T) T { ... }` | `func Max(dict *dictionary, a, b go.shape.int) { ... }` |
| `Max(1, 2)`                             | `main.Max[go.shape.int](dict.Max[int], 1, 2)`           |
| `Max(uint(1), uint(2))`                 | `main.Max[go.shape.uint](dict.Max[uint], 1, 2)`         |
| `Max(FakeInt(1), FakeInt(2))`           | `main.Max[go.shape.int](dict.Max[main.FakeInt], 1, 2)`  |

Вместо генерации отдельной функции для каждого из `int`, `uint` или `FakeInt` (имеющего тот же underlying type, что и `int`), компилятор создаёт **единую реализацию**, используя общую shape — вроде `go.shape.int`. Dictionary (`dict.Max[int]`) предоставляет необходимую информацию для конкретного типа во время выполнения.

Здесь `int` и `FakeInt` имеют одинаковый underlying type, что позволяет им **разделять shape**. Напротив, Go генерирует **отдельную shape** для `uint`.

Это подчёркивает важное различие: хотя `int` и `uint` могут оба быть 64-битными integer (на 64-битных системах), Go трактует их как **разные shape** из-за различий в поведении и семантике типов. Дело не только в размере в памяти — **идентичность типа** и правила, которые ими управляют, тоже играют роль.

Напротив, alias-типы вроде `byte` и `rune` **разделяют ту же shape**, что и их underlying типы (`uint8` и `int32` соответственно). Поскольку их поведение идентично базовым типам, Go не трактует их как уникальные shape.

### Понимание Type Shape и Shape Inference

Компилятор Go использует внутренний placeholder-пакет `go.shape` для управления shape-типами. Этот пакет **недоступен** в обычном коде — он существует исключительно для использования компилятором:

```go
// Фиктивный пакет для shape-типов (см. typecheck.Shapify()).
var ShapePkg = NewPkg("go.shape", "go.shape")
// ShapePkg (src/cmd/compile/internal/types/type.go)_
```

Процесс назначения shape типу называется **shapify**. 
 Underlying type определяет shape.
  Два типа считаются имеющими одинаковую shape, только если их underlying представления **идентичны**. 
  Когда компилятор встречает тип, которому ещё не назначена shape, он извлекает его underlying структуру и назначает её в namespace `go.shape`.

Детали вроде именованных типов, type alias или другие синтаксические формы **не влияют** на shape. Важна только конкретная структура. Этот процесс также **рекурсивен** — при shapify составного типа вроде struct, массива или map компилятор рекурсивно определяет shape каждого компонента.

Например, рассмотрим передачу struct в generic-функцию:

```go
type User struct {
    ID   string
    Name string
    Age  int
}

func Print[T any](t T) {
    fmt.Println(t)
}

Print(User{})
// -> Print[go.shape.struct {ID string; Name string; Age int}](dict, User{})
```

Go генерирует версию `Print`, используя shape struct `User`. Любой другой тип с **точно такой же структурой** разделит эту shape и может **переиспользовать** ту же скомпилированную версию функции.

### Краткая шпаргалка по Shape

|Объект|Shape|
|---|---|
|`type User struct { ID string; Name string; Age int }`|`go.shape.struct {ID string; Name string; Age int}`|
|`[]int{}`|`go.shape.[]int`|
|`type Worker interface { Work() }`|`go.shape.interface { Work() }`|
|`*User`|`go.shape.*uint8` или `go.shape.*main.User`|

**Pointer** заслуживают особого внимания. Им часто назначается **общая shape** вроде `go.shape.*uint8`, независимо от того, на какой тип они указывают. Эта обобщённая shape используется, когда компилятору **не требуются** дополнительные детали о типе, на который указывает pointer:

```go
func Print[T any](t T) {
    fmt.Println(t)
}

Print(&Developer{})
// -> Print[go.shape.*int8](dict, &Developer{})
```

Поскольку constraint — `any`, generic-функция не выполняет операций, зависящих от точного типа за pointer. В таких случаях все pointer-типы трактуются одинаково. Компилятор выбирает **минимальную, универсальную** pointer shape вроде `go.shape.*int8`. Это позволяет переиспользовать сгенерированный код для разных pointer-типов (`*Developer`, `*int`, `*MyStruct` и т.д.), пока constraint остаётся общим.

### Pointer shape при наличии методов

Но что произойдёт, если мы изменим constraint на interface **с методами** и вызовем один из них?

```go
type Worker interface { Work() }

func Print[T Worker](t T) {
    t.Work()
}

Print(&Developer{})
// -> Print[go.shape.*int8](dict, &Developer{})
```

Мы обновляем constraint до basic interface `Worker` и вызываем `t.Work()` на type parameter. Удивительно, но shape **всё ещё не меняется**. Хотя `T` теперь удовлетворяет interface `Worker`, компилятор продолжает использовать обобщённую shape `go.shape.*int8`.

Как тогда вызов `t.Work()` успешно выполняется?

Ответ кроется в **dictionary** (`dict`), передаваемом функции. Когда вызывается `t.Work()`, компилятор не генерирует прямой вызов метода. Вместо этого он использует dictionary для **поиска и вызова** соответствующего метода `Work` для фактического типа, который `T` представляет во время выполнения.

> **Примечание: Basic Interface**
> 
> **Basic interface** определяется исключительно своими методами. Он не включает никаких type constraint.
> 
> **Non-basic interface** включают type restriction. Например, `interface{ ~int }` — не basic interface, потому что он ограничивает underlying type. Эти constraint определяют набор допустимых типов и могут разрешать дополнительные операции.

В случае non-basic interface компилятор **не может** полагаться только на dictionary. Ему нужно сохранить более детальную информацию о типе в shape:

```go
type DeveloperPtr interface { *Developer }

func Print[T DeveloperPtr](t T) {
    *t = Developer{Name: "Phuong Le", Skill: "Go"}
}

Print(&Developer{})
// -> Print[go.shape.*main.Developer](dict, &Developer{})
```

Поскольку `DeveloperPtr` включает **конкретный type constraint**, компилятор выбирает **более точную shape**: `go.shape.*main.Developer`. Это гарантирует, что сгенерированный код имеет доступ к полной информации о типе, необходимой для операций помимо method dispatch, таких как разыменование pointer или присвоение нового значения.

### Dictionary Passing и Function Dispatch

Вот как всё соединяется на практике:

```go
type Developer struct {
    Name  string
    Skill string
}

func (d Developer) Work() {
    fmt.Printf("%s is coding in %s\n", d.Name, d.Skill)
}

type Worker interface {
    Work()
}

//go:noinline
func CallWork[T Worker](worker T) {
    worker.Work()
}

func main() {
    worker := Developer{Name: "Phuong Le", Skill: "Go"}
    CallWork(worker)
}
```

Когда вызывается `CallWork(worker)`, компилятор генерирует **две версии** generic-функции: **wrapper-функцию** и **shaped-функцию**.

> **Иллюстрация 72.** Shaped-версия generic-функции _(Рисунок не загружен)_

#### Wrapper-функция

Начнём с wrapper: `main.CallWork[main.Developer]`. Эта версия создаётся для каждого конкретного типа, используемого с generic-функцией. Она **не содержит** основную логику — она просто настраивает вызов shaped-функции:

```go
// WRAPPER
func main.CallWork(arg1 main.Developer) {
    dict := main.getDict(main.Developer)
    main.CallWork(dict, arg1)
}
```

Wrapper получает type-specific dictionary и передаёт его вместе с аргументами функции в shaped-версию.

Assembly для wrapper:

```asm
main.CallWork[main.Developer] STEXT dupok size=144 args=0x20 locals=0x38 funcid=0x16
  0x0000 TEXT main.CallWork[main.Developer](SB), DUPOK|WRAPPER|ABIInternal, $64-32
  ...
  0x0038 00056 MOVD $main..dict.CallWork[main.Developer](SB), R0
  0x0040 00064 CALL main.CallWork[go.shape.struct { Name string; Skill string }](SB)
  ...
```

Wrapper-функция отмечена флагом `WRAPPER` и служит **точкой входа** для вызовов с конкретными типами. Её задача прямолинейна: получить dictionary для типа `Developer` и передать его вместе с фактическим аргументом в shaped-версию:

```asm
0x0038 00056 MOVD $main..dict.CallWork[main.Developer](SB), R0
```

Здесь компилятор загружает адрес dictionary (`main..dict.CallWork[main.Developer](SB)`) в регистр R0 из main package и вызывает shaped-функцию:

```asm
0x0040 00064 CALL main.CallWork[go.shape.struct { Name string; Skill string }](SB)
```

#### Shaped-функция

Эта shaped-версия содержит **фактическую логику** generic-функции. Она компилируется **один раз** для любого типа с одинаковой shape. В данном случае shape соответствует struct с двумя string-полями.

```asm
main.CallWork[go.shape.struct { Name string; Skill string }] STEXT dupok size=112
  0x0000 TEXT ...(SB), DUPOK|ABIInternal, $48-40
  ...
  0x0020 00032 MOVD (R0), R5   // Загружает method function pointer из dictionary
  0x0024 00036 MOVD R0, R26    // Сохраняет dictionary (type info) в R26
  0x0028 00040 MOVD R1, R0     // Developer.Name.data как первый аргумент
  0x002c 00044 MOVD R2, R1     // Developer.Name.len
  0x0030 00048 MOVD R3, R2     // Developer.Skill.data
  0x0034 00052 MOVD R4, R3     // Developer.Skill.len
  0x0038 00056 CALL (R5)       // Вызов метода Developer.Work({R0,R1,R2,R3})
  ...
```

Каждый string в Go состоит из pointer и length, всего **16 байт**. Struct `Developer`, имеющий два string, занимает **32 байта**. Добавив 8-байтный dictionary pointer, получаем общий размер аргументов функции — **40 байт**.

Это отражено в metadata assembly:

```asm
0x0000 TEXT ...(SB), DUPOK|ABIInternal, $48-40
```

Нотация `$48-40` указывает, что stack frame использует 48 байт всего, из которых 40 байт зарезервированы для аргументов. На этом этапе R0 содержит **dictionary pointer**, помещённый туда wrapper-функцией.

Pseudo-код shaped-функции:

```go
// DUPOK
func main.CallWork(dict *dictionary, arg1 go.shape.struct {Name string; Skill string}) {
    methodPtr := dict.typeParamMethodExprs[0] // получаем function pointer
    methodPtr(arg1)                           // вызываем функцию с arg1
}
```

#### Структура Dictionary

Концептуально layout dictionary выглядит так:

```go
// Упрощённое представление layout dictionary
type Dictionary struct {
    typeParamMethodExprs []uintptr  // Method pointer для type parameter
    subdicts             []uintptr  // Вложенные dictionary для inner generic типов
    rtypes               []uintptr  // Runtime type information
    itabs                []uintptr  // Interface table pointer
}
```

Ключевое поле — `typeParamMethodExprs`, хранящее pointer на методы, определённые на типе, используемом как type parameter. В нашем примере это включало бы `main.Developer.Work`. Эти pointer используются для выполнения вызовов методов во время выполнения, на основе фактического типа, переданного в generic-функцию.

Если вы помните, как struct расположены в Go — **первое поле** struct разделяет **тот же адрес** памяти, что и сам struct. Это поведение применимо здесь: dictionary pointer, хранящийся в R0, указывает непосредственно на первое поле dictionary, которое содержит method expression:

> **Иллюстрация 73.** Dictionary хранит runtime type information _(Рисунок не загружен)_

Поскольку первое поле начинается с базового адреса struct, оно **точно совпадает** с адресом, хранящимся в R0.

```asm
0x0020 00032 MOVD (R0), R5
...
0x0038 00056 CALL (R5)
```

Когда assembly выполняет `MOVD (R0), R5`, он загружает 8-байтное значение из адреса, хранящегося в R0, который является **базой dictionary**. Это значение помещается в регистр R5 и соответствует `dict.typeParamMethodExprs[0]` — pointer на метод `Work` в `main.Developer`.

Следующая инструкция `CALL (R5)` использует этот pointer для вызова метода, фактически выполняя `Developer.Work()`.

Вот как shaped-функция выполняет **method dispatch через dictionary**. То, что выглядит как простой вызов метода, на самом деле поддерживается более сложным процессом, связывающим generic type handling, method resolution и runtime-поведение.

## Итоги главы

### Структуры

Структуры организуют разные типы данных в единый блок. 
Определение struct указывает компилятору, как расположить данные в памяти для эффективного доступа. 
Критический аспект — **memory alignment**: компилятор добавляет padding между полями для оптимального выравнивания. Порядок полей влияет на использование памяти — расположение от большего к меньшему обычно даёт наиболее эффективный layout.

Embedding позволяет включать один struct в другой, продвигая его поля. 
Хотя это похоже на наследование, это **композиция** — embedded struct сохраняют свои методы в своём контексте.

### Интерфейсы

Interface определяют поведение через набор методов. 
Если тип реализует все указанные методы, он **автоматически** удовлетворяет interface. 
Под капотом interface состоят из type pointer и data pointer. 
Для непустых interface Go использует **itab** для маппинга методов interface на реализации конкретного типа.

### Generics

Generics решают проблему написания гибкого, переиспользуемого кода без дублирования логики. Go реализует generics через комбинацию **shape stenciling и dictionaries**. 
Компилятор генерирует **wrapper-версию** для каждого конкретного типа и **shaped-версию**, работающую с любым типом одинаковой shape. Dictionary содержит metadata вроде method pointer и runtime type information.

## Xulosa (Mening xulosalarim)

Bu bob Go'dagi generics'ning nafaqat **sintaksisi**, balki **kompilator ichki tuzilishi** — shape stenciling, dictionary passing va method dispatch mexanizmlarini chuqur ochib berdi. Mana asosiy xulosalarim:

**1. Generics — bu «har bir tip uchun funksiya yozish» muammosining elegant yechimi.** Go 1.18 dan oldin ikkita yo'l bor edi: `interface{}` + type assertion (runtime overhead, type safety yo'q) yoki har bir tip uchun alohida funksiya (code duplication). Generics bu ikkala muammoni hal qiladi — **bitta funksiya, type safety bilan, compile-time'da tekshiriladi**. Lekin generics'dan foydalanishda ehtiyot bo'lish kerak — murakkab constraint'lar kodni o'qishni qiyinlashtiradi.

**2. Constraint — bu interface'ning evolyutsiyasi.** Klassik Go'da interface faqat **method set** bilan ishlardi: «bu tip nima qila oladi?». Generics bilan endi interface ham **type identity** ni tekshiradi: «bu tip nima?». `~int | ~float64` deb yozganingizda, siz nafaqat behaviour, balki **underlying type** ni ham cheklamoqdasiz. Bu kuchli, lekin eslash kerak: constraint — bu oddiy interface emas, uni `var` da ishlatib bo'lmaydi.

**3. Shape stenciling — full monomorphization va type erasure o'rtasidagi oltin o'rta.** C++ template'lari to'liq monomorphization qiladi — har bir tip uchun yangi kod. Java generics type erasure qiladi — runtime'da tip ma'lumoti yo'qoladi. Go **o'rtada** turadi: bir xil underlying type'ga ega tiplar (masalan, `int` va `type FakeInt int`) **bitta shape** ni share qiladi. Bu binary size'ni kamaytiradi, lekin dictionary orqali kichik runtime overhead qo'shadi.

**4. Dictionary — bu generics'ning runtime «yordamchisi».** Shaped funksiya **qaysi method'ni chaqirish kerakligini** bilmaydi — u dictionary'ga qaraydi. `dict.typeParamMethodExprs[0]` — bu birinchi method'ning function pointer. Assembly'da ko'rdik: `MOVD (R0), R5` dictionary'dan method pointer'ni yuklaydi, `CALL (R5)` esa uni chaqiradi. Bu interface'dagi `itab.fun[0]` bilan juda o'xshash — **ikkalasi ham indirect method dispatch**.

**5. Wrapper + Shaped = ikki bosqichli dispatch.** `CallWork(developer)` chaqirganingizda, kompilator **ikkita funksiya** yaratadi:

- `main.CallWork[main.Developer]` — **WRAPPER**, dictionary'ni tayyorlab shaped funksiyaga uzatadi
- `main.CallWork[go.shape.struct{...}]` — **SHAPED**, asosiy logikani bajaradi, dictionary orqali method dispatch qiladi

Bu Go'dagi auto-generated pointer receiver wrapper (`DUPOK|WRAPPER`) bilan **o'xshash pattern** — kompilator developer uchun boilerplate kod yaratadi.

**6. Pointer shape — eng qiziqarli optimizatsiya.** Basic interface constraint bo'lganda (`[T Worker]`), **barcha pointer tiplar** `go.shape.*int8` shape'ni share qiladi. Kompilator pointer'ning qaysi struct'ga ko'rsatishini bilmasa ham, dictionary orqali to'g'ri method'ni topa oladi. Faqat non-basic interface bo'lganda (masalan, `interface{ *Developer }`) kompilator aniqroq shape ishlatadi — `go.shape.*main.Developer`.

**7. Type inference — generics'ni «ko'rinmas» qiladi.** Eng yaxshi generics — bu **siz bilmagan** generics. `NewBox("hello", 123)` yozganingizda, hech qanday type parameter ko'rsatmaysiz. Constraint type inference esa yanada kuchliroq: `K = []T` degan constraint'dan kompilator `T` ni bilsa `K` ni, `K` ni bilsa `T` ni **avtomatik chiqaradi**. `FilterDefault[T ~[]E, E comparable]` misoli — real-world'da bu qanday foydali ekanligining eng yaxshi ko'rsatkichi.

**Amaliy maslahatlar:**

- Oddiy holatlar uchun generics'dan foydalanmang — `MaxInt` yozish `Max[T Ordered]` dan ko'ra osonroq o'qiladi
- Constraint'larni **kichik va modular** qiling — `Ordered = Integer | Float | ~string` pattern'ini takrorlang
