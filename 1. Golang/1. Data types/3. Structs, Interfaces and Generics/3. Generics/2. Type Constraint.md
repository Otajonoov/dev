Type constraint — bu generic funksiya yoki type'ga qaysi type'lar bilan ishlash mumkinligini cheklab qo'yuvchi mexanizm. Go'da type constraint sifatida **interface** ishlatiladi.

### Nima uchun kerak?

Type constraint'siz generic funksiya **hamma** type'ni qabul qiladi, lekin ko'p operatsiyalar (`+`, `>`, `<`) barcha type'larda ishlamaydi. Masalan, `string`'ni `*` bilan ko'paytira olmaysiz. Constraint aynan shu muammoni hal qiladi — kompilyatorga "faqat mana shu type'lar keladi, ularda bu operatsiya ishlaydi" deb kafolat beradi.

```go
func AnyUsage[T any](t T) {
    // Присвоить переменной
    a := t

    // Использовать со встроенными функциями
    b := new(T)

    // Получить pointer на переменную
    c := &t

    // Использовать в composite literal
    d := []T{t}

    // Проверить тип в type switch
    var e interface{} = a
    switch t := e.(type) {
    case int:
        ...
    case T:
        ...
    default:
        ...
    }
}
```

Эта функция принимает значение любого типа и предлагает широкий диапазон использования. Вы можете присвоить его переменной, создать pointer, передать встроенным функциям или включить в составные типы вроде slice. Также можно использовать type switch для проверки конкретного типа во время выполнения.

Хотя `any` предоставляет большую гибкость, он **не гарантирует** возможности типов, которые допускает.

Например, не все типы можно сравнивать на равенство:

```go
func Equal[T any](a, b T) bool {
    return a == b // invalid operation: a == b (incomparable types in type set)
}
```

Это **не скомпилируется**, потому что некоторые типы не могут сравниваться с помощью `==` или `!=`. Когда нужно гарантировать, что тип поддерживает определённые операции, необходимо определить **границы**. 
Именно здесь constraint становятся необходимыми:

```go
func Equal[T comparable](a, b T) bool {
    return a == b
}
```

Constraint `comparable`, введённый в Go 1.18 вместе с generics, ограничивает type parameter типами, поддерживающими **операторы равенства**. 
Сужая диапазон допустимых типов, constraint позволяют вашему коду безопасно зависеть от конкретных операций.

Итак, constraint определяют, какие типы допускаются в качестве type argument в generic-функции или типе. Они также контролируют, как эти типы могут использоваться **внутри тела функции**.

Constraint используют стандартный синтаксис interface. 
Например, рассмотрим constraint `comparable`:

```go
// comparable — это interface, реализуемый всеми сравнимыми типами
// (boolean, int, string, pointer, channel, массивы сравнимых типов,
// struct, все поля которых сравнимы).
// Interface comparable может использоваться только как constraint type parameter,
// не как тип переменной.
type comparable interface{ comparable }
// comparable (src/builtin/builtin.go)
```

---
---

### Union типов в constraint

Распространённое использование constraint включает создание **union** (объединений) конкретных типов:

```go
type StringOrInt interface {
    string | int
}

func MustBeStringOrInt[T StringOrInt](t T) {
    fmt.Println(t)
}

func main() {
    MustBeStringOrInt("Hello, World!")
    MustBeStringOrInt(42)
    MustBeStringOrInt(1.23) 
// error: float64 does not satisfy StringOrInt float64 missing in string | int)
}
```

В данном случае `StringOrInt` — пользовательский constraint, допускающий только типы `string` или `int`. Любая попытка передать другой тип, например `float64`, приводит к **ошибке компиляции**.

Символ `|` называется **union operator**. Он позволяет объединять несколько типов в один constraint. При использовании union имейте в виду следующие правила:

- Элементы union должны быть **непересекающимися**. Нельзя включить и `int`, и `~int`, поскольку они перекрываются.
- Union может содержать **до 100** элементов, но не более.
- Constraint `comparable` **не может быть** частью union. Например, `interface{int | comparable}` недопустим.

---
---
### Constraint Ordered

Ранее мы использовали constraint `constraints.Ordered` в примере с функцией `Max`. Этот constraint определён в экспериментальном пакете constraint Go. Он гарантирует, что type parameter поддерживает **упорядочение** — то есть допускает сравнения с помощью `<`, `>`, `<=` или `>=`.

Constraint определён следующим образом:

```go
type Ordered interface {
    Integer | Float | ~string
}

type Float interface {
    ~float32 | ~float64
}

type Integer interface {
    Signed | Unsigned
}

type Signed interface {
    ~int | ~int8 | ~int16 | ~int32 | ~int64
}

type Unsigned interface {
    ~uint | ~uint8 | ~uint16 | ~uint32 | ~uint64 | ~uintptr
}
```

Структура прямолинейна. Constraint вроде `Ordered` **составляются** путём комбинирования меньших, переиспользуемых constraint, таких как `Float` и `Integer`. Этот модульный подход делает ограничения типов более выразительными без ненужной сложности.

---
---

### Оператор ~ (underlying type)

Символ `~`, известный как **underlying type operator**, вносит дополнительную гибкость. Он позволяет constraint сопоставлять тип, имеющий **определённый underlying type**. Например, `~string` означает, что тип может быть встроенным `string` или любым пользовательским типом, определённым с `string` в качестве основы — как `type MyString string`.

Однако тильду **нельзя применять** к именованным типам или interface напрямую. Например, если вы определите `type MyInt int`, вы не можете написать `~MyInt`.

---
---

### Комбинированные constraint

При работе с constraint в Go вы заметите, что они могут комбинировать **методы, interface, union типов и underlying типы**. 
Эта многослойная система позволяет выражать сложные правила о поведении и структуре типов:

```go
type MyConstraint interface {
    Print() string
    ~int | ~float64
}

type MyInt int

func (mi MyInt) Print() string {
    return fmt.Sprintf("MyInt: %d", mi)
}

type MyFloat float64

func (mf MyFloat) Print() string {
    return fmt.Sprintf("MyFloat: %f", mf)
}
```

В этом примере `MyInt` и `MyFloat` — пользовательские типы на основе `int` и `float64` соответственно. Оба реализуют метод `Print`, требуемый `MyConstraint`.

Этот подход выходит за рамки традиционных interface. 
В классическом Go interface были сосредоточены вокруг **method set**. 
Например, interface `Stringer` только ожидает, что тип реализует один метод: `String() string`. 
Если тип удовлетворяет этому требованию, он fulfills interface.

Generics **расширили роль interface**. 
Теперь они определяют и поведение через методы, и совместимость типов через constraint. 
Речь уже не только о том, что тип **может делать** — но и о том, **чем он является** по сути.

Это изменение не заменяет оригинальную философию Go об interface как поведенческих контрактах. Оно **надстраивается** над этим фундаментом, давая разработчикам более точный и выразительный способ моделирования требований к типам.

---
---
### Constraint — это не Interface

Хотя constraint используют синтаксис interface, они **не ведут себя** как традиционные interface при объявлении переменных:

```go
type Stringer interface {
    String() string
}

type IntConstraint interface {
    ~int | ~int8 | ~int16 | ~int32 | ~int64
}

type AnInt int

func (i AnInt) String() string {
    return fmt.Sprintf("AnInt: %d", i)
}

var s Stringer = AnInt(42)     // VALID
var i IntConstraint = AnInt(42) // error: cannot use type IntConstraint outside
                                // a type constraint: interface contains type constraints
```

Здесь `AnInt` наследуется от базового integer типа и удовлетворяет `IntConstraint`. 
Однако вы **не можете** использовать `IntConstraint` для объявления переменной. 
Constraint не предназначены для этой цели. Запись вроде `var x interface{~int}` тоже недопустима. Constraint определяют правила для type parameter, **а не** для объявления переменных.

Итак, несмотря на то, что interface используются для выражения constraint, **сами constraint — не обычные interface**.

Как только вы добавляете метод к constraint, он становится **комбинацией** type constraint и behavioral constraint:

```go
type MyConstraint interface {
    Print() string
    int | float64
}
```

Теперь constraint делает **две вещи**: требует метод (`Print`) и принуждает к определённой структуре типа (`int | float64`).

---
---

### Inline (анонимные) constraint

Go также поддерживает **inline или анонимные constraint**, где вы определяете constraint непосредственно в месте использования:

```go
func Execute[T interface {
    Print() string
    ~int | ~float64
}](t T) {
    result := t.Print()
    fmt.Println(result)
}
```

В этой функции `T` должен удовлетворять **обоим условиям**: иметь метод `Print` и его underlying type должен быть `int` или `float64`. 
Это комбинация **логического AND и OR** — `T` должен соответствовать всем критериям, определённым в блоке constraint.

Интересно, что когда constraint основан **только на типах**, можно опустить обёртку `interface{}` полностью:

```go
func Execute[T ~int | ~float64](t T) {
    ...
}
```

Эта версия работает так же и избегает ненужного синтаксиса, результируя в более чистый код.

---
---

### Пересечение (intersection) типов

Теперь пойдём дальше с немного более продвинутым примером:

```go
type Constraint interface {
    int | string
    string | float64
}
```

С участием логики union и intersection этот constraint вычисляет **пересечение** двух множеств. 
Поскольку `string` присутствует в обоих, он становится **единственным** допустимым типом, удовлетворяющим constraint.

Этот подход позволяет определять точные и гибкие constraint. 
Пересекая type set, вы уменьшаете допустимые типы **только до тех**, которые соответствуют каждому перечисленному требованию. 
В данном случае пересечение `int | string` и `string | float64` даёт только `string`:

```go
type Constraint interface {
    int | string
    string | float64
}

func Intersecting[T Constraint](t T) {
    fmt.Println(t)
}

// VALID
Intersecting("Hello, World!")

// error: cannot use 42 (untyped int constant) as string value in argument to Intersecting
Intersecting(42)              
```

---
---
### Пустое пересечение

Но что происходит, если два type set **не имеют общих элементов** — результируя в пустое пересечение?

Go обрабатывает этот сценарий изящно. 
Объявление constraint, приводящего к **пустому typeset**, не вызывает ошибку немедленно. 
Это потому, что определение, действительно ли constraint пуст, потребовало бы проверки каждого типа во всех пакетах и зависимостях. Хотя Go может распознать простые случаи, он **не выдаёт ошибку** компиляции, пока вы не попытаетесь фактически использовать функцию или тип, ограниченный этим пустым set.

Даже значение типа `any` не удовлетворит пустой constraint:

```go
type EmptyConstraint interface {
    float32
    int
}

func Empty[T EmptyConstraint](t T) {}

func main() {
   // compile error: cannot satisfy EmptyConstraint (empty type set)
    Empty(1) 
    
   // compile error: cannot satisfy EmptyConstraint (empty type set)    
    Empty(any(1))
}
```

Это иллюстрирует, как Go работает с пустыми type set, и завершает раздел о constraint.

## 3.3 Type Inference: от аргументов к сложным взаимосвязям

**Type inference** позволяет компилятору определять типы автоматически, чтобы нам не приходилось указывать их каждый раз:

|Без Type Inference|С Type Inference|
|---|---|
|`var a int = 10`|`a := 10`|
|`Max[int](a, 20)` — T явно указан как int|`Max(a, 20)` — T выведен как int|

С inference компилятор анализирует предоставленные значения и определяет соответствующие типы самостоятельно. Он применяет эту логику и к объявлениям переменных, и к вызовам функций.

Эта возможность играет ключевую роль в сохранении **лаконичности** generic-кода. Без inference generics стали бы многословными и повторяющимися — того, чего Go старается избегать. Вместо этого сложность **поглощается разработчиком**, проектирующим generic-функцию, а не тем, кто её использует:

go

```go
package main

import "fmt"

// Box — generic struct с двумя type parameter: T1 и T2.
type Box[T1, T2 any] struct {
    First  T1
    Second T2
}

// NewBox — generic constructor для struct Box.
func NewBox[T1, T2 any](first T1, second T2) *Box[T1, T2] {
    return &Box[T1, T2]{First: first, Second: second}
}

func main() {
    box := NewBox("hello", 123)  // T1 выведен как string, T2 как int
    box2 := NewBox(3.14, true)   // T1 выведен как float64, T2 как bool
    ...
}
```

В примере выше мы определяем generic struct `Box` с двумя type parameter, `T1` и `T2`. Функция `NewBox` действует как constructor, создавая экземпляр `Box` из двух входных значений. Типы для `T1` и `T2` **выводятся автоматически** на основе переданных аргументов.

Хотя написание `NewBox` требует больше усилий под капотом, использование остаётся чистым и прямолинейным. Читая функцию `main()`, вы даже не осознаете, что `NewBox` generic. В этом **преимущество type inference** — он упрощает использование, убирая generic-детали из поля зрения.

### Вывод типов в сложных сценариях

До сих пор мы рассматривали type inference в аргументах функций. Но компилятор Go способен обрабатывать **более сложные ситуации**, включая случаи с function parameter или составными типами:

go

```go
func Complex[T comparable, K any](t map[T]K, f func(K) T) T {
    for k, v := range t {
        if f(v) == k {
            return k
        }
    }
    var def T
    return def
}

func main() {
    firstArg := map[string]int{"a": 1}
    secondArg := func(i int) string { return "a" }
    _ = Complex(firstArg, secondArg)
}
```

Нет нужды углубляться в то, что `Complex` делает. Важно то, как гладко компилятор выводит типы `T` и `K` из входных данных — даже когда функция включает map, function argument и множество слоёв логики. Компилятор обрабатывает эти связи **автоматически**.

### Constraint Type Inference

Type inference в Go выходит за рамки простого вычисления типов из аргументов — вы также можете **вывести один type parameter на основе другого**:

go

```go
func Infer[T any, K []T](t T, k K) K {
    return append(k, t)
}

Infer(1, nil)           // = func Infer(t int, k []int) []int
Infer(1, []float64{})   // = func Infer(t float64, k []float64) []float64
```

Это называется **constraint type inference**. Рассмотрим, как это работает под капотом.

В функции `Infer` мы определяем два type parameter: `T` и `K`. Ключевая связь в том, что `K` ограничен быть **slice от T**, или `K = []T`. Это создаёт связь между ними — знание одного помогает определить другой.

**Первый вызов: `Infer(1, nil)`**

> **Иллюстрация 70.** Компилятор Go выводит типы из constraint _(Рисунок не загружен)_

1. Компилятор видит нетипизированную константу `1`. Без дополнительного контекста он откладывает присвоение конкретного типа `T`.
2. Второй аргумент — `nil`, который может соответствовать любому slice-типу. Всё ещё недостаточно информации для разрешения `K`.
3. Поскольку дополнительных подсказок нет, компилятор присваивает нетипизированной константе **тип по умолчанию** `int`, обновляя маппинг типов до `{"T": int}`.
4. С установленным `T` и зная, что `K` должен быть slice от `T`, компилятор выводит `K` как `[]int`.

**Второй вызов: `Infer(1, []float64{})`**

> **Иллюстрация 71.** Вывод типа из существующего slice типа _(Рисунок не загружен)_

Здесь второй аргумент — **конкретный slice**. Компилятор немедленно выводит `K` как `[]float64`. Из constraint `K = []T` он выводит, что `T` должен быть `float64`. Затем проверяет, может ли константа `1` быть присвоена `float64`, что допустимо. Type inference снова успешен, но на этот раз **вывод идёт в обратном направлении**.

Хотя это полезная демонстрация, на практике такое не пишут. Более простая версия полностью убрала бы второй type parameter:

go

```go
func Infer[T any](t T, k []T) []T {
    return append(k, t)
}
```

Тем не менее, constraint type inference — **мощная возможность** при правильном использовании. Она позволяет более продвинутые generic-паттерны и предлагает чистый способ выражения связей между типами.

### Сохранение типов в Generics

Команда Go часто приводит классический пример, демонстрирующий полезность этой возможности. Представьте, что вам нужна функция, фильтрующая zero value из slice любого типа:

go

```go
func FilterDefault[T comparable](s []T) []T {
    var result []T
    var def T
    for _, v := range s {
        if v != def {
            result = append(result, v)
        }
    }
    return result
}
```

Это работает хорошо в типичных случаях. Однако ситуация становится менее прямолинейной с **пользовательскими slice-типами**:

go

```go
type StringSlice []string

func (s StringSlice) Dummy() {
    fmt.Println("Dummy")
}

func main() {
    s := StringSlice{"a", "b", "c", ""}
    noZeroS := FilterDefault(s)
    noZeroS.Dummy() // compiler error: noZeroS.Dummy undefined
                    // (type []string has no field or method Dummy)
}
```

В чём проблема: при вызове `FilterDefault(s)` входной `StringSlice` выводится как `[]string`, потому что `T` становится `string`. Возвращаемый тип оказывается **обычным** `[]string`, и пользовательский метод `Dummy` **теряется** в процессе.

Несмотря на то, что вы передаёте пользовательский slice, результат — базовый slice. Это нарушает ожидание, что output должен **сохранить** оригинальный тип и связанное с ним поведение.

Для исправления нужно обновить `FilterDefault` так, чтобы он возвращал **тот же тип**, что и input. Type parameter `T` должен представлять **весь slice-тип**, и мы вводим другой type parameter `E` для типа элемента. Ключ — ограничить `T` так, чтобы его underlying type был `[]E`:

go

```go
// до: func FilterDefault[T comparable](s []T) []T {}
func FilterDefault[T ~[]E, E comparable](s T) T {
    var result T
    var def E
    for _, v := range s {
        if v != def {
            result = append(result, v)
        }
    }
    return result
}
```

Эта версия более точная, хотя немного сложнее. Если передать `StringSlice`, вы получите обратно `StringSlice`. Метод `Dummy` остаётся доступным, и функция ведёт себя **точно как ожидалось**.

Этот пример показывает, как generics могут выявить тонкие проблемы. Хотя поверхностный синтаксис может казаться простым, более глубокие аспекты вроде type inference и constraint могут привести к **сюрпризам**, если ими не управлять тщательно.

## 3.4 Внутренности компилятора: как Go реализует Generics

Наконец, рассмотрим, как компилятор Go **реализует generics**. Он использует стратегию, называемую **monomorphization** — термин может звучать причудливо, но концепция проста.

Когда вы пишете generic-функцию, компилятор анализирует конкретные типы, которые вы используете, и **генерирует специфическую версию** этой функции для каждого типа.

Возьмём функцию `Max` как пример:

- Если вы вызываете её с integer, компилятор генерирует версию для `int`.
- Если позже вы используете её с `float64`, создаётся отдельная версия для `float64`.

В переводе на обычный Go это выглядело бы примерно так:

go

```go
func Max(a, b int) int {
    if a > b {
        return a
    }
    return b
}

func Max(a, b float64) float64 {
    if a > b {
        return a
    }
    return b
}
```

Это как написать каждую версию вручную, но компилятор генерирует их за вас, подобно **function overloading**.

**Почему этот подход?**

Генерируя конкретные реализации на этапе компиляции, Go **избегает runtime overhead**. В других языках generic-функции могут полагаться на virtual dispatch для определения нужной версии во время выполнения. Это добавляет стоимость в runtime. Go обходит это, **разрешая всё во время компиляции**.

Однако этот метод имеет **компромисс: code bloat**.

Каждый новый тип, используемый с generic-функцией, приводит к отдельной скомпилированной версии. Это может увеличить **размер бинарника**, добавить время компиляции и использовать больше памяти.

Для mitigation Go применяет технику, известную как **shape stenciling**, часто в комбинации с **dictionaries**. Этот подход представляет золотую середину — избегая ненужного дублирования при сохранении преимуществ type safety и производительности.

### Генерация кода: Stenciling и Dictionaries

Развивая monomorphization, Go идёт более эффективным путём, комбинируя **shape stenciling** с **dictionaries** для управления generics.

- **Shape Stenciling:** Вместо генерации полностью специализированной версии функции для каждого типа, компилятор создаёт **generic shape** во время компиляции. Эта общая shape поддерживает любой тип, соответствующий constraint. Вместо дублирования машинного кода для каждой вариации Go **переиспользует единую реализацию** где возможно.
- **Dictionaries:** Наряду со stenciled-кодом Go передаёт **type-specific metadata** во время выполнения через структуры, известные как dictionaries. Эти dictionaries включают method set и детали поведения для каждого конкретного типа, позволяя generic-коду работать корректно с множеством типов.

Вместе эти механизмы помогают уменьшить дублирование кода и сохранить **размеры бинарников меньшими** по сравнению с полной monomorphization. Хотя передача dictionaries вносит небольшую runtime-стоимость, компромисс приемлем для сохранения производительности и компактных бинарников.

### Пример с функцией Max

|Go|Pseudo Go Code|
|---|---|
|`func Max[T Ordered](a, b T) T { ... }`|`func Max(dict *dictionary, a, b go.shape.int) { ... }`|
|`Max(1, 2)`|`main.Max[go.shape.int](dict.Max[int], 1, 2)`|
|`Max(uint(1), uint(2))`|`main.Max[go.shape.uint](dict.Max[uint], 1, 2)`|
|`Max(FakeInt(1), FakeInt(2))`|`main.Max[go.shape.int](dict.Max[main.FakeInt], 1, 2)`|

Вместо генерации отдельной функции для каждого из `int`, `uint` или `FakeInt` (имеющего тот же underlying type, что и `int`), компилятор создаёт **единую реализацию**, используя общую shape — вроде `go.shape.int`. Dictionary (`dict.Max[int]`) предоставляет необходимую информацию для конкретного типа во время выполнения.

Здесь `int` и `FakeInt` имеют одинаковый underlying type, что позволяет им **разделять shape**. Напротив, Go генерирует **отдельную shape** для `uint`.

Это подчёркивает важное различие: хотя `int` и `uint` могут оба быть 64-битными integer (на 64-битных системах), Go трактует их как **разные shape** из-за различий в поведении и семантике типов. Дело не только в размере в памяти — **идентичность типа** и правила, которые ими управляют, тоже играют роль.

Напротив, alias-типы вроде `byte` и `rune` **разделяют ту же shape**, что и их underlying типы (`uint8` и `int32` соответственно). Поскольку их поведение идентично базовым типам, Go не трактует их как уникальные shape.

### Понимание Type Shape и Shape Inference

Компилятор Go использует внутренний placeholder-пакет `go.shape` для управления shape-типами. Этот пакет **недоступен** в обычном коде — он существует исключительно для использования компилятором:

go

```go
// Фиктивный пакет для shape-типов (см. typecheck.Shapify()).
var ShapePkg = NewPkg("go.shape", "go.shape")
```

_Figure 73. ShapePkg (src/cmd/compile/internal/types/type.go)_

Процесс назначения shape типу называется **shapify**. Underlying type определяет shape. Два типа считаются имеющими одинаковую shape, только если их underlying представления **идентичны**. Когда компилятор встречает тип, которому ещё не назначена shape, он извлекает его underlying структуру и назначает её в namespace `go.shape`.

Детали вроде именованных типов, type alias или другие синтаксические формы **не влияют** на shape. Важна только конкретная структура. Этот процесс также **рекурсивен** — при shapify составного типа вроде struct, массива или map компилятор рекурсивно определяет shape каждого компонента.

Например, рассмотрим передачу struct в generic-функцию:

go

```go
type User struct {
    ID   string
    Name string
    Age  int
}

func Print[T any](t T) {
    fmt.Println(t)
}

Print(User{})
// -> Print[go.shape.struct {ID string; Name string; Age int}](dict, User{})
```

Go генерирует версию `Print`, используя shape struct `User`. Любой другой тип с **точно такой же структурой** разделит эту shape и может **переиспользовать** ту же скомпилированную версию функции.

### Краткая шпаргалка по Shape

|Объект|Shape|
|---|---|
|`type User struct { ID string; Name string; Age int }`|`go.shape.struct {ID string; Name string; Age int}`|
|`[]int{}`|`go.shape.[]int`|
|`type Worker interface { Work() }`|`go.shape.interface { Work() }`|
|`*User`|`go.shape.*uint8` или `go.shape.*main.User`|

**Pointer** заслуживают особого внимания. Им часто назначается **общая shape** вроде `go.shape.*uint8`, независимо от того, на какой тип они указывают. Эта обобщённая shape используется, когда компилятору **не требуются** дополнительные детали о типе, на который указывает pointer:

go

```go
func Print[T any](t T) {
    fmt.Println(t)
}

Print(&Developer{})
// -> Print[go.shape.*int8](dict, &Developer{})
```

Поскольку constraint — `any`, generic-функция не выполняет операций, зависящих от точного типа за pointer. В таких случаях все pointer-типы трактуются одинаково. Компилятор выбирает **минимальную, универсальную** pointer shape вроде `go.shape.*int8`. Это позволяет переиспользовать сгенерированный код для разных pointer-типов (`*Developer`, `*int`, `*MyStruct` и т.д.), пока constraint остаётся общим.

### Pointer shape при наличии методов

Но что произойдёт, если мы изменим constraint на interface **с методами** и вызовем один из них?

go

```go
type Worker interface { Work() }

func Print[T Worker](t T) {
    t.Work()
}

Print(&Developer{})
// -> Print[go.shape.*int8](dict, &Developer{})
```

Мы обновляем constraint до basic interface `Worker` и вызываем `t.Work()` на type parameter. Удивительно, но shape **всё ещё не меняется**. Хотя `T` теперь удовлетворяет interface `Worker`, компилятор продолжает использовать обобщённую shape `go.shape.*int8`.

Как тогда вызов `t.Work()` успешно выполняется?

Ответ кроется в **dictionary** (`dict`), передаваемом функции. Когда вызывается `t.Work()`, компилятор не генерирует прямой вызов метода. Вместо этого он использует dictionary для **поиска и вызова** соответствующего метода `Work` для фактического типа, который `T` представляет во время выполнения.

> **Примечание: Basic Interface**
> 
> **Basic interface** определяется исключительно своими методами. Он не включает никаких type constraint.
> 
> **Non-basic interface** включают type restriction. Например, `interface{ ~int }` — не basic interface, потому что он ограничивает underlying type. Эти constraint определяют набор допустимых типов и могут разрешать дополнительные операции.

В случае non-basic interface компилятор **не может** полагаться только на dictionary. Ему нужно сохранить более детальную информацию о типе в shape:

go

```go
type DeveloperPtr interface { *Developer }

func Print[T DeveloperPtr](t T) {
    *t = Developer{Name: "Phuong Le", Skill: "Go"}
}

Print(&Developer{})
// -> Print[go.shape.*main.Developer](dict, &Developer{})
```

Поскольку `DeveloperPtr` включает **конкретный type constraint**, компилятор выбирает **более точную shape**: `go.shape.*main.Developer`. Это гарантирует, что сгенерированный код имеет доступ к полной информации о типе, необходимой для операций помимо method dispatch, таких как разыменование pointer или присвоение нового значения.

### Dictionary Passing и Function Dispatch

Вот как всё соединяется на практике:

go

```go
type Developer struct {
    Name  string
    Skill string
}

func (d Developer) Work() {
    fmt.Printf("%s is coding in %s\n", d.Name, d.Skill)
}

type Worker interface {
    Work()
}

//go:noinline
func CallWork[T Worker](worker T) {
    worker.Work()
}

func main() {
    worker := Developer{Name: "Phuong Le", Skill: "Go"}
    CallWork(worker)
}
```

Когда вызывается `CallWork(worker)`, компилятор генерирует **две версии** generic-функции: **wrapper-функцию** и **shaped-функцию**.

> **Иллюстрация 72.** Shaped-версия generic-функции _(Рисунок не загружен)_

#### Wrapper-функция

Начнём с wrapper: `main.CallWork[main.Developer]`. Эта версия создаётся для каждого конкретного типа, используемого с generic-функцией. Она **не содержит** основную логику — она просто настраивает вызов shaped-функции:

go

```go
// WRAPPER
func main.CallWork(arg1 main.Developer) {
    dict := main.getDict(main.Developer)
    main.CallWork(dict, arg1)
}
```

Wrapper получает type-specific dictionary и передаёт его вместе с аргументами функции в shaped-версию.

Assembly для wrapper:

asm

```asm
main.CallWork[main.Developer] STEXT dupok size=144 args=0x20 locals=0x38 funcid=0x16
  0x0000 TEXT main.CallWork[main.Developer](SB), DUPOK|WRAPPER|ABIInternal, $64-32
  ...
  0x0038 00056 MOVD $main..dict.CallWork[main.Developer](SB), R0
  0x0040 00064 CALL main.CallWork[go.shape.struct { Name string; Skill string }](SB)
  ...
```

Wrapper-функция отмечена флагом `WRAPPER` и служит **точкой входа** для вызовов с конкретными типами. Её задача прямолинейна: получить dictionary для типа `Developer` и передать его вместе с фактическим аргументом в shaped-версию:

asm

```asm
0x0038 00056 MOVD $main..dict.CallWork[main.Developer](SB), R0
```

Здесь компилятор загружает адрес dictionary (`main..dict.CallWork[main.Developer](SB)`) в регистр R0 из main package и вызывает shaped-функцию:

asm

```asm
0x0040 00064 CALL main.CallWork[go.shape.struct { Name string; Skill string }](SB)
```

#### Shaped-функция

Эта shaped-версия содержит **фактическую логику** generic-функции. Она компилируется **один раз** для любого типа с одинаковой shape. В данном случае shape соответствует struct с двумя string-полями.

asm

```asm
main.CallWork[go.shape.struct { Name string; Skill string }] STEXT dupok size=112
  0x0000 TEXT ...(SB), DUPOK|ABIInternal, $48-40
  ...
  0x0020 00032 MOVD (R0), R5   // Загружает method function pointer из dictionary
  0x0024 00036 MOVD R0, R26    // Сохраняет dictionary (type info) в R26
  0x0028 00040 MOVD R1, R0     // Developer.Name.data как первый аргумент
  0x002c 00044 MOVD R2, R1     // Developer.Name.len
  0x0030 00048 MOVD R3, R2     // Developer.Skill.data
  0x0034 00052 MOVD R4, R3     // Developer.Skill.len
  0x0038 00056 CALL (R5)       // Вызов метода Developer.Work({R0,R1,R2,R3})
  ...
```

Каждый string в Go состоит из pointer и length, всего **16 байт**. Struct `Developer`, имеющий два string, занимает **32 байта**. Добавив 8-байтный dictionary pointer, получаем общий размер аргументов функции — **40 байт**.

Это отражено в metadata assembly:

asm

```asm
0x0000 TEXT ...(SB), DUPOK|ABIInternal, $48-40
```

Нотация `$48-40` указывает, что stack frame использует 48 байт всего, из которых 40 байт зарезервированы для аргументов. На этом этапе R0 содержит **dictionary pointer**, помещённый туда wrapper-функцией.

Pseudo-код shaped-функции:

go

```go
// DUPOK
func main.CallWork(dict *dictionary, arg1 go.shape.struct {Name string; Skill string}) {
    methodPtr := dict.typeParamMethodExprs[0] // получаем function pointer
    methodPtr(arg1)                           // вызываем функцию с arg1
}
```

#### Структура Dictionary

Концептуально layout dictionary выглядит так:

go

```go
// Упрощённое представление layout dictionary
type Dictionary struct {
    typeParamMethodExprs []uintptr  // Method pointer для type parameter
    subdicts             []uintptr  // Вложенные dictionary для inner generic типов
    rtypes               []uintptr  // Runtime type information
    itabs                []uintptr  // Interface table pointer
}
```

Ключевое поле — `typeParamMethodExprs`, хранящее pointer на методы, определённые на типе, используемом как type parameter. В нашем примере это включало бы `main.Developer.Work`. Эти pointer используются для выполнения вызовов методов во время выполнения, на основе фактического типа, переданного в generic-функцию.

Если вы помните, как struct расположены в Go — **первое поле** struct разделяет **тот же адрес** памяти, что и сам struct. Это поведение применимо здесь: dictionary pointer, хранящийся в R0, указывает непосредственно на первое поле dictionary, которое содержит method expression:

> **Иллюстрация 73.** Dictionary хранит runtime type information _(Рисунок не загружен)_

Поскольку первое поле начинается с базового адреса struct, оно **точно совпадает** с адресом, хранящимся в R0.

asm

```asm
0x0020 00032 MOVD (R0), R5
...
0x0038 00056 CALL (R5)
```

Когда assembly выполняет `MOVD (R0), R5`, он загружает 8-байтное значение из адреса, хранящегося в R0, который является **базой dictionary**. Это значение помещается в регистр R5 и соответствует `dict.typeParamMethodExprs[0]` — pointer на метод `Work` в `main.Developer`.

Следующая инструкция `CALL (R5)` использует этот pointer для вызова метода, фактически выполняя `Developer.Work()`.

Вот как shaped-функция выполняет **method dispatch через dictionary**. То, что выглядит как простой вызов метода, на самом деле поддерживается более сложным процессом, связывающим generic type handling, method resolution и runtime-поведение.

## Итоги главы

### Структуры

Структуры организуют разные типы данных в единый блок. Определение struct указывает компилятору, как расположить данные в памяти для эффективного доступа. Критический аспект — **memory alignment**: компилятор добавляет padding между полями для оптимального выравнивания. Порядок полей влияет на использование памяти — расположение от большего к меньшему обычно даёт наиболее эффективный layout.

Embedding позволяет включать один struct в другой, продвигая его поля. Хотя это похоже на наследование, это **композиция** — embedded struct сохраняют свои методы в своём контексте.

### Интерфейсы

Interface определяют поведение через набор методов. Если тип реализует все указанные методы, он **автоматически** удовлетворяет interface. Под капотом interface состоят из type pointer и data pointer. Для непустых interface Go использует **itab** для маппинга методов interface на реализации конкретного типа.

### Generics

Generics решают проблему написания гибкого, переиспользуемого кода без дублирования логики. Go реализует generics через комбинацию **shape stenciling и dictionaries**. Компилятор генерирует **wrapper-версию** для каждого конкретного типа и **shaped-версию**, работающую с любым типом одинаковой shape. Dictionary содержит metadata вроде method pointer и runtime type information.

## Xulosa (Mening xulosalarim)

Bu bob Go'dagi generics'ning nafaqat **sintaksisi**, balki **kompilator ichki tuzilishi** — shape stenciling, dictionary passing va method dispatch mexanizmlarini chuqur ochib berdi. Mana asosiy xulosalarim:

**1. Generics — bu «har bir tip uchun funksiya yozish» muammosining elegant yechimi.** Go 1.18 dan oldin ikkita yo'l bor edi: `interface{}` + type assertion (runtime overhead, type safety yo'q) yoki har bir tip uchun alohida funksiya (code duplication). Generics bu ikkala muammoni hal qiladi — **bitta funksiya, type safety bilan, compile-time'da tekshiriladi**. Lekin generics'dan foydalanishda ehtiyot bo'lish kerak — murakkab constraint'lar kodni o'qishni qiyinlashtiradi.

**2. Constraint — bu interface'ning evolyutsiyasi.** Klassik Go'da interface faqat **method set** bilan ishlardi: «bu tip nima qila oladi?». Generics bilan endi interface ham **type identity** ni tekshiradi: «bu tip nima?». `~int | ~float64` deb yozganingizda, siz nafaqat behaviour, balki **underlying type** ni ham cheklamoqdasiz. Bu kuchli, lekin eslash kerak: constraint — bu oddiy interface emas, uni `var` da ishlatib bo'lmaydi.

**3. Shape stenciling — full monomorphization va type erasure o'rtasidagi oltin o'rta.** C++ template'lari to'liq monomorphization qiladi — har bir tip uchun yangi kod. Java generics type erasure qiladi — runtime'da tip ma'lumoti yo'qoladi. Go **o'rtada** turadi: bir xil underlying type'ga ega tiplar (masalan, `int` va `type FakeInt int`) **bitta shape** ni share qiladi. Bu binary size'ni kamaytiradi, lekin dictionary orqali kichik runtime overhead qo'shadi.

**4. Dictionary — bu generics'ning runtime «yordamchisi».** Shaped funksiya **qaysi method'ni chaqirish kerakligini** bilmaydi — u dictionary'ga qaraydi. `dict.typeParamMethodExprs[0]` — bu birinchi method'ning function pointer. Assembly'da ko'rdik: `MOVD (R0), R5` dictionary'dan method pointer'ni yuklaydi, `CALL (R5)` esa uni chaqiradi. Bu interface'dagi `itab.fun[0]` bilan juda o'xshash — **ikkalasi ham indirect method dispatch**.

**5. Wrapper + Shaped = ikki bosqichli dispatch.** `CallWork(developer)` chaqirganingizda, kompilator **ikkita funksiya** yaratadi:

- `main.CallWork[main.Developer]` — **WRAPPER**, dictionary'ni tayyorlab shaped funksiyaga uzatadi
- `main.CallWork[go.shape.struct{...}]` — **SHAPED**, asosiy logikani bajaradi, dictionary orqali method dispatch qiladi

Bu Go'dagi auto-generated pointer receiver wrapper (`DUPOK|WRAPPER`) bilan **o'xshash pattern** — kompilator developer uchun boilerplate kod yaratadi.

**6. Pointer shape — eng qiziqarli optimizatsiya.** Basic interface constraint bo'lganda (`[T Worker]`), **barcha pointer tiplar** `go.shape.*int8` shape'ni share qiladi. Kompilator pointer'ning qaysi struct'ga ko'rsatishini bilmasa ham, dictionary orqali to'g'ri method'ni topa oladi. Faqat non-basic interface bo'lganda (masalan, `interface{ *Developer }`) kompilator aniqroq shape ishlatadi — `go.shape.*main.Developer`.

**7. Type inference — generics'ni «ko'rinmas» qiladi.** Eng yaxshi generics — bu **siz bilmagan** generics. `NewBox("hello", 123)` yozganingizda, hech qanday type parameter ko'rsatmaysiz. Constraint type inference esa yanada kuchliroq: `K = []T` degan constraint'dan kompilator `T` ni bilsa `K` ni, `K` ni bilsa `T` ni **avtomatik chiqaradi**. `FilterDefault[T ~[]E, E comparable]` misoli — real-world'da bu qanday foydali ekanligining eng yaxshi ko'rsatkichi.

**Amaliy maslahatlar:**

- Oddiy holatlar uchun generics'dan foydalanmang — `MaxInt` yozish `Max[T Ordered]` dan ko'ra osonroq o'qiladi
- Constraint'larni **kichik va modular** qiling — `Ordered = Integer | Float | ~string` pattern'ini takrorlang
- `~T` operatorini ishlating agar pользовательский типlarni ham qo'llab-quvvatlash kerak bo'lsa
