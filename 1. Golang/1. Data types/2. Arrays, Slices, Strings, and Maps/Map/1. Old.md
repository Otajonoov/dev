Map — один из основных типов данных Go, предназначенный для хранения пар ключ-значение. Это неупорядоченная коллекция, где каждый ключ должен быть уникальным и **comparable** (сравнимым).

В данном контексте **comparable** означает, что Go должен иметь возможность определить, равны ли два ключа с помощью операторов `==` или `!=`. Хотя значения могут быть практически любого типа, ключи должны поддерживать такое сравнение.

```go
func main() {
    m := map[string]int{
        "key1": 1,
        "key2": 2,
    }
    fmt.Println(m) // map[key1:1 key2:2]
}
```

Существует два основных способа объявления map. Можно использовать литералы map, как показано выше, или функцию `make`, которая работает аналогично инициализации slice. В отличие от slice, map не имеет фиксированного размера. Вместо этого можно указать подсказку по ёмкости (capacity hint):

```go
func main() {
    m := make(map[string]int, 10)
}
```

Важная деталь: ключи должны быть уникальными. Если вы вставляете ключ, который уже существует, Go перезапишет текущее значение новым.

Чтобы map могла определить, существует ли ключ, тип ключа должен быть **comparable**. Поэтому некоторые типы данных — такие как slice, map и функции — не могут использоваться в качестве ключей, так как они не **comparable** и не поддерживают `==` или `!=`. В таких случаях Go не может определить, являются ли два значения одинаковыми.

```go
func main() {
    // int
    m1 := map[int]int{}
    // string
    m2 := map[string]int{}
    // array
    m3 := map[[1]int]int{}
    // slice
    // ошибка компиляции: invalid map key type []int
    m4 := map[[]int]int{}
    // map
    // ошибка компиляции: invalid map key type map[string]int
    m5 := map[map[string]int]int{}
    // function
    // ошибка компиляции: invalid map key type func()
    m6 := map[func()]int{}
}
```

Этот пример показывает, как Go обрабатывает различные типы в качестве ключей. Простые **comparable** типы, такие как integer, string и array, работают без проблем. Но при попытке использовать несравнимые типы вы столкнётесь с ошибками компиляции.

Несмотря на то что компилятор Go отлично ловит несравнимые ключи на этапе компиляции, всё же есть способ «протащить» несравнимые типы в map: используя `interface{}` в качестве типа ключа.

В этом случае Go не заблокирует вас при компиляции, потому что `interface{}` обходит обычные ограничения типов. По сути, это «свободный пропуск» для использования любого типа, включая те, которые обычно не являются **comparable**:

```go
func main() {
    m := map[interface{}]int{
        1:                          1,
        "key":                      2,
        [1]int{1}:                  3,
        // panic: runtime error: hash of unhashable type [1]func()
        [1]func(){func() {}}:       4,
        // panic: runtime error: hash of unhashable type []int
        []int{1, 2}:                5,
        // panic: runtime error: hash of unhashable type map[string]int
        map[string]int{"key": 1}:   6,
        // panic: runtime error: hash of unhashable type func() int
        func() int { return 1 }:    7,
    }
    fmt.Println(m)
}
```

Вместо того чтобы поймать проблему заранее, Go позволяет обнаружить несравнимость типа ключа во время выполнения. Приложение паникует, потому что некоторые из этих ключей не могут быть хешированы, а хеширование необходимо для работы ключей в map. Это происходит каждый раз, когда вы используете `interface{}` как ключ и пытаетесь добавить несравнимые значения — не только при объявлении.

Ещё одно важное поведение, которое нужно понять — как Go обрабатывает **nil map**. Интересно, что чтение из неё не вызывает ошибку: она ведёт себя как пустая map и просто возвращает нулевые значения для любых запрашиваемых ключей. Проблема возникает при попытке записи в nil map:

```go
func main() {
    var m map[string]int
    fmt.Println(m)        // map[]
    fmt.Println(m["key"]) // 0
    m["key"] = 1          // panic: assignment to entry in nil map
}
```

Это поведение отражает строгую обработку типов в Go. Nil map ведёт себя как map только для чтения. При выводе она отображается как пустая map (`map[]`).

Кроме того, при инициализации map значениями с самого начала не всегда нужно явно указывать типы ключей и значений:

```go
type Person struct {
    Name string
    Age  int
}

func main() {
    // Явное указание типов
    expM := map[Person][]string{
        Person{Name: "Alice", Age: 30}: []string{"Apple", "Banana"},
        Person{Name: "Bob", Age: 25}:   []string{"Cherry", "Date"},
    }
    // Неявное указание типов
    impM := map[Person][]string{
        {Name: "Alice", Age: 30}: {"Apple", "Banana"},
        {Name: "Bob", Age: 25}:   {"Cherry", "Date"},
    }
}
```

### Внутренняя структура и расположение в памяти

Под капотом Go map использует структуру, называемую **hashtable** (хеш-таблица). Runtime Go занимается хешированием ключей, управлением коллизиями, изменением размера и эффективным распределением пар ключ-значение.

Если вы не знакомы с хеш-таблицами, представьте их как структуры с фиксированным количеством **bucket** (корзин) и hash-функцией (`hash(key)`), которая определяет, где будет храниться каждая пара ключ-значение:

![](../../../../assets/obsidian-images/Pasted%20image%2020260211163028.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Представьте хеш-таблицу с 4 bucket, и вы добавляете ключи "cat", "dog", "fish" и "bird". Hash-функция может распределить их так:

- `hash("cat") % 4` → slot 2
- `hash("dog") % 4` → slot 1
- `hash("fish") % 4` → slot 3
- `hash("bird") % 4` → slot 1 (коллизия с "dog")

Когда происходит коллизия, Go использует **chaining** (цепочки). Вместо того чтобы ограничивать каждый bucket одним элементом, каждый bucket становится списком, который может хранить несколько элементов. Это позволяет и "dog", и "bird" делить slot 1, образуя простой связный список:

![](../../../../assets/obsidian-images/Pasted%20image%2020260211163251.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Чтобы найти "bird" в такой конфигурации, поиск начинается со slot 1 и проходит по списку, пока ключ не будет найден.

- В большинстве случаев Go map обеспечивают время поиска **O(1)** благодаря эффективному хешированию.
- В худшем случае, когда слишком много ключей попадают в один bucket, время поиска может деградировать до **O(n)** для этого конкретного bucket (n — количество элементов в этом конкретном bucket, а не общее количество элементов в map).

Хотя это даёт базовое понимание работы хеш-таблицы, реализация map в Go добавляет больше сложности. Runtime управляет большей частью этого автоматически, динамически подстраивая структуру map во время выполнения, а не применяя строгие правила на этапе компиляции.

Как и slice (`[]T`) и string (`string`), map (`map[K]V`) тоже имеют внутреннее представление. Мы можем использовать `println` для проверки:

```go
func main() {
    m := map[string]int{
        "key1": 1,
        "key2": 2,
    }
    println(m)
}
// Output: 0x1400004e728
```

Вместо содержимого map вы видите адрес памяти. Это показывает, что тип `map[K]V`, который вы обычно используете, на самом деле является **pointer** (указателем) на более глубокую структуру. Эта структура называется `hmap` и управляет внутренней механикой map:

```go
// Заголовок Go map.
type hmap struct {
    count     int
    flags     uint8
    B         uint8          
    noverflow uint16       
    hash0     uint32        
    buckets   unsafe.Pointer 
    oldbuckets unsafe.Pointer 
    nevacuate uintptr       
    extra     *mapextra  
}
```

Структура `hmap` управляет массивом bucket, и количество bucket всегда следует степени двойки: 1, 2, 4, 8, 16 и так далее. Каждый раз, когда map растёт из-за слишком большого количества элементов, число bucket удваивается, обеспечивая достаточно пространства для эффективной обработки новых элементов.

Вот краткое описание каждого поля:

- **count**: Отслеживает текущее количество элементов в map. Поскольку это первое поле, Go может быстро получить размер map при вызове `len()`.
- **flags**: Хранит информацию о состоянии map: идёт ли изменение размера, выполняется ли итерация, или происходит одновременный доступ.
- **B**: Представляет степень двойки для количества bucket. Если B равно 4, map имеет 2⁴ = 16 bucket.
- **noverflow**: Даёт оценку количества используемых overflow bucket. Не точно, но достаточно для отслеживания.
- **hash0**: Уникальный seed, используемый для хеширования ключей. У каждой map свой hash seed для равномерного распределения.
- **buckets**: Указывает на массив bucket, где хранятся записи map.
- **oldbuckets**: Хранит ссылку на предыдущий массив bucket при изменении размера, позволяя переносить данные постепенно.
- **nevacuate**: Действует как счётчик прогресса для перемещения данных при изменении размера.
- **extra**: Хранит необязательные поля для дополнительного хранения или конфигурации.

При более детальном рассмотрении структуры `hmap` выявляются важные моменты:

- Каждый экземпляр map имеет отличающееся значение seed (`hash0`) для hash-функции, что помогает равномерно распределять ключи по bucket.
- Каждый bucket имеет лимит на количество элементов. Когда bucket заполняются из-за высокого уровня коллизий, overflow bucket подключаются как временное хранилище для переполненных элементов.

### Bucket и управление коллизиями

Go эти слоты называются **bucket**. Каждый bucket может хранить до **8 пар** ключ-значение.

![](../../../../assets/obsidian-images/Pasted%20image%2020260211165953.png)

Когда bucket A заполняется и ещё одна пара ключ-значение попадает в тот же слот, происходит коллизия. Вместо немедленного изменения размера map Go создаёт **overflow bucket**, связанный с исходным bucket A. Эта цепочка overflow может расти, при этом overflow bucket ссылаются на дополнительные overflow bucket по мере необходимости

![](../../../../assets/obsidian-images/Pasted%20image%2020260211170341.png)


Каждый bucket управляется структурой `bmap` (bucket map):

```go
// Bucket для Go map.
type bmap struct {
    tophash [bucketCnt]uint8
}
```

Помимо пар ключ-значение, каждый bucket содержит массив байтов, называемый **tophash**, который хранит верхний байт hash-значения каждого ключа.

Подождите — мы видим tophash, массив из 8 байтов (`tophash [bucketCnt]uint8`), но где ключи и значения в структуре `bmap`?

Оказывается, ключи и значения хранятся непосредственно **после** массива tophash в памяти. Однако эти данные недоступны напрямую — к ним нужно обращаться с помощью арифметики указателей (unsafe pointer arithmetic):

![](../../../../assets/obsidian-images/Pasted%20image%2020260211170707.png)|![](../../../../assets/obsidian-images/Pasted%20image%20.png)

Причина кроется в том, как была спроектирована реализация map в Go. 
Встроенный тип map (`map[K]V`) был разработан задолго до появления generics в Go 1.18, поэтому ему требовался способ динамически обрабатывать различные типы ключей и значений. Вот почему в структуре `bmap` нельзя определить конкретные типы ключей и значений.

Если представить, как могла бы выглядеть более универсальная структура bucket с использованием современных возможностей Go:

```go
type bmap[K comparable, V any] struct {
    tophash  [bucketCnt]uint8
    keys     [bucketCnt]K      // или [bucketCnt]*K
    values   [bucketCnt]V      // или [bucketCnt]*V
    overflow *bmap[K, V]
}
```

Это **не** то, как Go на самом деле реализует map внутри, но иллюстрирует, как generics теоретически могли бы работать. При работе с большими ключами Go не хранит весь ключ непосредственно в bucket. Вместо этого он хранит **pointer** на ключ, чтобы сохранить управляемый размер bucket.

Что Go считает «большим»? 
Это определяется на этапе компиляции. Если ключ или значение превышает **128 байт**, Go хранит pointer в bucket вместо фактических данных.

Следующий код показывает, как Go принимает это решение:

```go
func MapOf(key, elem Type) Type {
    ...
    if ktyp.Size_ > maxKeySize { // maxKeySize сейчас равен 128
        mt.KeySize = uint8(goarch.PtrSize)
        mt.Flags |= 1 // indirect key
    } else {
        mt.KeySize = uint8(ktyp.Size_)
    }
    if etyp.Size_ > maxValSize { // maxValSize сейчас равен 128
        mt.ValueSize = uint8(goarch.PtrSize)
        mt.Flags |= 2 // indirect value
    } else {
        mt.MapType.ValueSize = uint8(etyp.Size_)
    }
    ...
}
```

Флаги 1 и 2 в `MapType` указывают, хранит ли map pointer на ключ или значение вместо самих данных. Тем временем `maxKeySize` и `maxValSize` действуют как пороговые значения (в настоящее время 128 байт) для определения, когда ключ или значение считается «большим».

### Оптимизация с помощью tophash

Когда runtime Go хеширует ключ, он извлекает **верхний байт** hash-значения для использования в качестве сокращённого идентификатора — это и есть **tophash**.

Show Image

Поиск ключа в bucket может быть медленным, особенно если ключи большие, например struct с множеством полей. Именно здесь tophash вступает в игру — он действует как быстрый фильтр для сужения потенциальных совпадений. Он не гарантирует точного совпадения, но быстро исключает нерелевантные записи, делая процесс поиска гораздо эффективнее.

Представьте сценарий, где у map есть только один bucket, и два ключа, "dog" и "cat", попали в один bucket из-за коллизии:

```go
func main() {
    m := map[string]int{"dog": 1, "cat": 2}
    println(m["dog"]) // поиск "dog"
}
```

При поиске ключа "dog" (`println(m["dog"])`), Go не сравнивает "dog" напрямую с каждым ключом в bucket. Сначала runtime хеширует ключ, извлекает верхний байт hash и сравнивает его с tophash-значениями в bucket.

Если tophash совпадает, Go затем проверяет фактический ключ для подтверждения. Но что если tophash пустого слота равен 0, а tophash ключа тоже 0? В этом случае runtime Go **резервирует** tophash-значения от 0 до 4 для специальных целей. Если tophash ключа равен 0, он корректируется до 5:

go

```go
const minTopHash = 5

func tophash(hash uintptr) uint8 {
    top := uint8(hash >> (goarch.PtrSize*8 - 8))
    if top < minTopHash {
        top += minTopHash
    }
    return top
}
```

Помимо представления tophash ключа, эти значения также указывают **состояние** каждого элемента в bucket — например, пуст ли он, был ли перемещён и т.д.

Конкретные состояния определены в runtime Go как константы:

go

```go
const (
    emptyRest      = 0 // Ячейка пуста, и нет непустых ячеек в старших индексах или overflow.
    emptyOne       = 1 // Ячейка пуста.
    evacuatedX     = 2 // Запись эвакуирована в первую половину большей таблицы.
    evacuatedY     = 3 // Запись эвакуирована во вторую половину большей таблицы.
    evacuatedEmpty = 4 // Ячейка пуста, и bucket был эвакуирован.
    minTopHash     = 5 // Минимальный tophash для нормально заполненной ячейки.
)
```

## 4.2 Операционная семантика и паттерны использования

Большинство операций с map обрабатываются за кулисами пакетом runtime. Компилятор Go перезаписывает определённые части кода, связанного с map, в вызовы конкретных runtime-функций.

Таблица ниже показывает, как это работает:

|Go-код|Псевдокод|
|---|---|
|`m = make(map[K]V, hint)`|`runtime.makemap(m, hint)` → `hmap`|
|`m[key] = value`|`runtime.mapassign(m, key, value)`|
|`v = m[key]`|`runtime.mapaccess1(m, key, &value)`|
|`v, ok = m[key]`|`runtime.mapaccess2(m, key, &value, &ok)`|
|`delete(m, key)`|`runtime.mapdelete(m, key)`|
|`clear(m)`|`runtime.mapclear(m)`|

Go также адаптирует поведение в зависимости от типа ключа. Например, `map[string]V` ведёт себя немного иначе, чем `map[OtherType]V`. Мы сосредоточимся на общей версии операций с map, но основные концепции применимы и к другим типам ключей.

### Инициализация и выделение памяти для Map

При использовании `make(map[K]V, hint)` для создания новой map мы предоставляем runtime Go оценку необходимого пространства для хранения элементов. Эта оценка называется **hint** (подсказка), а не capacity или length, как для slice или array:

go

```go
func main() {
    m := make(map[string]int, 10)
    fmt.Println(len(m)) // 0
}
```

Хотя мы указали hint 10, вызов `len(m)` сразу после создания возвращает 0. Это потому, что длина map отражает фактическое количество хранимых пар ключ-значение, которое на данный момент равно нулю. Hint не влияет напрямую на сообщаемую длину или ёмкость.

Что он действительно влияет на — стратегию внутреннего выделения памяти. Предварительно выделяя bucket на основе hint, runtime уменьшает потребность в немедленном изменении размера, улучшая производительность.

Это поведение становится яснее при рассмотрении внутренней функции `runtime.makemap`:

go

```go
func makemap(t *maptype, hint int, h *hmap) *hmap {
    ...
    if h == nil {
        h = new(hmap)
    }
    // 1. Устанавливаем hash seed
    h.hash0 = uint32(rand())

    // 2. Определяем количество bucket на основе hint
    B := uint8(0)
    for overLoadFactor(hint, B) {
        B++
    }
    h.B = B

    // 3. Создаём bucket, возможно предварительно выделяя overflow bucket
    if h.B != 0 {
        var nextOverflow *bmap
        h.buckets, nextOverflow = makeBucketArray(t, h.B, nil)
        if nextOverflow != nil {
            h.extra = new(mapextra)
            h.extra.nextOverflow = nextOverflow
        }
    }
    return h
}
```

_Рисунок 37. map: makemap (src/runtime/map.go)_

Runtime Go использует hint для оценки начального количества bucket, устанавливая B так, чтобы map начиналась с 2^B bucket.

В предыдущем примере, где hint равен 10, Go создаёт 2 bucket. Поскольку каждый bucket может хранить до 8 пар, это даёт 16 слотов.

Однако дело не только в точном соответствии hint. Например, с hint в 14 элементов Go не останавливается на 2 bucket — даже если 16 слотов технически покрывают потребность. Вступает в силу **load factor** (коэффициент загрузки), определяющий, насколько заполненными должны быть bucket для поддержания производительности. В этом случае runtime выделяет 3 bucket, чтобы не достичь порога слишком рано.

Если hint не указан, runtime упрощает процесс, используя облегчённую версию `makemap` — `runtime.makemap_small`. Эта функция оптимизирована для создания небольших map и вызывается, когда hint отсутствует или достаточно мал (обычно менее 8 элементов):

go

```go

    h := new(hmap)

    return h
}
```

### Поиск и получение значений

При доступе к значению map с помощью `v := m[key]` компилятор Go перезаписывает операцию как вызов `runtime.mapaccess1`. Для эффективности существуют специализированные версии для разных типов ключей — такие как `runtime.mapaccess1_fast32`, `runtime.mapaccess1_fast64` и `runtime.mapaccess1_faststr`.

Цифра 1 в `mapaccess1` сигнализирует о существовании другого варианта. Функция `mapaccess2` возвращает и значение, и boolean, указывающий, существует ли ключ в map. Двойной return полезен, когда вам нужно знать не только значение, но и был ли найден ключ.

Это даёт два способа доступа к значениям map:

- `v := m[key]` — получает значение.
- `v, ok := m[key]` — получает значение и проверяет, существует ли ключ.

Если вы читаете `m[key]` из nil или пустой map (`h.count == 0`), Go немедленно возвращает нулевое значение для данного типа. Как упоминалось ранее, чтение из nil map не вызывает panic, но запись — вызывает. Есть одно исключение: если ключ не **comparable**, доступ вызовет panic, вне зависимости от наличия данных.

### Вставка и обновление записей

Добавление пары ключ-значение в map управляется функцией `mapassign`, которая обрабатывает как вставку новых пар, так и обновление существующих. Эта операция более сложна, потому что требует и чтения, и записи.

Сначала Go проверяет, существует ли ключ (чтение); только затем может вставить новое значение или обновить существующее (запись).

**1. Проверка состояния map**

Попытка записи в nil map вызывает **panic**. Но запись в map из нескольких goroutine одновременно — это серьёзнее и приводит к **fatal error**.

В чём разница между panic и fatal error?

Присваивание в nil map вызывает panic, потому что это проблема, ограниченная одной goroutine — её можно перехватить и обработать. Одновременная запись отражает фундаментальную проблему конкурентности. Go не допускает восстановления из таких race condition; вместо этого выполнение полностью останавливается:

go

```go
func mapassign(t *maptype, h *hmap, key unsafe.Pointer) unsafe.Pointer {
    if h == nil {
        panic(plainError("assignment to entry in nil map"))
    }
    ...
    if h.flags&hashWriting != 0 {
        fatal("concurrent map writes")
    }
    ...
    h.flags ^= hashWriting
}
```

Рост map не происходит разом; это постепенный процесс, разворачивающийся на протяжении нескольких операций. Каждый раз, когда добавляется новая запись, map делает шаг вперёд в изменении размера, если фаза роста идёт.

**3. Итерация по bucket**

Этот шаг повторяет то, что происходит в `mapaccess`: tophash ключа сравнивается с tophash в каждом слоте. Если совпадения нет, Go проверяет, пуст ли слот. Если да, и мы в режиме вставки, слот становится кандидатом для добавления новой пары ключ-значение.

Существует три состояния для пустых слотов, но только два допустимы для новых записей:

- **evacuatedEmpty (4)**: Слот уже мигрировал в новый bucket и не должен использоваться для новых записей.
- **emptyRest (0)**: Сигнализирует, что дальше нет непустых слотов, позволяя прекратить поиск.
- **emptyOne (1)**: Слот пуст и доступен для новых записей, но дальше могут быть ещё ключи. Нужно продолжить цикл.

- **Load factor** (коэффициент загрузки) map превышает лимит.
- Слишком много overflow bucket также указывают на необходимость перераспределения.

### Удаление и освобождение слотов

Для удаления пары ключ-значение из map в Go используется встроенная функция `delete`:

### Стратегия роста и rehashing

Overflow bucket предлагают быстрое временное решение для хранения дополнительных элементов при заполнении bucket, но они плохо влияют на производительность. Чем больше overflow bucket у map, тем дольше поиск пары ключ-значение, так как время поиска линейно увеличивается с каждым overflow. Хуже того, это замедление влияет на **все** операции map — поиск, вставку и удаление.

Для исправления этого map должна вырасти. Рост означает создание большего количества bucket и перераспределение существующих пар ключ-значение для балансировки нагрузки. Иногда map растёт без увеличения общего числа bucket — процесс, называемый **same-size grow** (рост того же размера).

Рост запускается только при **добавлении** новых элементов:

go

```go
func mapassign(t *maptype, h *hmap, key unsafe.Pointer) unsafe.Pointer {
    ...
    if !h.growing() && (overLoadFactor(h.count+1, h.B) || tooManyOverflowBuckets(h.noverflow, h.B)) {
        hashGrow(t, h)
        goto again
    }
    ...
}
```

Обратите внимание: **map в Go не уменьшаются**; они либо растут, либо остаются прежними. При удалении элементов через `delete()` счётчик уменьшается и слот помечается как пустой, но размер массива bucket (контролируемый `h.B`) никогда не уменьшается.

Существовал открытый issue «runtime: shrink map as elements are deleted» [shrinkmap], который оставался открытым до Go 1.23.

Рост не происходит разом; это постепенный процесс. Go изменяет размер map лениво, обрабатывая только необходимое при каждой операции. Два основных условия запускают рост:

1. **Высокий load factor**: Map слишком заполнена.
2. **Слишком много overflow bucket**: Признак плохого распределения, ведущего к частым коллизиям.

В первом случае рост добавляет больше bucket. Во втором — map перераспределяет существующие элементы без увеличения числа bucket — **same-size grow**.

**Load factor** определяет, когда map нужно расти. Это коэффициент, измеряющий, насколько заполненными могут быть bucket до начала изменения размера. В Go порог составляет около **80%**:

go

```go
const (
    loadFactorDen = 2
    loadFactorNum = loadFactorDen * bucketCnt * 13 / 16
)

func overLoadFactor(count int, B uint8) bool {
    return count > bucketCnt && uintptr(count) > loadFactorNum*(bucketShift(B)/loadFactorDen)
}
```

Хотя 2 bucket технически могут вместить 14 элементов, превышение порога 80% (13) побуждает map расшириться до 4 bucket для лучшего распределения.

Overflow bucket также могут запустить рост. Когда их накапливается слишком много — это сигнал увеличения коллизий и вероятной деградации производительности. Overflow bucket считаются чрезмерными, когда их количество совпадает или превышает число основных bucket, с ограничением в 15:

go

```go
func tooManyOverflowBuckets(noverflow uint16, B uint8) bool {
    if B > 15 {
        B = 15
    }
    return noverflow >= uint16(1) << (B & 15)
}
```

Когда рост начинается, поле `buckets` в `hmap` указывает на новый набор bucket, а `oldbuckets` хранит ссылку на старые. Процесс разворачивается постепенно, управляемый функциями `growWork()` и `evacuate()`. Это ленивое изменение размера распределяет нагрузку по нескольким операциям.

Два действия движут рост map: присваивание и удаление. При каждой из этих операций bucket полностью мигрирует в новый массив:

go

```go
func growWork(t *maptype, h *hmap, bucket uintptr) {
    evacuate(t, h, bucket & h.oldbucketmask())
    if h.growing() {
        evacuate(t, h, h.nevacuate)
    }
}
```

Каждая единица работы обрабатывает два типа эвакуации bucket:

1. **Целевая эвакуация**: Фокусируется на текущем bucket.
2. **Стабильная эвакуация**: Поддерживает продвижение роста, эвакуируя хотя бы один дополнительный старый bucket.

**1. Same-size Grow**

Со временем повторные вставки и удаления могут привести к накоплению overflow bucket из-за hash-коллизий. Даже если элементы удалены, overflow bucket остаются, потребляя память и снижая эффективность.

Для решения этого Go использует **same-size grow**:

Show Image

Этот процесс перераспределяет элементы в чистый набор bucket, оставляя старые overflow bucket позади. Garbage collector затем очищает эти заброшенные bucket.

**2. Double-Size Grow**

Когда map удваивается в размере, все существующие элементы должны переместиться из старых bucket в новые. Может показаться, что каждый ключ нужно перехешировать и вставить заново — но это не так.

Поскольку hash-функция и seed остаются прежними, у каждой записи из старой map есть только **два возможных** места назначения в большей map.

Представьте map с 4 bucket, которая растёт до 8:

Show Image

С 4 bucket используются только последние 2 бита hash-значения для определения bucket. Например, hash, заканчивающийся на `11₂`, попадёт в bucket 3 (11 = 3 в десятичной). При росте до 8 bucket Go начинает использовать последние 3 бита hash вместо 2.

- Если дополнительный бит равен 0, ключ остаётся в исходном bucket.
- Если дополнительный бит равен 1, ключ перемещается в новый bucket со смещением на размер старой map.

Таким образом, ключ, изначально попавший в bucket 3, имеет два возможных назначения: он либо остаётся в bucket 3, либо перемещается в bucket 7:

Show Image

После определения новых мест Go начинает перемещать элементы. Он итерирует по старому bucket и его overflow bucket, перемещает каждую пару ключ-значение и помечает старые bucket как эвакуированные.

### Цикл for-range по Map

Один из наиболее интересных аспектов Go map — как работает итерация. Порядок, в котором элементы появляются в цикле `for-range`, **не фиксирован** — он меняется от одного цикла к другому.

go

```go
func main() {
    m := make(map[string]int)
    m["key1"] = 1
    m["key2"] = 2
    m["key3"] = 3
    m["key4"] = 4
    m["key5"] = 5

    for k, v := range m {
        fmt.Println(k, v)
    }
}
// Пример вывода:
// key4 4
// key5 5
// key1 1
// key2 2
// key3 3
```

Порядок итерации не привязан к последовательности вставки. Однако вывод не полностью случаен — он следует определённому паттерну.

Каждый раз при выводе map итерация начинается в случайной точке и продолжается в порядке вставки, заворачиваясь в конце.

**Как итерация работает внутри**

Go использует **iterator** для управления текущей позицией при обходе hash-таблицы. Из-за особенностей хеширования map не располагаются в простом линейном порядке.

**1. Фаза инициализации**

При начале итерации Go инициализирует iterator, представленный структурой `hiter`:

go

```go
type hiter struct {
    key         unsafe.Pointer
    elem        unsafe.Pointer
    t           *maptype
    h           *hmap
    buckets     unsafe.Pointer
    bptr        *bmap
    overflow    *[]*bmap
    oldoverflow *[]*bmap
    ...
}
```

_Рисунок 54. map: hiter (src/runtime/map.go)_

Iterator захватывает текущую структуру map. Это не глубокая копия — скорее закладка расположения и состояния map на момент начала цикла.

Итерация **не начинается** с bucket 0. Вместо этого она начинается со **случайного bucket** и **случайного слота** внутри этого bucket. Эта случайность препятствует зависимости от предсказуемого порядка итерации.

Show Image

go

```go

    ...
    it.B = h.B
    it.buckets = h.buckets
    ...
    // Начинаем со случайного bucket и случайного слота
    r := uintptr(rand())
    it.startBucket = r & bucketMask(h.B)
    it.offset = uint8(r >> h.B & (bucketCnt - 1))
    ...
}
```

**2. Фаза итерации**

Go map не безопасны для конкурентного использования. Можно иметь несколько читателей или запускать несколько циклов `for-range` одновременно. Но при появлении записей нужна внешняя синхронизация.

Итерация начинается со случайного слота в случайно выбранном bucket и сканирует все непустые слоты. После завершения переходит к следующему bucket (`bucket++`). Интересно, что даже при переходе к новому bucket итерация **не начинается** с начала — она продолжается с того же случайного слота, что и начальный bucket.

Show Image

Впечатляющий аспект итерации Go map — обработка изменения размера во время итерации. Даже если map растёт в середине цикла, iterator отслеживает оба массива bucket (старый и новый), чтобы ни один элемент не был пропущен или посещён дважды.

**Эксперимент: Модификации map во время итерации**

Что произойдёт, если вставлять новые пары ключ-значение внутри цикла?

go

```go
func main() {
    m := map[int]int{0: 1}
    for i := range m {
        fmt.Println(i)
        for k := range 100 {
            m[k] = k + 1
        }
    }
}
```

Map начинается с одной записи `{0: 1}`, и при каждой итерации добавляются 100 новых пар. Какой результат? Сколько строк выведется?

Когда цикл начинается, iterator делает снимок текущего состояния map — количество bucket (`h.B`) и pointer на массив bucket. Изначально map имеет один bucket с 8 слотами, но только один занят.

Вывод зависит от того, где начнёт iterator. Два основных сценария:

- **Iterator начинает со слота 0**: выводится **8 строк**. После обработки слота 0 новые элементы заполняют оставшиеся слоты, и iterator продолжает через них.
- **Iterator начинает с другого слота**: выводится **1 строка**. Допустим, iterator начинает со слота 1 — в нём ничего нет, он проходит bucket, возвращается к слоту 0, выводит его и останавливается.

Show Image

## Xulosa (Мои выводы)

Go'dagi map — bu oddiy `map[K]V` sintaksisi ortida yashiringan murakkab va aqlli data structure. Men bu bobni o'qib, bir nechta muhim xulosalarga keldim:

**1. Map aslida pointer ekanligini tushunish juda muhim.** `map[K]V` yozganingizda, aslida `*hmap` pointer bilan ishlayapsiz. Shuning uchun map'ni funksiyaga argument sifatida berganingizda, u copy bo'lmaydi — o'sha aynan bitta map ustida ishlaysiz. Slice'dan farqli o'laroq, slice header copy bo'ladi, lekin map to'g'ridan-to'g'ri pointer.

**2. Tophash optimizatsiyasi — juda oqilona yechim.** Har bir bucket'da 8 ta key-value pair bo'lishi mumkin, va ularni solishtirish qimmat operatsiya bo'lishi mumkin (masalan, katta struct key bo'lsa). Tophash — hash'ning faqat bitta yuqori byte'i — "quick filter" vazifasini bajaradi. Bu 1 byte solishtirish orqali ko'p key'larni darhol "bu emas" deb rad qilish mumkin, to'liq key comparisonga hojat qolmaydi.

**3. Map hech qachon kichraymaydi — bu Go'ning mashhur muammolaridan biri.** `delete()` bilan elementlarni o'chirsangiz ham, bucket array o'lchami (`h.B`) kamaymaydi. Bu katta map'larni to'ldirib, keyin elementlarni o'chirsangiz, memory bo'shatilmasligini bildiradi. Go 1.23 gacha bu issue ochiq edi. Amalda buning yechimi — agar map juda katta bo'lib qolsa va elementlar o'chirilgan bo'lsa, yangi map yaratib, kerakli elementlarni ko'chirish.

**4. Concurrent access — fatal error, panic emas.** Bu juda muhim farq. Panic'ni `recover()` bilan ushlash mumkin, lekin fatal error'ni — yo'q. Go concurrent map access'ni dastur dizaynidagi fundamental xato deb hisoblaydi. Production'da `sync.RWMutex` yoki `sync.Map` ishlatish **shart**.

**5. Iteration randomness — bu Go'ning ataylab qilgan dizayn qarori.** Har safar `for-range` ishlatganingizda, iterator random bucket'dan va random slot'dan boshlaydi. Bu developer'larni iteration tartibiga tayanishdan qaytaradi. Agar sizga tartiblangan iteration kerak bo'lsa, key'larni alohida slice'ga solib, sort qilib, keyin iterate qilishingiz kerak.

**6. Growth strategiyasi — lazy va incremental.** Map bir vaqtning o'zida barcha elementlarni yangi bucket'larga ko'chirmaydi. Har bir `mapassign` yoki `mapdelete` operatsiyasida faqat 1-2 ta bucket evacuate qilinadi. Bu "amortized cost" yondashuvidir — har bir alohida operatsiya biroz sekinroq, lekin hech qachon katta "stop-the-world" pause bo'lmaydi.

**Amaliy maslahatlar:**

- Map yaratayotganda, agar taxminiy o'lchamni bilsangiz, `make(map[K]V, hint)` bilan hint bering — bu keraksiz resize'larni oldini oladi.
- Concurrent access uchun doim `sync.RWMutex` yoki `sync.Map` ishlating.
- Katta map'larni "shrink" qilish uchun yangi map yaratib, elementlarni ko'chiring.
- Key type sifatida string yoki integer ishlating — ular uchun Go'da maxsus optimizatsiyalar bor (`mapaccess1_faststr`, `mapaccess1_fast32`).
