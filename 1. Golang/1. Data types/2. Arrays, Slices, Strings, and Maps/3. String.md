## String'lar va slice'larni stream'larga moslash

`bytes` va `strings` paketlarining eng muhim xususiyati - ularda xotiradagi baytlar va string'lar bilan ishlash uchun `io.Reader` va `io.Writer` interface'lari amalga oshirilgan.

### In-memory reader'lar

Go standart kutubxonasida kam ishlatiluvchi ikkita funksiya - `bytes.NewReader` va `strings.NewReader`:

```go
func NewReader(b []byte) *Reader
func NewReader(s string) *Reader

```

Bu funksiyalar xotiradagi bayt slice yoki string atrofida wrapper bo'lgan `io.Reader` interface implementatsiyasini qaytaradi.

Men tez-tez bunday kodni ko'raman - bayt slice yoki string avval `bytes.Buffer`ga yoziladi, keyin buffer reader sifatida ishlatiladi:

```go
var buf bytes.Buffer
buf.WriteString("foo")
http.Post("<http://example.com/>", "text/plain", &buf)

```

Bu yondashuv ortiqcha xotira ajratishni talab qiladi va sekin bo'lishi mumkin. `strings.Reader` ishlatish ancha samarali:

```go
r := strings.NewReader("foobar")
http.Post("<http://example.com>", "text/plain", r)

```

Ko'plab string'lar yoki bayt slice'larni birlashtirish kerak bo'lsa, `io.MultiReader` ishlatish mumkin:

```go
r := io.MultiReader(
    strings.NewReader("HEADER"),
    bytes.NewReader([]byte{0,1,2,3,4}),
    myFile,
    strings.NewReader("FOOTER"),
)

```

Bayt slice'lar va string'lar Go'da fundamental primitivlardir. 
Ular xotiradagi baytlar yoki rune'larning vakilidir. `
bytes va `strings` paketlari ko'plab yordamchi funksiyalar, shuningdek `io.Reader` va `io.Writer` interface'lari uchun adapterlar taqdim etadi.

Go tilida `string` o'zgarmas(immutable) ma'lumotlar turi, u ikki qismdan iborat:

1. **Baytlar ketma-ketligiga pointer**: string hotiradagi baytlar ketma ketligining birinchi baytining manzilini saqlaydi
2. **Qator uzunligi**: baytlar ketma-ketligining uzunligini.

Go tilida "rune" tushunchasi — qatorlarni qanday ishlashini tushunishning asosiy kalitidir.

Biz charset va kodlash (encoding) o'rtasidagi farqni tushunishimiz kerak:

- **Charset (kodlash tizimi)** — bu belgilar to'plami. Masalan, Unicode kodlash tizimi 2^21 ta belgini o'z ichiga oladi.
- **Encoding (kodlash)** — bu belgilar ro'yxatini ikkilik kodga o’tirish jarayoni. Masalan, UTF-8 bu kodlash standarti bo'lib, Unicode’dagi barcha belgilarni ikkilik formatda kodlash imkoniyatiga ega.

UTF-8 belgilarni 1 dan 4 baytgacha kodlaydi, ya’ni maksimal 32 bitgacha. Shu sababli, Go tilida **rune** — bu `int32`

Quyidagi qatorda beshta belgi bor: **h, e, l, l** va **o**. Ushbu oddiy belgilar har biri bitta baytda kodlanadi. Shu sababli `len` funksiyasi 5 qaytaradi:

```go
s := "hello"
fmt.Println(len(s)) // 5
```

Ammo har bir belgi har doim bitta baytda kodlanmaydi. Masalan, **汉** belgisi UTF-8’da uchta baytda kodlanadi. Buni quyidagicha tekshirishimiz mumkin:

```go
s := "汉"
fmt.Println(len(s)) // 3
```

Bu yerda 1 emas, 3 qaytariladi. Chunki `len` funksiyasi qatorning uzunligini emas, balki undagi baytlar sonini qaytaradi.

По своей сути строка — это последовательность символов, но технически она ведет себя больше как read-only slice байтов.

Хотя строки имеют некоторые сходства со slice'ами в том, как они ссылаются на данные, есть фундаментальная разница: строки неизменяемы (immutable). После создания строки ее данные не могут быть изменены через обычные операции. Эта неизменяемость является ключевой особенностью и формирует то, как строки функционируют в Go.

Чтобы лучше понять это, давайте посмотрим на внутреннюю структуру строки в Go:

```go
type stringStruct struct {
  str unsafe.Pointer
  len int
}
// stringStruct (src/runtime/string.go)**
```


==str== — это pointer на первый байт данных строки в памяти — оно отмечает, где начинаются данные, аналогично тому, как работает slice. 
==len== содержит длину строки в байтах. В отличие от slice'ов, строки не имеют поля capacity в своем заголовке, что означает, что они занимают всего 16 байт на 64-битной системе.

При создании строк Go предлагает два типа литералов:

- **Raw string literals**: Заключенные в обратные кавычки (\`...\`) они захватывают все в точности, как написано, включая пробелы, табуляции и переносы строк. То, что вы видите в коде, — это именно то, что вы получаете.
- **Interpreted string literals**: Заключенные в двойные кавычки (""), они поддерживают escape-последовательности. Escape-последовательность начинается с обратной косой черты (\) и представляет определенные символы, такие как \n для новой строки или \t для табуляции.

```go
func main() {
  raw := `String is a
  sequence of characters`
  
  interpreted := "String is a \nsequence of characters"
}
```

Теперь, когда вы присваиваете строку другой переменной или делаете slice строки, разделяют ли они один и тот же базовый массив байтов, или Go дублирует данные?

```go
func main() {
  a := "Hello, World!"
  b := a[:2]
  
  println(unsafe.StringData(a))  // 0x16f8fb4d8
  println(unsafe.StringData(b))  // 0x16f8fb4d8 (same)
}
```

В этом примере как a, так и b указывают на одно и то же местоположение в памяти. Это показывает, что присваивание строки другой переменной — или ее slicing — не дублирует базовые данные, независимо от размера строки. Копируется только заголовок строки.

Это может показаться рискованным, поскольку разделяемая память обычно означает, что изменения одной переменной могут повлиять на другую. Однако со строками это не проблема, потому что любая операция, которая изменяет строку, создает новую строку. Go обеспечивает неизменяемость, предотвращая прямые изменения данных строки через стандартные операции.

Интересная деталь заключается в том, что вы не можете использовать &a[0] для получения адреса первого байта, как вы бы делали со slice. Разрешение этого открыло бы базовые байты и открыло бы дверь для непреднамеренных модификаций, нарушая неизменяемость.

Вместо этого Go предоставляет unsafe.StringData, который безопасно возвращает адрес первого байта как *byte. Префикс unsafe — это четкий сигнал о том, что вы обходите обычные проверки безопасности Go. Тем не менее, даже с unsafe.StringData вы все равно не можете изменить содержимое строки:

```go
func main() {
  a := "Hello, World!"
  bptr := unsafe.StringData(a)
  *bptr = 'h'
}

// unexpected fault address 0x102f5c7bc
```

Это происходит потому, что строковые литералы хранятся в выделенной секции памяти только для чтения.

### **UTF-8 кодирование и внутреннее представление**

**ASCII** (American Standard Code for Information Interchange) был одним из самых ранних стандартов кодирования символов. Он использовал 7 бит на символ, что позволяло 2⁷ (128) уникальных символов.

Возьмем строку "Hi" в качестве примера. Каждый символ ASCII имеет соответствующий десятичный код: 
	"H" отображается на 72,
	 "i" отображается на 105.

![](../../../assets/obsidian-images/Pasted%20image%2020260204162043.png)

В памяти "Hi" хранится как последовательность байтов 01001000 01101001₂ причем каждый символ занимает один байт.

ASCII хорошо работал для английского текста, сохраняя каждый символ в диапазоне 0-127. Печать символов ASCII в Go проста:

```go
func main() {
  println(string(65))  // A is the 65th character in ASCII
  println(string(97))  // a
  println(string(48))  // 0
}
```

По мере того как компьютеры распространялись по всему миру, стало ясно, что нужна более гибкая система для поддержки языков помимо английского. 
Это привело к **Unicode**, разработанному для размещения действительно глобального набора символов.

==Unicode== теперь включает более 140,000 символов и продолжает расти, охватывая все, от английских букв до китайских иероглифов, эмодзи и специализированных символов.

Unicode присваивает каждому символу уникальный номер, известный как ==code point==. Однако сам Unicode не диктует, как эти code point'ы хранятся в памяти. Вот тут и появляется UTF-8.

Мы уже видели, как работает ASCII — он использует один байт для каждого символа, поскольку есть только 128 символов для представления. Десятичное значение каждого символа просто преобразуется в двоичное для хранения.

Теперь давайте возьмем что-то более сложное, например, эмодзи "😊". Это улыбающееся лицо имеет Unicode code point U+1F60A.

![](../../../assets/obsidian-images/Pasted%20image%2020260204162804.png)

Выделение 4 байт для каждого символа Unicode может показаться простым решением, поскольку Unicode покрывает более 140,000 символов. Но это не так просто — или эффективно.

Проблема в том, что большинству символов Unicode не нужны 4 байта. Возьмем букву "H" из примера выше — ей требуется только 1 байт. Фактически, для большинства повседневного текста, особенно на английском или других языках на основе латиницы, символы могут храниться, используя всего 1 или 2 байта. Принуждение каждого символа к 4 байтам приведет к значительной трате памяти.

Вот где сияет **UTF-8**. Он экономит место, корректируя размер хранилища на основе того, что на самом деле нужно каждому символу. Вместо использования подхода "один размер для всех", UTF-8 использует систему кодирования переменной ширины. В зависимости от символа он может использовать от 1 до 4 байт:

- Базовые символы ASCII (U+0000 to U+007F) помещаются в 1 байт.
- Многие европейские языковые символы используют 2 байта.
- Большинство восточноазиатских символов нуждаются в 3 байтах.
- Дополнительные символы, такие как эмодзи, требуют 4 байта.

Для ясности, в U+0000 to U+007F, "U" означает Unicode, а "+" отделяет префикс Unicode от шестнадцатеричного code point'а.

Вы можете увидеть это поведение в действии с простой программой Go:

```go
func main() {
  char1 := "A"
  char2 := "ñ"
  char3 := "字"
  char4 := "😊"
  
  fmt.Printf("Character: %s, Bytes: %d\n", char1, len(char1))  // 1 byte
  fmt.Printf("Character: %s, Bytes: %d\n", char2, len(char2))  // 2 bytes
  fmt.Printf("Character: %s, Bytes: %d\n", char3, len(char3))  // 3 bytes
  fmt.Printf("Character: %s, Bytes: %d\n", char4, len(char4))  // 4 bytes
}

// Output:
// Character: A, Bytes: 1
// Character: ñ, Bytes: 2
// Character: 字, Bytes: 3
// Character: 😊, Bytes: 4
```

Еще одно преимущество UTF-8 заключается в том, что он обратно совместим с ASCII. Символы ASCII остаются закодированными в одном байте, как всегда. 
Интересно, что UTF-8 был совместно разработан Кен Томпсоном и Роб Пайком — двумя из тех же умов, стоящих за языком Go.

UTF-8 следует четкому набору битовых паттернов для эффективного отображения Unicode code point'ов в байты:

- **1-byte character**: Для code point'ов в диапазоне U+0000 to U+007F, UTF-8 использует один байт. Битовый паттерн следует 0xxxxxxx₂, где первый бит — 0, а остальные 7 бит хранят символ.
- **Multi-byte characters**: Символы, нуждающиеся в 2, 3 или 4 байтах, следуют определенным паттернам: 
- 110xxxxx₂ для 2-byte characters 
- 1110xxxx₂ для 3-byte characters
- 11110xxx₂ для 4-byte characters
- Для multi-byte последовательностей, continuation  байты начинаются с 10xxxxxx₂

![](../../../assets/obsidian-images/Pasted%20image%2020260204165037.png)

Ключевое слово range делает цикл по строкам интуитивным — оно итерируется по Unicode code point'ам вместо сырых байтов. Это позволяет вам обрабатывать символы напрямую без запутывания в деталях байтового уровня:

```go
func main() {
  s := "Hello, 世界"
  
  for i, r := range s {
	  fmt.Printf("%#U starts at byte position %d\n", r, i)
  }
}

// U+0048 'H' starts at byte position 0
// U+0065 'e' starts at byte position 1
// U+006C 'l' starts at byte position 2
// U+006C 'l' starts at byte position 3
// U+006F 'o' starts at byte position 4
// U+002C ',' starts at byte position 5
// U+0020 ' ' starts at byte position 6
// U+4E16 '世' starts at byte position 7
// U+754C '界' starts at byte position 10
```

Этот вывод показывает Unicode code point каждого символа вместе с его начальной позицией байта в строке. 
Переменная r — это rune — тип Go для представления Unicode code point'а.

Что происходит, когда вы вызываете len(..) на строке? Точнее, что на самом деле считает поле len заголовка строки?

```go
func main() {
  s := "Hello, 世界"
  
  println(len(s))                      // 13
  println(utf8.RuneCountInString(s))   // 9
  println(len([]rune(s)))              // 9
}
```

Вызов len на строке возвращает количество байтов, а не количество символов. 

Есть нюанс, который нужно иметь в виду: некоторые символы, такие как определенные эмодзи, на самом деле могут состоять из нескольких Unicode code point'ов.

Возьмем иконку "👨‍👩‍👧‍👦" в качестве примера:

```go
len(s)                      // 25
utf8.RuneCountInString(s)   // 7
len([]rune(s))              // 7
```

Хотя визуально это выглядит как один эмодзи, он состоит из нескольких Unicode code point'ов, объединенных для создания одного символа. Вот почему длина в байтах намного больше, чем количество rune'ов.

В Go цикл for-range не итерирует через сырые байты — он разработан для работы напрямую с Unicode code point'ами. Это позволяет обрабатывать полные символы без беспокойства о базовой структуре байтов.

За кулисами каждая итерация цикла включает несколько дополнительных шагов. Go создает три временные переменные во время каждого цикла:

- **hv1**: текущая позиция байта.
- **hv1t**: копия hv1, отмечающая начальную позицию байта текущего символа.
- **hv2**: значение текущего символа Unicode (rune).

Вот упрощенная версия того, что происходит внутренне — это примерно то, как Go обрабатывает цикл for-range для обработки полных символов вместо сырых байтов:

```go
for hv1 := 0; hv1 < len(s); {
  hv1t := hv1
  hv2 := rune(s[hv1])
  
  if hv2 < utf8.RuneSelf {
    hv1++
  } else {
    hv2, hv1 = decoderune(s, hv1)
  }
  
  v1, v2 = hv1t, hv2
  // the loop body
}
```

Цикл начинается с hv1 = 0 и продолжается, пока hv1 < len(s). Если байт в hv1 представляет однобайтный символ, hv1 увеличивается на 1. 
Для многобайтных символов Go использует decoderune для декодирования всего символа Unicode, возвращая как rune, так и индекс следующего символа.

### **3.2 Преобразование строк и семантика памяти**

Go позволяет преобразовывать несколько типов в строки:
- int
- rune
- slice'ы байтов
- slice'ы rune'ов.

При преобразовании целого числа в строку речь идет не о превращении числа 10 в "10". Вместо этого преобразование отображает целое число на его соответствующий символ Unicode. Та же логика применяется к rune'ам:

```go
func main() {
  println(string(65))                           // A
  println(string([]byte{72, 101, 108, 108, 111}))  // Hello
  println(string('H'))                          // H
  println(string([]rune{72, 101, 108, 108, 111}))  // Hello
}
```

Как string(65), так и string('H') преобразуют число или rune в символ на основе его Unicode code point'а. За кулисами эти преобразования полагаются на функцию Go runtime.intstring():

```go
func intstring(buf *[4]byte, v int64) (s string) {
  var b []byte
  if buf != nil {
    b = buf[:]
    s = slicebytetostringtmp(&b[0], len(b))
  } else {
    s, b = rawstring(4)
  }
  
  if int64(rune(v)) != v {
    v = runeError
  }
  
  n := encoderune(b, rune(v))
  return s[:n]
}
// intstring (src/runtime/string.go)**
```

Под капотом компилятор настраивает 4-байтный буфер на stack'е и передает его в intstring(). Эта функция записывает UTF-8 байты целого числа в буфер, который хранится в поле stringHeader.str.

Другие типы — такие как slice'ы байтов или rune'ов — обрабатываются аналогичным образом, каждый со своей оптимизированной функцией runtime. Например, преобразование slice байтов в строку использует runtime.slicebytetostring, в то время как преобразование slice rune'ов полагается на runtime.slicerunetostring.

При преобразовании байтового slice в строку в Go runtime использует буфер, который может обрабатывать до 32 байт. Если slice, который вы преобразуете, больше этого, он должен выделить новый буфер на heap, чтобы вместить все:

```go
func slicebytetostring(buf *tmpBuf, ptr *byte, n int) string {
  if n == 0 {
    return ""
  }
  ...
  
  if n == 1 {
    p := unsafe.Pointer(&staticuint64s[*ptr])
    if goarch.BigEndian {
      p = add(p, 7)
    }
    return unsafe.String((*byte)(p), 1)
  }
  
  var p unsafe.Pointer
  if buf != nil && n <= len(buf) {
    p = unsafe.Pointer(buf)
  } else {
    p = mallocgc(uintptr(n), nil, false)
  }
  
  memmove(p, unsafe.Pointer(ptr), uintptr(n))
  return unsafe.String((*byte)(p), n)
}
// slicebytetostring (src/runtime/string.go)**
```


Эта функция обрабатывает четыре различных сценария при преобразовании байтового slice в строку с определенными оптимизациями:

1. **Пустой байтовый slice**: Немедленно возвращает пустую строку. Эта ранняя проверка оптимизирована для распространенных случаев в обработке текста.
2. **Однобайтный slice**: Использует предварительно выделенный массив под названием staticuint64s для быстрого преобразования одного байта в строку без дополнительного выделения памяти.
3. **Малый байтовый slice (≤ 32 байта)**: Использует буфер, предоставленный вызывающей стороной, или предварительно выделенный stack-буфер для прямого хранения строки.
4. **Большой байтовый slice (> 32 байта)**: Выделяет новый буфер на heap, если нет доступного существующего буфера или slice превышает 32 байта.

Ключевая деталь здесь заключается в том, что даже если базовые байты выделены на heap, сама строка не всегда выделяется на heap. Если выполняются определенные условия, строка остается stack-allocated, с stringHeader.str (unsafe.Pointer), указывающим на heap-буфер:

```go
// staticuint64s используется для избежания выделения в convTx для малых целочисленных значений.
var staticuint64s = [...]uint64{
  0x00, 0x01, 0x02, 0x03, 0x04, 0x05, 0x06, 0x07,
  0x08, 0x09, 0x0a, 0x0b, 0x0c, 0x0d, 0x0e, 0x0f,
  // ... (256 значений всего)
  0xf8, 0xf9, 0xfa, 0xfb, 0xfc, 0xfd, 0xfe, 0xff,
}
// staticuint64s (src/runtime/iface.go)**
```

Массив staticuint64s — это умная оптимизация. Он содержит предварительно вычисленные значения для каждого возможного однобайтного целого числа (0-255). Когда Go нужно преобразовать однобайтный slice в строку, он может полностью пропустить выделение памяти, извлекая из этого массива напрямую. Это ускоряет преобразования для малых значений и избегает ненужных накладных расходов памяти.

Подводя итог, когда вы присваиваете или создаете новую строку из байтового slice, Go обычно выделяет свежую память и копирует байты во время присваивания. После создания строки она полностью изолирована от любых изменений исходного байтового массива — благодаря ее неизменяемости.

|Go|Pseudo Code|
|---|---|
|`var s1 []byte`  <br>`s2 := string(s1)`|`s2Bytes := make([]byte, len(s1))`  <br>`copy(s2Bytes, s1)`  <br>`s2 = stringHeader{&s2Bytes[0], len(s1)}`|

Однако не каждое преобразование из байтового slice в строку требует копирования байтов в новую память. В некоторых случаях Go может повторно использовать исходный байтовый slice напрямую как базовые данные строки. Эта оптимизация происходит, когда runtime может гарантировать, что байтовый slice остается неизменным в течение жизни строки, делая безопасным ссылку на него без копирования.

Это поведение часто появляется в сценариях, таких как сравнения или операторы switch:

```go
func main() {
  ...
  switch string(anyBytes) {
  case "Hello":
    print("World")
  case "World":
    print("Hello")
  default:
    print("default")
  }
  
  if string(anyBytes) == "Hello" {
    print("Hello")
  }
}
```

В этом примере anyBytes преобразуется в строку внутри условия switch или if. Компилятор Go распознает, что anyBytes не изменяется в этом контексте, поэтому он безопасно использует исходный байтовый slice без выделения новой памяти.

Вместо вызова runtime.slicebytetostring, Go использует runtime.slicebytetostringtmp, который пропускает выделение памяти, когда это возможно:

```go
// slicebytetostringtmp возвращает "string", ссылающийся на фактические []byte
// Вызывающие стороны должны гарантировать, что возвращенная строка не будет использоваться после того, как вызывающая goroutine изменит исходный slice или синхронизируется с другой goroutine.
// Функция вызывается только при инструментировании
// и в противном случае встраивается компилятором.
//
// Некоторые внутренние оптимизации компилятора используют эту функцию.
// - Используется для m[T1{... Tn{..., string(k), ...} ...}] и m[string(k)]
//   где k — []byte, T1 до Tn — вложение struct и array литералов.
// - Используется для "<"+string(b)+">" конкатенации, где b — []byte.
// - Используется для string(b)=="foo" сравнения, где b — []byte.
func slicebytetostringtmp(ptr *byte, n int) string {
  ...
  return unsafe.String(ptr, n)
}
// slicebytetostringtmp (src/runtime/string.go)**
```

### **Эффективные техники конкатенации строк**

Конкатенация строк — это процесс объединения двух или более строк в одну. 
В Go более эффективный способ обработки множественных конкатенаций — использование strings.Builder вместо оператора `+`. 
Поскольку строки в Go неизменяемы, каждый раз, когда вы используете `+` для конкатенации, выделяется новая строка.

Компилятор пытается оптимизировать конкатенацию, особенно при работе с константными строками:

```go
func main() {
  const c1 = "Hello"
  str1 := "normal string"
  str2 := c1 + " World" + str1 + "!"
  print(str2)
}
```

| Go Assembly                                                                                                                                                                                                                                                                                                                                 | Pseudo-code                                                                                                                                                                                                     |
| ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `MOVD $main..autotmp_2-48(SP), R0`  <br>`MOVD $go:string."Hello World"(SB), R1`  <br>`MOVD $11, R2`  <br>`MOVD $go:string."normal string"(SB), R3`  <br>`MOVD $13, R4`  <br>`MOVD $go:string."!"(SB), R5`  <br>`MOVD $1, R6`  <br>`CALL runtime.concatstring3(SB)`  <br>`MOVD R0, main.str2.ptr-8(SP)`  <br>`MOVD R1, main.str2.len-16(SP)` | `{R1, R2} = string{"Hello World", 11}`  <br>`{R3, R4} = string{"normal string", 13}`  <br>`{R5, R6} = string{"!", 1}`  <br>`R0, R1 = concatstring3(R1, R3, R4)`  <br>`main.str2 = R0`  <br>`main.str2.len = R1` |

Во время компиляции Go оптимизирует код, объединяя константные или литеральные строки заранее. Например, компилятор объединяет c1 ("Hello") и " World" в одну строку, уменьшая необходимость в нескольких шагах конкатенации во время выполнения. Вместо объединения четырех строк компилятор упрощает операцию в один вызов runtime.concatstring3, который эффективно объединяет три строки.

Причина, по которой "normal string" не объединяется с другими константами, заключается в том, что Go объединяет только строки, известные во время компиляции — константы и литералы. Поскольку str1 — переменная, ее значение может измениться во время выполнения, и Go избегает объединения ее для обеспечения корректности.

Функция concatstring3 — это оптимизация, разработанная для эффективного объединения трех строк. Однако реальная работа происходит внутри runtime.concatstrings:

```go
func concatstring3(buf *tmpBuf, a0, a1, a2 string) string {
  return concatstrings(buf, []string{a0, a1, a2})
}

func concatstrings(buf *tmpBuf, a []string) string {
  ...
  // Шаг 1: Пропустить пустые строки
  for i, x := range a {
    n := len(x)
    if n == 0 {
      continue
    }
    ...
  }
  
  if count == 0 {
    return ""
  }
  
  // Шаг 2: Если есть только одна строка и условия памяти выполнены, вернуть ее напрямую
  if count == 1 && (buf != nil || !stringDataOnStack(a[idx])) {
    return a[idx]
  }
  
  // Шаг 3: Выделить или повторно использовать буфер
  s, b := rawstringtmp(buf, l)
  
  // Шаг 4: Скопировать байты в буфер
  for _, x := range a {
    copy(b, x)
    b = b[len(x):]
  }
  
  return s
}
// concatstring3 & concatstrings (src/runtime/string.go)**
```

Эта функция берет каждую строку, объединяет их вместе и записывает их в предварительно выделенный буфер на stack'е, последовательно копируя байты в это пространство.

Когда Go обнаруживает, что временная строка, такая как созданная из преобразования []byte в string или быстрой конкатенации, не нужно сохранять за пределами немедленной операции, он избегает выделения на heap. Вместо этого компилятор настраивает небольшой 32-байтный буфер на stack'е и передает его в функции, такие как slicebytetostring или concatstrings.

На данный момент Go предлагает несколько встроенных оптимизаций для конкатенации строк, каждая из которых адаптирована для эффективной обработки разного количества строк:

```go
func concatstring2(buf *tmpBuf, a0, a1 string) string {
  return concatstrings(buf, []string{a0, a1})
}

func concatstring3(buf *tmpBuf, a0, a1, a2 string) string {
  return concatstrings(buf, []string{a0, a1, a2})
}

func concatstring4(buf *tmpBuf, a0, a1, a2, a3 string) string {
  return concatstrings(buf, []string{a0, a1, a2, a3})
}

func concatstring5(buf *tmpBuf, a0, a1, a2, a3, a4 string) string {
  return concatstrings(buf, []string{a0, a1, a2, a3, a4})
}
// concatstring2 to concatstring5 (src/runtime/string.go)**
```



Компилятор Go автоматически выбирает наиболее эффективную функцию на основе количества объединяемых строк:

```go
func walkAddString(n *ir.AddStringExpr, init *ir.Nodes) ir.Node {
  ...
  var fn string
  if c <= 5 {
    // малое количество строк использует прямые runtime helper'ы.
    // примечание: order.expr знает этот cutoff тоже.
    fn = fmt.Sprintf("concatstring%d", c)
  } else {
    // большое количество строк передается runtime как slice.
    fn = "concatstrings"
    
    t := types.NewSlice(types.Types[types.TSTRING])
    // args[1:] для пропуска buf arg
    slice := ir.NewCompLitExpr(base.Pos, ir.OCOMPLIT, t, args[1:])
    slice.Prealloc = n.Prealloc
    args = []ir.Node{buf, slice}
    slice.SetEsc(ir.EscNone)
  }
  ...
}
// walkAddString (src/cmd/compile/internal/walk/expr.go)**
```

При работе с пятью или менее строками (c <= 5) компилятор вызывает специализированные вспомогательные функции, такие как concatstring3 или concatstring5. 

Если строк больше пяти, стратегия меняется. Вместо вызова фиксированной вспомогательной функции Go строит slice, содержащий все строки, и передает его в concatstrings.

**7. len() vs utf8.RuneCountInString():**

- If too large → heap allocation
- 32 bytes optimal for most temporary strings

**MEMORY OPTIMIZATION:**

**16. String literal storage:**

- Stored in read-only memory section
- Multiple identical literals share same memory
- Attempting write → crash

**17. String slicing - no copy:**

```go
s := "Hello, World!"
sub := s[0:5]  // "Hello"
// No copy - sub points to same memory!
```

**18. Raw vs Interpreted strings:**

```go
raw := `Line 1
Line 2`        // Preserves formatting

interpreted := "Line 1\nLine 2"  // Escape sequences
```

**PRACTICAL TIPS:**

**19. Common mistakes:**

1. Confusing len() (bytes) with character count
2. Modifying string (impossible - immutable!)
3. Using + in loops (multiple allocations)
4. Forgetting UTF-8 (1 char ≠ 1 byte)

**20. Best practices:**

- Use `strings.Builder` for concatenation in loops
- Use `utf8.RuneCountInString()` for character count
- Use `for-range` to iterate over characters
- Preallocate builder capacity when size known
- Remember strings are immutable - no defensive copying needed

**21. decoderune function:**

- Part of runtime package

- Called automatically in for-range

**KEY TAKEAWAYS:**

1. **String = header + immutable byte array**
2. **len() counts BYTES, not characters**
3. **UTF-8 = variable width (1-4 bytes)**
4. **for-range iterates RUNES, not bytes**
5. **String immutability = memory sharing safe**
6. **Compiler optimizes constants + small strings**
7. **Use strings.Builder for loops**
8. **Conversion optimization context-dependent**
