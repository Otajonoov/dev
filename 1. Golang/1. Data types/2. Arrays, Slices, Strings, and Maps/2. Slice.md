Slice - bu **o'zgaruvchan o'lchamli** struktura. Unga element qo'shish va o'chirish mumkin.

### Slice'ning ichki tuzilishi

```go
type slice struct {
    array unsafe.Pointer  // asosiy massivga havola
    len   int             // elementlar soni
    cap   int             // sig'im
}

```

Slice aslida **massivga havola** saqlovchi struktura. Unda uchta maydon bor:

- `array` - asl massivning qaysi elementidan boshlanishini ko'rsatadi
- `len` - hozirgi elementlar soni
- `cap` - umumiy sig'im (hajm)

### Slice yaratish

```go
// To'g'ridan-to'g'ri
a := []int{1, 2, 3}

// Massivdan
m := [3]string{"Birinchi", "Ikkinchi", "Uchinchi"}
b := m[0:2]  // 0-indexdan 2-elementgacham: []string{"Birinchi", "Ikkinchi"}

// make() yordamida
s := make([]int, 2, 3)  // len=2, cap=3

```

### Slice - bu havola!

Bu juda muhim: slice aslida massivga havola bo'lgani uchun, uni funktsiyaga "qiymat bo'yicha" (by value) yuborilsa ham, asl massiv o'zgaradi:

```go
func main() {
    m := [...]int{1, 2, 3}
    s := m[:]

    func(l []int) {
        l[0] = 42  // asl massivni o'zgartiradi!
    }(s)

    fmt.Printf("%#v\\\\n", m)  // [3]int{42, 2, 3}
}

```

**Real hayotiy misol:** Go'ning `sort` paketi aynan shu tamoyil asosida ishlaydi - slice'ni qabul qilib, hech narsa qaytarmaydi, lekin berilgan slice ichidagi qiymatlarni saralaydi.

### len va cap farqi

```go
a := []int{1, 2, 3}

fmt.Println(len(a))  // 3 (elementlar soni)
fmt.Println(cap(a))  // 3 (sig'im)

// Bitta elementni "o'chiramiz"
a = append(a[:1], a[2:]...)
fmt.Println(len(a))  // 2
fmt.Println(cap(a))  // 3 (sig'im o'zgarmadi!)

// Ko'p element qo'shamiz
a = append(a, 10, 11, 12, 13, 14, 15)
fmt.Println(len(a))  // 8
fmt.Println(cap(a))  // 8 (avtomatik kattalashdi)

```

### Capacity oshganda nima bo'ladi?

Slice'ning sig'imi yetmay qolganda Go quyidagilarni qiladi:

1. **Yangi, kattaroq massiv yaratadi**
2. Barcha qiymatlarni yangi massivga **ko'chiradi**
3. Yangi elementlarni qo'shadi
4. Slice'ning `array` maydonini yangi massivga yo'naltiradi

```go
a := []string{"Bir"}
fmt.Println(cap(a))  // 1

a = append(a, "Ikki")
fmt.Println(cap(a))  // 2 (yangi massiv yaratildi)

a = append(a, "Uch")
fmt.Println(cap(a))  // 4 (optimizatsiya: 1 emas, 2 ta qo'shimcha joy)

```

### **Выражения Slice и семантика индексации**

Slicing создает новый slice, который содержит определенную секцию существующего slice или массива. Полный синтаксис — a[low : high : cap], где a — массив или slice, low — начальный индекс, high — конечный индекс (исключен из результата), а cap устанавливает предел capacity нового slice.

```go
func main() {
  var a = [5]int{0, 1, 2, 3, 4}
  
  fmt.Println(a[1:4])    // [1, 2, 3]
  fmt.Println(a[1:4:5])  // [1, 2, 3]
  fmt.Println(a[1:4:4])  // [1, 2, 3]
}
```

Хотя slice, такие как a[1:4], a[1:4:5] и a[1:4:4], выглядят идентичными при печати, их базовые capacity различаются:
![](../../../assets/obsidian-images/Pasted%20image%2020260130192806.png)

В этом случае a[1:4:4] уже достиг своей максимальной capacity (len == cap), что означает, что его нельзя расширить дальше с помощью slicing. Попытка сделать это приведет к ошибке времени выполнения. Единственный способ расширить его — с помощью append — но это связано с концепцией роста slice, которую мы рассмотрим в следующем разделе.

С другой стороны, a[1:4:5] все еще имеет место для роста, позволяя дальнейшее расширение через slicing:
```go
func main() {
  var a = [5]int{0, 1, 2, 3, 4}
  b := a[1:4:5]  // [1, 2, 3]
  c := a[1:4:4]  // [1, 2, 3]
  
  fmt.Println(b[:4])  // [1, 2, 3, 4]
  
  // runtime error: slice bounds out of range [:4] with capacity 3
  fmt.Println(c[:4])
}
```

Вы не можете напрямую получить доступ к элементу за пределами длины slice (b[4] не сработает). Однако при использовании операции slice вы можете расширить значение high за пределы текущей длины — пока оно остается в пределах capacity slice.

Мы не часто выписываем полную форму выражения slice, но каждая операция slice расширяется в свою полную форму за кулисами:

| Go Sugar Syntax | Underlying Syntax         |
| --------------- | ------------------------- |
| `b := a[1:4]`   | `b := a[1:4:cap(a)]`      |
| `b := a[:4]`    | `b := a[0:4:cap(a)]`      |
| `b := a[1:]`    | `b := a[1:len(a):cap(a)]` |
| `b := a[:]`     | `b := a[0:len(a):cap(a)]` |

---
---
Когда дело доходит до того, как компилятор обрабатывает slicing, нет особой сложности. Slice'ы в Go имеют четкую структуру под капотом, поэтому роль компилятора просто заключается в настройке нового заголовка slice с правильным pointer'ом, длиной и capacity.

Вот как компилятор обрабатывает операцию slice, такую как a[i:j:k]:

1. Компилятор определяет тип a — является ли это slice, строкой или pointer'ом на массив.
2. Если i, j или k не указаны, они по умолчанию устанавливаются в 0, len(a) и cap(a).
3. Компилятор проверяет, что индексы остаются в валидных границах (i <= j && j <= k).
4. Он вычисляет новую длину как j - i и новую capacity как k - i.
5. Устанавливается новый базовый pointer для slice, и создается заголовок slice с этим pointer'ом, новой длиной и capacity. Этот заголовок затем возвращается как новый slice.

До этого момента мы рассмотрели, как работает slicing и как компилятор управляет этим внутренне. Но capacity — это больше, чем просто ограничение того, сколько базового массива вы можете получить доступ.

Capacity также играет решающую роль в том, как slice растут — и когда этот рост происходит, он может молча разорвать соединение между slice'ами и их исходными массивами. Это тонкое поведение действительно важно отметить.

## **Определение новой capacity**

Go проверяет, имеет ли slice достаточно capacity для новых элементов, сравнивая целевую длину после добавления (newLen) с текущей capacity slice. Если достаточно места, slice просто расширяется путем обновления его длины, добавляет новые элементы и продолжает.

Если newLen превышает текущую capacity, Go запускает процесс роста через runtime.growslice. На этом этапе он вычисляет новую capacity, используя функцию runtime.nextslicecap(newLen, oldCap int), которая обрабатывает логику изменения размера:

Если newLen более чем в два раза больше текущей capacity (oldCap), Go устанавливает новую capacity, чтобы соответствовать newLen. В противном случае capacity изначально удваивается. Но как только capacity достигает 256, темп роста замедляется, используя эту формулу:

```go
cap += (cap + 3 * 256) / 4
```

Процесс на этом не заканчивается. Новая capacity затем округляется вверх, чтобы соответствовать размерам, которые аллокатор памяти предназначен обрабатывать. Аллокатор памяти Go не выделяет произвольные размеры; вместо этого он использует предопределенный набор классов размеров.

Например, если вы запрашиваете 7 байт, он округляет вверх и выделяет 8 байт вместо этого. Если вам любопытно перед погружением в Главу 7: Память, вот краткое описание того, как это работает:

- Для малых выделений Go округляет размер памяти вверх, чтобы поместиться в определенный класс размера. Эти предопределенные размеры разработаны, чтобы быть малыми и эффективными — думайте о блоках в 16 байт, 32 байта и так далее.
- Для больших выделений, которые не помещаются в эти малые классы, Go выделяет память полными страницами. В Go страница обычно составляет 8 КБ, что отличается от страницы ОС.

Различные типы slice растут с разными темпами. Slice значений int64 будет расти медленнее, чем slice byte, потому что int64 в 8 раз больше. Таблица ниже показывает, как различные типы масштабируются по мере роста:

При росте с нуля вы не увидите slice всего с одним элементом для []int8 или []int32, потому что аллокатор памяти Go использует минимальный класс размера в 8 байт. Это означает, что вместо увеличения только на один элемент, slice []int8 перепрыгнет сразу с 0 до 8 элементов, чтобы соответствовать наименьшему размеру выделения.

Имейте в виду, что Go имеет жесткий предел на размер выделения памяти, ограниченный 2⁴⁸ байтами (не элементами) на большинстве 64-битных архитектур. Если slice превышает этот предел во время роста, Go вызовет panic. Это ограничение применяется не только к slice'ам; оно также влияет на другие выделения, включая channel'ы, map'ы и аналогичные структуры.

```go
func append(src []Type, elements ...Type) []Type {
  ...
  // Инициализировать slice для добавления
  s := src
  
  // Увеличить slice, если нужно
  newLen := len(s) + len(elements)
  if newLen <= cap(s) {
    s = s[:newLen]
  } else {
    s = growSlice(s, newLen)
  }
  
  // Добавить новые элементы, копируя их в конец slice
  copy(s[len(src):], elements)
  
  return s
}

func growslice(oldPtr unsafe.Pointer, newLen, oldCap, num int, et *_type) slice {
  ...
  newcap := nextslicecap(newLen, oldCap)
  ...
  
  var p unsafe.Pointer
  if et.PtrBytes == 0 {
    p = mallocgc(capmem, nil, false)
    memclrNoHeapPointers(add(p, newlenmem), capmem-newlenmem)
  } else {
    p = mallocgc(capmem, et, true)
    ...
  }
  
  memmove(p, oldPtr, lenmem)
  return slice{p, newLen, newcap}
}

func nextslicecap(newLen, oldCap int) int {
  newcap := oldCap
  doublecap := newcap + newcap
  
  if newLen > doublecap {
    return newLen
  }
  
  const threshold = 256
  if oldCap < threshold {
    return doublecap
  }
  
  for {
    newcap += (newcap + 3*threshold) >> 2
    if uint(newcap) >= uint(newLen) {
      break
    }
  }
  
  if newcap <= 0 {
    return newLen
  }
  
  return newcap
}


```

### **Стратегии выделения и схема памяти**

Когда компилятор определяет, что slice нужно убежать на heap — либо из-за его размера, либо потому что он переживет функцию, где был создан — он заменяет вызов make([]T) вызовом runtime к runtime.makeslice. Независимо от того, пишете ли вы make([]T, len) или make([]T, len, cap), процесс выделения работает одинаково за кулисами.

Фактически, make([]T, len) автоматически переводится компилятором в make([]T, len, len). Это гарантирует, что как длина, так и capacity устанавливаются равными, если не указано иное:

```go
func makeslice(et *_type, len, cap int) unsafe.Pointer {
  mem, overflow := MulUintptr(et.Size_, uintptr(cap))
  if overflow || mem > maxAlloc || len < 0 || len > cap {
    mem, overflow := MulUintptr(et.Size_, uintptr(len))
    if overflow || mem > maxAlloc || len < 0 {
      panicmakeslicelen()
    }
    panicmakeslicecap()
  }
  
  return mallocgc(mem, et, true)
}
// makeslice (src/runtime/slice.go)**
```

Если происходит overflow или требуемая память превышает 2⁴⁸ байт (или возникают любые другие невалидные условия, как проверки выше), функция вызывает panic. То, является ли проблема с len или cap, определяет конкретный вызванный panic.

Если все проверки проходят, выделение памяти продолжается с использованием mallocgc. Эта функция принимает вычисленный размер памяти, тип элемента и флаг, который указывает, нужно ли обнулять выделенную память. И вот тут все становится интересным.

Обычно, когда создается slice, базовый массив обнуляется для обеспечения безопасности и предотвращения утечки данных. Однако дела меняются, когда вы создаете slice и немедленно заполняете его, копируя данные из другого массива или slice:

```go
func main() {
  a := [5]int{1, 2, 3, 4, 5}
  b := make([]int, 5)
  copy(b, a)
}
```

В этом случае вы можете ожидать, что Go создаст slice b с makeslice, обнулит память, а затем выполнит memmove для копирования значений из a. Однако есть более умная оптимизация для типов slice без pointer'ов, таких как []int. Поскольку slice будет немедленно перезаписан значениями из a, обнуление памяти сначала не нужно.

Go распознает этот паттерн и полностью пропускает обнуление, используя специализированную функцию: runtime.makeslicecopy. Эта функция выделяет память без очистки ее, используя mallocgc(mem, et, false), чтобы сэкономить время и ресурсы:

```go
func makeslicecopy(et *_type, tolen int, fromlen int, from unsafe.Pointer) unsafe.Pointer {
  ...
  if !et.Pointers() {
    to = mallocgc(tomem, nil, false)
    if copymem < tomem {
      memclrNoHeapPointers(add(to, copymem), tomem-copymem)
    }
  } else {
    to = mallocgc(tomem, et, true)
    ...
  }
  ...
  
  memmove(to, from, copymem)
  return to
}
// makeslicecopy (src/runtime/slice.go)**
```

Если тип элемента slice не содержит pointer'ов, Go полностью пропускает обнуление памяти. Поскольку типы без pointer'ов не представляют риска быть неправильно интерпретированными garbage collector'ом, эта оптимизация ускоряет выделение памяти без ущерба для безопасности.

Однако, если целевой slice больше, чем источник, любые дополнительные байты, которые не заполнены memmove, все еще нужно обнулить с использованием memclrNoHeapPointers. Это предотвращает оставление неинициализированной памяти, что могло бы привести к неожиданному поведению.

Для slice'ов, тип элемента которых содержит pointer'ы (например, *int, string или struct'ы с полями pointer'ов), Go должен обнулить память во время выделения.

Между выделением (mallocgc) и копированием данных (memmove) есть короткое окно, где garbage collector может видеть вновь выделенную память. Если эта память не обнулена, garbage collector может принять случайные, неинициализированные данные за валидные pointer'ы, что приведет к непредсказуемому поведению.

Это поведение применяется конкретно к slice'ам, выделенным на heap. Когда slice выделяется на stack, дела проще — память сначала обнуляется, а затем заполняется memmove как обычно.

Для массивов все, что больше 10 МБ, автоматически выделяется на heap. Однако slice'ы следуют другому правилу. Когда размер определяется во время компиляции, их базовый массив ограничен 64 КБ перед запуском выделения на heap:

```go
type A struct {
  slice [64 * 1024 + 1]byte
}

func main() {
  stackSlice := make([]byte, 64 * 1024)
  heapSlice := make([]byte, 64 * 1024 + 1)
  heapStruct := new(A)
}
```


```bash
$ go build -gcflags=-m
./main.go:4:20: make([]byte, 65536) does not escape
./main.go:5:19: make([]byte, 65537) escapes to heap
./main.go:10:19: new(A) escapes to heap
```

В этом примере stackSlice остается на stack, потому что его размер ровно 64 КБ. Но превышение этого предела даже на один байт заставляет выделение на heap, как видно с heapSlice. Это применяется не только к slice'ам — struct A также оказывается на heap из-за его большого размера.

Это поведение контролируется константой runtime с именем MaxImplicitStackVarSize, которая определяет наибольший размер переменной, разрешенный на stack:

```go
// MaxImplicitStackVarSize — максимальный размер неявных переменных, выделенных на stack'е.
// p := new(T) выделение T на stack'е
// p := &T{} выделение T на stack'е
// s := make([]T, n) выделение [n]T на stack'е
// s := []byte("...") выделение [n]byte на stack'е
// Примечание: флаг smallframes может настроить это значение.
MaxImplicitStackVarSize = int64(64 * 1024)
```

## **XULOSAM**

Bu slice'lar haqidagi qism Go'ning eng muhim va qiziqarli qismlaridan biri. Slice'lar array'dan ko'ra ko'proq ishlatiladi va ularning ichki mexanizmini bilish juda muhim. Asosiy xulosalar:

**SLICE STRUCTURE:**

**1. Slice = 3 component (24 bytes on 64-bit):**

```go
type slice struct {
  array unsafe.Pointer  // 8 bytes - pointer to underlying array
  len   int            // 8 bytes - current length
  cap   int            // 8 bytes - capacity
}
```

**2. Slice assignment = header copy, NOT array copy:**

```go
a := []int{1, 2, 3}
b := a              // Faqat 24 byte copy! Array emas!
b[0] = 10           // a ham o'zgaradi: [10, 2, 3]
```

**3. Multiple slice'lar bir array'ni share qilishi mumkin:**

```go
a := [5]int{0, 1, 2, 3, 4}
b := a[:3]   // [0, 1, 2]
c := a[2:]   // [2, 3, 4]
a[2] = 10    // b va c ham o'zgaradi!
```

**SLICING SYNTAX:**

**4. Full slicing syntax: a[low : high : cap]**

- `low` - boshlang'ich index
- `high` - oxirgi index (excluded)
- `cap` - yangi slice'ning max capacity

**5. Sugar syntax expansion:**

```go
a[1:4]   → a[1:4:cap(a)]
a[:4]    → a[0:4:cap(a)]
a[1:]    → a[1:len(a):cap(a)]
a[:]     → a[0:len(a):cap(a)]
```

**APPEND MECHANICS:**

**6. Append capacity'ni oshganda yangi array yaratadi:**

````go
s := make([]int, 0, 1)
s = append(s, 0)  // 0x...738 - same address
s = append(s, 1)  // 0x...000 - NEW address!
```
**Connection breaks!** Eski array endi irrelevant.

**7. Growth strategy - two phases:**

a) **Small slices (oldCap < 256)**: Double capacity
```
0 → 1 → 2 → 4 → 8 → 16 → 32 → 64 → 128 → 256
````

b) **Large slices (oldCap >= 256)**: Formula

```go
newcap += (newcap + 3*256) >> 2
// Yoki: newcap += (newcap + 768) / 4
```

**8. Memory size class rounding:** Go memory allocator har doim round up qiladi:

- 7 bytes → 8 bytes
- []int64 slower growth than []byte
- Minimum size class: 8 bytes

**CRITICAL PITFALLS:**

**9. Slice in function - connection can break:**

```go
func appendTo(s []int) {
  s = append(s, 3)  // Agar cap yetarli bo'lmasa - yangi array!
}

a := []int{1, 2}
appendTo(a)
fmt.Println(a)  // [1, 2] - O'ZGARMAYDI!
```

**Solution:** Return the slice yoki pointer pass qiling:

```go
func appendTo(s []int) []int {
  return append(s, 3)
}
```

**OPTIMIZATION TIPS:**

**10. Preallocate when size known:**

```go
// BAD
a := []int{}
for i := 0; i < 1000; i++ {
  a = append(a, i)  // Ko'p reallocation!
}

// GOOD
a := make([]int, 0, 1000)
for i := 0; i < 1000; i++ {
  a = append(a, i)  // Reallocation yo'q!
}
```

**11. make() variants:**

```go
make([]int, 1000)       // len=1000, cap=1000
make([]int, 0, 1000)    // len=0, cap=1000 (better for append)
make([]int, len(other)) // Copy length from another
```

**MEMORY ALLOCATION:**

**12. Stack vs Heap thresholds:**

- **Slice:** 64 KB compile-time size limit for stack
- **Array:** 10 MB limit for stack
- Exceeding → heap allocation

**13. makeslice vs makeslicecopy optimization:**

```go
a := [5]int{1, 2, 3, 4, 5}
b := make([]int, 5)
copy(b, a)
```

Go aqlli: zeroing skip qiladi agar darhol copy bo'lsa (pointer-free types uchun)!

**14. Pointer-free vs Pointer-containing types:**

- **Pointer-free** ([]int, []byte): mallocgc(..., false) - no zeroing
- **Pointer-containing** ([]string, []*int): mallocgc(..., true) - must zero Sababi: GC random data'ni pointer deb o'ylashi mumkin!

**ADVANCED INSIGHTS:**

**15. Slice literal workaround:**

```go
largeSlice := []byte{1024*1024*1024: 10}  // 1GB!
```

Array heap'ga ketmaydi, lekin stack overflow yoki "frame too large" error!

**16. Hard limits:**

- Max allocation: 2⁴⁸ bytes (256 TB)
- Bu slice, map, channel uchun ham amal qiladi
- Exceeding → panic

**PRACTICAL WISDOM:**

**18. Common mistakes:**

1. Slice'ni function'ga pass qilib append qilish
2. Shared underlying array'ni unutish
3. Capacity'ni ignore qilish
4. Pre-allocation qilmaslik (known size bo'lganda)

**19. Best practices:**

- Return slice from function (don't pass pointer to slice)
- Preallocate when size is known or predictable
- Be careful with shared underlying arrays
- Use full slice syntax when you need capacity control

**20. Debug tricks:**

```go
println(slice)  // [len/cap]address - very useful!

// unsafe method (educational only)
sliceArray := uintptr(unsafe.Pointer(&mySlice))
sliceLength := unsafe.Pointer(sliceArray + 8)
sliceCapacity := unsafe.Pointer(sliceArray + 16)
```

**KEY TAKEAWAYS:**

1. **Slice ≠ Array** - Slice is a descriptor (header), not the data itself
2. **Copy = Cheap** - Only 24 bytes copied, not the entire data
3. **Shared arrays = Dangerous** - Multiple slices can point to same data
4. **Growth breaks connection** - append() can silently create new array
5. **Preallocate = Fast** - Avoid multiple reallocations
6. **Return, don't modify** - Safer pattern for functions
7. **Capacity matters** - It's not just a limit, it controls growth

Bu bobning eng muhim xulosa: **Slice Go'da eng ko'p ishlatiladigan data structure. Uning internal structure (header + underlying array) va growth mechanism'ini yaxshi tushunish production bug'larni oldini oladi va performance'ni yaxshilaydi!**
