1. Massiv - bu **o'zgarmas o'lchamli** struktura bo'lib, bir xil turdagi elementlarni saqlaydi.
2. Xotirada massiv - bu ketma-ket joylashgan bir xil o'lchamdagi qiymatlar.
3. Yaratilganidan keyin unga yangi elementlar qo'shib yoki olib bo'lmaydi.
4. Bo'sh massivda ham elementlar mavjud (default qiymatlar bilan)
5. Когда вы создаете массив, вы определяете его размер заранее, и этот размер фиксирован; он не может быть изменен позже.

Фактически, размер массива также является частью его типа:
```go
// 1
var a [5]int  // тип a — [5]int

// 2
a := [3]int{1, 2, 3}

// 3
a := [...]int{1, 2, 3}  // kompilator o'zi uzunlikni aniqlaydi
```

```go
var a [5]int  // тип a — [5]
var b [3]int
a = b  // XATO! Turlar mos kelmaydi

a == b  // invalid operation: a == b (mismatched types [5]int and [3]int)
```
В этом примере [5]int означает не просто 'массив целых чисел' — он означает 'массив из ровно 5 целых чисел'. Число 5 встроено в сам тип.

Это приводит к нескольким важным деталям, которые могут показаться знакомыми, но заслуживают более пристального взгляда:

- Поскольку размер является частью типа, массивы разных размеров считаются полностью разными типами. Вот почему в приведенном выше примере вы не можете сравнивать или присваивать a и b напрямую — они обрабатываются компилятором как различные типы.
- При передаче массива в функцию размер должен быть включен в тип параметра. Вы не можете передать [5]int функции, ожидающей [3]int.

**С точки зрения компилятора массив может быть представлен так:

```go
type Array struct {
    len  int64   // uzunlik
    elem Type    // element tipi
}

```

Это не точная структура, используемая Go под капотом, а скорее упрощенное представление того, как компилятор интерпретирует массив во время компиляции (не во время выполнения). Фактически элементы массива хранятся в непрерывном блоке памяти.

### **1.1 Понимание схемы памяти массива**

Массивы — это непрерывные последовательности элементов, хранящихся непосредственно рядом друг с другом в памяти. Нет пробелов — каждый элемент следует за предыдущим по прямой линии:

```go
func main() {
  a := [5]byte{0, 1, 2, 3, 4}
  
  println("a", &a)
  println("a[0]", &a[0])
  println("a[1]", &a[1])
  println("a[2]", &a[2])
  println("a[3]", &a[3])
  println("a[4]", &a[4])
}

// Output:
// a 0x1400004e723
// a[0] 0x1400004e723
// a[1] 0x1400004e724
// a[2] 0x1400004e725
// a[3] 0x1400004e726
// a[4] 0x1400004e727
```

Адреса памяти элементов массива выстраиваются последовательно, каждый ровно на 1 байт друг от друга. Адрес всего массива совпадает с адресом его первого элемента, a[0]. Это поведение похоже на то, как обрабатывается память для struct'ов в Go.

Эта схема имеет важное последствие: присваивание одного массива другому или передача массива в функцию приводит к копированию всего массива — а не только ссылки.

Когда вы присваиваете массив, Go создает поверхностную копию исходного массива под капотом:
```go
func main() {
  var a [3]int  // [0 0 0]
  b := a
  b[0] = 10
  
  fmt.Println(a)  // [0 0 0]
  fmt.Println(b)  // [10 0 0]
}
```

В этом примере изменение b не влияет на a, потому что b — это копия a, а не ссылка на него. Каждый элемент в b начинается с того же значения, что и соответствующий элемент в a. Однако, поскольку b — это отдельная копия, любые изменения b оставляют a неизменным.

Когда массив создается, все элементы автоматически инициализируются их нулевым значением, которое зависит от типа элемента.

### **1.3 Работа с литералами массивов**

Литералы значений в Go предлагают несколько способов инициализации массивов при одновременной установке их элементов:

```go
// Инициализация на основе значений
a := [3]int{1, 2, 3}  // [1 2 3]

// Выведенный размер
b := [...]int{1, 2, 3}  // [1 2 3]

// Инициализация на основе индекса
c := [3]int{0: 1, 2: 3}  // [1, 0, 3]

// Смешанный: Выведенный размер и на основе индекса
d := [3]int{2: 2, 0: 1, 1}  // [1, 1, 2]

// Смешанный: Выведенный размер и на основе значений
e := [...]int{0: 1, 2: 3}  // [1, 0, 3]

// Смешанный: Выведенный размер, на основе индекса и на основе значений
f := [...]int{0: 1, 2: 3, 1}  // [1, 0, 3, 1]
```

Этот пример подчеркивает гибкость, которую Go предлагает при создании массивов:

- Массив ==b== использует синтаксис `...`, позволяя Go вывести размер массива на основе количества предоставленных элементов.
- В массиве ==c== инициализация на основе индекса позволяет присваивать значения конкретным позициям. Здесь позиция 0 содержит 1, позиция 2 содержит 3, а любые неуказанные позиции, такие как индекс 1, автоматически устанавливаются в 0.
- Массив ==d== смешивает методы, присваивая значения по индексу, используя непоследовательный порядок.
- Компилятор сначала обнуляет память, затем устанавливает значение каждого элемента индивидуально.

---
---
Большие массивы выделяются на heap, но это поведение не эксклюзивно для массивов. Любой большой блок памяти — будь то struct, массив или другой тип данных — может быть перемещен на heap, если его размер превышает определенный порог.

В текущей версии Go этот порог составляет около 10 МБ:
```go
var (
  // MaxStackVarSize — максимальный размер переменной, которую мы будем выделять на stack'е.
  // Этот предел для явных объявлений переменных, таких как "var x T" или "x := ...".
  // Примечание: флаг smallframes может обновить это значение.
  MaxStackVarSize = int64(10 * 1024 * 1024)
)
```


```go
func main() {
  a := [10 * 1024 * 1024]byte{}      // Выделено на stack'е
  b := [10*1024*1024 + 1]byte{}      // Превышает предел на 1 байт
}
```

```cmd
$ go build -gcflags="-m" main.go
./main.go:5:2: moved to heap: b
```

Используя команду `-gcflags="-m"`, мы можем подтвердить, что b перемещается на heap — превышение порога всего на один байт вызывает это поведение.


Есть еще одна деталь, заслуживающая внимания: вы можете напрямую брать адрес литералов значений массивов, что не то, что вы можете делать с другими базовыми типами, такими как int, string и т.д.
```go
a := &[3]int{1, 2, 3}  // tmp = [3]int{1, 2, 3}; a = &tmp
b := &1                // invalid
c := &"hello"          // invalid
```

Эта способность напрямую ссылаться на литералы массивов — и это распространяется на slice'ы, map'ы и struct'ы — это удобный синтаксический сахар, предлагаемый компилятором Go. Это упрощает работу с pointer'ами на композитные структуры данных, которые инициализируются при объявлении.

### **1.5 Длина и capacity массива**

Длина и capacity массива всегда одинаковы и остаются постоянными. Это имеет смысл, поскольку длина массива фиксирована и определена во время объявления:

```go
a := [5]int{1, 2, 3, 4, 5}
len(a)  // 5
cap(a)  // 5
```

Оба len(a) и cap(a) разрешаются как константные значения во время компиляции, конкретно как int-константы. Чтобы увидеть, как компилятор обрабатывает эти значения без прямого погружения во внутренности, рассмотрим этот пример:

```go
a := [5]int{1, 2, 3, 4, 5}

// валидно
const b = len(a)

// валидно
var _ int = b

// невалидно: cannot use b (constant 5 of type int) as int64 value in variable declaration
var _ int64 = b
```

Присваивая len(a) константе b, мы можем видеть, что len(a) должен обрабатываться как константа. Его можно присвоить переменной типа int, но не напрямую int64 — потому что len(a) конкретно типа int, а не нетипизированное целое число.

С точки зрения компилятора оба len(array) и cap(array) обрабатываются как константы и встраиваются непосредственно в сгенерированный код.

**ARRAY BASICS:**

**1. Array size - type'ning bir qismi**

```go
var a [5]int  // type: [5]int
var b [3]int  // type: [3]int
a == b        // ERROR: different types!
```

Bu juda muhim: [5]int va [3]int - bu **turli type'lar**!

**2. Contiguous memory layout** Array element'lari xotirada yonma-yon joylashgan. Array'ning address'i birinchi element'ning address'iga teng.

**3. Array assignment = COPY**

```go
a := [3]int{1, 2, 3}
b := a        // COPY, not reference!
b[0] = 10     // a hali ham [1, 2, 3]
```

Bu slice'dan katta farq!

**MEMORY INITIALIZATION STRATEGIES:**

**4. Uch xil zeroing strategy:**

a) **Small blocks (0-79 bytes)**: Direct instructions

- 8 bytes: `MOVD` (Move Doubleword)
- 16 bytes: `STP` (Store Pair)
- Overlapping mumkin (17 byte uchun STP + MOVD)

b) **Medium blocks (80-1039 bytes)**: Duff's Device

- `runtime.duffzero` function
- 64 ta STP instruction (har biri 16 byte)
- Loop overhead yo'q!
- Formula: `offset = 4 * (64 - size/16)`

c) **Large blocks (1040+ bytes)**: Loop

- Har iteratsiyada 16 byte (STP.P instruction)
- Qoldiq uchun MOVD yoki STP

**5. Assembly primer:**

- `SP` - Stack Pointer
- `ZR` - Zero Register (8 bytes of zeros)
- `MOVD` - Move 8 bytes
- `STP` - Store Pair (16 bytes)
- `STP.P` - Store Pair with Post-increment

**ARRAY INITIALIZATION:**

**6. Small vs Large array initialization:**

- ≤4 elements: har bir element uchun alohida instruction
- >4 elements: packed values yoki static read-only block

**7. Packed value trick:**

```go
[5]byte{1, 2, 3, 4, 5}
// 67305985 = 1 | (2 << 8) | (3 << 16) | (4 << 24)
```

Birinchi 4 byte bir qiymatda!

**8. Static read-only block:** Katta yoki complex array'lar uchun compiler `.rodata` section'da block yaratadi va undan copy qiladi.

**9. Stack vs Heap:**

- Stack limit: **10 MB**
- `10*1024*1024 + 1` byte heap'ga ketadi
- Bu faqat array uchun emas, har qanday katta struct uchun ham

**ARRAY SPECIFICS:**

**10. Array literal address:**

```go
a := &[3]int{1, 2, 3}  // VALID
b := &10               // INVALID
```

Syntactic sugar - compiler tmp variable yaratadi.

**11. len() va cap():**

- Array uchun doim teng
- Compile-time constant (agar function call yo'q bo'lsa)
- Constant folding: natija code'ga embed qilinadi
- `len(getArray())` - NOT constant (function call bor)

**PERFORMANCE INSIGHTS:**

**12. Nima uchun bu optimization'lar?**

- Loop overhead expensive (counter, condition check, branch)
- Direct instructions faster (small blocks uchun)
- Duff's Device eliminates loop overhead (medium blocks)
- Branch misprediction yo'q!

**13. CPU optimization:**

- Sequential memory access - cache-friendly
- Unrolled loops - fewer branch instructions
- Register usage - no memory access needed

**AMALIY MASLAHATLAR:**

1. **Array copy expensive** - katta array'lar uchun pointer/slice ishlatish
2. **Stack allocation better** - 10MB dan kichik bo'lsa
3. **Zero initialization automatic** - explicit qilish kerak emas
4. **Index-based init efficient** - sparse array'lar uchun
5. **Compiler optimization kuchli** - trust the compiler!

**QIZIQ FAKTLAR:**

- Duff's Device 1983 yilda Tom Duff tomonidan ixtiro qilingan
- Go assembly ARM64/AMD64'da farq qiladi
- Stack frame SP (Stack Pointer) ga nisbatan address'lanadi
- Zero register (ZR) hardware level'da mavjud
- Compiler 17 byte uchun ham 16 byte write qiladi (overlap)

**KO'P UCHRAYDIGAN XATOLAR:**

1. Array size type'ning qismi ekanligini unutish
2. Array assignment reference deb o'ylash (actually COPY!)
3. Function'ga array pass qilish - butun array copy bo'ladi
4. len(getArray()) constant deb o'ylash
