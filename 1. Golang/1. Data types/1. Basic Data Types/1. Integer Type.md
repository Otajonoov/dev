
```go
int8: -128 до 127
int16: -32,768 до 32,767
int32: -2,147,483,648 до 2,147,483,647
int64: -9,223,372,036,854,775,808 до 9,223,372,036,854,775,807

uint8: 0 до 255
uint16: 0 до 65,535
uint32: 0 до 4,294,967,295
uint64: 0 до 18,446,744,073,709,551,615
```

**1. int va uint - architecture-dependent** Bu juda amaliy qaror. 32-bit tizimda int=4 byte, 64-bit'da int=8 byte. Sababi - CPU'ning word size'iga mos kelish. Bu performance uchun muhim, chunki CPU o'z register size'ida ishlashni yaxshi ko'radi.

Типы int и uint работают немного иначе по сравнению с другими типами. Они не имеют фиксированного размера:
```go
fmt.Println("- GOARCH:", runtime.GOARCH)
fmt.Println("- int size:", unsafe.Sizeof(int(0)))
```

**64-битные системы:**
```
- GOARCH: arm64
- int size: 8 bytes
```

**32-битные системы:**
```
- GOARCH: 386
- int size: 4 bytes
```

Скорее, их размер определяется архитектурой системы:

- На 32-битной системе: int занимает 4 байта, соответствуя int32 и разделяя тот же диапазон значений.
- На 64-битной системе: int занимает 8 байт, как int64, что позволяет ему обрабатывать гораздо большие числа.

Такое же поведение применяется к uint.
Причина, по которой int и uint зависят от системной архитектуры, сводится к размеру слова (также известному как native size). Это относится к самому большому фрагменту данных, который CPU может обработать в одной операции, а также к объему памяти, который он может обработать за одно чтение или запись.

- На 32-битной системе размер слова составляет 32 бита (4 байта).
- На 64-битной системе размер слова увеличивается до 64 бит (8 байт).

**Целочисленные литералы и форматы чисел**

Вы не ограничены записью чисел в стандартной десятичной форме при работе с целыми числами. Go позволяет представлять числа в различных форматах, в зависимости от того, что имеет наибольший смысл для вашего кода. Они называются int-литералами, и их можно записать как:

- **Decimal**: Стандартный формат base-10, просто напишите число напрямую.
- **Binary**: Base 2, начинающийся с 0b или 0B, за которым следует последовательность 0 и 1.
- **Octal**: Base 8, начинающийся с 0o или 0O, за которым следуют цифры от 0 до 7. Вы также можете использовать ведущий 0, но это может быть запутанным и обычно избегается.
- **Hexadecimal**: Base 16, начинающийся с 0x или 0X, используя цифры 0-9 и буквы a-f (или A-F).
```go
a := 10           // Decimal
b1 := 0b1010      // Binary
b2 := 0B1010      // Binary
o1 := 0o12        // Octal
o2 := 0O12        // Octal
o3 := 012         // Octal (не рекомендуется)
h1 := 0xface      // Hexadecimal
h2 := 0XFaCe      // Hexadecimal

// Использование Binary
num := 29                    // 11101 в binary
mask := 0b11                 // 00011: маска для последних 2 битов
lastTwoBits := num & mask    // 01: извлечение последних 2 битов с помощью bitwise AND

// Использование Octal для прав доступа к файлам
permission := 0o755          // Владелец: rwx, Группа: r-x, Остальные: r-x

// Использование Hexadecimal
address := 0x1400004e740     // Адрес памяти
color := 0x0000FF            // Код синего цвета
```

Различные форматы служат разным целям:

- Binary полезен для битовых манипуляций, установки флагов или работы непосредственно с битами.
- Octal часто используется для прав доступа к файлам.
- Hexadecimal идеален для адресов памяти, значений цветов в графике и низкоуровневого программирования.

---
Валидные
```go
a := 1_000_000
b := 0b_1010_1010
c := 0x_1_2_3_4_5_6_7_8
```

Невалидные
```go
d := _1234         // Нельзя начинать с подчеркивания
e := 0_x1234       // Нельзя ставить подчеркивание после префикса
g := 1_000_000_    // Нельзя заканчивать подчеркиванием
h := 1__0          // Нельзя использовать последовательные подчеркивания
```

Overflow и Underflow

Когда значение превышает пределы своего типа данных, это вызывает либо overflow (когда значение слишком велико), либо underflow (когда оно слишком мало).

Возьмем int8 в качестве примера, он может содержать значения от -128 до 127. Если вы выходите за этот диапазон, значение оборачивается:
```go
var a int8 = 127
a++  // Оборачивается до -128 (overflow)
a--  // Оборачивается обратно до 127 (underflow)
```

**Overflow/underflow handling** Go'da integer overflow compile-time'da iloji boricha tutiladi, lekin runtime'da wraparound qiladi (error bermaydi!). Bu hardware'ning tabiy xususiyati. Masalan: `int8(127) + 1 = -128`. Bu potensial bug manbasi bo'lishi mumkin.

**Деление на ноль: защита во время компиляции и выполнения**
```go
func main() {
  a := 0
  println(a / 0)   // compile error: invalid operation: division by zero 
                   // (compile-time error)
  println(10 / a)  // panic: runtime error: integer divide by zero
}
```

На большинстве процессоров деление на ноль вызывает аппаратный сбой и отправляет сигнал приложению. Runtime ловит это через обработчик сигналов и преобразует в panic, вызывая panicdivide(). На платформах без надежных аппаратных сбоев для деления на ноль компилятор вставляет явную проверку: если делитель равен нулю, выполнение переходит напрямую к пути panicdivide() вместо попытки деления.
```go
package runtime

func panicdivide() {
  panicCheck2("integer divide by zero")
  panic(divideError)
}
```
