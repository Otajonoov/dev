В Go переменные обычно объявляются с использованием ключевого слова var. Вы также можете сразу присвоить начальное значение:
```go
var a int
var h = 2

var (
  b int
  c string
  d float64
)

var e int = 1
var f, g int = 2, 3

g := 10 
h, i := 20, "example"
h, k := 30, 40
```


Короткое присваивание простое и удобное. Если переменная на левой стороне еще не существует в текущей области видимости, := создает ее. Если переменная уже существует, := переназначит ее. Это называется переобъявлением, при условии, что хотя бы одна из переменных на левой стороне новая.

Имейте в виду, что короткие присваивания могут использоваться только внутри функций. Причина проста: "Вне функции каждое выражение начинается с ключевого слова (var, func и т.д.), поэтому конструкция := недоступна." Это утверждение взято непосредственно из The Tour of Go [tour].

С самого начала короткие присваивания могут показаться похожими на var-объявления с выведенными типами, но есть важные различия:

|Аспект|Короткое объявление переменной (:=)|Объявление переменной с var (без типа)|
|---|---|---|
|Где можно использовать|Только внутри тела функции|Разрешено на уровне пакета и внутри функций|
|Аннотация типа|Не разрешена (тип выводится с правой стороны)|Разрешена (можно указать тип или опустить его)|
|Групповые объявления|Не поддерживаются|Поддерживаются с блоками var (...)|
|В коротком выражении перед if/for/switch|Разрешено (например, if x := f(); x > 0 {})|Не разрешено (нельзя написать if var x = f(); {})|
|Требование новой переменной|Должно вводить хотя бы одно новое не-пустое имя; иначе ошибка|Всегда объявляет новые привязки; нет правила "должно быть новым"|
|Переобъявление в том же блоке|Может повторно использовать существующие имена только если хотя бы одно имя слева новое|Переобъявление существующего имени в том же блоке — ошибка|
|Типичная ловушка|Случайное затенение внешней переменной|Также может затенять, но обычно используется с явными типами, где намерение яснее|

Легко предположить, что := позволяет переобъявлять переменные, если одна переменная слева новая. Но есть подвох: все элементы слева должны быть простыми именами переменных. Если нет, компилятор выбросит ошибку плохого объявления.
```go
person.Age, name := 20, "John"  // non-name person.Age on left side of :=
a[0], b := 1, 2                 // non-name a[0] on left side of :=
m["key"], n := 1, 2             // non-name m["key"] on left side of :=
*ptr, c := 1, 2                 // non-name *ptr on left side of :=
```

### ==**Затенение переменных и разрешение области видимости**==

Компилятор Go обрабатывает затенение, отслеживая области видимости. Каждая функция, блок if, блок else, выражение switch или автономный блок кода определяет свою собственную область видимости благодаря использованию фигурных скобок в Go:
```go
if condition {   // новая область видимости
}

for condition {  // новая область видимости
}

{                // ручной блок области видимости
}
```

Компилятор строит иерархическую структуру областей видимости. Каждая область видимости поддерживает 3 основных компонента: 
1. родительскую область видимости (внешнюю область видимости)
2. любые дочерние области видимости (блоки, вложенные внутрь)
3. map, которая связывает имена переменных с их соответствующими объектами.
```go
type Scope struct {
  parent   *Scope
  children []*Scope
  elems    map[string]Object
  isFunc   bool
  ...
}
```

![](../../../assets/obsidian-images/Pasted%20image%2020260125153317.png)

Когда компилятор ищет переменную, он начинает с current scope видимости. Если он не находит переменную там, он перемещается к parent scope видимости и продолжает, пока не найдет переменную или не достигнет верхнего уровня. Это гарантирует, что самое последнее объявление всегда имеет приоритет в current scope видимости, прежде чем смотреть наружу.

```go
// Lookup возвращает объект в области видимости s с заданным именем, если такой
// объект существует; в противном случае результат nil.
func (s *Scope) Lookup(name string) Object {
  return resolve(name, s.elems[name])
}

func (s *Scope) LookupParent(name string, pos syntax.Pos) (*Scope, Object) {
  for ; s != nil; s = s.parent {
    if obj := s.Lookup(name); obj != nil && ... {
      return s, obj
    }
  }
  return nil, nil
}
```
Эта структура естественно приводит к затенению переменных. Когда новая переменная с существующим именем объявляется в текущей области видимости, она имеет приоритет над любыми внешними объявлениями. Результат в том, что внутренняя переменная эффективно скрывает — или "затеняет" — внешнюю.

### ==**Применение компилятором правила неиспользуемых переменных**==

Если вы объявляете переменную и не используете ее в этой области видимости, компилятор остановится и выбросит ошибку. Это правило применяется не только к переменным — Go также помечает неиспользуемые импортированные пакеты и метки.

Неиспользуемые переменные сами по себе не всегда проблема, но они могут быть признаком чего-то большего, например, логической ошибки или незавершенного кода:
```go
var global int = 10

func (receiver AnyType) checkUnused(arg int) (named int) {
  var local int = 20  // Ошибка компилятора: local declared and not used
  {
    var local int = 30  // Ошибка компилятора: local declared and not used
  }
  return
}
```

В этом случае global, arg, receiver и named также не используются, но компилятор помечает только local. Почему это происходит?

Это сводится к тому, когда и где компилятор ожидает использования. За кулисами каждая переменная в Go представлена структурой ==Var== с полем ==used== — простым булевым значением, которое отслеживает, была ли переменная использована:
```go
type Var struct {
  object
  
  // Установлено, если переменная была использована или аргументы функции были     доступны
  used     bool
  
  // Установлено, если переменная является полем struct
  isField  bool 
  
  // Установлено, если переменная является встроенным полем struct, с именем как   именем типа 
  embedded bool
  ...
}

type PkgName struct {
  object
  imported *Package
  used     bool  // Установлено, если пакет был использован
}

type Label struct {
  object
  used bool  // Установлено, если метка была использована
}
```

Go применяет то же правило к импортированным пакетам и меткам — они также не могут оставаться неиспользуемыми. Как структуры PkgName, так и Label имеют поле used, точно так же, как Var.

Как обсуждалось ранее, компилятор Go отслеживает каждую область видимости и каждый объект в этой области видимости, используя map, которая связывает имена с объектами. Эта система также помогает компилятору идентифицировать неиспользуемые переменные. Он проверяет каждую переменную в своей области видимости — и любых дочерних областях видимости — проходя через этот список и проверяя поле used:
```go
func (check *Checker) usage(scope *Scope) {
  ...
  // Собрать все неиспользуемые переменные в текущей области видимости
  for name, elem := range scope.elems {
    elem = resolve(name, elem)
    if v, _ := elem.(*Var); v != nil && !v.used {
      unused = append(unused, v)
    }
  }
  ...
  // Выбросить ошибку для каждой неиспользуемой переменной, но продолжить проверку
  for _, v := range unused {
    check.softErrorf(v.pos, UnusedVar, "%s declared and not used", v.name)
  }
  
  // Рекурсивно проверить дочерние области видимости, но пропустить области видимости функций
  for _, scope := range scope.children {
    if !scope.isFunc {
      check.usage(scope)
    }
  }
}
```

Итак, компилятор Go находит неиспользуемые переменные, помечает их ошибкой, а затем переходит к проверке дочерних областей видимости. Области видимости функций пропускаются здесь, потому что, когда функция определена, ее тело обрабатывается отдельно. Другой метод, funcBody, заботится о проверке использования переменных в области видимости функции.
Не каждая неиспользуемая переменная вызывает ошибку. Компилятор не помечает глобальные переменные или параметры функций во время этой проверки.

Все переменные — будь то глобальные или локальные — начинаются с их поля used, установленного в false. Однако, поскольку эта проверка неиспользуемых применяется только в областях видимости функций, глобальные переменные игнорируются.
Параметры функций автоматически помечаются как использованные. Даже если они не ссылаются явно в теле функции, они могут потребоваться для удовлетворения интерфейса.

Вот как Go создает переменные с NewVar (для обычных переменных) и NewParam (для аргументов функций):
```go
func NewVar(...) *Var {
  return &Var{object: object{...}}
}

func NewParam(...) *Var {
  return &Var{object: object{...}, used: true}  // Параметры всегда 'used'
}
```

Получатели функций и именованные возвращаемые значения также создаются с использованием NewParam(..), которое автоматически помечает их как "использованные":
```go
func (receiver AnyType) doSomething(arg int) (named int) {
  return 0
}
```

В этом примере компилятор не помечает receiver, arg или named как неиспользуемые, даже если ни один из них не ссылается в теле функции, потому что они автоматически считаются использованными.

### ==**Константы: фиксированные значения, семантика типов и оптимизация**==

Константы — это фиксированные значения, которые не могут быть изменены после установки.
```go
// not bat
func calculatePrice(price float64) float64 {   
   // hard-coded value with no context
   return price + (price * 0.07) 
}

// better
const taxRate = 0.07 // clear, named constant for the tax rate
func calculatePrice(price float64) float64 {   
	return price + (price * taxRate) 
}
```

Когда код компилируется в бинарный файл, константы помещаются в сегменты памяти только для чтения, и инструкции ссылаются на них с помощью pointer'ов. Однако это в основном применяется к большим константам, таким как строковые литералы.

Для меньших констант компилятор часто оптимизирует производительность, встраивая их значения напрямую в поток инструкций, когда это возможно:
![](../../../assets/obsidian-images/Pasted%20image%2020260125215709.png)
В функции foo выше, которая просто возвращает константу 20, компилятор пропускает сохранение значения в памяти. Вместо этого он встраивает константу напрямую в инструкцию, делая функцию более эффективной.

Константы могут быть определены только с использованием базовых типов: ==int==, ==float==, ==bool== и ==string==. Вы не можете объявлять константы для более сложных структур, таких как ==array==, ==slice==, ==map== или ==struct==.

## **O'ZIMNING XULOSAM**

Bu bob Go'dagi o'zgaruvchilar va konstantalarning chuqur mexanizmlarini ochib berdi. Compiler qanday ishlashi va Go'ning dizayn qarorlari ortida nima borligini ko'rsatdi. Men uchun eng muhim xulosalar:

**O'ZGARUVCHILAR HAQIDA:**

**1. var deklaratsiyasining uchta komponenti**

- `obj` (o'zgaruvchi nomi)
- `typ` (tip)
- `init` (boshlang'ich qiymat)

Agar `typ` ham `init` ham yo'q bo'lsa, compiler `Typ[Invalid]` deb belgilaydi va xato beradi.

**2. := (short assignment) vs var** `:=` juda qulay, lekin bir nechta cheklovlar bor:

- Faqat function ichida ishlaydi
- Kamida bitta yangi o'zgaruvchi bo'lishi kerak
- Faqat oddiy o'zgaruvchi nomlari (`person.Age`, `a[0]`, `*ptr` kabi narsalar ishlamaydi)

**3. Variable shadowing - katta muammo!** Ichki scope'da bir xil nomli o'zgaruvchi e'lon qilish tashqi o'zgaruvchini "shadow" qiladi. Bu juda xavfli bug manbai:

go

```go
var discount int
if condition {
    discount, err := ...  // BU YANGI O'ZGARUVCHI! Tashqarisini shadow qildi
}
// discount hali ham 0
```

**4. Scope hierarchy** Compiler har bir scope uchun structure saqlaydi:

- `parent` - tashqi scope
- `children` - ichki scope'lar
- `elems` - map[string]Object (o'zgaruvchi nomlari)

Variable lookup scope hierarchy bo'ylab yuqoriga ko'tariladi.

**5. Unused variables** Compiler faqat **local** o'zgaruvchilarni tekshiradi:

- Global variables - tekshirilmaydi
- Function parameters - auto `used=true` bo'ladi
- Named return values - auto `used=true` bo'ladi

**KONSTANTALAR HAQIDA:**

**6. Typed vs Untyped constants**

- **Typed**: `const a int = 10` - qat'iy type'ga ega
- **Untyped**: `const b = 10` - context'ga qarab moslashadi

**7. Constant folding** Compiler konstantalarni compile-time'da hisoblaydi. Shuning uchun overflow/truncation xatolari compile-time'da tutiladi:

go

```go
const a int8 = 127
const b int8 = a + 1  // ERROR: overflows int8
```

**8. Untyped constant limits**

- Default type'lar: int, float64, complex128, string, bool
- Lekin ko'p type'larga assign bo'lishi mumkin
- Size limit: 512 bit (kelajakda 1100 bit bo'lishi mumkin)

**9. iota - smart counter**

- Har const block boshida 0 dan boshlanadi
- Har satrda 1 ga oshadi
- Expression'larda ishlatilishi mumkin: `2*iota + 1`
- Implicit repetition: qiymat yozmasang, oldingi pattern takrorlanadi

**10. iota implementation**

go

```go
iota := constant.MakeInt64(int64(index - first))
```

iota qiymati faqat const block ichidagi pozitsiyaga bog'liq.

**AMALIY MASLAHATLAR:**

1. **Shadowing'dan ehtiyot bo'ling** - `golangci-lint` yoki shunga o'xshash tool'lardan foydalaning
2. **:= dan ehtiyotkorlik bilan foydalaning** - reassignment kerakmi yoki yangi o'zgaruvchi kerakmi deb o'ylab ko'ring
3. **Constant'larni prioritet qiling** - magic number'lar o'rniga
4. **iota'dan samarali foydalaning** - enums va bitflag'lar uchun
5. **Typed constant faqat kerak bo'lganda** - ko'pincha untyped yetarli va moslashuvchan

**QIZ IQ FAKTLAR:**

- Go compiler code'i o'qish mumkin va instructive!
- `var` keyword tashqarida ishlatish uchun majburiy, chunki har bir statement keyword bilan boshlanishi kerak
- `iota` keyword emas, identifier (variable nomi sifatida ishlatish mumkin, lekin tavsiya etilmaydi!)
