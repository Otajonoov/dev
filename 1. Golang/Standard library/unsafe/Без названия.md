Прежде чем углубиться в то, что делает что-то 'unsafe' (небезопасным), важно понять, что Go считает 'safe' (безопасным). Обычные pointer'ы в Go, будь то *int, *string, *struct или другие, безопасны из-за двух основных правил, встроенных в дизайн Go.

1. Go не позволяет прямое преобразование между типами pointer'ов. Если у вас есть pointer типа *int64, вы не можете просто преобразовать его в *float64. Компилятор обеспечивает типобезопасность и выбросит ошибку времени компиляции. Вы также не можете сравнивать pointer'ы разных типов с помощью == или !=, и не можете напрямую присвоить pointer одного типа другому типу. На первый взгляд это может показаться очевидным — особенно если вы знакомы со строгой типизацией Go. Но по сути pointer — это просто число — целое число, представляющее адрес памяти. Так почему вы не можете присвоить его адресу памяти другого типа, как вы бы делали с любым другим числом?
```go
func main() {
  a := new(int)
  b := new(float64)
  
  // cannot convert a (variable of type *int) to type *float64
  b = (*float64)(a)
  
  // invalid operation: a == b (mismatched types *int and *float64)
  if a == b { ... }
  
  // cannot convert b (variable of type *float64) to type *int
  if (*int)(b) == a { ... }
}
```

2. Go не позволяет арифметику на pointer'ах. Вы не можете вручную сдвигать или корректировать адреса памяти, как это можно делать в таких языках, как C:
```go
func main() {
  a := new(int)
  
  // invalid operation: a++ (non-numeric type *int)
  a++
  
  // invalid operation: a += 1 (mismatched types *int and untyped int)
  a += 1
}
```

Пакет unsafe меняет это. С unsafe.Pointer вы получаете гораздо больше свободы — но эта свобода идет с риском. Используя unsafe.Pointer, вы можете преобразовывать между типами pointer'ов без каких-либо ограничений, обходя обычные проверки безопасности Go и полностью нарушая первое правило.

Давайте рассмотрим пример, где эта свобода становится рискованной. В этом случае мы обходим типобезопасность Go, присваивая pointer другому типу pointer'а:
```go
import "unsafe"

func main() {
  a := new(int64)
  b := new(float64)
  
  // Преобразовать a в unsafe.Pointer
  b = (*float64)(unsafe.Pointer(a))
  
  println("a", a, *a)
  println("b", b, *b)
}

// a 0x1400004e740 0
// b 0x1400004e740 +0.000000e+000
```

Когда память используется таким образом, это открывает дверь для повреждения данных. Хотя int64 и float64 оба занимают 8 байт, они хранят и интерпретируют эти байты по-разному. Чтобы увидеть это в действии, давайте установим значение float64 в 1 и посмотрим, что произойдет с int64:
```go
func main() {
  a := new(int64)
  b := new(float64)
  
  // Преобразовать a в unsafe.Pointer
  b = (*float64)(unsafe.Pointer(a))
  *b = 1
  
  println("a", a, *a)
  println("b", b, *b)
}

// Output
// a 0x1400004e740 4607182418800017408
// b 0x1400004e740 +1.000000e+000
```

Используя unsafe.Pointer, мы уже нарушили первое правило безопасности pointer'ов Go. Чтобы обойти второе правило, которое запрещает арифметику pointer'ов, нам нужно преобразовать pointer в uintptr.

Go 1.17 представил unsafe.Add, который помогает избежать преобразования pointer'а в uintptr. Эта функция обрабатывается как внутренняя функция компилятора, которая выполняет побайтовую арифметику pointer'ов:
```go
// Функция Add добавляет len к ptr и возвращает обновленный pointer
// [Pointer](uintptr(ptr) + uintptr(len)).
// Аргумент len должен быть целочисленного типа или нетипизированной константой.
// Константный аргумент len должен быть представим значением типа int;
// если это нетипизированная константа, ей присваивается тип int.
// Правила для валидного использования Pointer все еще применяются.
func Add(ptr Pointer, len IntegerType) Pointer
```

uintptr — это целочисленный тип, который представляет адрес памяти как число. Однако ему не хватает свойств pointer'а: его нельзя разыменовать, и он не имеет возможностей управления памятью. Проще говоря, это просто число. Это преобразование позволяет нам обойти ограничения компилятора, позволяя действия, такие как доступ к приватным (неэкспортированным) полям в struct. Предположим, у нас есть struct User в другом пакете, и мы хотим получить доступ к его неэкспортированному полю password:
```go
type User struct {
  Username string
  password string
}
```

Давайте скажем, мы напишем функцию exploitPassword(), которая извлекает поле password без прямого доступа к нему:
```go
func main() {
  u := User{
    Username: "func25",
    password: "123456",
  }
  
  println(&u, &u.Username, &u.password)
  println(exploitPassword(&u))
}

func exploitPassword(u *User) string {
  ... // здесь вступает в игру uintptr
}

// Output:
// 0x1400004e700 0x1400004e700 0x1400004e710
// 123456
```

Чтобы завершить эту функцию, нам нужно понять, как struct'ы размещаются в памяти. Когда вы определяете struct, Go выделяет память для всего struct'а как единый блок, размещая каждое поле по порядку, с возможным padding'ом между ними. Это означает, что поля Username и password хранятся рядом друг с другом в памяти, причем каждое поле находится на фиксированном смещении на основе его порядка и размера.

Например, если u начинается с 0x1400004e700, Username может находиться прямо по адресу 0x1400004e700, в то время как password будет непосредственно следовать по адресу 0x1400004e710. Это точно соответствует тому, что мы видим в выводе:

Под капотом string в Go фактически является struct'ом с двумя полями: pointer на базовый массив байтов, который указывает на первый байт строки, и int, который хранит длину строки. Это означает, что string занимает 8 байт для pointer'а и еще 8 байт для длины, в общей сложности 16 байт. Вот почему u.password находится на расстоянии 16 байт от начала struct User.

Также важно отметить, что первое поле struct разделяет тот же адрес памяти, что и сам struct. Это делает возможным эксплуатацию схемы памяти всего одной строкой кода с использованием пакета unsafe:
```go
type User struct {
  Username string
  password string
}

func main() {
  u := User{
    Username: "func25",
    password: "123456",
  }
  
  passwordUintptr := uintptr(unsafe.Pointer(&u)) + 16
  passwordPtr := (*string)(unsafe.Pointer(passwordUintptr))
  
  println("password:", *passwordPtr)
  println("-- debug --")
  println("u:", &u)
  println("u.Username:", &u.Username)
  println("u.password:", &u.password)
}

// Output:
// 123456
// -- debug --
// u: 0x1400004e6e8
// u.Username: 0x1400004e6e8
// u.password: 0x1400004e6f8
```

В этом примере мы не обращаемся к u.password напрямую. Вместо этого мы используем адрес памяти u для вычисления того, где должен находиться password. Поскольку password находится на расстоянии 16 байт от начала struct User, мы можем вычислить его адрес и получить доступ к его значению через unsafe.Pointer.
```go
passwordUintptr := uintptr(unsafe.Pointer(&u)) + 16
```

Результат был бы таким же, если бы мы использовали &u.Username вместо &u, потому что первое поле struct разделяет тот же адрес, что и сам struct.

В отличие от этого, uintptr обрабатывается исключительно как целое число, не более того. Runtime Go не отслеживает значения uintptr как ссылки на память, что означает, что garbage collector не знает, что uintptr может все еще ссылаться на важные данные. Это может привести к неожиданной очистке памяти, так как runtime может неправильно предположить, что данные больше не используются.

Безопасное использование unsafe: руководства и лучшие практики

unsafe.Pointer может быть мощным инструментом для оптимизации производительности в высокоэффективных приложениях. Один из распространенных случаев использования — преобразование между []byte и string без запуска дополнительных выделений памяти:
```go
func String(b []byte) string {
  return unsafe.String(unsafe.SliceData(b), len(b))
}

func Bytes(s string) []byte {
  return unsafe.Slice(unsafe.StringData(s), len(s))
}

func StringSlicing(data []byte, start, end int) string {
  n := int(end - start)
  return unsafe.String((*byte)(unsafe.Pointer(uintptr(unsafe.Pointer(unsafe.SliceData(data))) + uintptr(start))), n)
}
```

Эти функции преобразуют между []byte и string без копирования данных.
Обычно преобразование []byte в string требует создания новой строки и копирования данных байтов. Это гарантирует, что строки остаются неизменяемыми. Однако использование unsafe позволяет пропустить это копирование. Вместо создания новой памяти эти функции просто переинтерпретируют существующие данные, делая операцию быстрее, но также более рискованной.

**UNSAFE PACKAGE:**

**1. Go'ning ikki xavfsizlik qoidasi:**

- Pointer type'lar orasida direct conversion yo'q
- Pointer arithmetic yo'q (pointer + 1 kabi)

**2. unsafe.Pointer - kuchli lekin xavfli** Bu ikki qoidani ham buzishga imkon beradi:

- Har qanday pointer type'ga convert qilish mumkin
- uintptr orqali pointer arithmetic qilish mumkin

**3. uintptr vs unsafe.Pointer**

- `unsafe.Pointer` - hali ham Go runtime tomonidan pointer sifatida taniladi
- `uintptr` - oddiy integer, GC va stack growth'da track qilinmaydi
- **Stack growth paytida uintptr invalid bo'lib qoladi!**

**4. Unsafe'ning 5 ta qoidasi:**

a) **Conversion _T1 to _T2__: Xotira layout compatible bo'lishi kerak

b) **Pointer to uintptr**: One-way (qaytib pointer'ga convert qilmaslik kerak)

c) **Pointer to uintptr and back with arithmetic**: Bitta expression'da qilish kerak

```go
// VALID
p = unsafe.Pointer(uintptr(p) + offset)

// INVALID 
u := uintptr(p)
p = unsafe.Pointer(u + offset)
```

d) **System calls**: `syscall.Syscall` ichida inline convert qilish

e) **reflect.Value.Pointer**: Darhol convert qilish kerak

**5. Unsafe use case'lar:**

- []byte ↔ string conversion (zero-copy)
- Private struct field'larga kirish
- Performance optimization
- Lekin **string immutability buziladi!**

**ADDRESSABLE VS UNADDRESSABLE:**

**6. Addressable values:**

- Package/function level variables
- Struct fields (agar struct addressable bo'lsa)
- Array elements (agar array addressable bo'lsa)
- Slice elements (har doim!)

**7. Unaddressable values:**

- Function return values
- Constants
- Literals (10, "hello")
- Expressions (10 + 20)
- Map elements

**8. Function return unaddressable sababi:**

- Kichik qiymatlar register'da saqlanadi (address yo'q)
- Katta qiymatlar caller'ning stack frame'iga yoziladi
- Function return qilganidan keyin, o'sha stack frame invalid bo'ladi

**9. Map elements unaddressable sababi:** Map internal storage'ni reorganize qilishi mumkin. Address olsangiz, pointer invalid bo'lib qolishi mumkin.

**10. Slice element exception:** Slice o'zi unaddressable bo'lsa ham, element'lari addressable. Sababi - slice underlying array'ga pointer, va array xotirada bor.
