
SSA pipeline'ining oxirida, barcha optimizatsiyalar va register allocation'dan keyin, bizda to'liq SSA-transformatsiya qilingan funksiya (`*ssa.Func`) bor. Bu funksiya endi yuqori darajali yoki abstrakt emas — u ma'lum tartibda rejalashtirilgan, fizik register'lar tayinlangan va bajarilishga tayyor **aniq operatsiyalar**dan iborat. Yakuniy qadam — bu `ssa.Func` ustidan yurib, **haqiqiy mashina kodi** generatsiya qilish.

## Step 1: SSA'dan instruktsiyalarga — `genssa`

SSA backend `ssa.Func` ni kod generatsiya fazasiga uzatadi:

```go
func genssa(f *ssa.Func, pp *objw.Progs) {}
```

Compiler SSA funksiyasidagi har bir basic block'ni ko'rib chiqadi va value'larni (SSA instruktsiyalar) ketma-ket qayta ishlaydi. Har bir value uchun arxitekturaga xos mantiq (`Arch.SSAGenValue`) ishlatib, value'ni bir yoki bir nechta **mashina instruktsiyasiga** tarjima qiladi.

> **obj.Prog** — instruktsiyaning tuzilmali ifodalanishi. Bu hali xom baytlar emas — yakuniy binary'ga chiqariladigan narsani yaqindan aks ettiradigan strukturalar.

Block qayta ishlangandan keyin, `Arch.SSAGenBlock` chaqiriladi — arxitekturaga xos block-darajadagi kod generatsiya qilish uchun.

**Nima uchun bunday qilingan:** SSA — hali abstrakt ko'rinish. Haqiqiy CPU unda to'g'ridan-to'g'ri bajara olmaydi. Har bir SSA operatsiyani aniq mashina instruktsiyasiga aylantirish kerak.

## Step 2: Bosqichma-bosqich misol

Oldingi misolimizga qaytaylik:

```go
func example(a int, b int) int {
    var y int
    if b > 0 {
        y = a * 2
    } else {
        y = b * 2
    }
    return y + b
}
```

SSA (register'lar tayinlangan holda):

```
b1:-
  v8 = ArgIntReg <int> {b+0} [1] : R1 (b[int])
  v7 = ArgIntReg <int> {a+0} [0] : R0 (a[int])
  v2 = CMPconst <flags> [0] v8
GT v2 → b3 b4

b3: ← b1-
  v19 = SLLconst <int> [1] v7 : R2
Plain → b2

b4: ← b1-
  v5 = SLLconst <int> [1] v8 : R2
Plain → b2

b2: ← b3 b4-
  v16 = Phi <int> v19 v5 : R2 (y[int])
  v1 = InitMem <mem>
  v18 = ADD <int> v16 v8 : R0
  v20 = MakeResult <int,mem> v18 v1 : <>
Ret v20
```

Bu SSA'da uchta block bor: entry block (b1), true branch (b3), false branch (b4) va merge/exit block (b2). Register allocation allaqachon bajarilgan — har bir value'da tayinlangan fizik register'lar bor.

### 2.1. Entry block — b1

```
v8 = ArgIntReg <int> {b+0} [1] : R1 (b[int])
v7 = ArgIntReg <int> {a+0} [0] : R0 (a[int])
v2 = CMPconst <flags> [0] v8
GT v2 → b3 b4
```

`v8` va `v7` — input argumentlar `b` va `a`, R1 va R0 register'lariga tayinlangan. Compiler bu argumentlarni shu bosqichda tan oladi, lekin ular uchun mashina kodi generatsiya qilishni **keyinga qoldiradi**. Argumentlarni register'larga joylashtirish mas'uliyati **chaqiruvchi funksiyaga** tegishli.

> **ABI qoidasi:** A funksiyasi B funksiyasini chaqirganida, A argumentlarni R0, R1 va h.k. ga tayyorlaydi. Chaqirilgan funksiya (callee) B faqat ABI konventsiyalariga ko'ra argumentlarni **qaerdan topishni** bilishi kerak.

Keyingi instruktsiya — `v2`:

```
v2 = CMPconst <flags> [0] v8
```

Bu `b` qiymatini (R1 register'da) 0 konstanta bilan **solishtiradi**. `genssa` funksiyasi bu instruktsiyani ARM64-ga xos kod generatoriga (`Arch.SSAGenValue`) uzatadi va u tegishli mashina instruktsiyasini chiqaradi:

Bu `CMP $0, R1` ga aylanadi — immediate qiymat 0 ni R1 register'ning tarkibi bilan solishtiradi va processor'ning **condition flag**'larini yangilaydi.

### 2.2. Shartli branch va inversiya

```
GT v2 → b3 b4
```

Ma'nosi: «Agar v2 natijasi noldan katta bo'lsa — b3 blokga o'tish. Aks holda — b4 ga.»

Lekin **layout pass** eslanganmi? U bloklarni ishlashni yaxshilash uchun qayta tartiblagan. Bu holatda b3 (b > 0 holati) b1 dan **keyin joylashtirilgan**. Ya'ni dastur b3 ga **o'z-o'zidan o'tishi** kerak va faqat shart false bo'lganda jump qilishi kerak.

Lekin yuqoridagi mantiq b > 0 bo'lganda jump qilish — bu biz xohlagan narsaning **aksi**. Buni to'g'rilash uchun compiler **teskari** shartli branch instruktsiyasini chiqaradi: **BLE** (branch if less than or equal):

Tarjimasi: «Agar b <= 0 bo'lsa — b4 blokga jump qil. Aks holda — b3 ga o'z-o'zidan o'tsin (fallthrough).»

**Nima uchun bunday qilingan:** Jump'lar CPU uchun **qimmat** — I-cache va prefetcher'ni buzadi. Eng ko'p bajariladigan yo'lni **fallthrough** qilish — ishlashni yaxshilaydi.

### 2.3. b3 blok — chapga siljitish

b3 dagi birinchi instruktsiya — chapga siljitish (left shift). Lekin `SLLconst` — bu **pseudo-instruktsiya**. ARM64 backend (`Arch.SSAGenValue`) uni haqiqiy mashina instruktsiyasiga aylantiradi:

Bu `LSL $1, R0, R2` ga aylanadi — R0 dagi qiymatni (ya'ni `a`) bitta bitga chapga siljitadi va natijani R2 register'ga saqlaydi. b3 bloki b2 ga **shartsiz jump** bilan tugaydi:
### 2.4. b4 blok

b4 bloki shunga o'xshash operatsiya bajaradi, lekin R0 dagi v7 o'rniga R1 dagi **v8** ishlatadi:

`LSL $1, R1, R2` — R1 dagi `b` qiymatini chapga siljitib, R2 ga saqlaydi.

### 2.5. b2 blok — merge va return

```
b2: ← b3 b4
  v16 = Phi v19 v5 : R2
  v18 = ADD v16 v8 : R0
  v1 = InitMem <mem>
  v20 = MakeResult v18 v1
Ret v20
```

**Phi node'lar** to'g'ridan-to'g'ri instruktsiyalarga tarjima **qilinmaydi**. Buning o'rniga, oldingi compiler pass'lari allaqachon register tayinlashni shunday boshqarganki, Phi ga input'lar (v19 va v5) va natija (v16) hammasi **bir xil fizik registrni** (R2) ulashadi. Natijada, qaysi boshqaruv yo'li olingan bo'lsa ham, siljitilgan qiymat allaqachon R2 da **mavjud**.

`v18` — yakuniy qo'shish: `y + b`. Backend buni **ADD** instruktsiyasiga tarjima qiladi:

Bu `ADD R1, R2, R0` ga aylanadi — R2 dagi qiymatni (y) R1 dagi qiymatga (b) qo'shib, natijani R0 ga saqlaydi. Go ABI'da R0 — integer return qiymatlari uchun **standart zaxiralangan** register.

b2 — return block. Backend haqiqiy **RET** instruktsiyasini chiqaradi — boshqaruvni caller'ga qaytaradi va R0 dagi qiymatni funksiya natijasi sifatida ishlatadi.

### 2.6. Yakuniy Go assembly ko'rinishi

```
00000 TEXT main.example(SB), ABIInternal
...
00006 CMP $0, R1
00007 BLE 10
00008 LSL $1, R0, R2
00009 JMP 11
00010 LSL $1, R1, R2
00011 ADD R1, R2, R0
00012 RET
00013 END
```

Bular hali **obj.Prog** tuzilmalari — hali xom mashina kodi emas.

## Step 3: genssa yakuniy bosqichi

Barcha block'lar va value'lar qayta ishlangandan keyin, `genssa` bir nechta muhim vazifalarni bajaradi:

### 3.1. Debug ma'lumotlari generatsiya qilish

O'zgaruvchilar va qator raqamlarini mashina kodi joylashuvlariga xaritalaydigan **debug ma'lumotlari** (location list'lar) generatsiya qilinadi. Bu debugger'lar va stack trace'lar kabi tool'lar uchun zarur.

### 3.2. Branch target'larni hal qilish

Har bir block'ning tegishli boshlang'ich `obj.Prog` instruktsiyasi bor. Oldin chiqarilgan `BLE` va `JMP` kabi branch instruktsiyalari target'lari **belgilanmagan** qoldirilgan edi. Endi compiler ularning `prog.target` field'larini maqsad blokni ifodalovchi aniq `obj.Prog` ga ko'rsatadi.

### 3.3. Stack frame hajmini hisoblash

Lokal o'zgaruvchilar, argument spill slot'lar va chaqiruvlar uchun kerak bo'lgan vaqtinchalik qiymatlar uchun joy hisobga olinadi. Bu oddiy funksiyada frame hajmi **minimal**, lekin bu qadam baribir zarur.

**Nima uchun bunday qilingan:** Branch target'lar SSA generatsiya paytida noma'lum — faqat barcha block'lar qayta ishlangandan keyin aniq bo'ladi. Debug ma'lumotlari esa production'da xatolarni tuzatish uchun zarur.

## Step 4: obj.Prog → xom mashina kodi

Barcha `obj.Prog` tuzilmalari hali xom mashina kodi emas. Yakuniy qadam — **Flush** funksiyasi:

```go
func (pp *Progs) Flush() {
    plist := &obj.Plist{Firstpc: pp.Text, Curfn: pp.CurFunc}
    obj.Flushplist(base.Ctxt, plist, pp.NewProg)
}
// Flush (src/cmd/compile/internal/objw/prog.go)_
```

`Flushplist` `obj.Prog` ro'yxatini bosib o'tib, har bir instruktsiyani arxitekturaga xos backend'ga uzatadi. Bu handler'lar target platformaga qarab `cmd/internal/obj/x86/` yoki `cmd/internal/obj/arm64/` kabi package'larda joylashgan. Har bir instruktsiya tegishli **binariy mashina kodi** bayt ketma-ketligiga aylantiriladi.

**Nima uchun bunday qilingan:** `obj.Prog` — hali platform-agnostik tuzilma. Har bir CPU arxitekturasi o'zining **binariy formatiga** ega — shu formatga yakuniy konversiya shu yerda bo'ladi.

## Step 5: Assembler — object fayl (.o) generatsiya qilish

Assembler har bir global element (funksiyalar, o'zgaruvchilar) uchun **symbol** yaratadi. Lekin instruktsiya bu symbol'lardan biriga murojaat qilganida, assembler uning yakuniy memory adresini **hali bilmaydi**. Shuning uchun u symbol'ga havola qiluvchi **placeholder** qo'yadi.

> **Relocation entry** — «X joylashuvda kodda Y symbol'ga havola bor. Linker, Y uchun yakuniy adresni hisoblang va X joylashuvga shu adresni yozing» degan yozuv.

**Nima uchun assembler o'zi yakuniy adreslarni tayinlay olmaydi:** Yakuniy memory layout'ni **linker** aniqlaydi, assembler emas. Assembler faqat bitta package'ni ko'radi — butun dasturning layout'ni bilmaydi.

Go assembler to'liq object fayllar (`.o`) hosil qiladi. Object fayl odatda quyidagi asosiy section'larni o'z ichiga oladi:

|Section|Vazifasi|
|---|---|
|**`.text` (TEXT)**|Bajariladigan kod uchun haqiqiy mashina instruktsiyalari|
|**`.data` (DATA)**|Initsializatsiya qilingan global o'zgaruvchilar|
|**`.bss` (BSS)**|Initsializatsiya **qilinmagan** global o'zgaruvchilar. Source code'da boshlang'ich qiymat tayinlanmagan|
|**`.rodata` (RODATA)**|Faqat o'qiladigan data: konstantalar va string literal'lar|
|**Symbol table'lar**|Barcha e'lon qilingan symbol'lar (funksiyalar, global o'zgaruvchilar) va ularning metadata'si. Har bir symbol nisbiy adres va tegishli section bilan qayd etiladi|
|**Relocation table'lar**|Linkage paytida o'zgartirish talab qiladigan kod yoki data'dagi pozitsiyalar. Assembly vaqtida yakuniy joylashuvi noma'lum bo'lgan symbol'larga havolalarni belgilaydi|
|**DWARF Debug ma'lumotlari**|Ixtiyoriy debugging qo'llab-quvvatlash section'i: mashina kodidan source qatorlarga xaritalash, o'zgaruvchi turlari, scope'lar va h.k.|

## Step 6: Export data generatsiya qilish va archive fayl

Bu section'lar generatsiya qilingandan keyin, assembler ishi **tugagan**. Lekin Go compiler'ning yana bitta vazifasi bor: joriy package uchun **export data** generatsiya qilish.

> **Export data** — boshqa package'lar ushbu package'ni import qilganida kerak bo'ladigan type ma'lumotlari, funksiya signature'lari va konstantalar. Bu alohida kompilatsiya imkonini beradi va dependency'larning to'liq qayta kompilatsiyasidan qochadi.

Nihoyat, Go assembler natijasini (`.o` fayl) va export data'ni bitta **archive fayl**ga birlashtiradi (odatda `.a` kengaytmali):

|Section|Tarkibi|
|---|---|
|**`__.PKGDEF`**|Export data|
|**`go.o`**|Kompilatsiya qilingan object fayl (kod va data)|

Buni Go build cache'da to'g'ridan-to'g'ri kuzatish mumkin:

```bash
$ ar -tv /path/to/cache-file
rw-r--r-- 0/0  37542   Jan 1 08:00 1970 __.PKGDEF
rw-r--r-- 0/0  1137368 Jan 1 08:00 1970 _go_.o
```

`__.PKGDEF` — export data, `_go_.o` — kompilatsiya qilingan kod va data bilan object fayl.

## Step 7: Linking — yakuniy bajariladigan faylni yaratish

Bu nuqtada **linker** ishga tushadi. U barcha input fayllarni o'qiydi — archive fayllar (`.a`) va mumkin bo'lgan alohida object fayllar (`.o`). Uning asosiy vazifasi — **yakuniy bajariladigan fayl** (executable) generatsiya qilish.

> **Linker** — barcha alohida kompilatsiya qilingan qismlarni bitta yaxlit dasturga birlashtiruvchi tool. U symbol'larni hal qiladi, memory layout'ni aniqlaydi va relocation'larni patch qiladi.

Linker quyidagi qadamlarni bajaradi:

### 7.1. Symbol table qurish

Input fayllardan kod section'lar, data section'lar, symbol table'lar va relocation entry'larni oladi. Keyin barcha input'lardan symbol'larni yig'ib **birlashtirilgan symbol table** quradi. Har bir symbol **qaerda aniqlangan**ligini tekshiradi va barcha havola qilingan symbol'lar **hal qilingan**ligini tasdiqlaydi.

Agar biror symbol ishlatilgan, lekin hech qanday input faylda aniqlanmagan bo'lsa — linker **xato** xabar qiladi.

### 7.2. Section'larni birlashtirish

Linker barcha object fayllardan `.text`, `.data` va boshqa section'larni output fayldagi **yagona section'larga** birlashtiradi. Masalan, barcha `.text` section'larni yakuniy bajariladigan faylning kod segment'iga birlashtiradi.

Bu qadam tugagach, linker bajariladigan fayl uchun **to'liq memory layout**ni ko'radi.

### 7.3. Relocation'larni patch qilish

Assembly paytida yaratilgan relocation entry'larni qayta ishlaydi. Har bir relocation entry uchun linker target symbol'ning **yakuniy memory adresini** ishlatib, to'g'ri qiymatni hisoblaydi. Bu qiymatni kod yoki data section'dagi tegishli joyga **patch qiladi**.

Bu patching barcha funksiya chaqiruvlari, data havolalari va jump target'lar yakuniy bajariladigan faylda **to'g'ri absolyut adreslar** ishlatishini ta'minlaydi.

**Natija:** Bitta bajariladigan binary fayl — barcha kodlar, data'lar va runtime birlashtirilgan, barcha adreslar to'g'ri belgilangan.

## Xulosa — Butun Kompilatsiya Jarayoni

Go kompilatsiya jarayoni source code'ni bajariladigan mashina instruktsiyalariga aylantiradi. `go build -n` bilan Go build tizimidagi qadamlar ketma-ketligini ko'rish mumkin.

|Bosqich|Nima bo'ladi|Natija|
|---|---|---|
|**Parsing**|Source code → token → syntax tree|AST — dasturning tuzilmaviy ifodalanishi|
|**Type checking**|Tiplar tekshiriladi, konstantalar hisoblanadi, scope'lar quriladi|Semantik xatolar aniqlanadi, export data yuklanadi|
|**IR Construction**|AST + type ma'lumotlari → IR node'lar, export data kodlanadi|Relocation'lar orqali package ma'lumotlari serializatsiya qilinadi|
|**Optimization**|Devirtualization, inlining, escape analysis|Tezroq, samaraliroq IR|
|**Walk**|For-range → for, switch → if+goto, channel → runtime call|SSA uchun tayyor pastlatilgan IR|
|**SSA**|IR → SSA shakli, 50+ optimizatsiya pass'lari|Bounds/nil check elimination, constant folding, CSE|
|**Lowering**|Umumiy SSA → arxitekturaga xos operatsiyalar|CPU-specific pseudo-instruktsiyalar|
|**Layout + Schedule**|Block'lar va instruktsiyalar tartibi optimallashtiriladi|Kam jump, yaxshi I-cache utilization|
|**Register Allocation**|Virtual register'lar → fizik register'lar|Spilling, Phi alignment|
|**Machine Code Gen**|SSA → obj.Prog → xom mashina kodi|`.o` object fayllar|
|**Export Data**|Package uchun type/func/const ma'lumotlari|`.a` archive fayl (__.PKGDEF + go.o)|
|**Linking**|Symbol'lar hal qilinadi, section'lar birlashtiriladi, relocation'lar patch qilinadi|**Yakuniy bajariladigan binary**|

**Asosiy tushunchalar:**

- **Export data** Go'ning alohida kompilatsiya qobiliyatining asosidir — package ma'lumotlari, tiplar va object tafsilotlari relocation'lar orqali kodlanadi
- **Devirtualization** ma'lum interface chaqiruvlarini to'g'ridan-to'g'ri funksiya chaqiruvlariga almashtiradi
- **Inlining** funksiya tanasini chaqiruv joyiga joylashtiradi — funksiya murakkabligi va chaqiruv kontekstini baholashni talab qiladi
- **Walk fazasi** yuqori darajali Go konstruktsiyalarini (for-range, switch, channel) soddaroq operatsion ketma-ketliklarga pasaytiradi
- **SSA backend** (Go 1.7 dan) data flow'ni aniqlashtirdi va bounds/nil check elimination kabi samarali optimizatsiyalarni ta'minladi
- **Lowering** umumiy SSA operatsiyalarini arxitekturaga xos instruktsiyalarga tarjima qiladi
- **Linker** barcha compilation unit'lardan symbol'larni hal qiladi, yakuniy memory layout'ni aniqlaydi va relocation entry'larni patch qiladi

Dastlabki source parsing'dan yakuniy binary linking'gacha — Go compiler transformatsiya va optimizatsiyalarning **tuzilmali ketma-ketligini** qo'llaydi. Bu Go'ning soddalik, samaradorlik va ishonchlilik xususiyatlarini ta'minlaydi.
