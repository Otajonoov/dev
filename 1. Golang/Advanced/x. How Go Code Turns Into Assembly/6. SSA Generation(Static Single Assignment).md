
> **SSA (Static Single Assignment)** — har bir o'zgaruvchi **faqat bir marta** tayinlanadigan oraliq ko'rinish. Bu data flow va kod dependency'larni **aniqroq va ko'rinarliroq** qiladi, ko'plab optimizatsiyalarni osonlashtiradi.

SSA'dan oldin ba'zi optimizatsiyalar qiyin edi. SSA buni o'zgartiradi — bound check elimination, nil check elimination, dead code elimination, constant propagation kabi optimizatsiyalarni amalga oshirish mumkin bo'ladi.

Keith Randall (Go jamoasi a'zosi) SSA backend'ni taklif qilgan va qurgan — u birinchi marta **Go 1.7** da paydo bo'lgan, dastlab AMD64 arxitekturasini qo'llab-quvvatlagan.

GopherCon 2017 da Keith Randall sezilarli ishlash yaxshilanishlarini xabar qilgan: Go 1.7 benchmark'larida bajarilish tezligida **12% oshish** va kod hajmida **13% qisqarish**. Go 1.8 SSA qo'llab-quvvatlashini ARM, MIPS va PPC arxitekturalariga kengaytirgan — ARM'da ilovalar **20% tezlashgan**, kod hajmi **18% kamaygan**.

## Step 1: SSA asosiy g'oyasi

SSA shaklining fundamental printsipi — har bir o'zgaruvchi **aniq bir marta** tayinlanadi. Ya'ni, agar `a` o'zgaruvchisiga `10` qiymati berilsa, keyinchalik **yangi qiymat berib bo'lmaydi**. SSA da o'zgaruvchilar dastlabki tayinlashdan keyin amalda **faqat o'qiladigan**.

| Go                      | Go SSA                     |
| ----------------------- | -------------------------- |
| `x = 1; ...; x = x + 2` | `x1 = 1; ...; x2 = x1 + 2` |

`x` ni qayta ishlatish o'rniga, SSA shakli kerak bo'lganda `x1` va `x2` kabi **yangi versiyalar** yaratadi. Bu o'zgaruvchi kodning ma'lum nuqtasida nimani saqlayotgani haqidagi har qanday **noaniqlikni yo'qotadi**.

Birinchi misolga qarang: compiler darhol `x = x + 2` ni `x = 3` ga aylantirib **bo'ladimi**? Unchalik emas — chunki ikki tayinlash orasida `x` qiymati **o'zgargan bo'lishi mumkin**. Compiler oradagi har bir qatorni skanerlashi kerak edi.

SSA shaklida ancha sodda: `x2` faqat `x1` ga bog'liq ekanligi aniq ko'rinadi, va `x1` = 1 qilib belgilangan. SSA qoidalari tufayli `x1` **hech qachon o'zgartirilmaydi** — compiler ishonch bilan `x2 = x1 + 2` ni `x2 = 3` ga qayta yozishi mumkin.

**Nima uchun bunday qilingan:** Data flow'ni aniq qilish orqali compiler optimizatsiyalarni **xavfsiz va samarali** amalga oshiradi.

## Step 2: SSA sintaksisi

Go kodida o'zgaruvchi nomlari, scope'lar va bloklar bilan ishlaymiz. SSA bu tushunchalarni **olib tashlaydi**. U o'zgaruvchi nomlari yoki an'anaviy scope'lar ishlatmaydi — faqat yagona ketma-ket ID'lar bilan aniqlangan **qiymatlar va bloklar**:

![](../../../assets/obsidian-images/Pasted%20image%2020260310130200.png)

Barcha qiymatlar ID'lar bilan belgilangan (`v1`, `v2`, `v3` va h.k.) va bloklar `entry`, `b1`, `b2` kabi nomlangan. Funksiya `entry` blokdan boshlanadi — `a > 10` tekshiradi, true bo'lsa `b1` ga, aks holda `b2` ga o'tadi. **Yashirin scope yoki implicit flow yo'q**.

## Step 3: Phi funksiyasi — boshqaruv oqimi birlashish nuqtasi

SSA ko'pchilik holatlar uchun yaxshi ishlaydi, lekin klassik muammo bor: turli boshqaruv oqimi yo'llari bo'ylab tayinlanishi mumkin bo'lgan o'zgaruvchilarni qayta ishlash:

| Go                                     | Pseudo SSA                                  |
| -------------------------------------- | ------------------------------------------- |
| `x = 1; if condition { x = 2 }; y = x` | `x1 = 1; if condition { x2 = 2 }; y1 = ???` |
|                                        |                                             |

`y` `x` ning qiymatini oladi, lekin bu nuqtada `x` ning **qaysi versiyasi** ishlatilishi kerakligi noaniq. 
SSA har bir qiymat faqat bir marta tayinlanishini talab qiladi — bir o'zgaruvchida olingan yo'lga qarab turli qiymatlar bo'lishi mumkin bo'lgan holatni qanday hal qiladi?

Bu yerda **phi operatsiyasi** (φ-node) ishga tushadi — dastur orqali olingan yo'lga qarab to'g'ri qiymatni **tanlaydi**:

```
v6 = Const64 <int> [1]        // x1
v7 = Const64 <int> [2]        // x2
v8 = Phi <int> v6 v7          // y1 — boshqaruv yo'liga qarab tanlanadi
```

> **Phi node** — SSA'ning «ko'p ta'riflar» muammosini yechish usuli. U boshqaruv oqimi bo'linib, keyin birlashganda ham har bir qiymatni faqat bir marta tayinlash qoidasini saqlash imkonini beradi.

## Step 4: Memory — SSA da xotira modeli

Ko'pchilik hisob-kitoblar **register**'larda bo'ladi — processor to'g'ridan-to'g'ri kirishi mumkin bo'lgan juda tez saqlash joylari. Lekin o'zgaruvchilar, massivlar va struct'lar **memory**'da uzoq muddatli saqlanadi.

```go
a = b + c
```

SSA shakli:

```
v1 = load <int> {b}       // b ni memory'dan register'ga yuklash
v2 = load <int> {c}       // c ni memory'dan register'ga yuklash
v3 = v1 + v2              // qo'shish
store <int> {a} v3         // natijani a ning memory joylashuviga saqlash
```

### Memory state token'lari — tartibni ta'minlash

Compiler ko'pincha ishlashni yaxshilash uchun instruktsiyalarni **qayta tartiblaydi**. Vaqtinchalik qiymatlar bilan bu xavfsiz. Lekin memory operatsiyalari uchun **tartib muhim** — dastur `x` ga yozib, keyin `x` dan o'qisa, yozilgan qiymatni olishni kutadi.

> **Memory state token** — dastur bajarilishining ma'lum nuqtasidagi barcha memory holatini ifodalovchi abstrakt qiymat. Bu haqiqiy memory joylashuvi yoki data emas — memory operatsiyalari to'g'ri tartibda bajarilishini ta'minlovchi compiler buxgalteriya mexanizmi.

```
m0 = InitMem                      // boshlang'ich memory holati
v1 = Const64 <int> [10]
m1 = Store <mem> {*p} v1 m0       // *p ga saqlash, m0 → m1
v2 = Load <int> {*q} m1           // *q dan yuklash, m1 kerak
v3 = Mul64 <int> v2 2
m2 = Store <mem> {*p} v3 m1       // *p ga saqlash, m1 → m2
```

Bu zanjir compiler'ning memory operatsiyalarini dastur to'g'riligini buzadigan tarzda **qayta tartiblamasligini** ta'minlaydi.

Uchta asosiy memory operatsiyasi:

|Operatsiya|Vazifasi|Memory state|
|---|---|---|
|**Load**|Memory'dan qiymat yuklash, o'zgartirmaydi|Input oladi, yangi **hosil qilmaydi**|
|**Store**|Memory'ga qiymat yozish|Input oladi, yangi **hosil qiladi**|
|**Function call**|Memory'ni o'qishi va o'zgartirishi mumkin|Input oladi, yangi **hosil qiladi**|

**Nima uchun bunday qilingan:** Pointer aliasing muammosi — compiler `p` va `q` bir xil memory joylashuviga ko'rsatmasligini **isbotlay olmasa**, memory operatsiyalar tartibini o'zgartirish **xavfli**. Memory state token'lar bu tartibni **majburiy** saqlaydi.

## Step 5: Block va Function — SSA tuzilmasi

Go compiler backend'ida uchta asosiy SSA komponenti bor: **function** (`ssa.Func`), **block** (`ssa.Block`) va **value** (`ssa.Value`).

- **Function** — bitta Go funksiyasini ifodalaydi. Barcha basic block'larni va muhim metadata'ni saqlaydi.
- **Block** — **basic block** (asosiy blok) — ichki boshqaruv oqimi bo'lmagan to'g'ri chiziqli instruktsiyalar ketma-ketligi. Aniq bitta kirish va bitta chiqish nuqtasiga ega. Bloklar predecessor va successor munosabatlari orqali bog'lanadi.
- **Value** — hisoblashning asosiy birligi. Bir marta aniqlangandan keyin **hech qachon o'zgarmaydi**.

Block turlari:

|Block turi|Vazifasi|
|---|---|
|**BlockPlain**|Bitta successor, maxsus boshqaruv oqimi yo'q — to'g'ri chiziqli kod|
|**BlockIf**|Ikkita successor: true yo'li va false yo'li. Boolean shart qiymatiga asoslanadi|
|**BlockDefer**|Go'ning defer mexanizmi bilan bog'liq mantiqni boshqaradi|
|**BlockRet**|Funksiya caller'ga qaytadigan nuqta|
|**BlockRetJmp**|Tail call optimizatsiyasini qo'llab-quvvatlaydi|
|**BlockExit**|Panic yoki tiklash imkonsiz xato tufayli funksiyadan chiqish|
|**BlockJumpTable**|Switch statement'larni amalga oshirish uchun jump table'lar|

Munosabatlar:

- Function → Block'lar: funksiya bir yoki ko'p block'dan iborat
- Block → Value'lar: har bir block hisoblashlarni ifodalovchi value'larni o'z ichiga oladi
- Value → Block: har bir value **aniq bitta** block'ga tegishli
- Block → Block: predecessor va successor qirralari orqali bog'langan

## Step 6: IR Node'lardan SSA qurish

`ssagen.buildssa` funksiyasi IR funksiya node'ni SSA funksiyaga aylantiradi.

Boshlang'ich sozlash:

```go
func buildssa(fn *ir.Func, worker int, isPgoHot bool) *ssa.Func {
    ...
    s.f.Entry = s.f.NewBlock(ssa.BlockPlain)  // entry blok
    s.sp = s.entryNewValue0(ssa.OpSP, ...)    // virtual stack pointer
    s.sb = s.entryNewValue0(ssa.OpSB, ...)    // static base pointer
    s.startmem = s.entryNewValue0(ssa.OpInitMem, types.TypeMem) // boshlang'ich memory
}
```

Bu uchta muhim narsani sozlaydi:

- **SP (Stack Pointer)** — joriy bajarilish frame'iga ko'rsatadi
- **SB (Static Base)** — global o'zgaruvchilar va konstantalarga kirish uchun reference nuqtasi
- **InitMem** — funksiyaning boshlang'ich memory holati

> **ABI (Application Binary Interface)** — funksiyalar binariy darajada qanday o'zaro ta'sir qilishini aniqlaydigan qoidalar to'plami: caller argumentlarni qanday uzatadi, callee qiymatlarni qanday qaytaradi. Go 1.17 dan boshlab ishlashni yaxshilash uchun **register-based ABI** kiritilgan.

Keyin compiler har bir parametr uchun: «Bu to'g'ridan-to'g'ri SSA qiymati sifatida ishlatilishi mumkinmi?» deb so'raydi:

- Oddiy va lokal bo'lsa (`int`, `float64`, pointer'lar; 4 yoki kamroq pointer-hajmli qiymat; adresi olinmagan; heap'ga escape qilmagan) → **SSA-eligible** — register'da saqlash mumkin
- Aks holda → **memory** orqali

## Step 7: SSA Pass'lar — optimizatsiya pipeline'i

Compiler dastlab IR'ni SSA'ga aylantirganda, asl source code'ning barcha xavfsizlik kafolatlari va xatti-harakatlarini saqlaydigan **konservativ** ifodalash yaratadi. Keyin **50 dan ortiq** turli pass'lar orqali optimizatsiya qilinadi:

```go
var passes = [...]pass{
    {name: "opt", fn: opt, required: true},
    {name: "sccp", fn: sccp},
    {name: "writebarrier", fn: writebarrier, required: true},
    {name: "lower", fn: lower, required: true},
    {name: "cse", fn: cse},
    {name: "schedule", fn: schedule, required: true},
    {name: "regalloc", fn: regalloc, required: true},
    // ...
}
```

Pass'lar **to'rtta asosiy fazaga** bo'linadi: **Optimization**, **Lowering**, **Layout** va **Register Allocation**.

### 7.1. Optimization fazasi — Rewrite Rules

> **Rewrite rules** — SSA kodidagi ma'lum pattern'larni aniqlash va ularni tezroq yoki soddaroq alternativalar bilan almashtirish uchun ishlatiladigan qoidalar. Ular `cmd/compile/internal/ssa/_gen` katalogidagi maxsus sintaksisda aniqlangan.

Oddiy misol — 1 ga ko'paytirish:

```
(Mul(8|16|32|64) (Const(8|16|32|64) [1]) x) => x
```

Bu qoida: har qanday 1 ga ko'paytirish **xavfsiz olib tashlanishi** mumkin. `y := x * 1` → `y := x`.

-1 ga ko'paytirish → negation:

```
(Mul(8|16|32|64) (Const(8|16|32|64) [-1]) x) => (Neg(8|16|32|64) x)
```

`return x * -1` → `return -x`. Ko'paytirish **kommutativ** bo'lgani uchun `-1 * x` va `x * -1` ikkalasi ham ishlaydi.

Ikki darajali ko'paytirish → bit shift:

```
(Mul64 <t> n (Const64 [c])) && isPowerOfTwo64(c)
=> (Lsh64x64 <t> n (Const64 <typ.UInt64> [log64(c)]))
```

`b * 8` → `b << 3`. Ikki darajali songa ko'paytirish — shu sonning logarifmiga **chapga siljitish** bilan ekvivalent.

#### Constant folding:

```
(Add8  (Const8  [c]) (Const8  [d])) => (Const8  [c+d])
(Mul32 (Const32 [c]) (Const32 [d])) => (Const32 [c*d])
(Sub64 (Const64 [c]) (Const64 [d])) => (Const64 [c-d])
```

Agar ikkala operand ham konstanta bo'lsa — butun operatsiyani **bitta konstanta natija** bilan almashtirish.

**Nima uchun rule fayllari ishlatiladi (Go kodi o'rniga):** Pattern-matching transformatsiyalari orqali optimizatsiyalarni **aniq aniqlash** imkonini beradi — mantiqni Go'da qo'lda amalga oshirish shart emas.

### 7.2. Deadcode Elimination (SSA darajasida)

Deadcode elimination dastur xatti-harakatiga **ta'sir qilmaydigan** kodni olib tashlaydi. Hozirgi Go compiler'da bu optimizatsiya kompilatsiya davomida **kamida 8 ta turli joyda** bajariladi:

go

```go
{name: "early deadcode", fn: deadcode},
{name: "pre-opt deadcode", fn: deadcode},
{name: "opt deadcode", fn: deadcode, required: true},
// ... jami 7+
```

Bu **ataylab** takrorlanadi — ko'p optimizatsiya pass'lari dead code elimination uchun **yangi imkoniyatlar** yaratishi mumkin.

Misol: constant propagation ikki qoida orqali shart bosqichma-bosqich soddalashtiriladi:

|Qadam|Oldin|Keyin|
|---|---|---|
|Constant folding (addition)|`if x + y > 100` (x=10, y=20)|`if 30 > 100`|
|Constant folding (comparison)|`if 30 > 100`|`if false`|
|Deadcode elimination|`if false { ... }`|_(butunlay olib tashlanadi)_|

**IR darajasidagi early deadcode elimination nima uchun buni qilmadi?** IR darajasida compiler faqat cheklangan statik tahlil bajaradi — o'zgaruvchi qiymatlarini tahlil **qilmaydi**. SSA esa barcha boshqaruv yo'llariga **to'liq ko'rinish** beradi va qiymatlar dastur bo'ylab qanday oqishini kuzatadi.

### 7.3. Common Subexpression Elimination (CSE)

> **CSE** — takrorlangan hisob-kitoblarni aniqlash va ortiqcha narsalarni olib tashlash. Agar `E` expression bir marta hisoblanib, uning operand'lari qayta paydo bo'lganda o'zgarmagan bo'lsa — oldin hisoblangan qiymatni **qayta ishlatish** mumkin.

|Oldin|Keyin|
|---|---|
|`x := a + b; if condition { return x }; return a + b + c`|`x := a + b; if condition { return x }; return x + c`|

### 7.4. Bounds Check Elimination

> **Bounds check elimination** — Go'da har bir index kirish avtomatik bounds check bilan himoyalangan. Agar compiler `len(a)` har doim index'dan katta ekanini **isbotlasa** — tekshiruvni xavfsiz olib tashlashi mumkin.

go

```go
// Bounds check'siz — compiler isbotlashi mumkin
func WithoutBoundsChecks(a []int, b []int) int {
    if len(a) != len(b) { panic("lengths must be equal") }
    sum := 0
    for i := range a {
        sum += a[i] + b[i]  // bounds check kerak EMAS
    }
    return sum
}
```

Compiler quyidagi faktlarni bilib oladi:

1. `len(a) == len(b)` (aniq tekshiruv)
2. `i` doimo `a` uchun valid index (`range` kafolati)
3. Demak `i` `b` uchun ham valid → **bounds check kerak emas**

Benchmark: bounds check'siz versiya o'rtacha **~7% tezroq** (`len(a) == len(b) == 1_000_000` bilan).

bash

```bash
go build -gcflags=-d=ssa/check_bce/debug  # bounds check'larni ko'rish
```

### 7.5. Nilcheck Elimination

Bounds check elimination'ga o'xshash, lekin **pointer dereference**'larga qo'llaniladi. Compiler har bir pointer dereference'dan oldin nil check qo'shadi. Agar pointer **hech qachon nil emasligini** isbotlasa — tekshiruvni olib tashlashi mumkin.

Ko'pchilik zamonaviy hardware'da nil pointer kirish address 0 da memory'ni o'qishga urinishda **hardware fault** hosil qiladi — Go runtime buni ushlaydi va panic'ka aylantiradi.

## Step 8: Lower Pass'lar — arxitekturaga moslashtirish

> **Lowering** — umumiy SSA operatsiyalarini **arxitekturaga xos** operatsiyalarga aylantirish jarayoni. `Add32`, `Load`, `Store` kabi umumiy operatsiyalar hech qanday CPU instruktsiyalar to'plamiga bog'liq emas — **portativ IR**.

Compiler yana rewrite rules ishlatadi, lekin arxitekturaga xos fayllarda (`ARM64.rules`, `AMD64.rules` va h.k.):

```
(Add64 x y) => (ADD x y)          // ARM64 ADD instruktsiyasi
(Mul64 x y) => (MUL x y)          // ARM64 MUL instruktsiyasi
(Add64 x (Const64 [5000])) => (ADDconst [5000] x)  // immediate konstanta
```

`ADDconst` — oraliq qadam. Keyingi lowering pass konstanta to'g'ridan-to'g'ri immediate sifatida ishlatilishi yoki register'ga yuklanishi kerakligini **hal qiladi** (ARM64 ning immediate qiymatlari uchun maxsus qoidalari bor).

**Nima uchun bunday qilingan:** Umumiy IR turli CPU'larda **bir xil** ishlashi kerak — faqat lowering bosqichida aniq hardware farqlari hisobga olinadi.

## Step 9: Layout va Schedule Pass'lar

### Layout — bloklar tartibi

> **Layout pass** — assembly generatsiyasi davomida funksiyaning basic block'lari memory'da qanday **tartibda** joylashtirilishini aniqlaydi.

**Nima uchun tartib muhim:** CPU instruktsiyalarni **ketma-ket** bajaradi. Agar mantiqiy jihatdan ketma-ket keladigan ikkita blok memory'da **yonma-yon** joylashtirilsa, CPU o'tish (jump) qilmasdan davom etishi mumkin — bu bajarilishni tezlashtiradi.

```go
func f(x int) int {
    if x != 0 { return x }
    panic("BUG: x is 0")
}
```

Compiler panic yo'li **kamdan-kam** olinishini biladi. Agar panic bloki entry blokdan keyin joylashtirilsa — CPU **har safar** return blokga o'tish (jump) qilishi kerak. Buning o'rniga layout pass bloklarni qayta tartiblab, **ehtimoliy yo'llar** memory'da ketma-ket joylashadi:

```
TEXT main.f(SB), ABIInternal
CBZ R0, 8          // R0 == 0 bo'lsa → panic'ga o'tish
RET                // oddiy return — jump KERAK EMAS
MOVD $type:string(SB), R0   // panic kodi — funksiya OXIRIDA
CALL runtime.gopanic(SB)
```

> **Branch probability** — compiler qaysi yo'llar ko'proq olinishini bilishi kerak. Ikki usul: **taxminlar** (standart pattern'larga asoslangan) va **PGO** (haqiqiy profiling data). Layout pass taxminlardan foydalanadi.

Kamroq ehtimoliy branch'lar pattern'lari:

- **Panic/exit yo'llari** — doimo kam ehtimoliy
- **Loop'dan chiqish** (`break`) — odatda kam ehtimoliy
- **Branch ichidagi funksiya chaqiruvlari** — chaqiruvli branch kamroq ehtimoliy
- **Error handling** — error return yo'li kam ehtimoliy
- **Defer ishlatish** — defer li branch'lar kamroq ehtimoliy

> **CPU optimizatsiyasi tushuntirish:** Zamonaviy CPU'lar I-cache (instruktsiya keshi) va hardware prefetching tufayli ketma-ket instruktsiya bajarishga optimallashtirilgan. Jump — I-cache'ni buzadi va prefetcher'ni to'xtatadi.

### Schedule — instruktsiyalar tartibi

Layout tugagach, **schedule pass** ishga tushadi — har bir blok **ichidagi** instruktsiyalar (value'lar) tartibini belgilaydi. Maqsad — boshqaruv va data flow uchun muhimroq operatsiyalarni **ustuvorlashtirish**.

Scheduler baholash tizimi (eng muhimdan kamroq muhimga):

|Score|Nima|Sabab|
|---|---|---|
|`ScorePhi`|Phi node'lar|Blok boshida bo'lishi kerak|
|`ScoreArg`|Argumentlar|Entry blok boshida — debug uchun|
|`ScoreNilCheck`|Nil tekshiruvlar|Erta bajarish — xatolarni tez aniqlash|
|`ScoreMemory`|Store operatsiyalar|Erta bajarish — register bosimini kamaytirish|
|`ScoreDefault`|Standart|Ko'pchilik operatsiyalar|
|`ScoreControl`|Boshqaruv|Blok oxirida — branch/return|

## Step 10: Register Allocation — virtual'dan fizikga

> **Register allocation** — dastur o'zgaruvchilarini cheklangan sonli fizik CPU register'lariga tayinlash. SSA shaklida compiler **cheksiz** virtual register'lar bilan ishlaydi. Register allocator bu virtual register'larni target arxitektura qo'llab-quvvatlaydigan **sobit to'plam**ga xaritalash uchun javobgar.

### ARM64 register'lar

ARM64 da Go compiler jami **64 ta register** bilan ishlaydi:

- **32 ta umumiy maqsadli integer register** (R0-R31): R0-R30 — integer qiymatlar va adreslar; R31 — stack pointer (SP)
- **32 ta floating-point register** (F0-F31): IEEE 754 arifmetikasi uchun

Ajratish uchun mavjud **bo'lmagan** register'lar:

- R18 — platform registri (reserved)
- R27 (REGTMP) — assembler vaqtinchalik registri
- R29 — frame pointer
- SP, SB, g — stack pointer, static base, goroutine holati

### Linear scan algoritmi

Go compiler **linear scan** algoritmidan foydalanadi — dastur bo'ylab yuradi va har bir qiymatni qaerda saqlash haqida **qadam-baqadam** qaror qabul qiladi:

1. Allocator har bir instruktsiyani ketma-ket ko'rib chiqadi. Har bir nuqtada qaysi register'lar **bo'sh** va qaysilari **band** ekanini biladi.
2. Har bir instruktsiya uchun — avval input qiymatlar, keyin output qiymat:
    - Qiymat allaqachon register'da bo'lsa — shu register ishlatiladi
    - Aks holda — bo'sh, mos register topib tayinlaydi (**eng kichik raqamli** register afzal)
    - Register bo'sh bo'lmasa — yaqin orada **ishlatilmaydigan** qiymatni saqlaydi (spilling) va register'ni bo'shatadi
3. Instruktsiya qayta ishlangach — ichki holatni yangilaydi

### Bosqichma-bosqich misol

go

```go
func example(a int, b int) int {
    var y int
    if b > 0 { y = a * 2 } else { y = b * 2 }
    return y + b
}
```

**Entry blok (b1):** ABI qoidalariga ko'ra `a` → R0, `b` → R1. Bu allocator'ning **tanlovi emas** — ABI talab qiladi.

> **ABI talab:** Argumentlar va return qiymatlari qaysi register'larga joylashtirilishi ABI tomonidan **majburiy** belgilanadi. Allocator bunda moslashuvchanlikka ega emas.

**b3 blok (b > 0):** `v19 = a * 2`. R0 ABI tufayli return qiymati uchun **zaxiralangan**, R1 `v8` (b) ni saqlaydi va keyinchalik kerak. Shuning uchun `v19` → **R2** tayinlanadi.

**b4 blok (else):** Allocator b2 dagi Phi operatsiyasiga **oldinga qaraydi** — `v19` allaqachon R2 da, shuning uchun `v5` ham **R2** ga tayinlanadi — Phi node uchun izchillik.

**b2 blok (merge):** Phi node natijasi R2 da qoladi (ikkala kiruvchi qiymat allaqachon shu register'da). Yakuniy return qiymati ABI talabiga ko'ra **R0** ga ko'chiriladi.

### Qo'shimcha bosqichlar

- **Spill placement:** Ajratish davomida spilling uchun belgilangan qiymatlar uchun `StoreReg` instruktsiyasi **qo'shiladi**
- **Stack allocation:** Register'larda saqlanmagan barcha SSA qiymatlarga **stack slot'lar** tayinlaydi
- **Shuffle fazasi:** Merge nuqtalarida register nomuvofiqliklarini to'g'rilaydi — masalan Phi R2 kutsa, lekin qiymat R3 da bo'lsa → `MOV R3 → R2`

### Critical Edge Splitting

> **Critical edge** — bir nechta successor'li blokdan (bo'linish nuqtasi) bir nechta predecessor'li blokga (birlashish nuqtasi) boradigan qirra. Bu register allocation uchun muammoli.

Misol: A → D qirrasi critical — A'da bir nechta successor (C, D), D da bir nechta predecessor (A, B).

**Muammo:** D da Phi funksiyasi bor. U A dan kelganda R1 da bitta qiymat, B dan kelganda R1 da boshqa qiymat kutadi. Register to'g'rilash kodini **qaerga** joylashtiramiz?

- A oxiriga qo'yib bo'lmaydi — A → C yo'liga ta'sir qiladi
- D boshiga qo'yib bo'lmaydi — B → D yo'liga ta'sir qiladi

**Yechim:** Critical edge'ni **yangi blok** qo'shib sindirish — «pad block». Bu yangi blok register to'g'rilash xavfsiz bo'ladigan **ixtisoslashtirilgan joy** yaratadi.
