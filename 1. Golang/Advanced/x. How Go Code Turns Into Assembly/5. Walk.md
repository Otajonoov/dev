
Walk fazasi — bu dasturimizning IR node'lari ustidagi **yakuniy o'tish**.

> **Walk fazasi** — yuqori darajali Go konstruktsiyalarini pastroq darajali operatsiyalarga aylantiradigan compiler bosqichi.

Oddiy misol — for-range tsikl qanday o'zgartiriladi:

```go
for v1 := range a { 
	// body
}

// Pseudo-Go
hv1 := 0; 
hn := len(a); 
for hv1 < hn; hv1 = hv1 + 1 { 
	v1 = hv1;
	// body 
}

```

Bu bosqich ikki asosiy qismdan iborat: 
1. **Ordering** (tartibga solish)
2. **Transformation** (o'zgartirish, walk deb ham ataladi).

---

## Step 1: Order fazasi — amallar tartibini ta'minlash

> **Order fazasi** — kodingizdagi hamma narsa to'g'ri ketma-ketlikda bajarilishini ta'minlaydigan bosqich. Uning asosiy vazifasi — statement'larni ko'rib chiqib, **side effect** (yon ta'sir) li qismlarni topish va ularni alohida statement'larga ko'tarish.

> **Side effect** — o'zgaruvchiga qiymat berish, funksiya chaqiruvi, memory'ga yozish kabi — boshqa narsalarga ta'sir qiladigan operatsiyalar.

Misol:

```go
x := f(g())
```

Uchta qismni ko'rib chiqamiz:

- **Assignment node (`:=`)** — tayinlash va e'lon qilishni bajaradi. `x` ni e'lon qilish initsializatsiyani o'z ichiga oladi — bu side effect.
- **Chap tomon (`x`)** — tayinlanadigan o'zgaruvchi. Oddiy nom — side effect yo'q.
- **O'ng tomon (`f(g())`)** — `f()` chaqiruvi `g()` argumenti bilan. `g()` funksiya chaqiruvi bo'lgani uchun side effect bo'lishi mumkin. `f()` `g()` bajarilguncha **kutishi** kerak.

Order fazasi side effect'larni alohida statement'larga ko'taradi va oraliq natijalar uchun **vaqtinchalik o'zgaruvchilar** ishlatadi:

```go
var x string
tmp := g()
x = f(tmp)
```

**Nima uchun bunday qilingan:** Baholash tartibini to'g'ri ta'minlash va oraliq qiymatlarni keyingi kompilatsiya bosqichlari uchun mavjud qilish.

### 1.1. For-Range Order — array nusxalash himoyasi

```go
func main() {
    arr := [3]int{1, 2, 3}
    for i, v := range arr {
        if i == 1 {
            arr[2] = 99
        }
        println(v)
    }
}
```

Ko'pchilik `1 2 99` deb kutishi mumkin, lekin haqiqiy natija — **`1 2 3`**. Nima uchun?

Compiler `n.X = o.copyExpr(n.X)` orqali **vaqtinchalik o'zgaruvchi** yaratadi — asl data'ning snapshot'ini saqlaydi:

| Go                                  | Pseudo-Go                                         |
| ----------------------------------- | ------------------------------------------------- |
| `for i, v := range arr { // body }` | `temp := arr; for i, v := range temp { // body }` |
|                                     |                                                   |

`v` **nusxadan** o'qiydi, asl `arr` dan emas. Shuning uchun `arr[2] = 99` o'zgarishi `v` ga ta'sir qilmaydi.

**Lekin slice uchun boshqacha:**

```go
slice := []int{1, 2, 3}
for i, v := range slice {
    if i == 1 { slice[2] = 99 }
    println(v)
}
// Output: 1 2 99
```

Chunki slice nusxalanganida faqat **header** nusxalanadi (pointer, length, capacity) — asosiy massiv **bir xil**. Shuning uchun o'zgarishlar ko'rinadi.

> **Muhim farq:** Array nusxalash — butun data nusxalanadi. Slice nusxalash — faqat header nusxalanadi, asosiy data ulashiladi.

Shuning uchun Go `*[N]T` (array'ga pointer) ustida `range` qilishga **ruxsat beradi**, lekin `*[]T` (slice'ga pointer) ustida **ruxsat bermaydi** — katta array nusxalashning oldini olish uchun.

### 1.2. Zero-copy optimization — `[]byte(str)` uchun

```go
for i, v := range []byte(str) {
    // loop body
}
```

Odatda `[]byte(str)` konversiyasi string'ning asosiy data'sining **yangi nusxasini** yaratadi (chunki string immutable, byte slice mutable). Lekin agar nusxa **hech qachon o'zgartirilmasa**, nusxalash **isrofgarchilik**.

Go compiler bu holatni aniqlaydi va `OSTR2BYTESTMP` operatsiya kodi bilan belgilaydi — yangi slice ajratish o'rniga asl data'dan foydalanish mumkin (**zero-copy**).

> **Zero-copy** — data'ni nusxalamasdan to'g'ridan-to'g'ri asl joyidan o'qish optimizatsiyasi.

Tekshirish:

```bash
$ go build -gcflags="-m" .
./main.go:4:27: zero-copy string->[]byte conversion
```

**Nima uchun bunday qilingan:** Keraksiz memory allocation va nusxalashni oldini olish — ishlashni yaxshilash.

## Step 2: Walk(Transformation) fazasi — yuqori darajali konstruktsiyalarni pasaytirish

Walk fazasi order fazasidan soddalashtrilgan va tartiblangan IR node'larni olib, SSA generatsiyasi uchun tayyorlash maqsadida **transformatsiya** qiladi. Uning asosiy roli — yuqori darajali Go konstruktsiyalarini SSA'ning qat'iy talablariga mos keladigan **soddaroq shakllar**ga pasaytirish.

> **Lowering** (pasaytirish) — yuqori darajali, abstrakt konstruktsiyalarni pastroq darajali, aniqroq operatsiyalarga aylantirish.

**Nima uchun kerak:** Go'ning yuqori darajali xususiyatlari (channel, interface, defer) ko'pincha SSA ifodalovchi pastroq darajali mashina modeliga to'g'ridan-to'g'ri xaritalanadigan darajada **juda abstrakt**. Walk fazasi bu muhim tarjima qatlami vazifasini bajaradi.

Walk fazasi qiladigan ba'zi misollar:

- Channel send operatsiyalari → `runtime.chansend1` runtime chaqiruvlariga
- `defer` statement'lar → ichki buxgalteriya mantiqiga
- Interface method chaqiruvlari → aniq dispatch kodiga
- Composite literal'lar (struct yoki array initsializatsiyalari) → oddiy assignment statement'larga

---
### 2.1. For-Range Lowering

```go
func walkRange(nrange *ir.RangeStmt) ir.Node {
    base.Assert(!nrange.DistinctVars)
    ...
    nfor := ir.NewForStmt(nrange.Pos(), nil, nil, nil, nil, nrange.DistinctVars)
    nfor.SetInit(nrange.Init())
    nfor.Label = nrange.Label
}
```

Yuqori darajali range tsikl **oddiy for tsikl** tuzilmasiga pasaytiriladi. Bu transformatsiya qanday ma'lumot turi ustida iteratsiya qilinishidan **qat'i nazar** sodir bo'ladi.

---
#### Integer ustida range:

```go
// Go code 
for i := range n { // Body }

// Preudo-Go
hv1 := 0;
hn := n;
for ; hv1 < hn; hv1++ { i = hv1; // Body }
```

---
#### Slice ustida range:

```go
// Go code
for i, v := range slice {
	println(i, v) 
}

// Pseudo-Go
sliceTmp := slice;
hv1 := 0;
hn := len(sliceTmp);

for hv1 < hn {
	tmp := sliceTmp[hv1];
	i, v := hv1, tmp;
	println(i, v); 
	hv1++
}
```

**Bu savolga javob beradi:** nima uchun for-range tsikl ichida slice'ga `append` qilish cheksiz tsiklga **olib kelmaydi**?

Pastlatilgan kodda tsikl slice'ning **asl uzunligidan** foydalanadi — u tsikl **boshlanishidan oldin** aniqlanadi. Shuning uchun yangi elementlar qo'shish iteratsiyalar soniga **ta'sir qilmaydi**.

---
#### Map ustida range:

```go
// Go code
for k, v := range myMap {
	// body
}

// Pseudo code
ha := myMap;
it := new(hiter);
runtime.mapiterinit(typeOf(map[string]int), ha, &it);
for it.key != nil {
	k := *(*string)(it.key);
	v := *(*int)(it.elem);
	// body; 
	runtime.mapiternext(&it)
}
```


> **hiter** — map iteratsiyasi uchun runtime'ning ichki iterator tuzilmasi. 
> `runtime.mapiterinit` uni initsializatsiya qiladi, `runtime.mapiternext` keyingi elementga o'tkazadi.

---
#### String ustida range:

String iteratsiyasi **UTF-8 dekodlashni** aniq boshqaradi — bayt va rune pozitsiyalarini qo'lda boshqaradi:

```go
s := "Hello, 世界"
sCopy := s
hv1 := 0       // Joriy bayt pozitsiyasi
hv1t := 0      // Joriy rune boshlanish pozitsiyasi (i)
hv2 := rune(0) // Joriy rune qiymati (r)

for hv1 < len(sCopy) {
    hv1t = hv1
    hv2 = rune(sCopy[hv1])
    if hv2 < utf8.RuneSelf {  // ASCII bo'lsa (RuneSelf = 128)
        hv1++                  // bitta bayt oldinga
    } else {
        hv2, hv1 = runtime.decoderune(sCopy, hv1)  // multibyte rune dekodlash
    }
    i, r = hv1t, hv2
    // ... body
}
```

---
#### Switch pasaytirish:

```go
switch x {
case 1, 2: println("one or two")
case 3:    println("three"); fallthrough
case 4:    println("three or four")
default:   println("something else")
}
```

Compiler buni **if statement'lar va labeled jump'lar** ketma-ketligiga pasaytiradi:

```
.autotmp_1 := x
goto .switch_dispatch

.s0: println("one or two"); goto .switch_end
.s1: println("three")  // fallthrough — goto yo'q, keyingisiga davom
.s2: println("three or four"); goto .switch_end

.switch_dispatch:
    if .autotmp_1 >= 1 && .autotmp_1 <= 2 { goto .s0 }
    if .autotmp_1 == 3 { goto .s1 }
    if .autotmp_1 == 4 { goto .s2 }
    goto .switch_default

.switch_default: println("something else"); goto .switch_end
.switch_end:
```

### 2.2. Loop-var muammosi va `DistinctVars`

> **Loop-var muammosi** — Go'da tsikl o'zgaruvchilari har bir iteratsiyada **qayta yaratilmasligi** sababli yuzaga keladigan kutilmagan xatti-harakatni ifodalaydi. Ular bir xil memory joylashuvini qayta ishlatadi — bu closure yoki goroutine'larda muammoga olib keladi.

```go
funcs := make([]func(), 3)
for i := 0; i < 3; i++ {
    funcs[i] = func() { fmt.Println(i) }
}
for _, f := range funcs { f() }
// Output: 3 3 3 (kutilgan: 0 1 2)
```

`i` barcha iteratsiyalarda **qayta ishlatiladi**. Har bir closure **bir xil** memory joylashuviga havola qiladi. Closure'lar bajarilganida tsikl allaqachon tugagan va `i` yakuniy qiymatga ega.

**Yechim:** Go 1.22 da `DistinctVars` flag kiritildi — compiler har bir iteratsiya uchun **yangi o'zgaruvchi** yaratadi:

|For|For-Range|
|---|---|
|`for i := 0; i < len(a); i++ { i2 := i; // ... i2 ishlat }`|`for k, v := range a { k2 := k; v2 := v; // ... k2, v2 ishlat }`|

> **Muhim:** Loop-var muammosi walk fazasida emas, **IR optimizatsiya fazasida** (inlining va devirtualization'dan keyin) hal qilinadi. Walk fazasiga kelguncha `DistinctVars` flag allaqachon `false` bo'lishi kerak.
