
# Go Compilation Jarayoni

Go — **kompilyatsiya qilinadigan til**. Bu shuni anglatadiki, siz yozgan kod bajarilishdan oldin mashina kodiga tarjima qilinadi. 
Go compiler, rasmiy Go toolchain'ning bir qismi bo'lib, bu tarjimani amalga oshiradi. 
U Go source fayllarni (.go) olib, binariy bajariladigan fayllarga kompilatsiya qiladi. 
Go shuningdek **cross-compilation**ni qo'llab-quvvatlaydi — o'z mashinangizdan turib boshqa OS va arxitekturalar uchun kompilatsiya qilishingiz mumkin.

---

## Step 1: `$WORK/b...` — Vaqtinchalik build katalogi

Go build jarayonida birinchi navbatda `$WORK/b...` nomli **vaqtinchalik katalog** yaratadi.

Bu katalog ichida tizim har bir **action** uchun nomerlangan papkalar yaratadi (`b001/`, `b002/`, ...).

> **Action** — bajarilishi kerak bo'lgan har qanday ish birligi. Masalan: bitta package'ni kompilatsiya qilish, yoki link qilish.

**Nima uchun bunday qilingan:** Oraliq natijalarni (object fayllar, export data) izolyatsiyalangan muhitda saqlash uchun. Build tugagach katalog o'chiriladi.

**Vazifasi:** Har bir action o'z papkasida mustaqil ishlaydi — parallel build va caching imkonini beradi.

---

## Step 2: `importcfg` — Dependency xaritasi

Kompilatorga kodingiz import qilgan package'larni **qayerdan topishni** ko'rsatadigan xarita.

Boshqacha qilib aytganda: `B` package'ini import qiladigan `A` package'ini kompilatsiya qilishda, kompilator export qilinadigan type, func, const larga kirish uchun `B` package'ining kompilatsiya qilingan versiyasini **qayerdan topishni** bilishi kerak.

> **importcfg** — oddiy matn fayli, har bir qatorda `packagefile paket_nomi=fayl_yoli` formatida yozilgan. Kompilatorga «qaysi package qayerda» degan ma'lumotni beradi.

**Nima uchun bunday qilingan:** Go'da package'lar bir-biriga bog'liq. Kompilator `import "fmt"` ko'rganida, `fmt` ning kompilatsiya qilingan `.a` faylini topishi kerak. `importcfg` bu manzilni aniq ko'rsatadi.

**Vazifasi:** Kompilatsiya vaqtida dependency'larni tez va aniq topish.

---

## Step 3: Import package'lari kompilatsiya qilinadi

`main` package kompilatsiya qilinishidan oldin, uning **barcha dependency'lari** avval kompilatsiya qilinadi.

> **Package** — Go'dagi kompilatsiya birligi. C/C++ da har bir `.c`/`.cpp` fayl alohida kompilatsiya qilinadi. Go'da esa **butun package** birdan kompilatsiya qilinadi.

Har bir package `.a` archive fayliga aylanadi, va bu archive ichida ikkita asosiy qism bor:

- `__.PKGDEF` — **export data**: tiplar, funksiyalar, konstantalar haqida metadata
- `go.o` — **mashina kodi**

**Nima uchun bunday qilingan:** `A` package'i `B` ni import qilsa, kompilator `B` ning tiplarini bilishi kerak. Shuning uchun avval `B` kompilatsiya qilinadi, keyin `A`.

**Vazifasi:** Dependency grafidagi eng chuqur package'lardan boshlab, **pastdan yuqoriga** kompilatsiya qilish. Har bir package faqat o'z dependency'lari tayyor bo'lgandan keyin kompilatsiya qilinadi.

## Step 4: Parsing va Syntax Tree qurilishi

Birinchi bosqichda kompilator source code'ni o'qiydi, odatda bir nechta fayllarga taqsimlangan. U **cheklangan concurrency** bilan `syntax.Parse()` funksiyasi orqali syntaktik tahlil qiladi. Bu jarayon ikkita asosiy komponentdan iborat: **scanner** va **parser**.

### 4.1. Scanner (Lexer) — tokenizatsiya

> **Scanner (lexer)** — so'rovlarni qayta ishlashda keng tarqalgan va muhim komponent, ayniqsa SQL engine'lar, compiler'lar va interpreter'larda. U xom source code'ni o'qib, kichikroq ma'noli qismlarga — **token**'larga ajratadi.

> **Token** — tilning asosiy qurilish bloklari. Ular quyidagilarni o'z ichiga oladi: kalit so'zlar (`func`, `for`, `if`), operatorlar (`+`, `-`, `<=`), ajratgichlar (`;`, `:`, `.`), nomlar va literallar.

Scanner **ahamiyatsiz** deb hisoblagan belgilarni **e'tiborsiz qoldiradi**: bo'sh joylar, tablar va yangi qator belgilari. Ularni whitespace sifatida ko'rib, token yaratmasdan o'tkazib yuboradi.

Scanner shuningdek kommentariyalarni ikki shaklda uchratadi: `//` bilan boshlanadigan qatorli kommentariyalar va `/*` `*/` orasidagi blokli kommentariyalar. Ko'pchilik kommentariyalar whitespace sifatida e'tiborsiz qoldiriladi. Lekin maxsus kategoriya bor — **directive**'lar, Go compiler uchun aniq ma'no tashiydi:

- `//go:noinline` yoki `/*go:noinline*/` — kompilatorga funksiyani inline qilmaslikni ko'rsatadi
- `//go:generate` — tashqi tool orqali kod generatsiya qilish uchun ko'rsatma beradi

**Nima uchun bunday qilingan:** Kompilator matn bilan ishlay olmaydi — unga tuzilmali ma'lumot kerak. Scanner matnni tuzilmali token oqimiga aylantiradi.

**Vazifasi:** Source code → token oqimi. Parser uchun input tayyorlash.

### 4.2. Parser — daraxt qurilishi

Source code tokenizatsiya qilingandan keyin, parser natijadagi token ketma-ketligini oladi. Uning asosiy vazifasi — **syntax tree** qurish, kodning grammatikasini aks ettiruvchi ierarxik tuzilma. Bu daraxt kodning tashkil etilishini ko'rsatadi: expression'lar statement'lar ichida qanday joylashgani va statement'lar funksiyalar ichida qanday guruhlangani.

> **Syntax tree** — kodning ierarxik xaritasi. Har bir tugun kodning bir qismini ifodalaydi.

Natijadagi syntax tree tafsilotlarni o'z ichiga oladi: function declaration (`FuncDecl`), if statement (`IfStmt`) va `fmt.Println` chaqiruvi (`CallExpr` va `SelectorExpr` orqali ifodalanadi).

Ko'p tugunlarga qaramay, ularning aksariyati **uch asosiy kategoriya**ga tushadi:

- **Expression (`syntax.Expr`)** — qiymatlar va hisob-kitoblarni ifodalaydi: nomlar, literallar, funksiya chaqiruvlari, operatorlar, type assertion, composite literal. Expression'lar dastur bajarilishida qiymat olish uchun hisoblanishi mumkin.
- **Statement (`syntax.Stmt`)** — kodning bajariladigan qismlari: tayinlashlar, tsikllar, shartlar, expression statement, return statement.
- **Declaration (`syntax.Decl`)** — o'zgaruvchilar, konstantalar, tiplar va funksiyalarni aniqlaydi. Declaration'lar — dasturga yangi nomlar kiritish usuli.

**Nima uchun bunday qilingan:** Tekis token oqimi bilan optimallashtirish va kod generatsiya qilish mumkin emas. Daraxt tuzilmasi kodning **semantik munosabatlarini** saqlaydi.

**Vazifasi:** Token oqimi → syntax tree. Keyingi bosqichlar uchun tuzilmali input.

### Syntax tree'ni dasturiy tarzda yaratish

Syntax tree'ni generatsiya qilish va tekshirish uchun dasturchilar Go standart kutubxonasining `go/*` oilasidagi paketlardan foydalanishi mumkin (`go/parser`, `go/ast`). Bu paketlar Go source code'ni tahlil qilish, tekshirish yoki manipulyatsiya qilish uchun mo'ljallangan — ko'pincha linter, formatter yoki code generator kabi toollar yaratish uchun.

Bu ommaviy paketlar compiler'ning ichki `syntax` paketidan farq qilsa-da, ular Go kodini syntax tree sifatida ifodalashning **bir xil fundamental prinsiplariga** asoslangan.

Source code'ni parse qilib syntax tree'ni chop etadigan Go dasturi:

```go
package main

import (
    "go/ast"
    "go/parser"
    "go/token"
    "log"
)

const src = `
package main

import "fmt"

func main() {
    num := 10
    if num > 5 {
        fmt.Println("The number is greater than 5")
    }
}`

func main() {
    // FileSet pozitsiya ma'lumotlarini kuzatish uchun kerak.
    fset := token.NewFileSet()

    // Source code satrini AST tuguniga parse qilish.
    fileNode, err := parser.ParseFile(fset, "", src, 0)
    if err != nil {
        log.Fatalf("Failed to parse source: %v", err)
    }

    // Generatsiya qilingan AST'ni stdout'ga chop etish.
    ast.Print(fset, fileNode)
}
```

Bu jarayondagi asosiy paketlar:

- **go/token** — leksik token'larni ifodalovchi konstantalar (kalit so'zlar, operatorlar) va source code'dagi pozitsiyalarni boshqarish uchun tiplarni aniqlaydi (`token.FileSet`, `token.Pos`).
- **go/scanner** — source matnni o'qib, `go/token` dagi ta'riflarga asosan token ketma-ketligiga aylantiradi. (Parser scanner'ni ichki ishlatadi.)
- **go/parser** — scanner generatsiya qilgan token ketma-ketligini olib, AST quradi.
- **go/ast** — Go kodining tuzilmasini AST ichida ifodalash uchun tugun tiplarini aniqlaydi (`ast.File`, `ast.FuncDecl`, `ast.IfStmt`).

## Step 5: Type Checking va Scope Resolution

> **Type checking** — kompilator barcha o'zgaruvchilar, expression'lar va funksiya chaqiruvlari Go til qoidalariga amal qilishini **tekshiradigan** bosqich. U shuningdek dasturda har bir expression va o'zgaruvchining **tipini aniqlaydi**.

Bu jarayon syntax tree qurilishidan keyin sofort bo'lgani uchun, **syntax tree type checking uchun input sifatida ishlatiladi**. Type checker daraxtni obxod qilib, qo'shimcha ma'lumot bilan **boyitadi**, xom sintaksisni yanada tuzilmali, scope-aware ko'rinishga aylantiradi.

**Nima uchun bunday qilingan:** Go — statik tipli til. Kompilator `string`'ni `int`'ga tayinlash, noto'g'ri argument berish kabi xatolarni **kompilatsiya vaqtida** aniqlashi kerak.

**Vazifasi:** Sintaktik jihatdan to'g'ri bo'lgan, lekin semantik jihatdan noto'g'ri kodlarni aniqlash va har bir ifodaning aniq tipini belgilash.

### 5.1. Global ob'yektlarni yig'ish

Keyingi qadam — **global ob'yektlarni** yig'ish: o'zgaruvchilar, konstantalar, tiplar, funksiyalar va boshqa top-level declaration'lar. Bu yerda «global» deganda funksiyadan **tashqarida** e'lon qilingan hamma narsa nazarda tutiladi.

Bu ob'yektlarni tashkil etish va boshqarish uchun kompilator **scope** (ko'rinish sohalari) ishlatadi.

> **Scope** — e'lon qilingan nomlarni kuzatuvchi konteyner. Kompilator har bir nom **qayerda** va **qanday** ishlatilganini scope orqali biladi. Uning vazifasi — nomlarning ko'rinish doirasini aniqlash va nom ziddiyatlarini oldini olish.

Ikki asosiy scope turi:

- **Package scope** — package'ning barcha fayllarini qamrab oladi. Bu yerdagi e'lonlar **butun package bo'ylab** ko'rinadi.
- **File scope** — bitta faylga xos va asosan nomlarni **shu fayl ichida** lokal hal qilish uchun ishlatiladi.

Kompilator syntax tree'ning **ikkinchi darajasini** obxod qiladi — `File` tugunining ostidagi — va barcha top-level declaration'larni yig'adi. Keyin ular tegishli scope'larga joylashtiriladi. Har bir fayl uchun kompilator faylga xos scope yaratadi. Lekin top-level declaration'lar — global konstantalar, tiplar yoki funksiyalar topilganda — ularni **package scope**'ga joylashtiradi. Shuning uchun bitta package'ning turli fayllarida ikki xil global o'zgaruvchi yoki funksiyani bir xil nomda elon qlish **mumkin emas** — ularning hammasi bitta package scope'ni baham ko'radi.

Bu bosqichning maqsadi — barcha global ob'yektlarni yig'ish.

**Nima uchun bunday qilingan:** Funksiya tanalarini tekshirishdan **oldin** barcha global nomlarning «skeleti» quriladi — chunki funksiya tanasi global e'lonlarga murojaat qilishi mumkin.

### 5.2. File scope va import konfliktlari

File scope bitta **faylga cheklangan** ob'yektlar uchun ishlatiladi. File-scope ob'yektlarga misol — **import declaration**'lar. Go faylida `fmt` kabi package import qilganingizda, u faqat **shu fayl ichida** mavjud bo'ladi.

Bu bosqichda kompilator **metodlarni** ham yig'ishni boshlaydi. Ular `check.methods` map'ida saqlanadi. Boshqa top-level declaration'lardan farqli, metodlar file yoki package scope'ga **to'g'ridan-to'g'ri qo'shilmaydi**. Buning o'rniga, ular receiver'ining **bazaviy tipi** bilan **assotsiatsiya qilinadi**.

Barcha global ob'yektlar yig'ilib, o'z scope'lariga joylashtirilgandan keyin, keyingi qadam — ularning har birini **type-check** qilish. Kompilator har bir global ob'yektni tekshirib, tipini aniqlaydi va ziddiyatlar yo'qligini tasdiqlaydi.

### 5.3. Ranglar tizimi — holatni kuzatish

> **Ranglar tizimi (color system)** — type checking jarayonini boshqarish uchun kompilator har bir ob'yektning holatini kuzatadigan mexanizm. Uning vazifasi — tsiklik (circular) dependency'larni aniqlash.

Har bir ob'yektning holati:

|Rang|Qiymat|Ma'nosi|
|---|---|---|
|**White**|0|Ob'yekt hali tekshirilmagan|
|**Black**|1|Ob'yekt to'liq tekshirilgan, valid yoki invalid tipga ega|
|**Grey**|2 va yuqori|Ob'yekt **hozir tekshirilmoqda**|

**Nima uchun bunday qilingan:** Agar DFS (depth-first search) jarayonida grey ob'yekt qayta uchrasa — **tsikl** aniqlangan. Lekin har bir tsikl xato emas: 
```go
var a int = b;
var b = a
```
`
bu **type cycle emas** (chunki `a` ning tipi ma'lum), bu **initialization cycle** — keyinroq boshqa bosqichda aniqlanadi.

### 5.4. Dependency'larni hal qilish — misol

Package-level variable declaration'larni ko'rib chiqamiz:

```go
var a = b
var b = d + c
var c = 10
var d int64
```

Kompilator o'zgaruvchilar tiplarini initialization dependency'larini hal qilib aniqlaydi. `a` o'zgaruvchisi `b` ga bog'liq, `b` esa `c` va `d` ga bog'liq. `c` (to'g'ridan-to'g'ri initsializatsiya qilingan) va `d` (aniq tip ko'rsatilgan) boshqa **dependency'larga ega emas**.

To'g'ri initialization tartibini ta'minlash uchun kompilator bu declaration'larni **depth-first search** (DFS) yordamida qayta ishlaydi:

1. Jarayon `a` dan boshlanadi. Kompilator `a` ni grey (2) qilib belgilaydi va uning dependency'si `b` ni tekshiradi.
2. `b` ni grey (3) qilib belgilaydi va uning dependency'laridan biri — `d` ni tekshiradi.
3. `d` ni grey (4) qilib belgilaydi, `d` ning boshqa dependency'lari yo'qligini aniqlaydi. `d` tipi aniqlanadi va u black (1) qilib belgilanadi.

> **DFS (Depth-First Search)** — grafni «chuqurlik bo'yicha» obxod qilish usuli. Avval bitta yo'lni oxirigacha boradi, keyin orqaga qaytib boshqa yo'llarni tekshiradi. Bu yerda dependency zanjirini oxirigacha kuzatish uchun ishlatiladi.

### 5.5. `go` va `defer` statement'larini qayta ishlash

Ikkalasi ham `CallStmt` sifatida ishlanadi.

- `uint64(1)` — bu **type conversion**.
- `make`, `unsafe.Sizeof`, `new` — bu **builtin expression**.
- `copy` — sintaktik jihatdan call expression, lekin semantik jihatdan **statement** sifatida ishlanadi, shuning uchun `go`/`defer` da ruxsat etilgan.

Oddiy aniqlash qoidasi: agar builtin chaqiruv **qiymat qaytarsa** — u expression sifatida ishlanadi va `go`/`defer` bilan **ishlatib bo'lmaydi**. Agar **qaytarmasa** — statement sifatida ishlanadi va ruxsat etilgan.

## Step 6: IR Construction, Optimization va Code Generation

Yuqorida batafsil yoritilgan type checking bosqichidan keyin, kompilator quyidagi bosqichlardan o'tadi. Har bir bosqichda kodning yanada optimallashtirilgan ko'rinishi hosil bo'ladi:

### 6.1. IR Nodes Construction

Type checking tugagandan keyin kompilator o'zining **Intermediate Representation** (IR) ni quradi, syntax tree va tip ma'lumotlarini kompilatsiyaga moslashtirilgan formatga aylantiradi.

> **IR (Intermediate Representation)** — oraliq ko'rinish. Source code va mashina kodi o'rtasidagi format. Original AST keyingi bosqichlardagi transformatsiyalar uchun mos emas, shuning uchun IR keyingi tahlil va qayta yozishni qo'llab-quvvatlaydigan **mutable tuzilma** beradi.

**Nima uchun bunday qilingan:** Syntax tree — «nima yozilgan»ni ifodalaydi. IR — «qanday bajarish kerak»ni ifodalaydi. Optimizatsiya uchun IR format qulay.

### 6.2. IR Optimizations

IR'ga bir qator optimizatsiyalar qo'llaniladi:

- **Dead code elimination** — hech qachon bajarilmaydigan kodni olib tashlash
- **Inlining** — kichik funksiyalar tanasini chaqiruv joyiga ko'chirish (funksiya chaqiruvi overhead'ini yo'qotish)
- **Escape analysis** — heap va stack allocation o'rtasida qaror qabul qilish
- **Devirtualization** — interface chaqiruvlarni, iloji bo'lganda, to'g'ridan-to'g'ri funksiya chaqiruvlariga almashtirish

**Nima uchun bunday qilingan:** Yakuniy mashina kodini tezroq va samaraliroq qilish. Masalan, escape analysis orqali o'zgaruvchi stack'da qolsa — GC yuki kamayadi.

### 6.3. Walk Phase

IR yanada soddalashtiriladi — murakkab expression'lar **yanada primitiv operatsiyalar**ga qayta yoziladi, past darajali qayta ishlashga tayyorlaydi. Masalan, `m[key]` kabi map'ga kirish `runtime.mapaccess` runtime chaqiruviga qayta yozilishi mumkin.

**Nima uchun bunday qilingan:** SSA generatsiyasi uchun tayyor bo'lgan, soddalashtirilgan IR hosil qilish.

### 6.4. SSA Generation (Static Single Assignment)

IR SSA shaklga aylantiriladi — bu yerda har bir o'zgaruvchi **faqat bir marta** tayinlanadi.

> **SSA (Static Single Assignment)** — har bir o'zgaruvchi faqat bir marta qiymat oladigan oraliq format. Bu data flow'ni soddalashtiradi va optimizatsiya imkoniyatlarini oshiradi.

**Nima uchun bunday qilingan:** SSA shaklida kompilator register allocation, constant propagation va boshqa optimizatsiyalarni ancha oson qiladi.

### 6.5. Machine Code va Linker

Kompilator SSA shaklni **arxitekturaga xos assembly**ga tarjima qiladi. Assembler uni object fayllarga (`.o`) qayta ishlaydi, linker keyin ularni boshqa kompilatsiya qilingan package'lar va Go runtime bilan birlashtirib yakuniy bajariladigan faylni yaratadi.

**Nima uchun bunday qilingan:** Oxirgi qadam — haqiqiy mashina kodi. Linker barcha qismlarni bog'lab, bitta yaxlit binary hosil qiladi.

Umuman kompilator **aniq fazalarga** tashkil etilgan. Har bir faza kodning o'z ko'rinishiga ega bo'lishi mumkin, bu oldingi bosqich cheklovlarisiz ma'lum vazifaga e'tiborni osonlashtiradi.

## Step 7: Linking the Final Binary

Bu bir nechta kompilatsiya qilingan object fayllarni **bitta yaxlit bajariladigan fayl**ga birlashtirish jarayoni.

Turli fayllar yoki tashqi kutubxonalardagi funksiyalardan foydalanadigan dastur yozganimizda, har bir qism alohida kompilatsiya qilinadi.

> **Linker** — bu qismlarni ulash va ular orasidagi barcha bog'lanishlarni boshqarish vazifasini bajaradigan tool. Masalan, `main` package `fmt.Println` ni chaqirsa, linker bu chaqiruvni `fmt` package'ining haqiqiy mashina kodiga bog'laydi.

Link ikki xil rejimda ishlaydi:

- **Internal linking** — Go'ning o'z linker'i, tashqi tool kerak emas
- **External linking** — tizimning linker'i (masalan, `gcc` yoki `ld`) ishlatiladi, odatda cgo yoki tashqi C kutubxonalar bilan ishlashda

Sof Go dasturlari uchun internal linking — **standart va afzal** rejim.

**Nima uchun bunday qilingan:** Package'lar alohida kompilatsiya qilinganligi uchun, ularni bir-biriga bog'lash va yagona binary hosil qilish kerak.

**Vazifasi:** Barcha `.a` archive fayllar + Go runtime → bitta bajariladigan binary fayl.

|---|---|---|
|`$WORK/b...`|Vaqtinchalik build katalogi yaratiladi|Har bir action uchun izolyatsiyalangan papka|

|Import compile|Dependency'lar avval kompilatsiya qilinadi|`.a` archive fayllar (export data + object code)|
|Parsing|Source code → token → syntax tree|Kodning ierarxik tuzilmasi|
|Type checking|Tiplar tekshiriladi, scope'lar quriladi|Semantik xatolar aniqlanadi|
|IR + Optimization|Oraliq ko'rinish quriladi va optimizatsiya qilinadi|Tezroq va samaraliroq kod|

|Linking|Object fayllar + runtime birlashtriladi|Yakuniy bajariladigan binary|

**Asosiy tushunchalar:**

- Go'da kompilatsiya birligi — **package**, fayl emas
- **Build ID** (`ActionID/ContentID`) orqali Go o'zgarmagan package'larni qayta kompilatsiya **qilmaydi**
- Scanner semicolon'ni **avtomatik qo'shadi** — shuning uchun Go'da `;` yozish shart emas
- **Package scope** butun package uchun umumiy — bir xil nomdagi ikki funksiya turli fayllarda ham **taqiqlangan**
- `go`/`defer` da faqat qiymat **qaytarmaydigan** builtin'lar ruxsat etilgan
