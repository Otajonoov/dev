> **IR (Intermediate Representation)** — oraliq ko'rinish. Source code va mashina kodi o'rtasidagi format. Compiler bu formatda optimizatsiya va transformatsiya qiladi. Original syntax tree keyingi bosqichlardagi transformatsiyalar uchun mos emas.

Bu fazada compiler o'zining asosiy IR'ini ikki kalit qadam orqali quradi: **export data generatsiya qilish** va shu data'dan **IR node'lar konstruktsiya qilish**.

![](../../../assets/obsidian-images/Pasted%20image%2020260306150220.png)

Bu export data'ni generatsiya qilish uchun Go compiler'ga ikkita asosiy input kerak: **syntax tree** va **type-checked ob'yektlar**. Syntax tree tuzilmaviy ma'lumot beradi, type-checked ob'yektlar esa oldingi bosqichlarda yig'ilgan barcha type bilan bog'liq ma'lumotni o'z ichiga oladi.

![](../../../assets/obsidian-images/Pasted%20image%2020260306150919.png)

> **Relocation** — bu bir joyda saqlangan ma'lumotga boshqa joydan havola qilish mexanizmi. Maqsadi — **deduplikatsiya** (takrorlanishni yo'qotish) va **indirection** (bilvosita murojaat).

Tasavvur qiling, dasturingizda ikkita bir xil string qiymati bor — `"foo"`. Relocation'siz compiler `"foo"` ni memory'da **ikki marta** saqlaydi — bu isrofgarchilik. Buning o'rniga, compiler string'ni **bir marta** saqlaydi va har bir ishlatilishni shu yagona joyga **havola** bilan almashtiradi.

Relocation turlari

```go
const (
    RelocString  RelocKind = iota // 0 — string'lar
    RelocMeta                     // 1 — metadata
    RelocPosBase                  // 2 — fayl pozitsiyasi
    RelocPkg                      // 3 — package
    RelocName                     // 4 — nom
    RelocType                     // 5 — tip
    RelocObj                      // 6 — ob'yekt
    RelocObjExt                   // 7 — ob'yekt kengaytmasi
    RelocObjDict                  // 8 — ob'yekt lug'ati (generic uchun)
    RelocBody                     // 9 — funksiya tanasi
    numRelocs = iota
)
```

#### Go Compilation: Stage 4 — Optimizatsiyalar


> **Dead code elimination** — hech qachon bajarilmaydigan kodni olib tashlash optimizatsiyasi. Bu bosqichda syntax tree tuzilmasi va type checking davomida aniqlangan konstanta qiymatlarga asoslanadi.

> **Devirtualization** — compiler optimizatsiyasi bo'lib, interface orqali amalga oshiriladigan bilvosita (indirect) method chaqiruvini **to'g'ridan-to'g'ri** (direct) chaqiruvga almashtiradi. Interface chaqiruvlari runtime'da to'g'ri metodning qidirilishi kerak bo'lgani uchun kichik ishlash narxiga ega.

> **Inlining** — compiler optimizatsiyasi bo'lib, funksiya chaqiruvini chaqirilgan funksiyaning **haqiqiy body'si** bilan almashtiradi. Bu funksiya chaqiruvi overhead'ini (yangi stack frame yaratish va yo'qotish) bartaraf etadi.

> **Escape analysis** — o'zgaruvchi **stack**'da yoki **heap**'da ajratilishi kerakligini aniqlaydigan jarayon. Compiler har bir o'zgaruvchining qanday ishlatilishini tekshirib, u funksiya stack frame'ida qola oladimi yoki undan tashqarida ham mavjud bo'lishi kerakmi — qaror qabul qiladi. Compiler escape analysis'ni inlining va devirtualization'dan keyin qo'llaydi.
