
## Аллокатор памяти Go

Аллокатор памяти Go — это система, решающая, как память **выделяется и переиспользуется** в Go-программах. Изначально он был вдохновлён **TCMalloc** (Thread-Caching Malloc) от Google, но со временем Go **отошёл** от TCMalloc, чтобы лучше соответствовать потребностям Go-приложений и runtime.

Некоторые ключевые идеи сохранились: Go группирует allocation по **size class** и использует **уровни кеширования**, так что большинство мелких allocation выполняются быстро и обычно избегают глобальных lock. Но знание TCMalloc не требуется для понимания дальнейшего.

## Виртуальная память

Этот раздел вводит виртуальную память как основу для дальнейшего. Можно пропустить, если вы уже знакомы с виртуальной памятью и страницами.

На почти всех компьютерах общего назначения **физическая память** — это DRAM, установленная в машине. Представьте мир только с физической памятью: без виртуальной памяти, без page table, без аппаратной защиты. В таком мире каждый процесс и ОС **разделяли бы** одно большое адресное пространство, и каждый процесс просто занимал бы некоторую область RAM:

> **Иллюстрация 275.** Иллюстрация физической памяти без виртуальной изоляции адресов _(Рисунок не загружен)_

Это сделало бы систему **хрупкой**. С одним общим пространством RAM и без изоляции, баг в одном процессе мог бы прочитать или перезаписать память другого процесса. В худшем случае он мог бы даже повредить память ОС и **обрушить всю машину**.

> **Иллюстрация 276.** Фрагментация и отсутствие изоляции памяти _(Рисунок не загружен)_

Фрагментация также была бы видимой. По мере запуска и завершения процессов они оставляли бы **дыры** свободной RAM между используемыми областями. Суммарно свободной памяти может быть достаточно, но она разбита на мелкие куски, и **нет единого непрерывного блока**, достаточно большого для следующего процесса.

Виртуальная память **решает эти проблемы**. Физическая память по-прежнему одна, но CPU добавляет **уровень трансляции**, так что каждый процесс получает собственное **приватное** виртуальное адресное пространство:

> **Иллюстрация 277.** Виртуальная память предоставляет изолированные адресные пространства каждому процессу _(Рисунок не загружен)_

Всё, что мы обсуждаем в этой главе (segment TEXT и BSS, heap, stack, даже напечатанный адрес переменной), использует **виртуальные адреса**.

Процесс не может видеть или перезаписывать данные другого процесса, потому что ОС программирует page table так, что только определённые виртуальные диапазоны **валидны**, и каждый диапазон имеет конкретные разрешения (read/write/execute). Это защищает и другие пользовательские программы, и саму ОС.

Поскольку адреса виртуальные, ОС может размещать память процесса **где угодно** в физической памяти без необходимости процессу знать об этом. Фрагментация **скрыта**, потому что множество разбросанных физических страниц могут выглядеть как **один непрерывный** виртуальный диапазон:

> **Иллюстрация 278.** Виртуальная память скрывает фрагментацию через трансляцию адресов _(Рисунок не загружен)_

ОС также может менять, какие физические страницы поддерживают данный виртуальный диапазон. Например, она может **вытеснить** редко используемые страницы на диск и позже вернуть их, сохраняя виртуальные адреса процесса **неизменными**.

Обычный user-space процесс **никогда** не читает и не пишет «сырую» физическую RAM напрямую. Каждая загрузка или запись использует виртуальный адрес, и **MMU** (Memory Management Unit) CPU транслирует его в физический адрес, используя page table, которые ядро настроило для этого процесса. Эти таблицы отображают виртуальные страницы на физические.

Даже если два процесса запросят один и тот же адрес памяти, скажем `0x00025fff`, они **не будут мешать** друг другу, потому что это виртуальный адрес, привязанный к каждому процессу.

С точки зрения приложения оно видит **полное, приватное** виртуальное адресное пространство, которое выглядит полностью доступным. На 64-битной системе это пространство **огромно** (часто в терабайтах), гораздо больше фактической физической RAM в машине. Процесс может обращаться к любому адресу, который ОС пометила как валидный. Он не знает (и не должен знать), обеспечен ли этот адрес физической RAM, вытеснен на диск, или отображён на файл.

### Пример с Go

go

```go
buf := make([]byte, 1<<30) // выделяем 1 GiB slice
```

Когда вы выделяете буфер 1 GiB, Go-аллокатору нужен **непрерывный 1 GiB виртуальный** диапазон для этого slice. Он может уже иметь такой диапазон, или может вырастить Go heap, зарезервировав больше адресного пространства у ОС. В любом случае, наличие виртуального диапазона **не означает** автоматически, что машина выдала вам 1 GiB физической RAM.

Фактически Go и ОС **стараются** избегать обращения к памяти до тех пор, пока она не понадобится. Runtime может часто откладывать обнуление, и ОС обычно обеспечивает анонимную память физическими страницами **только при фактическом обращении**.

Приложение не знает, что ОС фактически обеспечило, пока вы не обратитесь к памяти, например присвоив `buf[0]`:

go

```go
buf[0] = 42
```

CPU выполняет запись по виртуальному адресу `buf[0]`. Если страница, содержащая этот адрес, ещё **не обеспечена** физической страницей, CPU вызывает **page fault**. Ядро затем выделяет физическую страницу (для анонимной памяти она начинается заполненной нулями), обновляет page table и повторяет инструкцию. ОС не выделяет физическую RAM побайтно: **единица обеспечения — страница**, поэтому обращение к одному байту может вызвать обеспечение всей страницы, его содержащей.

Эта страница — **наименьшая единица**, которую hardware и ОС обычно используют при отображении виртуальной памяти на физическую. Размер зависит от ОС и CPU (например, 4 KiB, 16 KiB и т.д.). Виртуальное адресное пространство приложения построено из **множества страниц**. При размере страницы 4 KiB, 256 TiB адресного пространства соответствует 2³⁶ страницам, что составляет 68,719,476,736 страниц (около 68,7 миллиарда).

> **Иллюстрация 279.** Виртуальная память процесса, организованная в блоки размером со страницу _(Рисунок не загружен)_

Каждая страница начинается с адреса, **кратного** размеру страницы. Для страницы 4 KiB это кратные 4096 байт (`0x1000` в hex); для 16 KiB — кратные 16384 байт (`0x4000`). Если вы обращаетесь к любому байту внутри страницы, ОС обеспечивает **всю страницу** физической памятью.

Проверить размер страницы на macOS (Apple Silicon, arm64):

bash

```bash
$ sysctl -n hw.pagesize
16384
$ getconf PAGE_SIZE
16384
$ pagesize
16384
```

### Трансляция виртуального адреса

Когда процесс обращается к виртуальному адресу, происходит следующая последовательность:

1. MMU проверяет **быстрый кеш трансляции** CPU (TLB), чтобы узнать, знает ли он уже, где этот адрес расположен в RAM.
2. Если кеш промахнулся, MMU идёт по **медленному пути** и обходит page table в памяти для нахождения отображения.
3. Если отображение существует и разрешения позволяют, MMU **транслирует** виртуальный адрес в физический, и доступ выполняется.
4. Если отображение не существует или разрешения не позволяют, MMU останавливает CPU и вызывает исключение **«page fault»**.

> **Иллюстрация 280.** MMU транслирует виртуальные страницы процесса в физические страницы _(Рисунок не загружен)_

В нашем Go-примере адрес попадает в область, зарезервированную приложением через `buf := make([]byte, 1<<30)`, поэтому доступ **валиден**. Ядро выделяет реальную RAM (zero-filled страницу для анонимной памяти), обновляет page table и продолжает выполнение. Это момент, когда использование памяти вашего allocation **слегка увеличивается**.

Этот механизм называется **demand paging**. Идея проста: ОС не предоставляет виртуальной странице реальную физическую память, пока программа **фактически не обратится** к ней. Первый доступ вызывает page fault, и ядро обрабатывает этот fault, обеспечивая страницу памятью.

Благодаря этому **lazy allocation** виртуальное адресное пространство процесса может быть **гораздо больше** доступной RAM. Только фактически используемые области получают физические страницы, и ОС может рекуперировать или вытеснить неактивные области при необходимости.

> **Swapping физической памяти**
> 
> Когда RAM под давлением, ОС может высвободить память, выбирая страницы, которые давно не использовались (выбирая «жертву»).
> 
> Если страница-жертва **чистая** (не модифицирована), ОС может её отбросить. File-backed страницы можно перезагрузить из файла позже, чистые анонимные страницы можно воссоздать как zero-filled.
> 
> Если страница **грязная** (модифицирована), ОС должна сохранить её содержимое перед переиспользованием физического frame. Это swap space на диске — запись страницы называется **swap-out**.
> 
> Если процесс позже обратится к этой странице снова, CPU вызывает page fault. ОС обрабатывает fault, читая страницу обратно из swap в свободный frame (**swap-in**), обновляя page table, и позволяя процессу продолжить.

### Размер виртуального адресного пространства

На x86-64 pointer имеют 64 бита, поэтому теоретически можно ожидать 2⁶⁴ байт виртуального адресного пространства. На практике hardware не обязательно использует все 64 бит для адресации. Традиционно x86-64 использует **48-битную** виртуальную адресацию с правилом «canonical address» (старшие биты должны быть sign-extended). Новые CPU могут поддерживать больше (например, 57-бит), но что фактически получает user-process, зависит от конфигурации ОС. В любом случае, даже 48-битная адресация уже **огромна**: 2⁴⁸ байт = 256 TiB.

На arm64 многие системы также используют **48-битный** пользовательский виртуальный адресный диапазон, и некоторые конфигурации поддерживают большие диапазоны (до 52-бит, что составляет 4 PiB). Go runtime обычно предполагает **48-битное** heap-адресное пространство на большинстве 64-битных платформ.

### Трансляция адреса — пример

Предположим, размер виртуальной страницы 4 KiB (2¹² байт). Младшие **12 бит** 48-битного виртуального адреса представляют **offset** внутри страницы, а верхние 36 бит указывают, **какой** виртуальной странице принадлежит адрес:

> **Иллюстрация 281.** Разбиение виртуального номера страницы и offset страницы _(Рисунок не загружен)_

> **Примечание.** Это упрощённый вид. Одна плоская page table для 48-битного виртуального адресного пространства с 4 KiB страницами потребовала бы сотни гигабайт памяти. Например, 2³⁶ страниц × 8-байтная запись page table = 2³⁹ байт = 512 GiB.
> 
> На практике 36 бит разделяются на несколько частей (часто 9-битные группы), используемых через **несколько уровней** page table, так что только реально необходимые части занимают память.

Возьмём виртуальный адрес `0x000000012340`:

1. С 4 KiB страницами младшие 12 бит (`0x340`, что равно 832 в десятичной) формируют **offset** — адрес указывает на байт 832 внутри страницы.
2. Всё выше этих 12 бит — **virtual page number**, который равен `0x12` (18 в десятичной).
3. Предположим, ОС отображает виртуальную страницу 18 на физическую страницу 163.
4. CPU формирует физический адрес, комбинируя физический номер страницы (163) с offset (832). Результирующий физический адрес: `0x0000000A3340`.

> **Иллюстрация 282.** Пошаговый пример отображения виртуального в физический адрес _(Рисунок не загружен)_

### Canonical адреса

На 64-битной системе виртуальный адрес имеет 64 бита. В дампах памяти, регистрах или pointer arithmetic вы всегда видите **полное 64-битное** значение, а не только 48 бит. Для чего оставшиеся 16 бит?

В обычном 48-битном режиме на amd64 верхние биты (биты 48–63) **не свободны** для произвольного использования. Они должны все совпадать с битом 47 по правилу sign-extension:

- Если бит 47 равен нулю — биты 48–63 должны быть **все нули**.
- Если бит 47 равен единице — биты 48–63 должны быть **все единицы**.

Любой адрес, нарушающий правило sign-extension, вызывает fault. ОС обычно завершает процесс (например, segmentation fault). Адрес, следующий правилу, называется **canonical address**.

Из-за правила canonical address используемое виртуальное адресное пространство разделено на **два диапазона** с большой неиспользуемой «дырой» между ними:

> **Иллюстрация 283.** Canonical и non-canonical области в виртуальной памяти amd64 _(Рисунок не загружен)_

Три области:

- **Lower canonical region** (бит 47 = 0): адресный диапазон от `0x0000000000000000` до `0x00007FFFFFFFFFFF` — полные 2⁴⁷ (128 TiB).
- **Upper canonical region** (бит 47 = 1): от `0xFFFF800000000000` до `0xFFFFFFFFFFFFFFFF` — ещё 2⁴⁷ байт валидного пространства.
- **Non-canonical region** между ними: любой адрес между `0x0000800000000000` и `0xFFFF7FFFFFFFFFFF` **невалиден**.

> **Примечание.** Правило «canonical address» — деталь amd64. Go heap-индексация должна обрабатывать это последовательно, поэтому runtime применяет фиксированный offset (`arenaBaseOffset`) перед преобразованием адреса в arena index.
> 
> На arm64 пользовательские адреса обычно в нижнем 48-битном диапазоне и zero-extended, поэтому проблема sign-extension не возникает. На AIX/ppc64 `mmap` часто возвращает адреса в высоком диапазоне, поэтому Go также использует offset для предсказуемости.

## Как Go управляет виртуальной памятью

Когда Go-программа запускается на 64-битной системе, ОС предоставляет ей **приватное** виртуальное адресное пространство. Go runtime не пытается управлять всем этим пространством напрямую. Вместо этого он предполагает ограниченное **«heap address space»** для Go heap (часто 48 бит, что составляет около 256 TiB виртуального диапазона) и строит структуры данных аллокатора вокруг этого предположения:

> **Иллюстрация 284.** Go heap, разделённый на 64 MiB виртуальные arena _(Рисунок не загружен)_

Heap организован в единицы фиксированного размера — **arena**. На большинстве 64-битных платформ каждая arena составляет **64 MiB**. С 256 TiB heap-адресного пространства существует **4,194,304 возможных** arena slot, распределённых по этому диапазону.

Go **не выделяет** metadata для всех arena заранее. Вместо этого он хранит большую **таблицу pointer**, по одной записи на arena. При запуске все записи `nil`. Когда heap растёт и runtime решает использовать новую arena, он резервирует адресное пространство для неё, выделяет один объект metadata `heapArena` и сохраняет его pointer в соответствующей записи таблицы.

### Резервирование arena

Когда runtime резервирует новую arena, он просит ОС **зарезервировать** кусок виртуального адресного пространства, **не делая его доступным**. На POSIX-системах это обычно делается анонимным `mmap` с `PROT_NONE`, что означает «зарезервируй диапазон, но любой доступ должен вызвать fault». ОС записывает этот диапазон в виртуальное адресное пространство процесса:

> **Иллюстрация 285.** Go резервирует виртуальные arena перед allocation _(Рисунок не загружен)_

Эта область **не читаема** и **не записываема**, и Go отслеживает её как **Reserved**; любая попытка чтения, записи или выполнения в этом диапазоне вызывает немедленный fault. Если ничто не обрабатывает сигнал, процесс завершается, и в Go-программе runtime обычно печатает `SIGSEGV: segmentation violation` и завершает работу.

> **Примечание.** Reserved, Prepared и Ready — имена, которые Go runtime использует для описания перехода памяти между «просто зарезервированным адресным пространством» и «безопасным для доступа». Разные ОС реализуют эти переходы по-разному, но высокоуровневые состояния одинаковы.

Прежде чем Go-код сможет использовать память из arena, runtime переводит этот адресный диапазон из «reserved» в «accessible» (обычно по пути Reserved → Prepared → Ready).

### Страницы runtime и palloc chunk

Когда программа нуждается в большем количестве памяти, runtime выдаёт её в единицах, называемых **page**. В Go runtime page обычно составляет **8 KiB** на большинстве архитектур. Это **концепция уровня runtime**, а не то же самое, что размер физической страницы ОС. Далее, когда мы говорим «page», мы имеем в виду Go runtime page, если явно не указано «physical page».

Компонент, запрашивающий page из Go heap, — **page allocator**. Он управляет памятью в единицах page, но не всегда запрашивает по одной page. Вместо этого он округляет запросы до ближайшего кратного **palloc chunk**, который составляет **4 MiB**:

> **Иллюстрация 286.** Go arena, подразделённая на 4 MiB palloc chunk _(Рисунок не загружен)_

Palloc chunk — это **блок учёта** page allocator, фиксированный фрагмент arena. Аллокатор всё ещё выдаёт память в runtime page, но отслеживает свободное пространство и состояние scavenging в **chunk-sized summary**, чтобы быстро отвечать на вопросы вроде «есть ли в этом chunk свободные page?».

Каждая arena разбита на 4 MiB palloc chunk, что означает **512 runtime page** по 8 KiB каждая. Для каждого palloc chunk runtime хранит **bitmap-состояние**, записывающее, какие page свободны, используются или уже возвращены ОС.

### Пример: запрос span в 6 MiB

Рассмотрим, что происходит, когда программа запрашивает новый span в 6 MiB:

1. Аллокатор сначала конвертирует размер в runtime page. Поскольку каждая runtime page = 8 KiB, 6 MiB = 6 MiB ÷ 8 KiB = **768 page**.
2. Если page allocator исчерпал запас, он просит heap allocator **вырасти**.
3. Runtime выравнивает запрос до **palloc chunk** (512 page = 4 MiB). Поскольку 768 page не кратно 512, аллокатор округляет до **1,024 page**. Запрос к heap allocator — 1,024 page, или **8 MiB** виртуальной памяти, хотя span изначально нуждался только в 6 MiB.
4. После роста heap, allocation 6 MiB span потребляет 768 page из этого 8 MiB региона. Оставшиеся **256 page** остаются свободными для будущих span.

После округления heap растёт, переводя этот 8 MiB slice из Reserved в Prepared. В runtime это делается через `sysMap`. На POSIX-системах это обычно соответствует отображению адресного диапазона как read-write:

> **Иллюстрация 287.** Reserved palloc chunk переводятся в Prepared при allocation _(Рисунок не загружен)_

Heap предоставляет page allocator 8 MiB нового Prepared адресного пространства и всё ещё хранит 56 MiB Reserved адресного пространства в arena. Этот процесс называется **heap growth**. Если текущая arena не может удовлетворить запрос, runtime резервирует новую arena у ОС и повторяет процесс.

Page allocator затем обновляет metadata для этого адресного диапазона. Вновь выросшая heap-память трактуется как **уже scavenged** (released), пока runtime фактически не нуждается в ней.

> **Что такое Scavenged Memory?**
> 
> Когда Go решает, что нужно больше памяти для allocation, он переводит часть адресного пространства в usable-состояние. Но позже, когда память снова становится свободной, Go хочет **уменьшить RSS** (физическую RAM, используемую процессом) без постоянного отказа от виртуальных адресов.
> 
> **Scavenging** — шаг, на котором runtime освобождает физическое обеспечение **без освобождения** виртуального адресного пространства. В терминах Go, он берёт heap-память, всё ещё отображённую в процессе, и сообщает ОС, что физические страницы **больше не нужны**. Виртуальный адресный диапазон остаётся частью Go heap, но ОС **может** рекуперировать RAM.
> 
> В модели состояний runtime scavenging — это переход из **Ready** в **Prepared**, выполняемый через `sysUnused`. На многих Unix-подобных системах `sysUnused` реализуется через `madvise`, например `MADV_FREE` или `MADV_DONTNEED`. После scavenging те же адреса могут быть использованы снова позже.

### Heap allocator vs Page allocator

Уточним взаимоотношение. **Heap allocator** (`mheap`) владеет Go heap, включая arena и span. **Page allocator** (`mheap.pages`) живёт внутри `mheap` и отслеживает свободную и используемую память на уровне runtime page с chunk-sized summary для быстрого поиска и scavenging. Когда page allocator не может удовлетворить запрос, runtime растит heap, резервируя или отображая больше arena space у ОС.

Palloc chunk — единица, которую page allocator использует для запроса большей памяти у heap allocator. Однако более фундаментальная единица Go для управления allocation — **span**.

## Span

**Span** — это непрерывная область page, управляемая вместе для allocation определённого **size class**. Вместо того чтобы думать об arena как состоящей из palloc chunk, думайте о ней как состоящей из **множества span**:

> **Иллюстрация 288.** Go heap, организованный в span из непрерывных page _(Рисунок не загружен)_

Размеры span **варьируются**. Span может быть маленьким, как одна runtime page, или покрывать множество page. Количество page span (`s.npages`) выбирается при allocation этого span runtime.

Когда нужны span, runtime запрашивает у page allocator **n page** для формирования span. Эти page могут включать некоторые, которые были ранее scavenged (Prepared). В таком случае runtime переводит scavenged часть обратно в **Ready** через `sysUsed()` перед использованием для allocation.

Ready означает, что runtime считает память **безопасной для доступа**. Фактически ли ОС выделяет физические страницы немедленно или только при первом обращении — зависит от платформы.

> **Примечание.** На Windows ОС требует **явный commit** прежде чем зарезервированная страница может быть доступна. Обращение к uncommitted странице невалидно и вызовет fault.

Span затем подразделяется на множество **равноразмерных object slot**. Когда программа выделяет объект, runtime выдаёт один из этих slot. Все объекты в данном span имеют **одинаковый размер** (span выделен size class), и разные span обслуживают разные размеры объектов:

> **Иллюстрация 289.** Span, подразделённые на равноразмерные allocation object _(Рисунок не загружен)_

### Упрощённый путь allocation

1. Для маленького объекта (обычный случай) аллокатор выбирает size class и использует текущий span, кешированный для этого размера.
2. Выделяет один свободный slot из этого span.
3. Если в span нет свободных slot, он пополняется, захватывая другой span для этого size class.
4. Если heap нуждается в росте для предоставления больше page, он растёт по **4 MiB** palloc-chunk increment.
5. Если в текущей arena недостаточно места, runtime резервирует новую **64 MiB** arena у ОС и продолжает.

> **Иллюстрация 290.** Иерархия heap allocation Go: object, span, palloc, arena _(Рисунок не загружен)_

Каждая arena в Go heap — **единый непрерывный** блок (например, 64 MiB), но heap в целом **не гарантированно** является одной непрерывной областью. Последовательные arena могут быть несмежными в виртуальном адресном пространстве.

## Система аллокации памяти в Go

В Go span представлен типом **mspan**. Mspan управляет непрерывным куском Go heap, состоящим из runtime heap page. Ключевые поля:

- **startAddr:** виртуальный адрес первого байта в span
- **npages:** сколько runtime heap page покрывает span
- **nelems:** сколько объектов может вместить span
- **allocBits:** bitmap, отслеживающий, какие object slot свободны или используются
- **elemsize:** размер каждого объекта в байтах в этом span

> **Иллюстрация 291.** Структура span внутри arena _(Рисунок не загружен)_

Упрощённая версия структуры `mspan`:

Когда компилятор решает, что значение должно жить на heap, он генерирует вызов в runtime для его allocation. Общая точка входа — `runtime.newobject()`, которая вызывает `runtime.mallocgc()`:

### Span Class

67 small-object size class плюс sentinel size class 0 для large allocation. Span далее различаются по тому, могут ли они **содержать pointer**. Runtime кодирует это как **span class**, комбинирующий size class с битом **noscan/scan** (нет pointer / есть pointer):

Span class вычисляется из номера size class плюс один дополнительный бит. Концептуально — size class, сдвинутый влево на один, и младший бит = 1 для noscan, = 0 для scan:

- **scan span class:** чётные значения 0, 2, 4, 6, ... до 134. Значение 0 зарезервировано для large allocation.

## Три главных компонента

Все эти единицы обрабатываются тремя основными игроками: **per-P кеш mcache**, **центральный аллокатор mcentral** и **heap аллокатор mheap**.

### mcache — Per-P кеш

```go
type mcache struct {
    ...
    tiny       uintptr
    tinyoffset uintptr
    tinyAllocs uintptr
    alloc      [numSpanClasses]*mspan
    stackcache [_NumStackOrders]stackfreelist
    ...
}
```

_Figure 146. mcache struct (src/runtime/mcache.go)_

Каждый логический processor (P) имеет собственный **локальный кеш** для small allocation. Этот кеш — `mcache`. Он содержит один активный `mspan` pointer на каждый span class, так что большинство small allocation завершаются **без обращения** к любой shared global структуре.

> **Примечание.** Две структуры управляют allocation внутри per-P кеша: **small-object allocation cache** (`alloc`) и **tiny-object allocation cache** (`tiny`). Tiny allocation будут рассмотрены позже.

Когда goroutine работает на P и нуждается в маленьком объекте, он обычно берёт его **напрямую** из mcache этого P. Этот путь обычно **не требует lock**, потому что mcache — per-P и используется только goroutine, работающей на этом P.

Кеш аллокатора (`mcache.alloc`) — массив, индексируемый span class; каждая запись указывает на **активный span** для этого class:

> **Иллюстрация 298.** Каждый P хранит локальные span для быстрых small allocation _(Рисунок не загружен)_

Span разделён на множество объектов. Каждый активный span предоставляет много **свободных slot** для small allocation. Количество slot зависит от size class:

- Span class 2 (size class 1, 8-байтные slot): **1,024 slot** в 8,192-байтном span
- Span class 4 (size class 2, 16-байтные slot): **512 slot** в 8,192-байтном span

> **Иллюстрация 299.** Каждый кешированный span содержит много fixed-size object slot _(Рисунок не загружен)_

Per-P кеш (`mcache.alloc[spc]`) хранит span для class только пока в нём есть свободный slot. Когда он становится **full**, runtime возвращает span в central lists и **пополняет** кеш другим span со свободным пространством из mcentral.

### mcentral — Центральные свободные списки

Центральный аллокатор, или **central free lists**, — shared pool для одного span class. Каждый span class имеет **ровно один** central. Каждый central хранит две группы span: span с хотя бы одним свободным объектом (**partial**) и span без свободных объектов (**full**):

> **Иллюстрация 300.** Центральный аллокатор, управляющий span, shared между P _(Рисунок не загружен)_

go

```go
type mheap struct {
    ...
    central [numSpanClasses]struct {
        mcentral mcentral
        ...
    }
}

type mcentral struct {
    ...
    spanclass spanClass
    partial   [2]spanSet  // списки span со свободным объектом
    full      [2]spanSet  // списки span без свободных объектов
}
```

Центральный аллокатор **shared** между processor, поэтому его операции должны быть safe при конкурентности. Go избегает единого глобального lock, **разделяя** каждый span class и используя `spanSet` как underlying-контейнер. В обычном случае push и pop span используют **atomic operation**. Маленький lock нужен только когда `spanSet` должен вырастить свой internal spine.

Каждый central **синхронизируется независимо** и хранится отдельно от других. Если сотни P борются за span size class n, allocation в других span class **не затрагиваются**.

Каждый central поддерживает два списка: **partial** и **full**. Каждый из них далее разделён на два span set: **swept** и **unswept**:

> **Иллюстрация 301.** mcentral разделяет span на partial/full и swept/unswept _(Рисунок не загружен)_

Что означают swept и unswept:

- **Partial-swept:** span с хотя бы одним свободным объектом, **уже swept** в текущем GC-цикле. Предпочтительны для allocation, потому что сразу usable.
- **Partial-unswept:** span со свободными объектами, **ещё не swept** в текущем GC-цикле.
- **Full-swept:** span без свободных объектов, уже swept.
- **Full-unswept:** span без свободных объектов, ещё не swept. Sweeping может освободить dead objects и переместить span в partial-swept.

### Порядок получения span из central

Central аллокатор пробует в следующем порядке:

1. Pop из **partial-swept** — span со свободными slot, уже swept.
2. Если нет — pop из **partial-unswept**, но **sweep span** перед использованием.
3. Если нет — проверить **full-unswept**. Sweeping может освободить dead objects.

> **Иллюстрация 303.** Порядок получения span в central allocator _(Рисунок не загружен)_

Если ничего usable — central просит heap о **новом span**.

Важная деталь: central **передаёт span** P's кешу полностью. Central больше не хранит ссылку на этот span; ownership переходит к mcache P до возврата span.

### mheap — Heap аллокатор

Heap аллокатор — **последняя остановка**. Он управляет heap page и предоставляет память higher-level аллокаторам.

```go
type mheap struct {
    ...
    lock    mutex
    pages   pageAlloc  // структура данных page allocation
    allspans []*mspan  // все span
    ...
    arenas  [1 << arenaL1Bits]*[1 << arenaL2Bits]*heapArena
    ...
}
```

_Figure 147. mheap struct (src/runtime/mheap.go)_

Heap-адресное пространство разделено на **fixed-size arena**. Двухуровневая arena map индексирует arena по heap-адресному пространству. На большинстве 64-битных non-Windows платформ `arenaL1Bits` = 0 — по сути одна большая L2-таблица.

В целом `mheap` держит меньшие кеши **хорошо заполненными** памятью, так что большинство allocation-запросов обрабатываются быстро.

В отличие от других аллокаторов, `mheap` **не выделяет** отдельные объекты или slot. Его базовая единица — **runtime heap page**. Он выделяет page, затем группирует их в span для mcentral.

### Per-P Page Cache

Каждый P также имеет маленький **per-P page cache**, содержащий до **64 runtime page** (512 KiB). Когда P нуждается в малом числе page, runtime пробует этот **локальный кеш** первым перед обращением к глобальному page allocator. Если в кеше есть место, он может выдать page **без взятия heap lock**:

> **Иллюстрация 306.** Lock-free path для page allocation через локальный кеш _(Рисунок не загружен)_

Page cache представляет непрерывный 64-page chunk в виртуальном адресном пространстве, но может иметь только некоторые свободные page. Runtime использует этот кеш только для малых запросов — **менее 16 page** (менее 128 KiB).

### Radix Tree — глобальный Page Allocator

Глобальный page allocator — **source of truth**. Он отслеживает состояние каждой page: какие page сейчас часть используемых span, какие свободны, какие scavenged и возвращены ОС.

Он отслеживает состояние page, используя **radix tree** — иерархическую summary-структуру:

```go
type pallocSum uint64

const summaryLevels = 5

type pageAlloc struct {
    summary [summaryLevels][]pallocSum
    ...
    // chunks
}
```

Каждая запись — упакованное 64-битное значение **palloc summary**. На 64-битных системах дерево имеет **5 уровней**:

> **Иллюстрация 307.** Структура дерева page allocator по heap space _(Рисунок не загружен)_

Каждый уровень разбивается на **8 меньших частей**:

- **Level 0:** каждая запись покрывает **16 GiB**
- **Level 1:** каждая запись покрывает **2 GiB**
- **Level 2:** каждая запись покрывает **256 MiB**
- **Level 3:** каждая запись покрывает **32 MiB**
- **Level 4 (leaf):** каждая запись — один chunk, **4 MiB** (512 page)

16,384 записи Level-0 суммируют **256 TiB** виртуального пространства.

`pallocSum` кодирует **три числа** — с ними аллокатор может быстро ответить на три вопроса:

1. Сколько **непрерывных свободных** page в **начале** этого региона?
2. Какой **максимальный** непрерывный run свободных page **где-либо** в регионе?
3. Сколько непрерывных свободных page в **конце** этого региона?

Каждый summary пакует **три поля по 21 бит** каждое:

> **Иллюстрация 308.** Каждый summary кодирует start, max и end free run _(Рисунок не загружен)_

Поля start и end **не избыточны** — они позволяют быстро **сшивать** run через границы соседних записей:

> **Иллюстрация 309.** Быстрое объединение свободного пространства через границы chunk _(Рисунок не загружен)_

### Пример: allocation 1 MiB (128 page)

1. Если Level-0 summary говорит, что в регионе **нет** free run ≥ 128 page — **пропускаем**. Иначе — спускаемся на Level 1.
2. Если Level-1 запись имеет max run ≥ 128 — продолжаем спускаться до leaf (4 MiB chunk).
3. На leaf — консультируемся с **512-битным bitmap** для нахождения точного run.

Пример 4 MiB chunk (512 page): 1 MiB allocated, 1.5 MiB free, 0.5 MiB allocated, 1 MiB free:

> **Иллюстрация 310.** Пример leaf chunk summary для 4 MiB _(Рисунок не загружен)_

`pallocSum` для этого chunk: start = 0, max = 192, end = 128.

### pallocData — per-page деталь

`pallocSum` записи — summary. Подробности per-page живут на leaf-уровне в `pallocData`:

```go
type pageAlloc struct {
    summary  [summaryLevels][]pallocSum
    chunks   [1 << pallocChunksL1Bits]*[1 << pallocChunksL2Bits]pallocData
}

type pallocData struct {
    pallocBits
    scavenged pageBits
}
```

Каждая `pallocData` покрывает один chunk из **512 page** (4 MiB) и отслеживает состояние page с **512-битными bitmap**.

Три категории, о которых заботится runtime:

- **Allocated pages:** page, обеспечивающие живой span. Они Ready и in use.
- **Free, unscavenged pages:** свободные page, всё ещё Ready. Могут быть **переиспользованы немедленно**.
- **Free, scavenged pages:** свободные page, помеченные как scavenged (**Prepared**). Runtime сообщил ОС, что физическая память сейчас не нужна. Перед переиспользованием runtime просит ОС сделать их снова usable.

> **Иллюстрация 311.** Переходы состояний OS-памяти в Go _(Рисунок не загружен)_

Если нет достаточного непрерывного свободного run — runtime сначала пытается **избежать роста heap**, завершая оставшуюся sweeping-работу из текущего GC-цикла и рекуперируя page. Если этого недостаточно — heap **растёт**:

> **Иллюстрация 312.** Рост heap путём подготовки новых arena/palloc _(Рисунок не загружен)_

## Large Object Allocation

Любая allocation больше ~32 KiB трактуется как **large allocation**. Large allocation **не используют** size class. Вместо этого runtime выделяет **выделенный span** из heap для одного объекта.

Внутренне large allocation используют **size class 0** с двумя span class: span class 0 для scan объектов и span class 1 для noscan объектов.

Аллокатор округляет запрошенный размер до **целого числа heap page**. Например, запрос 33 KiB округляется до 40 KiB — пять 8 KiB page. Любое оставшееся место — internal slack, которое **не может быть разделено** с другими allocation.

> **Иллюстрация 314.** Путь large-object allocation через heap allocator _(Рисунок не загружен)_

Large span немедленно помещаются в **full-swept** список central (`mcentral.full`). Это не для allocation, а для **bookkeeping** — чтобы background sweeper мог видеть large span.

## Tiny Object Allocation

Tiny object allocation — меньше **16 байт** и **не содержат pointer**. Они проходят через специальный **tiny allocator** внутри per-P кеша (mcache). Цель — **упаковать** множество tiny allocation в один 16-байтный блок.

```go
type mcache struct {
    ...
    tiny       uintptr  // pointer на начало текущего tiny-блока, или nil
    tinyoffset uintptr  // сколько байт уже использовано в текущем 16-байтном блоке
    tinyAllocs uintptr  // счётчик tiny allocation, выданных этим P
    ...
}
```

> **Иллюстрация 315.** Tiny allocator упаковывает sub-16-байтные объекты в Go _(Рисунок не загружен)_

Когда приходит tiny allocation запрос, аллокатор сначала **выравнивает** `tinyoffset` к требованию выравнивания объекта. Если запрос помещается после alignment, аллокатор продвигает `tinyoffset` и инкрементирует `tinyAllocs`.

Пример: 6 байт уже allocated, осталось 10 байт. Приходит запрос на 8-byte-aligned объект (8 байт). Хотя 10 байт осталось (`tinyoffset=6`), 8-байтный объект **не может** начаться с offset 6, потому что адрес не кратен 8. Он размещается по offset 8, используя последние 8 байт и заполняя блок:

> **Иллюстрация 317.** 8-байтный объект заполняет остаток из-за alignment _(Рисунок не загружен)_

Для следующего 2-байтного объекта tiny allocator берёт **свежий** 16-байтный блок из per-P кеша. Если кеш пуст — пополняет из central lists. Если те пусты — выделяет новый span из heap.

## Pointer Bitmaps и Malloc Header

Для garbage collector'а мало знать, **где** аллокатор получил память. Он должен знать, какие слова являются **pointer**, чтобы трассировать граф объектов.

Span class предоставляют быстрое различие. Объекты в noscan span **не содержат** heap pointer — GC может пропустить их сканирование. Объекты в scan span **могут содержать** pointer — GC нуждается в pointer layout. На 64-битной системе он решает для каждого 8-байтного machine word, является ли оно pointer или scalar data.

Layout поступает из **type metadata** программы. Runtime использует `GCData` для генерации pointer bitmap. `GCData` имеет **две формы**:

- **Simple bitmap:** один бит на word, где 1 = pointer, 0 = scalar.
- **Compact GC program:** для больших массивов или slice struct — bytecode, кодирующий повторяющийся pattern.

### Heap bits в span (объекты ≤ 512 байт)

Для span с объектами **≤ 512 байт** runtime читает `GCData` и преобразует в pointer bitmap, хранимый **в самом span** — в heap-bits area в конце span:

> **Иллюстрация 318.** Heap bitmap, хранимый в конце small-object span _(Рисунок не загружен)_

На 64-битной машине один 8 KiB span содержит 1024 word. Bitmap с одним битом на word нуждается в **1024 бит = 128 байт**. Runtime резервирует 128 байт в конце span.

Это оставляет `[base, base+8064)` для object slot. Последние 128 байт `[base+8192-128, base+8192)` зарезервированы для heap bitmap. В результате span обслуживает **168 объектов** вместо 170.

Это уменьшение происходит **только** для scan span — noscan span не нуждаются в heap bits.

### Malloc header (объекты > 512 байт, но small)

Для pointer-содержащих объектов **больше 512 байт**, но всё ещё small, runtime использует **malloc header** — одно 8-байтное word, хранимое **непосредственно перед** user object. Оно указывает на type descriptor:

> **Иллюстрация 319.** Объекты больше 512 байт используют malloc header _(Рисунок не загружен)_

GC следует по этому type pointer к `GCData` типа для сканирования объекта. Аллокатор **включает** 8-байтный header при выборе size class. Например, запрос 513 байт — аллокатор добавляет header и округляет на основе **521 байт**.

**Порог 512 байт** балансирует стоимость metadata: ниже него per-span heap bits дешевле; выше — 8-байтный header per object дешевле.

### Large objects

Large objects (больше ~32 KiB) получают собственный span. Они **не используют** per-object header или heap-bits-in-span scheme. Вместо этого span metadata (`mspan`) хранит **type pointer**. Если тип noscan — pointer = nil. Если тип использует GC program — runtime выделяет маленький side buffer и раскрывает program в bitmap.

> **Примечание.** Есть небольшое сглаживающее окно. Если добавление 8-байтного header вытолкнет pointer-содержащий объект за границу 32 KiB — он трактуется как large. Размеры в (32 KiB - 8 байт, 32 KiB] трактуются как large.

## Xulosa (Mening xulosalarim)

Bu bob Go'ning memory allocator'ini **barcha darajalarda** — virtual memory'dan tortib per-P cache'gacha, radix tree'dan tortib GC bitmap'gacha ochib berdi. Bu kitobning **eng texnik va eng muhim** boblaridan biri. Mana asosiy xulosalarim:

**1. Virtual memory — «ko'rgazma» (illuziya) tushunilmasa, Go allocator tushunilmaydi.** Go `make([]byte, 1<<30)` deb 1 GiB ajratganida — bu 1 GiB **virtual** address space. Fizik RAM **hali berilmagan**. Faqat `buf[0] = 42` qilganingizda page fault bo'ladi, yadro fizik page ajratadi. Bu **demand paging** — Go va OS ikkalasi ham lazy: imkon qadar keyin qoldiradi. Shuning uchun Go dasturining VSZ (virtual size) **juda katta**, lekin RSS (resident set size) **kichik** bo'lishi mumkin.

**2. Arena → Palloc Chunk → Page → Span → Object — 5 darajali ierarxiya.**

|Daraja|Hajm|Maqsad|
|---|---|---|
|**Arena**|64 MiB|Virtual address space'ning asosiy bloki|
|**Palloc Chunk**|4 MiB (512 page)|Page allocator'ning bookkeeping bloki|
|**Runtime Page**|8 KiB|Allocation'ning minimal birligi|
|**Span**|1+ page|Size class uchun ajratilgan page guruhi|
|**Object**|Size class ga qarab|Dastur foydalanuvchi qismi|

Bu ierarxiyani tushunish **barcha** keyingi GC va memory optimization tushunchalarining asosi.

**3. Size class — «yaxlitlash» narxi.** 33 bayt so'rasangiz — 48 bayt olasiz. 15 bayt **yo'qoladi** (internal fragmentation). Go 67 ta size class ishlatadi, `runtime/sizeclasses.go` da aniqlangan. Har bir class **o'z span**iga ega. Max waste 87.50% gacha yetishi mumkin (8-byte class, 1-byte object uchun), lekin amalda ko'p hollarda 10-20% atrofida.

**4. Span class = size class × 2 + noscan bit — GC uchun muhim.** Har bir size class **ikkiga** bo'linadi: scan (pointer bor, GC skanerlaydi) va noscan (pointer yo'q, GC **o'tkazib yuboradi**). Bu 134 small-object span class beradi. GC noscan span'larni butunlay ignorlashi mumkin — bu **katta** performance yutug'i.

**5. mcache → mcentral → mheap — uch darajali allocator.**

|Komponent|Lock?|Miqyos|Vazifa|
|---|---|---|---|
|**mcache**|❌ Lock-free|Per-P|Har bir P o'z span cache'iga ega, ko'p allocation shu yerda|
|**mcentral**|⚡ Atomic + rare lock|Per-span-class|Span'larni P'lar o'rtasida ulashadi: partial/full, swept/unswept|
|**mheap**|🔒 Heap lock|Global|Page ajratadi, arena boshqaradi, OS bilan aloqa qiladi|

Ko'p small allocation mcache'da tugaydi — **lock olmaydi**. Faqat span tugaganida mcentral'ga boriladi, va faqat yangi page kerak bo'lganda mheap'ga.

**6. Page allocator — 5 darajali radix tree.** Har bir daraja **pallocSum** — 3 × 21-bit packed field: start, max, end. Bu allocator'ga quyidagilarni **tezda** aniqlash imkonini beradi:

- Shu regionda 128 ta ketma-ket bo'sh page bormi? (max ≥ 128?)
- Agar yo'q — butun subtree'ni **o'tkazib yuborish** mumkin
- Start va end field'lar **chunk chegaralari** o'rtasida free run'larni birlashtirish uchun

Bu **O(log N)** qidirish, oddiy linear scan emas.

**7. Scavenging — RSS ni kamaytirish, virtual address'ni saqlash.**

- **Ready → Prepared** = scavenging (`sysUnused` → `madvise(MADV_DONTNEED)`)
- **Prepared → Ready** = commit (`sysUsed`)
- **Reserved → Prepared** = map (`sysMap`)

Scavenging — Go runtime'ning **eng muhim** memory management feature'laridan biri. U heap'ni kichik tutadi (past RSS), lekin virtual address space'ni saqlab qoladi (future allocation uchun).

**8. Tiny allocator — 16 baytlik blokka sig'dirish.** 16 baytdan kichik, pointer yo'q objectlar uchun: bitta 16-byte block ichiga **bir nechta** tiny object sig'diriladi. Alignment qoidasi: 8-byte-aligned object faqat 0 yoki 8 offset'da boshlana oladi. Blok to'lganda — yangi 16-byte block olinadi. Bu **juda ko'p** kichik allocation'larni (masalan, `bool`, `byte`, `int8`) samarali qiladi.

**9. Pointer bitmap va malloc header — GC object'ni qanday skanerlaydi.**

- Object ≤ 512 bayt, scan: **heap bits** span oxirida (128 bayt bitmap)
- Object > 512 bayt, scan: **malloc header** (8 bayt type pointer, object oldida)
- Large object: **mspan** da type pointer
- Noscan object: **hech qanday** bitmap kerak emas

512 bayt chegarasi — metadata cost'ni balanslash: pastda per-span bitmap arzonroq, tepada per-object header arzonroq.

**Amaliy maslahatlar:**

- `runtime.MemStats` yoki `runtime/debug.ReadMemStats()` bilan heap holat'ini kuzating — HeapInuse, HeapIdle, HeapReleased
- Object'laringiz pointer o'z ichiga olmasa — noscan span'da joylashadi, GC yuki kamayadi
- Tiny object'lar (bool, byte, int8) uchun alohida allocation **bo'lmaydi** — tiny allocator sig'diradi
- 33 KiB dan katta object'lar uchun **alohida span** ajratiladi — pool pattern foydalaning
- `GOGC` va `GOMEMLIMIT` bilan GC va heap o'sishini sozlang
- `go tool pprof -alloc_space` bilan qaysi funksiyalar ko'p allocation qilayotganini toping
